# PLAN.13 — Fix wildcard representation in OpenAPI spec

## Problem

`segmentToParam` in `openapi-context.ts` maps wildcard segments to
`{ type: 'string' }`, but at runtime wildcards produce `string[]`. The
generated OpenAPI spec will be wrong for any route using `*rest` patterns.

## Change

~~1. Update the `wildcard` case in `segmentToParam`.~~
~~2. Add a test asserting the array schema.~~

See Decision below — the Change section above is superseded.

## Decision

**Exclude wildcard routes from spec generation** rather than emitting a
structurally incorrect parameter type.

OpenAPI 3.1 path parameters have no native representation for a variable-length
path suffix (`/files/a/b/c`). Emitting `{ type: 'array' }` would be technically
wrong and likely break tooling that validates the spec. Emitting `{ type: 'string' }`
is a lie about the runtime type. Neither option produces a usable spec.

The right output is to skip the path entirely and emit a `console.warn` so
authors know the route was omitted:

```typescript
// in generateSpec, before building path parameters:
const hasWildcard = node.segments.some(s => s.kind === 'wildcard');
if (hasWildcard) {
  console.warn(`[openapi] skipping route with wildcard segment: ${node.path}`);
  continue;
}
```

## Notes

- Update the test in `openapi-context.test.ts` to assert that a wildcard route
  is absent from the generated spec (not that it produces an array parameter).
- The `segmentToParam` wildcard case can be removed — it's unreachable after
  the guard above.

## Risk

Low, but the OpenAPI representation may need a pragmatic compromise rather
than a "correct" one.
