# PLAN.13 — Fix wildcard representation in OpenAPI spec

## Problem

`segmentToParam` in `openapi-context.ts` maps wildcard segments to
`{ type: 'string' }`, but at runtime wildcards produce `string[]`. The
generated OpenAPI spec will be wrong for any route using `*rest` patterns.

## Change

1. Update the `wildcard` case in `segmentToParam`:
   ```typescript
   case 'wildcard':
     return { name: seg.name, in: 'path', required: true, schema: { type: 'array', items: { type: 'string' } } };
   ```
2. Add a test in `openapi-context.test.ts` with a wildcard route and verify
   the generated parameter schema.

## Notes

- OpenAPI 3.1 path parameters don't natively support array types well. The
  `style: 'simple'` serialization (default for path params) does handle
  comma-separated values, but a true catch-all wildcard (`/files/a/b/c`) is
  outside the OpenAPI path-parameter model. Consider whether wildcard routes
  should be excluded from spec generation with a warning, or mapped to a
  single `{ type: 'string' }` param with a note that it represents a path
  suffix.

## Risk

Low, but the OpenAPI representation may need a pragmatic compromise rather
than a "correct" one.
