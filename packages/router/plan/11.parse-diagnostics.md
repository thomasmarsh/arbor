# PLAN.11 — Add parse-failure diagnostics

## Problem

`parse()` returns `Result.failure("no route: /path")`. When debugging a route
mismatch in a complex tree, there's no way to know *why* it failed — did the
path match but a schema reject a query param? Did a numeric segment fail to
coerce? `walkParse` returns `null` and discards all context.

## Change

Option A — **verbose parse**:
Add a `parseDiag(url: URL)` method that returns the same `Result` on success,
but on failure returns a structured list of candidates that were tried and why
each was rejected (segment mismatch, schema validation error, etc.).

Option B — **breadcrumb accumulator**:
Thread an optional diagnostics accumulator through `walkParse` so the existing
`parse()` can optionally collect rejection reasons without a separate code path.

## Notes

- This should not add overhead to the happy path. If using an accumulator,
  make it opt-in (only allocated when diagnostics are requested).
- The schema `.safeParse()` calls already produce Zod error objects — surface
  those rather than discarding them.

## Risk

Low-medium. Needs care to avoid coupling diagnostics into the hot path.
