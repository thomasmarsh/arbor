# PLAN.11 — Add parse-failure diagnostics

## Problem

`parse()` returns `Result.failure("no route: /path")`. When debugging a route
mismatch in a complex tree, there's no way to know *why* it failed — did the
path match but a schema reject a query param? Did a numeric segment fail to
coerce? `walkParse` returns `null` and discards all context.

## Change

Use **Option B — breadcrumb accumulator**. Option A is rejected because a separate
`parseDiag` entry point duplicates the entire traversal and diverges from `parse()`
over time.

Thread an optional `diag: ParseDiag[]` accumulator through `walkParse`. When
provided, push a rejection record at each dead end:

```typescript
type ParseDiag =
  | { kind: 'segment-mismatch'; path: string; urlSegments: string[] }
  | { kind: 'schema-error'; path: string; issues: z.ZodIssue[] };
```

Expose it via a `parseDiagnostics(url: URL)` method on the router object:

```typescript
parseDiagnostics(url: URL): {
  result: Result<Route, string>;
  diagnostics: ParseDiag[];
}
```

`parse()` is untouched — it never allocates the accumulator.

## Notes

- Allocate the accumulator array only inside `parseDiagnostics`, never in `parse()`.
- The schema `.safeParse()` calls already return `ZodError` — surface `error.issues`
  directly rather than stringifying them.
- Do not push a diag record for section nodes (`schema === null`) — those aren't
  candidate endpoints and would clutter the output.

## Risk

Low-medium. Needs care to avoid coupling diagnostics into the hot path.
