# Plan 73 — Include `Allow` header in 405 responses

## Context

RFC 7231 §6.5.5 states: "The origin server MUST generate an Allow header field in a 405
(Method Not Allowed) response containing a list of the target resource's currently supported
methods."

Currently `resolveHandler()` returns:

```ts
{ ok: false, status: 405, body: { error: 'method not allowed' } }
```

No `Allow` header is present. Clients (browsers, API consumers, OpenAPI tooling) rely on
the `Allow` header to discover valid methods without reading documentation.

The information is already available: `methodMap[tag]` contains the registered HTTP method
for the matched tag. When the incoming method doesn't match, the allowed method is exactly
`methodMap[tag]`.

## Goal

405 responses include `headers: { Allow: '<method>' }`, e.g. `Allow: GET`.

## Change Surface

- `src/server/server.ts`
  - `resolveHandler()` return type: add `headers?: Record<string, string>` to the failure branch
  - `resolveHandler()` body: populate `Allow` on the 405 branch
  - `executeRoute()`: propagate `resolved.headers` into the returned `DispatchResult`
    (already done for the success path; the failure short-circuit `return { ...resolved, tag }`
    will carry it automatically once the type is updated)
- `src/server/server.test.ts` — add assertion that 405 response includes `Allow` header

## Dependencies

None.

## Success Criteria

- `server.handle(url, 'POST')` on a GET-only route returns
  `{ status: 405, headers: { Allow: 'GET' }, ... }`.
- Existing 405 tests updated to assert the header presence.
- `pnpm test && pnpm typecheck` green.
