# PLAN.49 — Spike: handler enrichers as an alternative to a middleware pipeline

## Context

Plan 44 proposes a `middleware: []` array on routes, drained by a `runMiddleware(pipeline, ctx)`
runner. Three structural problems with that design:

1. **Array accumulation is a type black hole.** TypeScript cannot straightforwardly reduce
   a heterogeneous `Middleware<In, Out>[]` to a single accumulated `Out` type — it requires
   deep conditional recursion that fights the compiler the same way `_child` did before plan 47.

2. **Architecture contradiction.** `CLAUDE.md` forbids implicit middleware arrays. A
   `middleware: [authMiddleware, rateLimit]` field on `RouteNode` bakes HTTP-specific
   cross-cutting concern ordering into the core type, violating the domain-independence
   constraint on `Context`.

3. **Opaque application.** A pipeline runner hides which concerns are applied to which routes
   and in what order. A reader cannot tell at a glance what a handler receives without
   tracing the pipeline.

**User preference**: higher-order operations that are **strategically applied** — visible,
typed, and composable at the callsite rather than accumulated in an invisible runner.

## Goal

Spike whether **handler enrichers** — typed HOFs that wrap a handler and extend its `ctx`
argument — can replace the pipeline entirely without touching `RouteNode` or `createServer`.

The canonical shape:

```typescript
// An enricher: given the base ctx, produce an augmented ctx or short-circuit.
type Enricher<BaseCtx, Extra> = (
  ctx: BaseCtx,
) => Promise<{ ok: true; ctx: BaseCtx & Extra } | { ok: false; response: Response }>;

// The HOF: wrap a handler that needs Extra, produce a handler that only needs BaseCtx.
function withEnricher<BaseCtx, Extra>(
  enricher: Enricher<BaseCtx, Extra>,
  handler: (ctx: BaseCtx & Extra) => Promise<Response>,
): (ctx: BaseCtx) => Promise<Response>;
```

Concrete example:

```typescript
const withSession = withEnricher(
  async (ctx) => {
    const session = await resolveSession(ctx.headers?.authorization);
    if (!session) return { ok: false, response: new Response(null, { status: 401 }) };
    return { ok: true, ctx: { ...ctx, session } };
  },
  async ({ session, params }) => {
    return { status: 200, body: session.user };
  },
);
```

Multiple enrichers compose explicitly, not via array:

```typescript
const handler = withRateLimit(withSession(async ({ session, rateInfo, params }) => {
  ...
}));
```

Each wrap site is visible. Each type addition is local. No runner needed.

## Change Surface

This is a **spike** — no production code changes. The deliverable is a self-contained
`src/server/enrichers.ts` with:

1. `Enricher<BaseCtx, Extra>` type definition.
2. `withEnricher(enricher, handler)` HOF implementation (~15 lines).
3. A `composeEnrichers` helper if chaining two enrichers is desirable without nesting
   (evaluate whether TypeScript can infer the composition cleanly).
4. Type-level tests (`expectTypeOf`) covering:
   - `Extra` fields appear in the wrapped handler's ctx.
   - Short-circuit path returns a response, not the handler result.
   - Two enrichers composed: both additions visible in the innermost handler.

No changes to `RouteNode`, `httpRoute`, `createServer`, or any existing test.

## Design questions to answer in the spike

1. **Does TypeScript infer `Extra` cleanly** when `Enricher` is defined as an inline
   arrow function, or does the generic need to be explicitly annotated?
2. **Can `composeEnrichers` be typed** without accumulator recursion? (Explore 2-arity
   and variadic options; stop if inference breaks.)
3. **Does this model short-circuiting well?** The `{ ok: false; response }` branch must
   prevent the handler from running — verify the runtime path is correct.
4. **Server integration**: can the wrapped handler be passed directly into `HandlerMap`
   without a signature mismatch, or does `HandlerCtx` need a minor adjustment?

## Dependencies

- Plan 22 (DI context) — base `HandlerCtx` shape. Already done.
- No dependency on plan 44 — this is the alternative.

## Success Criteria

- `withEnricher` compiles and the type tests pass with `pnpm run typecheck`.
- The innermost handler ctx reflects every enricher's additions without casts.
- A short-circuiting enricher prevents the handler from running (runtime test).
- No changes to `RouteNode`, `httpRoute`, or `createServer`.
- If `composeEnrichers` cannot be typed cleanly without array-accumulation tricks,
  document that finding and recommend explicit nesting instead.

## What replaces plan 44 if this succeeds

Plans 36, 37, 38, 39, 40 each get a concrete, named enricher (`withRateLimit`,
`withSession`, `withCors`, etc.) that handlers opt into explicitly. There is no
general pipeline runner. The plan 44 document is superseded.
