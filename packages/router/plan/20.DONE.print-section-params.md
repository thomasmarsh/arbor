# PLAN.20 — Type-safe section path params in `print()`

## Context

Issue 2 from plan 18: `print()` is typed as `(route: Route) => string` where
`Route` is a union of leaf routes. Section path params (e.g., `orgId` on a
section node) are not part of any leaf route type, so callers must pass them
via `as any` to avoid a type error — even though omitting them causes a silent
runtime failure (broken URL).

## Goal

Make omitting a required section param a compile-time error.

## Proposed API

```typescript
// Before
router.print({ child: { tag: 'project', projectId: 42 } } as any)

// After
router.print(
  { child: { tag: 'project', projectId: 42 } },
  { orgId: 'acme' }           // section params: typed, mandatory
)
```

## Change Surface

1. **`route-node.ts`**: Add a new phantom type `_sectionParams` (or derive from
   the existing `path` template) that captures which param names a section
   hierarchy requires.
2. **`define-routes.ts` / `walk.ts`**: Thread section param requirements up
   through `defineRoutes` so `print()` can expose them in its signature.
3. **`define-routes.ts`**: Update `print()` to accept a second argument typed
   as `Record<SectionParamKeys, string | number>`. Fall back to
   `Record<string, string | number>` if no section params exist (so leaf-only
   routers don't need to pass anything).
4. Update tests: remove both `as any` casts in `define-routes.test.ts:177,224`.

## Decision (Q1 resolved)

Use full phantom threading (option a). The weak fix (accepting
`Route & Record<string, unknown>`) is explicitly rejected — omitting a
required section param must be a compile-time error, not a silent runtime
failure. If threading proves intractable, open a spike rather than
silently falling back.

## Success Criteria

- `define-routes.test.ts:177,224` compile without `as any`.
- Omitting `orgId` when the section requires it is a type error.
- `npm test && npm run typecheck` clean.
