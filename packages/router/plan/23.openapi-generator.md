# PLAN.23 — Automated OpenAPI spec generator utility

## Context

`spec.enhancements.md` Phase 3, item 1. The `openapi-context.ts` already
accumulates `responseSchemas` per route. The missing piece is a top-level
`generateSpec(router)` function that walks the full route tree and emits a
standards-compliant `openapi.json` object.

## Goal

```typescript
import { generateSpec } from '@arbor/router/openapi';

const spec = generateSpec(router);
// spec is a valid OpenAPI 3.1 document object
```

## Change Surface

1. **New file `src/openapi/generate-spec.ts`**: Walk the route tree, collect
   path templates, methods, request body schemas, and response schemas, emit
   an OpenAPI 3.1 paths object.
2. **`src/openapi/index.ts`**: Export `generateSpec`.
3. **`src/index.ts`**: Re-export from `openapi/`.
4. Tests: snapshot test asserting the structure of the generated spec for a
   known router fixture.

## Notes

- The partial `generateSpec` used in existing tests (`openapi-context.test.ts`)
  is a test helper operating at the node level. The new utility operates at the
  router level and delegates down.
- `z.toJSONSchema` is already used; lean on it for request body and response
  schema conversion.

## Dependencies

- Plan 19 (phantom context split) — `_ctx.responseSchemas` must be accessible
  cast-free before building this utility.

## Success Criteria

- `generateSpec(router)` emits a valid OpenAPI 3.1 document for a fixture router.
- Snapshot test catches regressions.
- `npm test && npm run typecheck` clean.
