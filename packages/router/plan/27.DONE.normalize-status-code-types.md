# PLAN.27 — Normalize response status code key type (number vs string)

## Context

`httpRoute` and `openApiRoute` store response schemas inconsistently:

```typescript
// http-context.ts — keyed by number
responseSchemas: Record<number, z.ZodType>

// openapi-context.ts — keyed by string
responseSchemas: Record<string, z.ZodType>
```

HTTP status codes are numbers semantically, but JSON object keys are always
strings. The mismatch leaks into `walk.ts` (which uses `Object.entries` on
`responseSchemas`) and will cause subtle bugs in the OpenAPI generator
(plan 23) where status codes need to be emitted as string keys in the JSON
document but compared numerically at runtime.

## Goal

Pick one canonical representation and use it everywhere. **Recommended:
`Record<number, z.ZodType>` as the runtime type, with explicit
`String(statusCode)` conversion at JSON output boundaries** (i.e., inside
the OpenAPI generator).

This is the approach that keeps the type honest (status codes are numbers)
while acknowledging that JSON serialization requires strings.

## Change Surface

1. **`openapi-context.ts`**: Change `responseSchemas` key type from
   `Record<string, z.ZodType>` to `Record<number, z.ZodType>`.
2. **`walk.ts`**: Audit all `Object.entries(ctx.responseSchemas)` calls.
   Keys from `Object.entries` on a `Record<number>` are still strings at
   runtime (JS coerces numeric keys). Use `Number(key)` where the numeric
   value is needed.
3. **`openapi-context.test.ts`**: Update any assertions that check string
   status code keys.
4. **Future `generate-spec.ts`** (plan 23): Explicitly call `String(code)`
   when writing the OpenAPI paths object.

## Dependencies

- Should land **before plan 23** (OpenAPI generator) — normalizing this
  afterward would require touching the generator immediately.
- Independent of plans 19–22; can run in parallel.

## Success Criteria

- `httpRoute` and `openApiRoute` agree on `Record<number, z.ZodType>` for
  `responseSchemas`.
- No `Object.entries` call silently mixes string and number key assumptions.
- `npm test && npm run typecheck` clean.
