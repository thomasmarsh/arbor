# Plan 76 — `.use()` fluent builder on route nodes

## Context

Guards currently compose via higher-order factories that nest outward:

```typescript
withSession(withRbac(['admin'], httpRoute(GetAdmin, 'GET', 'admin', { ... })))
```

This reads inside-out (innermost = declared last, applied first) and is unfamiliar to
developers coming from Express, Hono, or Fastify. The ergonomic gap is real: teams prefer
`app.use(auth)` style because it reads in execution order. The current model is more
correct (explicit type threading, no implicit mutation) but harder to teach.

A `.use()` builder produces **identical types** to the nested factory form with left-to-
right reading. It is a pure ergonomics proxy — no new execution semantics, no new type
system machinery. The resulting `RouteNode` is the same type either way; `createServer`
sees no difference.

```typescript
// Equivalent to the nested form above:
httpRoute(GetAdmin, 'GET', 'admin', { ... })
  .use(withSession)
  .use(withRbac(['admin']))
```

A companion `pipeline(...guards)` applies the same set of guards to multiple routes at
once — the equivalent of `router.use('/admin', auth)`:

```typescript
const adminPipeline = pipeline(withSession, withRbac(['admin']));

defineRoutes([
  httpRoute(GetAdmin, 'GET', 'admin/users', { ... }).pipe(adminPipeline),
  httpRoute(DeleteAdmin, 'DELETE', 'admin/users/:id', { ... }).pipe(adminPipeline),
]);
```

## Goal

1. Route nodes returned by `httpRoute()`, `route()`, `section()`, and `openApiRoute()`
   have a `.use(guard)` method.
2. `.use(guard)` returns a new `BuildableRouteNode` with the guard's type enrichment
   applied — identical to calling the guard factory directly.
3. `pipeline(...guards)` returns a single combinator applicable via `.pipe(pipeline)`.

## Type design

```typescript
// Augmented return type — wraps any RouteNode with the builder methods.
type BuildableRouteNode<N extends RouteNode<any, any, any, any, any>> = N & {
  use<G extends (node: N) => RouteNode<any, any, any, any, any>>(
    guard: G,
  ): BuildableRouteNode<ReturnType<G>>;
  pipe<R extends RouteNode<any, any, any, any, any>>(
    combinator: (node: N) => R,
  ): BuildableRouteNode<R>;
};

function buildable<N extends RouteNode<any, any, any, any, any>>(
  node: N,
): BuildableRouteNode<N> {
  return Object.assign(node, {
    use: (guard: (n: N) => any) => buildable(guard(node)),
    pipe: (fn: (n: N) => any) => buildable(fn(node)),
  });
}

export function pipeline<N extends RouteNode<any, any, any, any, any>>(
  ...guards: Array<(node: any) => RouteNode<any, any, any, any, any>>
): (node: N) => RouteNode<any, any, any, any, any> {
  return (node) => guards.reduce((acc, g) => g(acc), node);
}
```

`httpRoute`, `route`, `section`, and `openApiRoute` each wrap their return value in
`buildable(...)`. The underlying node shape is unchanged.

## Change Surface

- `src/core/define-routes.ts` — `buildable()` helper; wrap return values of `route()` and
  `section()`
- `src/contexts/http-context.ts` — wrap `httpRoute()` return value
- `src/contexts/openapi-context.ts` — wrap `openApiRoute()` return value
- `src/server/guard.ts` — add `pipeline()` export
- `src/index.ts` — export `pipeline`
- Tests and examples updated to demonstrate the fluent form

## Dependencies

None. Pure ergonomic layer over existing types.

## Success Criteria

- `httpRoute(...).use(withSession).use(withRbac(['admin']))` compiles to the same type as
  `withSession(withRbac(['admin'], httpRoute(...)))`.
- `.pipe(pipeline(withSession, withRbac(['admin'])))` produces the same type.
- Existing nested composition still works — `.use()` is additive only.
- `pnpm test && pnpm typecheck && pnpm lint` green.
- At least one example updated to demonstrate the fluent form.
