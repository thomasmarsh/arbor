# PLAN.35 — Multipart / streaming body parsing (S1)

## Context

`spec.roadmap.md` Phase S1. The server currently calls `await req.json()` for
all request bodies. File uploads and multipart form payloads arrive as
`multipart/form-data` and cannot be parsed this way. Large binary payloads
need backpressure protection against memory exhaustion.

## Goal

Extend `createServer`'s request pipeline to inspect the `Content-Type` header
and dispatch to the appropriate parser:

| Content-Type              | Parser                          |
|---------------------------|---------------------------------|
| `application/json`        | `await req.json()`              |
| `multipart/form-data`     | `await req.formData()`          |
| `text/plain`              | `await req.text()`              |
| anything else             | 415 Unsupported Media Type      |

Zod schemas can declare file fields using `z.instanceof(File)` or
`z.instanceof(Blob)`. The parsed `FormData` entries are unpacked into a plain
object and fed into the existing `bodySchema.safeParse` path.

A server-level `maxBodySize` option (bytes) is enforced before parsing to
prevent memory-exhaustion attacks. If the `Content-Length` header exceeds the
limit, return `413 Content Too Large` immediately.

## Change Surface

1. **`server.ts`**: Replace the hard-coded `req.json()` call with a
   `parseBody(req, route)` helper that dispatches by `Content-Type`.
2. **New `src/server/parse-body.ts`**: The dispatcher. Handles JSON,
   formData, text. Returns a `{ ok, data, error }` discriminated result.
3. **`createServer` options**: Add `maxBodySize?: number` (default: 1 MB).
4. **`http-context.ts`**: No contract change needed — `bodySchema` already
   accepts any Zod object; `z.instanceof(File)` works as-is.
5. Tests:
   - multipart request with a `z.instanceof(File)` schema → 200, file in handler.
   - unknown content-type → 415.
   - payload exceeding `maxBodySize` → 413.

## Dependencies

- Plan 22 (DI context) — body value flows through `ctx.body`; must be
  settled first.
- Plan 25 (validation error boundary) — same 400-on-parse-failure pattern.

## Success Criteria

- Multipart form bodies with file fields are validated and available typed in
  the handler ctx.
- Oversized payloads return 413 without reading the full stream.
- `npm test && npm run typecheck` clean.
