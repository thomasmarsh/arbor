# Plan 51 — Spike: RouteCtx Design Consolidation

## Context

`RouteCtx` was introduced in plan 19 to separate phantom type information (`Context`) from runtime metadata (`_ctx`). The intention was to avoid baking domain-specific concerns into the core `RouteNode` shape.

That goal was not achieved. Today `RouteCtx` in `src/core/route-node.ts` lists:

```typescript
export interface RouteCtx {
  method?: string;
  bodySchema?: z.ZodType;
  responseSchemas?: Record<number, z.ZodType>;
  responseHeaderSchemas?: Record<number, z.ZodObject<any, any>>;
  querySchema?: z.ZodObject<any, any>;
  headerSchema?: z.ZodObject<any, any>;
  rateLimit?: { windowMs: number; maxRequests: number };
  meta?: Record<string, unknown>;
  [key: string]: unknown;
}
```

These are HTTP/OpenAPI domain concepts embedded directly in the core type file. The index signature `[key: string]: unknown` was meant to keep it extensible, but named fields were added anyway — defeating the purpose.

Meanwhile, `RouteNode` now has two mechanisms for associating data with a route:

| Field | Type | Runtime? | Purpose |
|-------|------|----------|---------|
| `context?: Context` | Phantom | No (`undefined as never`) | DI context typing for server handlers |
| `_ctx?: RouteCtx` | Concrete | Yes | HTTP metadata for routing, parsing, OpenAPI |

Two orthogonal mechanisms, different semantics, overlapping intent. The `_ctx` naming (`_` prefix implies internal) but it's accessed from `walk.ts`, `define-routes.ts`, `fetch-client.ts`, and `generate-spec.ts`.

## Goal

Resolve the design tension. One of two outcomes:

- **A**: Keep `_ctx` but remove all domain knowledge from `route-node.ts`. Core only knows the bag exists, plugins own their own shapes.
- **B**: Collapse `_ctx` into `context` where it makes sense, or establish a clear policy for when each is appropriate.

This plan documents the options and picks one for implementation.

## Change Surface

### Current read sites of `_ctx` fields

| Consumer | Fields read |
|----------|-------------|
| `src/core/walk.ts` | `querySchema` |
| `src/core/define-routes.ts` | `method`, `bodySchema`, `headerSchema`, `responseHeaderSchemas`, `rateLimit` |
| `src/client/fetch-client.ts` | `method`, `responseSchemas` |
| `src/openapi/generate-spec.ts` | `method`, `meta`, `headerSchema`, `bodySchema`, `responseSchemas`, `responseHeaderSchemas` |

### Current write sites of `_ctx`

| Producer | Fields written |
|----------|----------------|
| `src/contexts/http-context.ts` | `method`, `bodySchema`, `querySchema`, `headerSchema`, `responseSchemas`, `responseHeaderSchemas`, `rateLimit` |
| `src/contexts/openapi-context.ts` | `method`, `meta`, `bodySchema`, `responseSchemas` |

## Options

### Option A — Generic `_ctx` on `RouteNode`, plugin-owned typed accessors

Remove `RouteCtx` from `route-node.ts`. Add a generic `Ctx` type parameter (defaulting to `Record<string, unknown>`) so the core knows the bag exists but not its shape:

```typescript
interface RouteNode<
  R,
  C extends RouteNode<unknown, any, any, any, any>[] = [],
  Context = never,
  SectionParams extends string = never,
  Ctx extends Record<string, unknown> = Record<string, unknown>,
> {
  _ctx?: Ctx;
}
```

Each plugin owns its typed shape and exports typed accessors:

```typescript
// src/contexts/http-context.ts
export interface HttpCtxData {
  method?: string;
  bodySchema?: z.ZodType;
  querySchema?: z.ZodObject<any, any>;
  headerSchema?: z.ZodObject<any, any>;
  responseSchemas?: Record<number, z.ZodType>;
  responseHeaderSchemas?: Record<number, z.ZodObject<any, any>>;
  rateLimit?: { windowMs: number; maxRequests: number };
}
export function getHttpCtx(node: RouteNode<any, any, any, any, any>): HttpCtxData | undefined {
  return node._ctx as HttpCtxData | undefined;
}
```

Readers import `getHttpCtx` / `getOpenApiCtx` instead of accessing `node._ctx` directly.

**Pros:**
- `route-node.ts` becomes truly domain-agnostic
- Plugin schemas colocate with plugin logic
- No naming collision risk (each plugin reads/writes its own fields)
- Backward-compatible incremental refactor

**Cons:**
- Adds a 5th type parameter to `RouteNode` (already has 4)
- The `Ctx` parameter is mostly transparent (callers rarely specify it)
- Two separate HTTP-like ctx types (`HttpCtxData` vs `OpenApiCtxData`) still overlap on `method`, `bodySchema`, `responseSchemas`

### Option B — Namespace keys within `_ctx`

Keep `_ctx?: Record<string, unknown>` (no named fields in core). Plugins write to namespaced keys:

```typescript
// http-context.ts sets: node._ctx = { http: { method, bodySchema, ... } }
// openapi-context.ts sets: node._ctx = { openapi: { method, meta, ... } }
```

Typed accessors extract the namespaced sub-object:

```typescript
export function getHttpCtx(node: RouteNode<any>): HttpCtxData | undefined {
  return node._ctx?.['http'] as HttpCtxData | undefined;
}
```

**Pros:**
- No new type parameter on `RouteNode`
- Explicit namespacing prevents key collisions between plugins
- `route-node.ts` change is minimal (remove named fields, keep `_ctx?: Record<string, unknown>`)

**Cons:**
- Namespace string keys are stringly-typed
- Requires all existing write sites to wrap their data one level deeper
- Two plugins (http, openapi) both write `method` and `bodySchema` — they'll need to agree or diverge

### Option C — Make `context` concrete, remove `_ctx`

Currently `context?: Context` is phantom (`undefined as never`). The DI context types (`HttpContext<...>`) already encode all the schema information at the type level. If `context` carried runtime data too, `_ctx` becomes redundant.

```typescript
// httpRoute() sets: context: { method: 'GET', bodySchema: schema, ... }
// InferContext<N> still extracts the type
// node.context?.method replaces node._ctx?.method
```

**Pros:**
- Eliminates the dual mechanism entirely
- Single `context` field for both type inference AND runtime data
- No `RouteCtx`, no `_ctx`

**Cons:**
- Breaks the current phantom invariant (`undefined as never`)
- The `context` field serves *two* purposes now: DI handler typing AND route configuration metadata. These are not the same thing. `HttpContext<Method, Body, Response, Query, Header>` describes what a handler *receives*; `_ctx` describes how the *route* is configured. Merging conflates producer concerns (route factory) with consumer concerns (handler).
- `CtxMap` would need revisiting since it maps tags to Context types for server-side DI
- Higher blast radius

## Recommendation

**Option A** with a refinement: instead of adding a `Ctx` type parameter to `RouteNode`, keep `_ctx?: Record<string, unknown>` and rely on plugin-owned typed accessor functions that cast at the boundary. This avoids a 5th type param while still moving the named types to where they belong.

The two overlapping plugins (http-context and openapi-context) should be reconciled: `openApiRoute` likely should build on `httpRoute`'s metadata shape rather than defining parallel fields. Open question: does `openApiRoute` still make sense as a separate factory, or should it be an enricher on top of `httpRoute`? (See `plan/questions.md`.)

Option C should stay on the table as a long-term goal once the DI context story is clearer (plan 39 — JWT/session contracts — will inform this).

## Dependencies

- None blocking. Safe to execute independently.
- Loosely related to plan 39 (JWT/session contracts) which will flesh out what `Context` carries at runtime.

## Success Criteria

- `route-node.ts` exports no HTTP/OpenAPI field names
- `RouteCtx` interface either removed or reduced to `Record<string, unknown>`
- All existing tests pass
- Type-checking and linting pass
- Plugin-owned typed accessors replace all direct `node._ctx?.field` reads in `walk.ts`, `define-routes.ts`, `fetch-client.ts`, `generate-spec.ts`
