# PLAN.39 — JWT / session authentication contracts (SEC2)

## Context

`spec.roadmap.md` Phase SEC2. The compiler should prevent developers from
writing code that handles private data without proving authentication checks
have occurred. Right now auth is entirely by convention — nothing stops a
handler from returning sensitive data without checking an auth header.

## Goal

A `protectedRoute(node)` higher-order factory that mutates the route's context
type at compile time, making a `session: UserSession` argument mandatory in
the handler.

```typescript
const route = protectedRoute(
  httpRoute({ method: 'GET', path: 'me/', ... }),
);

// Without protectedRoute, ctx has no `session` field.
// With it, session is non-optional and typed.
handler: async ({ params, session }) => {
  return { status: 200 as const, body: { id: session.userId } };
},
```

Runtime enforcement in `createServer`:
1. If the matched route has an `{ authenticated: true }` context marker,
   call the user-supplied `resolveSession(req)` hook.
2. Hook decodes and verifies the JWT from `Authorization: Bearer <token>`.
3. Invalid or missing token → `401 Unauthorized`.
4. Valid token → `UserSession` injected into `ctx`.

The `resolveSession` hook is user-supplied (callers own JWT verification
logic and `UserSession` shape).

## Change Surface

1. **New `src/server/protected-route.ts`**: `protectedRoute<R, Child, C, Q>(node)`.
   Returns a `RouteNode` with context updated to `{ authenticated: true }`.
2. **`server.ts`**: After route match, check for `authenticated: true` in the
   context. If present, call `resolveSession`. On failure → 401.
3. **`createServer` options**: Accept optional `resolveSession(req): Promise<UserSession | null>`.
4. **`ServerHandlers` mapped type**: If route context includes
   `{ authenticated: true }`, `ctx` includes `session: UserSession` (type
   supplied by the caller via generic parameter).
5. Tests:
   - Unprotected route: no `session` in ctx (type-level).
   - Protected route with no/bad token → 401.
   - Protected route with valid token → `session` available in ctx.

## Dependencies

- Plan 22 (DI context) — `ctx` shape must be settled.
- Plan 30 (typed request headers) — `Authorization` header can optionally
  be declared in the contract; coordinate so it doesn't conflict.
- Plan 44 (middleware pipeline) — if the pipeline lands first, `protectedRoute`
  may be expressible as a built-in middleware rather than a factory function.

## Success Criteria

- Handlers on protected routes cannot compile without `session` in scope.
- Requests with missing/invalid tokens return 401.
- `npm test && npm run typecheck` clean.
