# Plan 78 — `IntoResponse`: domain object return from handlers

## Context

Every handler must currently call `respond(status, body)` even when the status is always
200 and the body is just a domain object:

```typescript
'get-user': async (ctx) => respond(200, { id: ctx.params.id, name: 'Alice' }),
```

For routes with a single declared success status code (the majority of read routes), the
`respond(200, ...)` is mechanical boilerplate. Handlers returning the domain object
directly would be shorter and easier to unit-test:

```typescript
'get-user': async (ctx) => ({ id: ctx.params.id, name: 'Alice' }),
```

The framework can infer the status code: if the declared response map has exactly one
non-error entry and the return value matches that body type, it wraps it automatically.

This is inspired by Rust/Axum's `IntoResponse` trait but scoped to the common case.
The explicit `respond()` helper remains valid for all other cases (multiple success codes,
non-200 success, error returns).

## Goal

When a handler's return type matches the single declared success-status body type, the
framework accepts it directly and wraps it in the appropriate `respond()` call.

## Approach

Add an overload to the handler function type — the existing `HandlerFn` type:

```typescript
type HandlerFn<Ctx, Resp extends HttpResponse> =
  | ((ctx: Ctx) => Promise<Resp>)                    // explicit respond()
  | ((ctx: Ctx) => Promise<InferSingleSuccessBody<Resp>>)  // direct domain object
  | ((ctx: Ctx) => InferSingleSuccessBody<Resp>);    // sync (edge case)

// InferSingleSuccessBody<R>: if R has exactly one 2xx status, return its body type.
// Otherwise: never (direct return not allowed; must use respond()).
type InferSingleSuccessBody<R extends HttpResponse> =
  Extract<R, { status: 200 | 201 | 202 | 204 }> extends { status: infer S; body: infer B }
    ? [S] extends [never] ? never
    : number extends S ? never  // more than one 2xx — exact count hard; use a helper
    : B
  : never;
```

The exact type machinery for "exactly one 2xx code" requires a count helper (similar to
`FlattenChildrenImpl`'s depth counter). If this proves too complex, scope to routes where
only one status code total is declared (simpler, covers the obvious case).

`createServer` / `executeRoute`: after the handler returns, if the return value is not
an `HttpResponse` (test for `'status' in result && 'body' in result`), wrap it as
`respond(successStatus, result)`.

## Design constraints

- **Pay for what you use**: this is entirely additive. Handlers that already use `respond()`
  are unaffected. The new overload only activates when the inferred single-success type is
  non-`never`.
- **No magic for multi-status routes**: routes with `{ 200: ..., 404: ... }` must still
  use `respond()`. Ambiguous returns are a type error, not a runtime guess.
- **Sync returns**: allow both `Promise<Body>` and `Body` direct returns since the handler
  wrapper already normalizes async.

## Change Surface

- `src/server/server.ts` — update `HandlerFn` type; add runtime wrap in `executeRoute`
- `src/contexts/http-context.ts` — `InferSingleSuccessBody` utility type
- `src/index.ts` — export `InferSingleSuccessBody` if useful externally
- `src/server/server.test.ts` — test direct-return handlers for single-status routes

## Dependencies

None. But this plan should follow Plan 76 (`.use()` builder) since the handler type changes
touch the same area of the codebase.

## Success Criteria

- Handler returning `{ id: '1', name: 'Alice' }` on a `{ response: { 200: UserSchema } }`
  route compiles and dispatches correctly.
- Handler returning a domain object on a multi-status route is a type error.
- Explicit `respond()` still works unchanged.
- `pnpm test && pnpm typecheck` green.
