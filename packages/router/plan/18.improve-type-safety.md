# PLAN.18 — Eliminate unjustified `as any` / `as unknown as` casts

## Context

Several type casts exist that either paper over structural mismatches in the
surface API or indicate real type-safety gaps. The `generateSpec` casts in
tests are accepted as inherent to inspecting raw JSON output. The three issues
below are genuine debt.

---

## Issue 1 (high): Runtime context shape doesn't match the type-level interface

**Files**: `src/contexts/http-context.ts:58`, `src/contexts/openapi-context.ts:70`

Both `httpRoute` and `openApiRoute` construct a runtime object:

```typescript
// runtime fields
{ method, bodySchema?, responseSchemas }
```

…and cast it to the type-level interface:

```typescript
// type-level fields
HttpContext<Method, Body, Response> = { method, body, response }
```

The field names differ (`bodySchema` → `body`, `responseSchemas` → `response`)
and the value kinds differ (Zod schema objects at runtime, inferred TypeScript
types at the type level). The `as unknown as HttpContext<...>` cast is the only
thing bridging this gap.

**Root cause**: `context` is treated as both a phantom type carrier (like
`_type`/`_child`) and a concrete runtime data bag. These two purposes have
incompatible shapes.

**Fix direction**: Choose one:

a. **Make `context` fully phantom** (like `_type`): store `undefined as never`,
   move runtime data to a separate concrete field (e.g. `_ctx`). `walkSpec`,
   `buildMethodMap`, etc. read from `_ctx`. No cast needed; the type-level
   `HttpContext` becomes a true phantom.

b. **Align the type-level interface to the runtime shape**: rename `body` →
   `bodySchema` and `response` → `responseSchemas` on `HttpContext`, and use
   `z.ZodType<Body>` / `Record<number, z.ZodType>` there. `InferContext` can
   then project to the inferred types. Public callers see Zod types, which is
   honest about what is actually stored.

Option (a) is cleaner because it keeps the existing ergonomic type-level
interface for callers (`InferContext` stays as-is) and keeps the phantom
pattern consistent.

**Secondary effect**: Fixing this also eliminates the cast in
`openapi-context.test.ts:40` (see Issue 1b below).

## Issue 1b (follows from 1): Test cast for internal `responseSchemas`

**File**: `src/contexts/openapi-context.test.ts:40`

```typescript
expect((r.context as unknown as { responseSchemas: unknown }).responseSchemas).toBeDefined();
```

The test wants to verify the runtime Zod schemas are stored, but
`r.context` is typed as `OpenApiContext<...>` which exposes `response`
(inferred types), not `responseSchemas` (Zod schemas). Once Issue 1 is
resolved (context becomes a true phantom and runtime data is in `_ctx`), this
test can read from `r._ctx.responseSchemas` directly without any cast.

## Issue 2 (moderate): `print()` doesn't accept section path params in its type

**Files**: `src/core/define-routes.test.ts:177,224`

```typescript
// eslint-disable-next-line @typescript-eslint/no-unsafe-argument
const url = sectionRouter.print({ child: { tag: 'project', projectId: 42, orgId: 'acme' } } as any);
```

`print()` is typed as `(route: Route) => string` where `Route` is the
discriminated union of leaf routes. Section path params (`orgId`) are not part
of any leaf route type, but the URL builder requires them at runtime to fill
the section's path template. Passing them requires bypassing the type system.

This is a real API gap: it is possible to call `print()` with a valid
type-checked argument and get a runtime error or a broken URL because a
required section param was omitted.

**Fix direction**: Thread section path params into the `print()` signature.
One approach: introduce a second argument `sectionParams?: Record<string, string | number>`
that is type-safe via a new phantom type tracking which params the section
hierarchy requires. This would make the omission a type error rather than a
silent runtime failure.

A simpler (but weaker) fix: make `print()` accept an intersection of `Route`
with `Record<string, unknown>` and document that section params must be
included. That removes the `as any` without fully closing the gap.

## Issue 3 (minor): `HttpContext<any, any, any>` in constraint positions

**Files**: `src/contexts/http-context.ts:1`, `src/client/fetch-client.ts:1`
(file-level `eslint-disable @typescript-eslint/no-explicit-any`)

Several places use `HttpContext<any, any, any>` as a "wildcard" constraint:

```typescript
type BodyArgs<Ctx extends HttpContext<any, any, any>> = ...
_ctxMap: Record<string, HttpContext<any, any, any>>
```

These could be tightened to `HttpContext<HttpMethod, unknown, Record<number, unknown>>`
which keeps the constraint meaningful without silencing the rule file-wide.
Not critical, but would allow both file-level disables to be removed.

---

## Accepted (not debt)

- `openapi-context.test.ts:250,263`: typed `Record` casts when inspecting
  deeply nested JSON schema output. `generateSpec` returns `Record<string, unknown>`;
  deep test assertions always require casting into the tree. No change needed.
