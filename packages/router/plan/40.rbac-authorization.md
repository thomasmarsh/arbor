# PLAN.40 — RBAC authorization (SEC3)

## Context

`spec.roadmap.md` Phase SEC3. After plan 39 proves authentication (identity is
known), authorization (permissions are checked) is a second compile-time and
runtime layer. Different endpoints need different clearance levels.

## Goal

An `authorizedRoute(node, roles)` factory that extends `protectedRoute` with
a required roles check:

```typescript
const route = authorizedRoute(
  httpRoute({ method: 'DELETE', path: 'admin/users/:id/', ... }),
  ['admin', 'super-user'],
);
```

At compile time, the route context carries `{ authenticated: true; requiredRoles: string[] }`.
At runtime:
1. The auth flow from plan 39 runs first (resolveSession).
2. After a valid session is obtained, the server checks
   `session.roles` intersects `requiredRoles`.
3. No matching role → `403 Forbidden`.

`authorizedRoute` composes with `protectedRoute` — it is a strict superset.
Callers should be able to use either without both.

## Change Surface

1. **New `src/server/authorized-route.ts`**: `authorizedRoute(node, roles)`.
   Wraps `protectedRoute` and adds `requiredRoles` to the context type.
2. **`server.ts`**: After session resolution (plan 39 flow), check roles.
   `403` on insufficient clearance.
3. Tests:
   - User with matching role → handler called.
   - User with no matching role → 403.
   - Unauthenticated request → 401 (hits plan 39 check first).

## Dependencies

- Plan 39 (JWT/session) — RBAC sits on top of the auth stack; must land first.
- Plan 50 (enrichers public API) — use `withEnricher` as the mechanism instead
  of a pipeline entry. Each plan delivers a named enricher function.

Enricher name: `withRbac`. Short-circuits with 403 on insufficient role.

## Success Criteria

- Requests from authenticated users without the required role return 403.
- Requests from unauthenticated users still return 401 (not 403).
- `npm test && npm run typecheck` clean.
