# PLAN.37 — Telemetry / metrics decorator (S2 part 2)

## Context

`spec.roadmap.md` Phase S2, split into two plans. This plan covers structured
telemetry. Rate limiting (plan 36) is the other half.

Observability is infrastructure-level: it should wrap the entire request loop
without touching individual route handlers. The pattern is a functional
decorator over `createServer`'s handler dispatch, not a per-route concern.

## Goal

A `withMetrics(server, emitter)` functional decorator that wraps the
`createServer` handler. For every request it measures wall-clock duration and
emits a structured event:

```typescript
interface RequestMetric {
  tag: string;         // matched route tag (or 'unmatched')
  method: string;
  status: number;
  durationMs: number;
  timestamp: number;
}

type MetricsEmitter = (metric: RequestMetric) => void;

const server = withMetrics(
  createServer(router, handlers),
  (m) => console.log(JSON.stringify(m)),
);
```

The emitter is synchronous and fire-and-forget. Callers can bridge it to
OpenTelemetry, Prometheus, Datadog, etc.

## Change Surface

1. **New `src/server/with-metrics.ts`**: `withMetrics(handler, emitter)`.
   The wrapper intercepts the WinterCG `fetch(req)` function, records start
   time, awaits the inner handler, records end time, and calls the emitter.
2. No changes to `createServer` — `withMetrics` wraps the output, not the
   internals.
3. Tests:
   - Emitter is called once per request.
   - `tag` is `'unmatched'` when no route matches.
   - `durationMs` is a non-negative number.
   - `status` matches the actual response status code.

## Dependencies

- Plan 22 (DI context) — `tag` comes from the matched route; must be
  available at the server dispatch layer.

## Success Criteria

- Every request (matched or not) calls the emitter with correct fields.
- Decorator is transparent: the `Response` from the inner handler is
  returned unchanged.
- `npm test && npm run typecheck` clean.
