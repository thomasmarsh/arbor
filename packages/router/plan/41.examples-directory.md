# PLAN.41 — Examples directory

## Context

Tests verify correctness but they are not written to be read. As the API
surface grows (servers, clients, auth, OpenAPI, multipart) we need living
examples that show how the pieces compose in realistic scenarios. These serve
as the primary human-facing documentation and act as integration tests for
the full stack.

An `examples/` directory co-located with the package is the right starting
point. It can be extracted to a top-level monorepo directory later if needed.

## Goal

Create `packages/router/examples/` with an initial set of self-contained
runnable examples. Each example is a single TypeScript file that can be run
directly with `tsx` (no build step).

### Initial examples

| File                | What it shows                                    |
| ------------------- | ------------------------------------------------ |
| `basic-server.ts`   | Minimal server with one route, no auth           |
| `basic-client.ts`   | `client.fetch` against the basic server          |
| `query-params.ts`   | Route with a query schema; parse + type-safe use |
| `nested-routes.ts`  | Multi-level route tree; `print()` output         |
| `openapi-output.ts` | `generateSpec(router)` → stdout JSON             |
| `auth-protected.ts` | `protectedRoute` with a mock `resolveSession`    |

Each example must:

- Be runnable: `tsx examples/<name>.ts` produces output or a live server.
- Be self-documenting: a short comment at the top explains what it demos.
- Import only from `../src` (relative) so no publish step is required.
- Not be test files — no `expect` / `describe`. Real code only.

## Change Surface

1. Create `packages/router/examples/` directory.
2. Add each example file listed above.
3. Add a `scripts.examples` entry to `package.json` (or a `Makefile` target)
   that runs all examples in sequence as a smoke test:
   `tsx examples/basic-server.ts && tsx examples/openapi-output.ts` etc.
4. Add an `examples/` entry to `.gitignore` patterns that should be excluded
   (node_modules etc.) — but track the `.ts` files themselves.

## Dependencies

- Examples are added progressively as their feature plans complete:
  - `basic-server.ts`, `basic-client.ts`: after plan 22.
  - `query-params.ts`, `nested-routes.ts`: after plan 20.
  - `openapi-output.ts`: after plan 23.
  - `auth-protected.ts`: after plan 39.
- First batch (`basic-server`, `nested-routes`, `query-params`) can land
  immediately after plan 22.

## Success Criteria

- All example files in the initial set run without errors.
- Each example produces visible output (printed JSON, a bound port, etc.).
- Examples are not duplicates of test fixtures — they tell a user story.
- CLAUDE.md includes instructions that help ensure the examples stay up to date as part of future work
