# Plan 75 — `matchResponse` exhaustive combinator

## Context

`client.fetch(route)` returns a typed union `{ status: 200; body: User } | { status: 404; body: Err }`.
Nothing in the type system currently requires callers to handle every declared status code.
An unhandled 404 silently produces `undefined` at runtime. TypeScript will not catch this — a
`switch` on `response.status` with only a `case 200:` branch compiles cleanly.

`matchResponse` closes this gap: it is the client-side mirror of the exhaustive
`switch(route.tag)` pattern already enforced on the server side. A missing branch is a type
error; the compiler makes unhandled status codes unrepresentable.

```typescript
const response = await client.fetch(userRoute);

matchResponse(response, {
  200: (body) => renderUser(body),      // body: User — inferred
  404: (body) => renderNotFound(body),  // body: { error: string } — inferred
  // omitting either branch is a compile error without a _ fallback
});

// Or with a catch-all for unexpected codes:
matchResponse(response, {
  200: (body) => renderUser(body),
  _:   (res)  => handleUnknown(res),
});
```

This also makes testing natural: `matchResponse` in a test asserts that all status
codes are exercised, which type-checks exhaustion at compile time rather than relying on
assertion counts.

## Goal

Export `matchResponse(response, handlers)` that:
1. Requires a handler for every status code in the response union, OR a `_` catch-all.
2. Returns the union of all handler return types.
3. Has zero runtime overhead beyond a function call — no reflection, no iteration.

## Type design

```typescript
type MatchHandlers<R extends { status: number; body: unknown }, Ret> =
  | { [S in R['status']]: (body: Extract<R, { status: S }>['body']) => Ret }
  | (
      Partial<{ [S in R['status']]: (body: Extract<R, { status: S }>['body']) => Ret }>
      & { _: (response: R) => Ret }
    );

export function matchResponse<R extends { status: number; body: unknown }, Ret>(
  response: R,
  handlers: MatchHandlers<R, Ret>,
): Ret;
```

Implementation: check `response.status` against the `handlers` object keys; if no
specific handler, fall through to `_`. Both branches call the handler with the body.
The runtime is five lines.

## Dependencies

None. Usable immediately with the existing client. Becomes more useful once Plan 72
(`createTestClient`) ships, since tests can assert exhaustive handling cheaply.

## Success Criteria

- `matchResponse(res, { 200: f })` on a `200 | 404` union is a type error.
- `matchResponse(res, { 200: f, _: g })` compiles cleanly.
- `matchResponse(res, { 200: f, 404: g })` compiles cleanly.
- Return type is correctly inferred as the union of handler return types.
- `pnpm test && pnpm typecheck` green.
