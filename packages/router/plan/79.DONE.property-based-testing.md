# Plan 79 — Property-based / fuzz testing from Zod schemas

## Context

`spec.enhancements.md` Phase 3 #3 describes an "E2E integration test rig that fires
intentional malformed inputs at your server." The right implementation is broader: because
every route's params, body, query, and headers are Zod schemas, the framework can generate
**arbitrary valid inputs** automatically and verify that handlers respond with one of their
declared response schemas. No manual test authoring required.

The gap this fills: unit tests cover happy paths; `createTestClient` (Plan 72) enables
integration paths. Property-based testing finds the edge cases neither covers — the
`undefined` field that panics a handler, the ID that produces a 500 when only 200/404
are declared, the body that passes Zod validation but triggers an unexpected code path.

This is a strong correctness signal specifically useful for adopters of Bombadil or
similar contract-testing frameworks: the route tree IS the contract; a property test
over it is an automated contract compliance check.

## Goal

Export `createRouteTests(router, handlers, options?)` that returns a test suite (framework-
agnostic: a map of test cases as plain functions) where each route gets:

1. A **valid-input fuzz test**: generates N random valid inputs via `fast-check` +
   `@fast-check/zod`, sends them through `createTestClient`, asserts the response status is
   one of the declared codes and the body parses against the matching response schema.

2. A **boundary / malformed-input test**: generates intentionally invalid inputs (wrong
   types, missing required fields, out-of-range values) and asserts the server returns 400.

## Pay-for-what-you-use design

The new dependency (`fast-check` + `@fast-check/zod`) must **not** enter the main
`@arbor/router` bundle. Options:

- **Separate package** `@arbor/router-test` (preferred — clean boundary, teams that don't
  test skip the dep entirely).
- **Peer dependency with dynamic import** (acceptable — no install-time cost, import error
  at runtime if missing rather than silent).

The output of `createRouteTests` is framework-agnostic: it returns an object where keys
are test names and values are async functions. Callers register them into Vitest, Jest, or
any other runner with a one-line shim.

```typescript
import { createRouteTests } from '@arbor/router-test';

const tests = createRouteTests(router, {
  'get-user': async (ctx) => respond(200, { id: ctx.params.id, name: 'Alice' }),
});

// Vitest integration:
for (const [name, fn] of Object.entries(tests)) {
  it(name, fn);
}
```

## Spike deliverable (before full implementation)

Before writing the full package, confirm:

1. `@fast-check/zod` (or `zod-fast-check`) can generate inputs for all Zod types used in
   this codebase: `z.string()`, `z.number()`, `z.literal()`, `z.object()`, nested
   objects, `z.optional()`, `z.array()`, `z.union()`.
2. Path param generation: params are string-typed; for `#id` routes the generated value
   must be an integer string. Zod's schema for `#id` is `z.number()` but the path value
   is a string representation. Confirm the generation + serialization round-trip works.

## Change Surface

- `packages/router-test/` — new package (or `src/testing/` if kept in-repo)
- `packages/router-test/src/create-route-tests.ts` — main export
- `packages/router-test/src/arbitraries.ts` — Zod-to-fast-check adapter helpers
- `packages/router/src/index.ts` — no changes (test utilities live in the test package)

## Dependencies

- Plan 72 (`createTestClient`) — the property tests use the in-memory server internally.
- Plan 75 (`matchResponse`) — the assertion logic uses `matchResponse` to verify
  exhaustive response coverage.
- External: `fast-check` + `@fast-check/zod` (dev-only in the test package).

## Success Criteria

- Spike confirms Zod → fast-check generation works for all schema types in the codebase.
- `createRouteTests(router, handlers)` produces tests that catch a deliberately broken
  handler (one that returns 500 instead of 404).
- The test package has zero runtime dependencies on `@arbor/router`'s server internals
  beyond the public API.
- `pnpm test` green in both packages.
