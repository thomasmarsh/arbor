# PLAN.29 — Typed response headers

## Context

Every plan through 23 treats `Response` bodies and status codes as the full
API contract. But HTTP responses routinely carry semantic headers:
`Location` on redirects, `ETag`/`Cache-Control` for caching, `X-Request-Id`
for tracing, pagination cursors in `Link` or `X-Next-Cursor`. Right now
handlers can set these freely but nothing enforces or documents them.

ts-rest and tRPC-openapi both treat response headers as first-class contract
members, which means they flow into OpenAPI output and the typed client. We
are missing the same.

## Goal

Allow a route to declare expected response headers via a Zod schema alongside
the existing per-status-code body schema. The type flows into:

1. The handler return type — TypeScript enforces which headers must be present.
2. The OpenAPI generator (plan 23) — headers appear in the spec automatically.
3. The typed client (plan 31 spike) — callers can read headers in a typed way.

```typescript
// Declaration
httpRoute({
  method: 'GET',
  path: 'users/:id/',
  schema,
  responses: {
    200: {
      body: UserSchema,
      headers: z.object({
        'x-request-id': z.string().uuid(),
        'cache-control': z.string().optional(),
      }),
    },
    302: {
      body: z.null(),
      headers: z.object({ location: z.string().url() }),
    },
  },
});

// Handler return — TypeScript requires the declared headers
handler: async (ctx) => ({
  status: 200 as const,
  body: user,
  headers: { 'x-request-id': crypto.randomUUID() },
}),
```

## Change Surface

1. **`http-context.ts`**: Extend the per-status response descriptor to accept
   an optional `headers` Zod schema alongside `body`.
2. **`server.ts`**: After the handler returns, validate the response headers
   against the declared schema (dev-mode assertion; not enforced in production
   for performance). Apply the headers to the `Response` object.
3. **`ServerHandlers` mapped type**: If a status code declares response
   headers, the handler return type requires a `headers` object with those
   keys.
4. **`plan/23.openapi-generator.md`** (addendum): Response headers appear in
   the generated spec.
5. Tests: handler that omits a required header fails type-check; runtime
   headers appear on the `Response`.

## Dependencies

- Plan 22 (DI context) — handler shape must be settled before adding headers
  to the return type.
- Plan 23 (OpenAPI) — headers need to be fed into the spec; coordinate rather
  than duplicate.

## Success Criteria

- Declaring a response header schema on a route makes it required in the
  handler return type.
- `Response` returned by `createServer` carries the declared headers.
- `npm test && npm run typecheck` clean.
