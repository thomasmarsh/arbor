# PLAN.46 ‚Äî Spike: restructure `Derive`/`ChildUnion` to eliminate mutual recursion

## Context

Plan 28 (`_child` removal) is blocked because `Derive` and `ChildUnion` are
mutually recursive type aliases. In the current design, `_child: Child` is a
structural anchor that lets `Derive<N>` recover the child union in O(1) by
reading a precomputed phantom field. Removing `_child` forces `Derive` to call
`ChildUnion<C>` directly, which calls `Derive` for each child node, which calls
`ChildUnion` again ‚Äî TypeScript hits `TS2589: Type instantiation is excessively
deep` even on the 3-level test trees already in the suite.

Plan 43 benchmarks *where* inference limits bite. This spike answers a different
question: *can we restructure the type algebra so the limits never apply?*

## Goal

Confirm, in a self-contained playground file, that a single-recursive type
approach can replace the `Derive` + `ChildUnion` pair without `_child` and
without hitting TypeScript's depth limit. Produce a working prototype that
passes `tsc --noEmit`, then decide whether to open a full implementation plan.

## Approach to explore

Replace the two mutually-recursive aliases with a single mapped type that
recurses through children directly:

```typescript
// Single recursive type: maps a children tuple to a route-shape tuple
type FlattenChildren<C extends RouteNode<unknown, unknown, any, any, any>[]> = {
  [K in keyof C]: C[K] extends RouteNode<infer R, any, infer GC, any, any>
    ? [R] extends [never]
      ? Flatten<{ child: FlattenChildren<GC>[number] }>
      : [FlattenChildren<GC>[number]] extends [never]
        ? Flatten<R>
        : Flatten<R & { child?: FlattenChildren<GC>[number] }>
    : never;
};

// Then:
type ChildUnion<C extends RouteNode<...>[]> = FlattenChildren<C>[number];
type Derive<N>                              = FlattenChildren<[N]>[0];
```

`FlattenChildren` is singly recursive (calls itself, not another alias that
calls back). TypeScript 4.1+ specifically added support for this pattern in
mapped types. Mutual recursion across two aliases (`A ‚Üí B ‚Üí A`) is the
pathological case; single self-recursion in a mapped type is handled better.

Secondary approaches to explore if the primary fails:

- **Property-access instead of conditional inference**: read `N['_type']` and
  `N['children']` directly rather than using `infer`, removing one layer of
  conditional type indirection.
- **Depth counter**: add a compile-time depth limiter tuple as a second type
  parameter to both aliases, keeping mutual recursion but capping instantiation
  depth explicitly.

## Change Surface

Spike only ‚Äî do **not** modify production code:

1. Create `src/core/__spike__/derive-no-mutual-recursion.ts` with the new type
   definitions (`FlattenChildren`, new `Derive`, new `ChildUnion`).
2. Write `expectTypeOf` assertions against the same scenarios as
   `route-node.test.ts` `Derive` and `ChildUnion` test suites, plus a 4-level
   tree to confirm depth is not a problem.
3. Run `tsc --noEmit` (not `pnpm test`) to validate types.
4. Delete the spike file when done ‚Äî do not commit it.

## Success Criteria

**Green** ‚Äî spike passes `tsc --noEmit` with no `TS2589` errors, all type
assertions hold for at least a 4-level route tree, and the `_child` field is
absent from the prototype definitions. Open a new implementation plan (28b)
immediately.

**Yellow** ‚Äî depth-counter approach works but adds a second type parameter to
public aliases. Document trade-offs in `plan/questions.md` (new question) and
let the user decide before opening 28b.

**Red** ‚Äî no approach avoids the depth limit. Close 28 permanently; add a note
to plan 43 to measure the current limit and document it as a known constraint.
