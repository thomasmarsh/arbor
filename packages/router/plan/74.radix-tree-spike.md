# Plan 74 — Radix tree router spike

## Context

`spec.enhancements.md` Phase 4 #2 and `spec.workflow.md` deferred section: "Upgrade the
underlying runtime parser from an O(N) loop over a flat array to an O(L) Radix/Patricia
Tree lookup (where L is path depth) to minimize routing overhead on hot backend pathways."

The current `walkParse()` iterates over all nodes in the tree depth-first. For a tree with
N leaf routes and a URL at depth L, the worst case is O(N × L) segment comparisons. For
small route trees (< 100 routes) this is irrelevant; at thousands of routes on hot paths
it becomes measurable.

The key constraint: any replacement must be **transparent to the public API**. `router.parse(url)`
returns a `Result<Route, string>` — callers cannot know or care what data structure is
underneath.

## Goal

Determine via benchmark whether the O(N) lookup is a real bottleneck in realistic route
trees, and if so, design a drop-in Patricia trie implementation that preserves the existing
API and segment semantics (including wildcards, optional params, and numeric params from
Plans 67–69).

## Spike deliverables

1. **Benchmark** — a standalone script that constructs a tree of N routes (50, 200, 500,
   1000) and measures `router.parse(url)` throughput (ops/sec) using `performance.now()`.
   Report: at what N does O(N) become measurable (>10% of total request handling budget)?

2. **Design sketch** — if the benchmark shows a bottleneck, document the trie structure:
   - Literal segments → trie edges (exact string match, no allocation)
   - Param segments (`:id`, `#id`) → special wildcard edges tried after all literal edges
   - Wildcard (`*rest`) → terminal catch-all edge
   - Optional params — how they are represented without backtracking

3. **Decision** — proceed to implementation or close as "not needed" based on benchmark.

## Implementation scope (if spike proceeds)

- New `src/core/route-trie.ts` — trie construction from `WalkNode[]`
- Modified `defineRoutes()` — build trie at definition time; `parse()` queries it
- Existing `walkParse()` retained for `walkPrint()` and `walkCollect()` (these are not
  hot-path; they run once at definition time)

## Dependencies

Plans 67–69 (segment correctness) must be complete before trie implementation, as the
trie edge semantics depend on the finalized segment kinds and matching rules.

## Priority

LOW — do not start until a benchmark confirms O(N) is measurable in a real workload.
Until then this plan exists to capture the design intent and spike scope.

## Success Criteria for spike

- Benchmark script committed under `benchmarks/` and runnable with `tsx`.
- Report logged to `plan/NOTES.md` with N-vs-ops/sec table.
- Decision recorded in this plan file before any implementation begins.
