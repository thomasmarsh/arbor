# PLAN.31 (Spike) — Typed HTTP API client architecture

## Context

ts-rest's most compelling feature is a typed client where
`client.users.get({ params: { id } })` is fully type-safe from the same
contract definition — no codegen, no duplication. The current project has a
rough `client.fetch` concept in the enhancements spec but no concrete plan or
design for a standalone typed client SDK.

This is the single largest gap relative to ts-rest and tRPC-openapi. Before
any implementation work starts, we need to understand two unresolved
architectural questions (Q8 below).

## Questions to Resolve

**Q8a — Package boundary**: Does the client live in this package (`@arbor/router`)
as an optional export, or in a separate package (`@arbor/router-client`) that
imports the route contract? The separation avoids pulling server deps into
browser bundles, but adds a package to maintain.

**Q8b — Generation strategy**: Does the typed client derive its types directly
from `RouteNode` / `ServerHandlers` (zero runtime cost, most type-safe), or
from the OpenAPI JSON output (more portable, works with third-party specs)?
The RouteNode approach gives tighter inference but couples client to internals;
the OpenAPI approach enables interop but loses some phantom-type precision.

## Spike Deliverable

A short decision memo (≤ 500 words) answering:

1. Can `ServerHandlers` (or a derived type) be used directly as the client's
   type source without leaking server-only code into browser bundles?
2. Which generation strategy preserves the most type fidelity for the cases
   that matter most (path params, discriminated response bodies by status code)?
3. Recommended package structure and a sketch of the client call signature.

Include a minimal prototype (10–20 lines of types only, no runtime) to verify
that the preferred shape is expressible before committing to implementation.

## Time-box

One session. If the prototype reveals a blocking type-system constraint,
escalate to a full design plan rather than forcing a solution.

## Dependencies

- Plans 22+23 should be complete — the `ctx` shape (22) and OpenAPI output
  (23) are both inputs to the design decision.
- Plans 29+30 (typed headers) should be at least drafted — the client needs
  to model request and response headers.

## Next Step (if spike succeeds)

A new implementation plan is opened from the spike memo's recommendation.
