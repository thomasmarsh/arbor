# PLAN.36 — Per-route rate limiting (S2 part 1)

## Context

`spec.roadmap.md` Phase S2, split into two plans. Rate limiting and telemetry
are independent concerns that share only the middleware pipeline (plan 44).
This plan covers rate limiting.

Declaring rate limits at the contract level (rather than in separate
infrastructure config) keeps the enforcement rules co-located with the route
definition and makes them visible in the OpenAPI spec.

## Goal

Allow routes to declare a `rateLimit` policy. `createServer` enforces it
before calling the handler.

```typescript
httpRoute({
  method: 'POST',
  path: 'auth/login/',
  rateLimit: { windowMs: 60_000, maxRequests: 5 },
  ...
});
```

The implementation uses an abstract `RateLimitStore` interface so callers
can supply an in-memory map (default) or a Redis-backed implementation:

```typescript
interface RateLimitStore {
  increment(key: string, windowMs: number): Promise<number>;
  reset(key: string): Promise<void>;
}
```

The request identifier defaults to `x-forwarded-for` → `req.url` IP parsing.
Callers can supply a custom `keyResolver(req)` function.

Exceeding the limit → `429 Too Many Requests` with a `Retry-After` header.

## Change Surface

1. **`http-context.ts`**: Add optional `rateLimit` field to the route context.
2. **New `src/server/rate-limit.ts`**: `RateLimitStore` interface + default
   in-memory implementation using a `Map<string, { count; resetAt }>`.
3. **`server.ts`**: After route match, if `rateLimit` is declared, call the
   store and return 429 if exceeded.
4. **`createServer` options**: Accept optional `rateLimitStore` and
   `rateLimitKeyResolver`.
5. Tests:
   - Requests within limit → 200.
   - Requests exceeding limit → 429 with `Retry-After`.
   - Custom key resolver is called.

## Dependencies

- Plan 44 (middleware pipeline) — rate limiting is a pre-handler concern;
  if middleware pipeline exists, rate limiting becomes a built-in middleware
  rather than ad-hoc code in `server.ts`. This plan may be simplified if 44
  lands first.

## Success Criteria

- Sixth request within a 5-request window returns 429.
- `Retry-After` header is set.
- `npm test && npm run typecheck` clean.
