# Plan 90 — WebSocket Route (`wsRoute`)

## Context

WebSocket is the bidirectional case of the session type story. Unlike SSE (unidirectional,
plan 89), a WebSocket channel carries two message streams simultaneously: client→server and
server→client. The session type encoding makes this explicit:

- Server type: `Recv<ClientMsg, Send<ServerMsg, S>>` — receive from client, send to client
- Client type (dual): `Send<ClientMsg, Recv<ServerMsg, S>>` — send to server, receive from server

This is the most direct application of the `Dual<S>` type from plan 88: the server declares
its session, and the client automatically gets the protocol-compatible counterpart. No
manually-maintained interface contract; the dual IS the guarantee.

The major obstacle is the WinterCG WebSocket API, which varies across runtimes. This plan
defines a pluggable adapter interface so the typed layer works identically on Node `ws`,
Bun native WS, Cloudflare Workers WS, and browser WS.

## Goal

Ship, end-to-end:

1. **`wsRoute(schema, path, opts)`** — factory; `opts.in` (client→server) and `opts.out`
   (server→client) are Zod schemas; returns a `RouteNode` with `WsMeta<In, Out>` in `_meta`
2. **Typed channel** — `WsChannel<S extends Session>` provides `.send(msg: ServerMsg)` and
   `messages: AsyncIterable<ClientMsg>` typed from `opts`
3. **Server dispatch** — recognises WS upgrade requests; hands the typed channel to the
   handler; serializes/parses messages against the Zod schemas
4. **Typed WS client** — `createClient().connect(route)` returns `WsChannel<Dual<S>>` so
   the client's `.send` type matches the server's `.recv` type and vice versa
5. **Pluggable transport adapter** — `WsAdapter` interface isolates runtime WS from the
   typed layer; ships with a Node `ws` reference adapter
6. **Example** — `examples/ws-chat.ts` demonstrates a typed bidirectional message exchange

## Success Criteria

- `wsRoute(S, 'chat', { in: z.object({ text: z.string() }), out: z.object({ reply: z.string() }) })`:
  - Server handler receives `WsChannel<Recv<{text: string}, Send<{reply: string}, S>>>`
  - Client `connect()` returns `WsChannel<Send<{text: string}, Recv<{reply: string}, S>>>`
  - Calling `channel.send({ reply: 42 })` on the server is a type error (wrong shape)
- In-memory loopback test: client sends a message, server echoes it back, client receives it
- Transport is injectable; a mock adapter enables tests without real network I/O
- `WsAdapter` interface documented with instructions for custom runtimes
- `pnpm run examples` shows typed WS chat exchange
- `pnpm test && pnpm typecheck && pnpm lint` green

## Design Notes

### WsAdapter Interface

Decouples runtime WS from the typed layer:

```typescript
interface WsAdapter {
  onMessage(handler: (raw: string) => void): void;
  onClose(handler: () => void): void;
  send(data: string): void;
  close(code?: number): void;
}
```

The server provides a `WsAdapter`; the typed channel wraps it. This is the same
pattern as `createServer`'s injectable `fetch` — no runtime coupling to Node or Bun.

### Session Type Granularity

`wsRoute`'s `{ in, out }` option is a convenience shorthand. Internally it is mapped to
a session type of the form `Recv<In, Send<Out, S>>` for a single request/response cycle,
or to a recursive type for a streaming exchange. For the first delivery, the simpler
`AsyncIterable<Out>` for the server stream and `(msg: In) => void` for sending is
sufficient; the full session type encoding comes from plan 88 foundations.

### Protocol Sequencing (future)

A more expressive `wsRoute` would accept a full session type:

```typescript
wsRoute(schema, path, {
  protocol: pipe(Recv(InSchema), Send(OutSchema), Recv(InSchema), End),
})
```

This is not in scope for plan 90. Defer to a follow-on plan once the bidirectional
`{ in, out }` API is validated.

### MPST Bridge (future)

Once plan 91 (MPST spike) validates multi-party projection, `wsRoute` becomes a
two-participant special case of the general MPST protocol. The `WsChannel` type already
maps to a local session type for one participant; upgrading it to carry an MPST projection
should require only a type-layer change, not a runtime change.
