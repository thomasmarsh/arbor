# Plan 53 — RouteCtx: Move Domain Fields to Plugin Modules

## Context

`RouteCtx` in `src/core/route-node.ts` lists HTTP-specific fields (`method`, `bodySchema`, `querySchema`, `headerSchema`, `responseSchemas`, `responseHeaderSchemas`, `rateLimit`, `meta`) directly in the core type file, violating the "core types should be domain-independent" principle.

After plan 52 (openApiRoute wraps httpRoute), there is effectively one `_ctx` shape: `HttpContextData` (already defined in `http-context.ts`), with OpenAPI routes adding `meta`. The cleanup can now proceed.

See plan 51 for the full option analysis. This plan implements Option A.

## Goal

- `route-node.ts` exports no HTTP/OpenAPI field names
- `RouteCtx` reduced to `Record<string, unknown>` (or removed entirely)
- `HttpContextData` in `http-context.ts` becomes the canonical typed shape
- All consumers import a typed accessor instead of reading `_ctx` fields directly

## Change Surface

### 1. `src/core/route-node.ts`

Remove `RouteCtx` interface. Change `_ctx` field type:

```typescript
// Before
_ctx?: RouteCtx;

// After
_ctx?: Record<string, unknown>;
```

Remove the `import type z from 'zod'` if it's only used by `RouteCtx` (check — it may also be used by `schema: z.ZodObject<any, any>`; if so, keep it).

### 2. `src/contexts/http-context.ts`

`HttpContextData` already exists (lines 9–16). Export a typed accessor:

```typescript
export function getHttpCtx(node: { _ctx?: Record<string, unknown> }): HttpContextData | undefined {
  return node._ctx as HttpContextData | undefined;
}
```

### 3. `src/contexts/openapi-context.ts`

Export a typed accessor for the OpenAPI superset shape:

```typescript
interface OpenApiCtxData extends HttpContextData {
  meta?: OpenApiMeta;
}

export function getOpenApiCtx(node: { _ctx?: Record<string, unknown> }): OpenApiCtxData | undefined {
  return node._ctx as OpenApiCtxData | undefined;
}
```

### 4. Consumer updates

| File | Current access | Replace with |
|------|---------------|--------------|
| `src/core/walk.ts` | `node._ctx?.querySchema` | `getHttpCtx(node)?.querySchema` |
| `src/core/define-routes.ts` | `node._ctx?.method`, `node._ctx?.bodySchema`, etc. | `getHttpCtx(node)?.method`, etc. |
| `src/client/fetch-client.ts` | `ctx?.method`, `ctx?.responseSchemas` | `getHttpCtx(node)?.method`, etc. |
| `src/openapi/generate-spec.ts` | `node._ctx as OpenApiContextData` | `getOpenApiCtx(node)` |

### 5. `src/index.ts`

Export `getHttpCtx` and `getOpenApiCtx` if they need to be part of the public API (e.g., for third-party plugin authors). Decide based on whether any public example or test needs them. If not, keep them package-internal.

## Dependencies

- Plan 52 must complete first (so openApiRoute's `_ctx` is always a superset of HttpContextData)

## Success Criteria

- `route-node.ts` contains no Zod field names, no HTTP method strings, no schema types beyond `z.ZodObject<any, any>` (used for the `schema` field itself)
- `RouteCtx` interface is gone
- `pnpm test && pnpm run typecheck && pnpm run lint` all pass
- No test files modified
- Public API surface unchanged from the consumer's perspective
