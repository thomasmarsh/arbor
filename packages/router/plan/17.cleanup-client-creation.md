# PLAN.17 — Replace createClient overloads with conditional return type

## Problem

`createClient` in `client/fetch-client.ts` uses three function overloads to express
the `validate: true` → `Result<...>` / default → `ResponseUnion<...>` branching.
The implementation overload is a lie (loosened types to satisfy both declared
overloads), requires `/* eslint-disable ... @typescript-eslint/no-redundant-type-constituents ... */`,
and does not compose — anyone wrapping `createClient` must reproduce the overloads.

## Change

Replace the three-overload pattern with a single generic signature that threads
`Validate extends boolean` as a type parameter and uses a conditional return type:

```typescript
export function createClient<
  Route,
  Map extends Record<string, HttpContext<any, any, any>>,
  Validate extends boolean = false,
>(
  baseUrl: string,
  router: RouterArg<Route> & { _ctxMap: Map },
  options?: { fetch?: FetchLike; validate?: Validate },
): {
  fetch<Tag extends keyof Map & string>(
    route: Extract<Route, { tag: Tag }>,
    ...args: BodyArgs<Map[Tag]>
  ): Promise<
    Validate extends true
      ? Result<ResponseUnion<Map[Tag]['response']>, z.ZodError>
      : ResponseUnion<Map[Tag]['response']>
  >;
}
```

TypeScript infers `Validate = true` when the caller passes `{ validate: true }` and
resolves the conditional at each call site. No implementation overload needed, no
suppression comments.

## Notes

- The `@typescript-eslint/no-redundant-type-constituents` disable in the file-level
  comment can be removed once this lands.
- IDE hover may show the unresolved conditional type rather than the concrete
  `Result<...>` or `ResponseUnion<...>`. Verify this is acceptable before closing.
- Existing tests cover both branches; no new tests should be needed beyond confirming
  `expectTypeOf` assertions still resolve correctly.

## Risk

Low. Pure type-level refactor; runtime behaviour is unchanged.
