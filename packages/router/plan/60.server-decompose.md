# Plan 60 — Decompose createServer god function

## Context

`server/server.ts` is 221 lines. The `createServer()` function's inner request handler
is a single sequential block handling eight distinct concerns:

1. HTTP method validation
2. Route lookup (tag extraction from URL)
3. Rate limit check
4. Handler resolution by tag
5. Request body validation against route schema
6. Request header validation
7. Request cookie parsing and validation
8. Handler execution + response header/cookie validation
9. Error handling and fallthrough

None of these steps are independently testable. The function also contains two
structural problems:

- `parseCookies` is re-implemented inline (lines ~136-142) even though it is already
  exported from elsewhere in the module.
- Response validation (lines ~182-199) only logs warnings and discards results,
  inconsistent with request validation which rejects with 422.
- Three validation steps (body, headers, cookies) share an identical
  `safeParse → 422 response` pattern but are written out separately rather than
  through a shared helper.

There are also pervasive unsafe casts: `router.children as WalkNode[]`,
`handlers as Record<string, UntypedHandler>`, `route as Record<string, unknown>` —
all of which hide real types from the compiler.

## Goal

Break `createServer()` into focused, named, independently-testable steps. Each step
should accept typed inputs and return a typed `Result` (or throw a typed error object
that the orchestrator catches), making the happy path and failure path explicit.

## Proposed Approach

### Step 1 — Extract request validation helpers

```typescript
function validateInput<T>(
  schema: z.ZodType | null,
  value: unknown,
  status: 422,
): Result<T, Response>
```

Used for body, headers, and cookies — replaces the three duplicated
`safeParse → 422 Response` blocks.

### Step 2 — Extract resolveHandler

```typescript
function resolveHandler(
  router: DefineRoutesResult,
  method: string,
  tag: string,
): Result<TypedHandler, Response>
```

Handles method validation and handler lookup. Returns typed `Result<Handler, 405|404 Response>`.
Eliminates the `handlers as Record<string, UntypedHandler>` cast.

### Step 3 — Extract validateResponse

```typescript
function validateResponse(
  route: HttpRoute,
  result: HandlerResult,
): Result<ValidatedResponse, Response>
```

Consolidates response header/cookie validation and makes the behavior consistent
with request validation (reject on failure rather than warn-and-continue).

### Step 4 — Remove inline parseCookies duplication

The inline cookie parsing block should simply call the already-exported `parseCookies`
function. One import, no duplication.

### Step 5 — Simplify createServer

After extraction, the request handler reads as a linear pipeline:

```
parse URL → resolveHandler → validateInput(body) → validateInput(headers)
→ validateInput(cookies) → execute → validateResponse → build Response
```

Each step returns `Result` and the handler short-circuits on `ok: false`.

Target: inner request handler under 40 lines.

### Typing cleanup

Remove `UntypedHandler` and replace with a properly typed `Handler<Ctx>` generic.
Remove `route as Record<string, unknown>` by threading the typed `HttpRoute` through
the execution flow.

## Change Surface

| File | Change |
|------|--------|
| `server/server.ts` | Extract helpers; simplify handler body; remove casts; fix response validation inconsistency |
| `server/server.test.ts` | Add isolated tests for `validateInput`, `resolveHandler`, `validateResponse` |

## Success Criteria

- The request handler body in `createServer` is under 40 lines.
- `validateInput`, `resolveHandler`, `validateResponse` each have dedicated unit tests.
- No `as WalkNode[]`, `as Record<string, UntypedHandler>`, or `as Record<string, unknown>`
  casts remain in `server.ts`.
- `parseCookies` is called via import, not re-implemented.
- Request validation and response validation use the same rejection behavior.
- `pnpm test && pnpm typecheck && pnpm lint` clean.
- No behavior change observable from tests.
