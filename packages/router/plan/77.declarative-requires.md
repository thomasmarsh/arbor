# Plan 77 — Declarative `requires` annotation on `httpRoute`

## Context

The current RBAC pattern requires explicit guard composition at the call site:

```typescript
withSession(withRbac(['admin'], httpRoute(GetAdmin, 'GET', 'admin', { ... })))
```

For 90% of real-world routes the question is simply "which roles can call this?" — the
full guard-composition chain adds noise that buries the answer. A `requires` annotation
puts the access policy where it belongs: next to the route declaration.

```typescript
httpRoute(GetAdmin, 'GET', 'admin', {
  requires: ['admin', 'super-user'] as const,
  response: { 200: AdminData },
})
```

This is **not** just syntactic sugar — it is a step toward the capability/permission system
outlined in Plan 80. `requires` is a typed declaration embedded in the route's `_meta`
bag; `createServer` reads it at startup and automatically applies `withSession` +
`withRbac` before the handler is invoked. The handler ctx type is narrowed to include
the session automatically, same as if the guards were applied manually.

Importantly, `requires` is **opt-in at the route level**. Routes without `requires` are
unaffected; the explicit guard-composition model remains valid in parallel.

## Goal

1. `httpRoute()` accepts an optional `requires: readonly string[]` option.
2. When present, `createServer` automatically enforces authentication + role check before
   the handler, with no extra code at the call site.
3. The handler `ctx` type reflects the session enrichment automatically — `ctx.session`
   is present and typed when `requires` is declared.

## Type design sketch

```typescript
// In HttpContextData / httpRoute options:
type HttpRouteOptions<Method, Requires extends readonly string[]> = {
  requires?: Requires;
  query?: z.ZodObject<any>;
  body?: ...;
  response: { [S in number]: z.ZodType };
};

// When Requires is present, ctx includes:
// session: { userId: string; roles: Requires[number][] }
// (same shape as withSession + withRbac produces)
```

`createServer` checks `_meta.httpCtx.requires` for each route tag at startup and wraps
the handler in the appropriate guard chain — the same runtime path as explicit composition,
just wired automatically.

## Design decisions to resolve before implementation

1. **Role type strictness**: should the `requires` type be `Requires[number]` (literal
   union from the array) or just `string[]`? Literal union enables `ctx.session.roles`
   to be narrowed to the declared set — more correct but more complex.

2. **Interaction with explicit guards**: if a route has both `requires` and is wrapped
   with `withSession`, should the automatic `requires` guard be skipped, composed, or
   an error? Simplest answer: explicit guards take precedence; `requires` is only applied
   when no manual session guard is detected. But "detecting" manual guards is fragile.
   Safer: document that using both is redundant (not an error); `requires` is preferred.

3. **Session shape**: the session type injected by `requires` should be identical to
   what `withSession` produces. Reuse the existing `SessionCtx` type.

## Change Surface

- `src/contexts/http-context.ts` — extend `HttpRouteOptions` with `requires`; store in
  `HttpContextData._meta`
- `src/server/server.ts` — `resolveHandler()` reads `requires` from the route's meta and
  wraps the handler in the guard chain before execution
- `src/contexts/http-context.test.ts` — `expectTypeOf` tests for ctx narrowing with and
  without `requires`
- `src/server/server.test.ts` — runtime tests confirming 401 and 403 responses

## Dependencies

- Plan 72 (`createTestClient`) makes runtime testing of the 401/403 paths simpler but is
  not required.
- Plan 76 (`.use()` builder) is a parallel ergonomics improvement; neither depends on
  the other.

## Success Criteria

- `httpRoute(S, 'GET', 'path', { requires: ['admin'] })` compiles; handler ctx includes
  `session` typed appropriately.
- Server returns 401 when no session is present on a `requires`-annotated route.
- Server returns 403 when session exists but lacks the required role.
- Routes without `requires` are unaffected.
- `pnpm test && pnpm typecheck` green.
