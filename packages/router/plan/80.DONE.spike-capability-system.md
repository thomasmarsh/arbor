# Plan 80 — Spike: typed capability / environment system (DIY, no Effect-TS)

## Context

The guard enrichment system (Plan 49/50) already proves one thing at the type level:
"this handler ran after authentication." `withSession` narrows `ctx` so handlers cannot
access `ctx.session` without the guard having run. This is a _capability proof_ in the
original sense — possession of the typed value IS the proof.

The natural extension: generalize this to arbitrary named capabilities (services,
resources, effect permissions). A handler that needs a database connection should declare
it; the type system should prove it was provided; and handlers should be unable to access
services they did not declare. This is what Effect-TS provides, but Effect-TS is a large,
opinionated runtime with its own fiber scheduler and error channels — adopting it changes
the character of the whole codebase.

**The hypothesis**: TypeScript's conditional types + the existing `_meta` bag + the guard
enrichment pattern may be sufficient to build a lean capability system without any
external dependency. This spike validates or refutes that hypothesis.

## Questions to answer

1. **Capability declaration**: Can `httpRoute(S, 'GET', '/path', { needs: ['db:read', 'cache'] as const })`
   narrow the handler's `ctx.services` to `{ 'db:read': DbReadService; cache: CacheService }`
   using the same typed meta mechanism as `requires` (Plan 77)?

2. **Service provision at startup**: Can `createServer(router, handlers, { services: { 'db:read': dbClient, cache: cacheClient } })`
   type-check that all routes' `needs` are satisfied by the provided service map — a
   compile error if a route needs `'db:write'` but the server wasn't given one?

3. **Isolation**: Can TypeScript _prevent_ a handler from accessing `ctx.services['db:write']`
   if the route only declared `['db:read']`? (The value would simply not be present at
   runtime, but ideally the _type_ also excludes it.)

4. **Composition with guards**: Does this conflict with or compose cleanly alongside the
   existing `withSession` / `withRbac` / `.use()` model?

5. **Effect tracking** (stretch): Is it feasible to annotate service interfaces with the
   side-effects they can produce (e.g., `DbWriteService` is tagged as `mutating`) and
   have the route type carry a proof that no `mutating` service was used on a GET route?

## Spike deliverable

A self-contained TypeScript file in `scripts/spike-capability.ts` that:

1. Defines a minimal `needs` option type and `createServer` service-provision type.
2. Demonstrates a handler that can access `ctx.services['db:read']` and cannot (type
   error) access `ctx.services['db:write']` when only `'db:read'` was declared.
3. Demonstrates a `createServer` call that fails to compile if a needed service is missing
   from the `services` map.
4. Notes where TypeScript's type system falls short (if it does) and what workarounds
   or additional machinery would be needed.

## Expected outputs

- **If feasible without new abstractions**: create Plan 81.capability-system-impl.md
  with the full implementation design.
- **If feasible with thin abstractions**: design those abstractions as a small internal
  module (no external deps); create implementation plan.
- **If Effect-TS or a comparable library is genuinely needed**: document specifically
  which TypeScript limitation requires it; create a spike for the minimal Effect-TS
  integration that only touches service injection (not fibers, not error channels).

## Success Criteria for the spike

- Spike file committed and runnable with `tsx scripts/spike-capability.ts`.
- Findings documented in `plan/NOTES.md` covering all five questions above.
- Decision recorded in this plan file: proceed / adjust / abandon, with rationale.

## Decision

**PROCEED** — all five questions answered affirmatively (see `plan/NOTES.md` for detail).
No Effect-TS required. The core mechanics (`const` generics + `Pick` + declaration merging)
are sufficient. One known limitation: `BaseCtx` cannot be inferred in `withServices()`
without a carrier argument; resolved by threading it through the `.use()` builder chain in
the full implementation.

Next step: Plan 81 (capability system implementation) — integrate `needs` into `httpRoute`,
`HandlerCtx`, and `createServer`, with the `ServiceRegistry` augmentable interface.
