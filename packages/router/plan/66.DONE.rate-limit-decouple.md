# Plan 66 — Decouple rate limiting from `executeRoute()`

## Context

`server.ts` `executeRoute()` (lines 134–175) does two fundamentally different things:

1. **Policy enforcement** (lines 142–148): rate-limit check with side-effects (store increment,
   key derivation, lazy store initialization).
2. **Request processing** (lines 149–174): handler resolution, body/header/cookie validation,
   handler execution, response validation, error mapping.

Rate limiting is structurally different from the other steps:
- It mutates external state (the rate limit store).
- It creates lazy infrastructure (`_defaultRateLimitStore ??= createMemoryStore()`).
- It uses server-level config (key resolver, store reference) rather than route-level schemas.

All other steps in `executeRoute()` are stateless schema validations or pure handler calls.

The symptom: `executeRoute()` holds two unrelated responsibilities, making it harder to test
rate limiting in isolation, reason about execution flow, or move rate limiting to a different
integration point (e.g., before parsing vs. after parsing).

## Goal

Extract the rate-limit logic from `executeRoute()` into a named helper function so the two
concerns are separate. The helper should be independently testable and `executeRoute()` should
read as a flat, ordered pipeline of delegated steps.

## Approach

Extract `applyRateLimit(policy, store, key): Promise<DispatchResult | null>` (or similar).
`executeRoute()` calls it at the top and short-circuits on a non-null return (meaning rate
limit exceeded). The lazy store initialization remains in the closure but is the only
cross-cutting reference.

Do NOT move rate limiting to a guard/middleware at this stage — that would require changing
the public API surface. The goal is internal refactoring only: same observable behavior,
better internal structure.

## Change Surface

- `src/server/server.ts` only — no public API changes.
- `src/server/server.test.ts` — no changes expected (behavior is identical).

## Dependencies

Depends on Plan 60 (server decomposition) being complete, which it is.

## Success Criteria

- `executeRoute()` delegates the rate-limit check to a named helper with a single `if` call.
- The helper is independently readable and could be unit-tested separately.
- `pnpm test && pnpm typecheck` pass.
- No observable behavior change (all existing rate-limit tests pass unmodified).
