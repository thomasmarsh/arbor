# Plan 57 — Rename Enricher → Guard and resolve Result overlap

## Context

`src/server/enrichers.ts` defines:

```typescript
type Enricher<BaseCtx, Extra> =
  (ctx: BaseCtx) => Promise<{ ok: true; ctx: BaseCtx & Extra } | { ok: false; response: Response }>;

function withEnricher(enricher, handler)   // run one guard before a handler
function composeEnrichers(first, second)   // chain two guards (2-arity only)
```

Three problems exist:

1. **Wrong name.** "Enricher" implies the function only _adds_ data. Every real use-case (`withSession`, `withRbac`, `withApiKey`) is actually a _guard_ — it either passes the request through (with extra context) or short-circuits with a response. The additive framing misrepresents the primary semantic.

2. **Duplicate discriminated union.** The return type `{ ok: true; ctx } | { ok: false; response }` is structurally identical to `Result<T, E>`'s internal `{ ok: true; value } | { ok: false; error }`, but with different field names (`ctx`/`response` vs `value`/`error`). This means:
   - The two abstractions cannot interop without adapting field names.
   - `composeEnrichers` re-implements Kleisli composition that `Result.flatMapAsync` already provides — but only for this bespoke union, not for `Result`.
   - Authors writing guards must learn a parallel set of conventions.

3. **2-arity cap on compose.** `composeEnrichers` is explicitly 2-arity. Three or more guards require ugly manual nesting. This is a consequence of fighting the compiler with a free-function accumulator instead of a builder or pipe.

## Structural analysis: Enricher IS Kleisli over Result

```
Guard<B, E> ≅ B → Promise<Result<B & E, Response>>
```

`composeEnrichers(a, b)` is `a >=> b` (Kleisli composition), which is exactly `Result.flatMapAsync` on the value axis. `withEnricher(e, handler)` is `e.flatMapAsync(ctx => handler_as_result(ctx))`.

The _reason_ they diverged is that `Result<T, E>` stores `value: T`, but guards want `ctx: BaseCtx & Extra` — a domain-meaningful name. That naming preference created a parallel type instead of an alias.

## Goal

1. Rename the concept from "Enricher" to "Guard" across all files.
2. Decide (with justification) whether to unify the return type with `Result`.
3. Optionally introduce a `pipe(a, b, c, ...)` helper to remove the 2-arity limitation.

## Options

### Option A — Rename only (recommended baseline)

Rename surface:
- `Enricher<B, E>` → `Guard<B, E>`
- `withEnricher` → `withGuard`
- `composeEnrichers` → `composeGuards`
- File: `enrichers.ts` → `guard.ts`

Keep the return union shape as-is (`{ ok: true; ctx } | { ok: false; response }`). No semantic or structural change — just a naming correction.

**Pros:** Zero risk, minimal diff, clear improvement to readability.  
**Cons:** Doesn't resolve the `Result` duplication.

### Option B — Align return type field names with Result

Change the success arm from `{ ok: true; ctx: ... }` to `{ ok: true; value: ... }`. This matches `Result`'s internal shape so authors familiar with `Result` face no impedance mismatch, and adapter code becomes trivial.

Call sites change from:
```typescript
return { ok: true, ctx: { ...ctx, session } };
```
to:
```typescript
return { ok: true, value: { ...ctx, session } };
```

`composeGuards` stays as a standalone combinator but now trivially wraps `Result`-shaped composition logic.

**Pros:** One canonical shape across the codebase; easier to write adapters.  
**Cons:** `value` is less descriptive than `ctx` in handler context. All existing guard implementations (`with-session`, `with-rbac`, `with-api-key`) need a field rename at return sites.

### Option C — Return `Result<BaseCtx & Extra, Response>` from `@arbor/common`

Replace the bespoke union with the actual `Result` class:

```typescript
type Guard<B, E> = (ctx: B) => Promise<Result<B & E, Response>>;
```

Composition collapses to `Result.flatMapAsync`. `composeGuards` becomes a thin wrapper or disappears entirely. `withGuard` becomes `result.fold(handler, identity)`.

**Pros:** Zero duplication; full `Result` method suite available in guards.  
**Cons:**  
- Adds runtime import of `@arbor/common` into every guard file.
- Call sites must use `Result.ok({ ...ctx, session })` — more verbose than `{ ok: true, ctx }`.
- Couples the hot-path guard execution to the class-based `Result`, which is heavier than a plain discriminated union.
- `flatMapAsync` chains don't accumulate the context type correctly (`B & E` widening) without extra plumbing — this is the same reason `composeEnrichers` is 2-arity.

### Option D — Fluent builder with `.pipe(a, b, c)`

Introduce a `guard(fn)` builder that wraps a guard function and exposes `.pipe(next)` for chaining:

```typescript
const authed = guard(withSession(resolveSession))
  .pipe(withRbac(['admin']))
  .pipe(withApiKey(...));

withGuard(authed, handler);
```

Each `.pipe` call composes into a single `Guard` (same type as before), so the output is still a plain `Guard` — no new runtime protocol. The builder is just a thin chainable wrapper that calls `composeGuards` internally.

**Pros:** Removes the nesting requirement for 3+ guards; ergonomic at call sites.  
**Cons:** Adds a builder class/object; TS must infer the accumulated generic correctly through each `.pipe` call — needs a spike to verify this doesn't hit instantiation limits.

## Recommendation

**Do Option A immediately** (rename). It's a mechanical change with a clear win and no risk.

**Defer Option B** (field name alignment). Useful if we ever want `Result`/`Guard` interop utilities, but not urgent.

**Do not do Option C** (full `Result` class). The coupling cost and call-site verbosity outweigh the deduplication gain. The bespoke union is fine for this use-case.

**Spike Option D** separately if the nesting ergonomics become a real pain point in usage. It's non-trivial to typecheck correctly.
