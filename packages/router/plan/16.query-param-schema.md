# PLAN.16 — Explicit query parameter schema in `httpRoute`

## Problem

Query parameters are currently implicit: `walkParse` reads every key in the route's
Zod schema shape that isn't already filled by a path segment and tries to pull it
from `URLSearchParams`. This means path-param schema and query-param schema are
merged into one object with no structural distinction.

Consequences:

- A handler can't tell at a glance which fields came from the path vs the query string.
- There is no way to declare query params on a `GET` route that has *no* path params.
- The client's `print()` serializes leftover fields as query params without any schema
  validation or coercion (e.g. a number field arrives as a string).

## Change

Add an optional `query` field to `httpRoute` options:

```typescript
httpRoute(schema, method, path, { body?, response, query? }, children?)
```

`query` is a `z.ZodObject`. Its keys are distinct from the path schema and are
validated separately.

### Runtime (server side)

In `walkParse`, when the matched node has `context.querySchema`, validate
`URLSearchParams` entries against it with `.safeParse()` instead of reading
them from the main schema shape. Merge the validated query result into the
returned object under a `query` key so handlers can destructure clearly.

### Type level

Extend `HttpContext` to carry the inferred query type:

```typescript
interface HttpContext<Method, Body, Response, Query = never> {
  method: Method;
  body: Body;
  response: Response;
  query: Query;
}
```

Update `HandlerMap` so handler arguments include `query: Ctx['query']` when
`Query` is not `never`.

### Client side

`buildUrl` already serializes non-path, non-tag keys as query params. After
this plan, the client should serialize the `query` sub-object explicitly rather
than relying on key diffing. No signature change needed for `createClient.fetch`
— the route object passed by the caller already carries query fields.

## Notes

- Keep backward compat: routes without a `query` option should behave exactly
  as today. The implicit fallback in `walkParse` (reading shape keys from
  `URLSearchParams`) stays for routes without an explicit `querySchema`.
- Start with the server-side validation and a test that confirms coercion
  (e.g. `?page=3` → `{ page: 3 }` via `z.coerce.number()`). Type-level
  changes to `HandlerMap` come after.
- The `query` key in the returned object must be stripped before Zod validates
  the main route schema, to avoid a "unrecognized key" error on strict schemas.

## Risk

Medium. Touches `walkParse`, `httpRoute`, `HttpContext`, and `HandlerMap`. Keep
changes incremental: server validation first, then type threading, then client.
