# PLAN.26 — Expand type-level test coverage

## Context

There are 200+ runtime tests but only 15 `expectTypeOf` assertions in
`route-node.test.ts`. Critical unsoundness cases are untested at the type
level: wrong tags compile, missing section params compile, the `query`
sub-object on a parsed route is unverified, and composition type safety has
no `expectTypeOf` coverage. If the phantom type machinery regresses, the
test suite would not catch it.

## Goal

Add type-level tests that cover the cases the runtime tests cannot. These
are `expectTypeOf` and `assertType` calls — they must produce compile errors
when the type contract is violated, not just runtime assertion failures.

## New Tests to Add

File: `src/core/route-node.test.ts` (extend existing file)

### 1. Tag discrimination exhaustiveness

```typescript
// Invalid tag must not compile
// @ts-expect-error
router.print({ tag: 'nonexistent' });
```

### 2. Section params required (after plan 20 lands)

```typescript
// Omitting a required section param must not compile
// @ts-expect-error
sectionRouter.print({ child: { tag: 'project', projectId: 42 } });

// Providing it must compile
sectionRouter.print(
  { child: { tag: 'project', projectId: 42 } },
  { orgId: 'acme' },
);
```

### 3. `parse()` result shape includes `query`

```typescript
const result = queryRouter.parse('/items?page=3');
if (result.ok) {
  expectTypeOf(result.value).toMatchTypeOf<{ tag: 'items'; query: { page: number } }>();
}
```

### 4. Composition type safety

```typescript
// Composed router preserves all leaf types
const composed = defineRoutes([aRouter, bRouter]);
type ComposedRoute = InferRoute<typeof composed>;
expectTypeOf<ComposedRoute>().toEqualTypeOf<ARoute | BRoute>();
```

### 5. Handler map must cover all tags

```typescript
// Missing a handler tag must not compile
// @ts-expect-error
createServer(router, { 'get-user': handler }); // 'create-user' missing
```

### 6. Body/query type narrowing in handlers

```typescript
createServer(router, {
  'create-user': (route, body) => {
    expectTypeOf(body).toEqualTypeOf<{ email: string; name: string }>();
    return Promise.resolve({ status: 201 as const, body: { id: '1' } });
  },
});
```

## Notes

- Tests 1 and 5 use `// @ts-expect-error` — they pass when TypeScript
  correctly rejects the call. These are regression guards for soundness.
- Test 2 is blocked on plan 20 landing first.
- Add in a dedicated `src/core/types.test.ts` file rather than growing
  `route-node.test.ts` further, to keep concerns separated.

## Success Criteria

- At least 20 new `expectTypeOf` / `@ts-expect-error` assertions added.
- All assertions are meaningful: each one guards a specific invariant that
  the runtime tests cannot catch.
- `npm test && npm run typecheck` clean (including the new type tests).
