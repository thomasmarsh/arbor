# Plan 58 — Architectural Smell Assessment

## Context

This document is a holistic review of the router package to surface long functions,
structural inconsistencies, ergonomic traps, and naming confusion. It serves as the
master backlog for smell-driven cleanup work and cross-references existing and new plans.

Completed context: Plans 56 (core/context decoupling) and 57 (Enricher → Guard rename)
addressed two of the highest-priority structural issues. This document captures what remains.

---

## Smell Inventory

### HIGH — fix before growing the API surface

| ID | Smell | Location | Plan |
|----|-------|----------|------|
| H1 | `createServer()` is a 221-line god function handling 8 distinct concerns: method validation, rate limiting, handler resolution, body/header/cookie validation, execution, response validation, and error handling. Individual concerns are untestable in isolation. | `server/server.ts` | Plan 60 |
| H2 | `walkParse()` is a 92-line function mixing segment matching, param accumulation, query schema resolution, Zod validation (twice), child recursion, and diagnostic collection. Six concerns in one function. | `core/walk.ts` | Plan 59 |
| H3 | `Enricher` is a misnomer for what is actually a guard/short-circuit; its discriminated union `{ ok: true; ctx } \| { ok: false; response }` duplicates `Result`'s `{ ok: true; value } \| { ok: false; error }` with different field names, creating two parallel abstractions that can't interop. `composeEnrichers` is 2-arity only. | `server/enrichers.ts` | Plan 57 (OPEN) |

### MEDIUM — structural correctness issues

| ID | Smell | Location | Plan |
|----|-------|----------|------|
| M1 | `_meta` is `Record<string, unknown>` everywhere; each consumer casts it to a different local shape (`HttpContextData`, `OpenApiCtxData`, `{ querySchema?: ZodObject }`). No central type contract. Easy to add a field in one place and forget it in another. | `core/route-node.ts`, all context files | Plan 61 |
| M2 | Three parallel response shapes serve overlapping roles: `DispatchResult` in `server.ts`, `HttpResponseUnion` in `http-context.ts`, and raw handler return objects. No single canonical `HttpResponse` type. | `server/server.ts`, `contexts/http-context.ts` | Plan 62 |
| M3 | `httpRoute()` descriptor unpacking uses fragile duck-typing: `'safeParse' in d` to detect a ZodType, `'body' in d` to detect a descriptor object. No type guard or discriminant. Silently misbehaves if descriptor is an object with no `body` key. | `contexts/http-context.ts:131-156` | Plan 62 |
| M4 | `walkParse`, `walkCollect`, `walkPrint` each repeat the same pattern: null-schema check, tag extraction, `Object.assign` recursion — DRY violation across three functions in one file. `buildMethodMap` and `buildResponseSchemaMap` in `fetch-client.ts` are two more near-identical walkers. | `core/walk.ts`, `client/fetch-client.ts` | Plan 59 |
| M5 | Unsafe type casts throughout: `router.children as WalkNode[]`, `handlers as Record<string, UntypedHandler>`, `route as Record<string, unknown>`. All hide real types and make refactoring invisible to the compiler. | `server/server.ts`, `core/walk.ts` | Plan 60 |

### LOW-MEDIUM — ergonomic friction for API users

| ID | Smell | Location | Plan |
|----|-------|----------|------|
| E1 | `status: 200 as const` required on every handler return for inference to work. Appears 30+ times across tests and examples. | All handler definitions | Plan 62 |
| E2 | Two response definition syntaxes: bare `{ 200: Schema }` vs descriptor `{ 200: { body: Schema, headers: H } }`. Same-looking, different behavior, no immediate error when you use the wrong form. | `http-context.ts:74-89`, examples | Plan 62 |
| E3 | Every enricher/guard requires explicit context threading in its type: `Enricher<BaseCtx & PriorExtra, NewExtra>`. No inference chain — authors must manually track accumulated context. | `server/enrichers.ts`, `examples/enrichers.ts` | Plan 57 (OPEN) |
| E4 | Query params are split across two places in the client: path/matrix params go on the route tag object, query params go in the options bag. Inconsistent data flow model. | `client/fetch-client.ts`, `examples/query-params.ts` | Plan 48 area |
| E5 | `TypedClient` requires manual type extraction at the call site: `TypedClient<Router['_type'], Router['_ctxMap']>`. No shorthand. | `examples/typed-client.ts:50-54` | Plan 48 area |

### LOW — clean-up / future-proofing

| ID | Smell | Location | Plan |
|----|-------|----------|------|
| L1 | Single-letter type params (`S`, `C`, `R`) carry non-obvious semantics. | `core/route-node.ts`, `contexts/http-context.ts` | cosmetic |
| L2 | OpenAPI segment-to-param mapping uses two parallel `switch`/`case` functions with no shared structure. Hard to extend for new segment kinds. | `openapi/generate-spec.ts:14-33` | low priority |
| L3 | `FlattenChildrenImpl` depth-counter (15-level tuple-counting to avoid TS2589) is a clever but brittle workaround. Magic number buried in the type system. | `core/route-node.ts` | Plan 47 (complete; no regression yet) |

## Recommended Execution Order

| Wave | Plans | Rationale |
|------|-------|-----------|
| 1 | Plan 57 | Rename only — zero risk, unblocks clear thinking about guards |
| 2 | Plan 59 | Walk decomposition — foundational; server and client both depend on walk |

## Goal

This document is a reference backlog, not an executable plan. Each referenced plan number
is the unit of work to execute.

## Change Surface

No code changes in this plan — it is documentation only.

## Dependencies

Plans 59–62 depend on Plan 57 completing first (cleaner guard semantics aid reasoning
about server decomposition).

## Success Criteria

- Plans 57, 59, 60, 61, 62 all land with green verification chains.
- No `as any` or unexplained casts remain in `server.ts` or `walk.ts`.
- The `_meta` cast pattern (`node._meta as HttpContextData`) is eliminated.
- `status: N as const` boilerplate is eliminated from all handler definitions.
- `pnpm test && pnpm typecheck && pnpm lint && pnpm run examples` clean.
