# Plan 85 — Fix `HttpContext` arity inconsistency; shared router contract type

## Context

Plan 72 introduced `createTestClient`, which had to use two `as unknown as` casts to
bridge `createClient` and `createServer`. The casts are not accidents — they reflect a real
inconsistency in how the API surface constrains `HttpContext`:

| Site | Constraint used | Cookies param |
|---|---|---|
| `createServer` | `HttpContext<HttpMethod, unknown, Record<number, unknown>, unknown, unknown, unknown>` | `unknown` (6th param) |
| `createClient` / `RouterArg` | `HttpContext<any, any, any, any, any>` | missing → defaults to `never` |
| `createTestClient` (Plan 72) | mixed — cast bridging the gap | — |

`Cookies` was the 6th parameter added to `HttpContext`. `createClient` and its local
`RouterArg` interface were never updated to include it, so their structural constraint
effectively claims "this router has no cookie-aware routes" (`never`). TypeScript therefore
rejects a real router — which can have `Cookies = something` — as structurally incompatible
with `RouterArg._ctxMap`, forcing a cast.

A secondary problem: each of the three consumer files defines its own local structural
requirement for the router argument (`RouterArg` in `fetch-client.ts`, `RouterShape` in
`test-client.ts`, an anonymous inline object in `server.ts`). There is no single source of
truth for "the shape a router must have."

## Goal

1. Fix the `HttpContext` arity bug in `createClient` (5 `any`s → 6 `any`s).
2. Extract a single shared `RouterContract<Route, Map>` interface that all three consumer
   sites use, living in `src/core/` (or `src/contexts/`) and exported from `src/index.ts`.
3. Delete the local `RouterArg` in `fetch-client.ts` and `RouterShape` in `test-client.ts`
   in favour of `RouterContract`.
4. Remove every `as unknown as` cast from `createTestClient`.
5. `pnpm test && pnpm typecheck && pnpm lint` green throughout.

## Root-cause diagnosis

```ts
// http-context.ts — six params, Cookies is param 6:
interface HttpContext<Method, Body, Response, Query, Headers, Cookies> { … }

// fetch-client.ts — five anys, Cookies silently becomes `never`:
interface RouterArg<Route> {
  _ctxMap: Record<string, HttpContext<any, any, any, any, any>>;  // ← BUG
}
export function createClient<
  …,
  Map extends Record<string, HttpContext<any, any, any, any, any>>,  // ← BUG
>( router: RouterArg<Route> & { _ctxMap: Map }, … )
```

Because `RouterArg._ctxMap` hard-codes the 5-param form, TypeScript widens `Map` to a
type where `Cookies = never`, which conflicts with the concrete router emitted by
`defineRoutes` (where `Cookies` can be `string` or a parsed object). The cast in
`test-client.ts` is a symptom; the fix belongs in `fetch-client.ts`.

## Proposed fix

### Step 1 — Introduce `RouterContract`

In `src/core/router-contract.ts` (new file):

```ts
// The structural contract every router object must satisfy.
// Used by createServer, createClient, and createTestClient.
import type { Result } from '@arbor/common';
import type { HttpContext, HttpMethod } from '../contexts/http-context.js';
import type { RouteNode } from './route-node.js';

export type AnyCtxMap = Record<
  string,
  HttpContext<any, any, any, any, any, any>  // all 6 params, keeps assignability open
>;

export interface RouterContract<
  Route extends { tag: string },
  Map extends AnyCtxMap,
> {
  _type: Route;
  _ctxMap: Map;
  children: RouteNode<unknown, any, any, any, any>[];
  parse(url: URL): Result<Route, string>;
  print(route: Route): string;
}
```

`AnyCtxMap` is the shared alias replacing every repeated
`Record<string, HttpContext<any,any,any,any,any>>` (5-param) occurrence. Using 6 `any`s is
correct: it keeps the bound as wide as possible so any concrete `Map` is assignable.

### Step 2 — Update `createClient`

Replace the local `RouterArg` with `RouterContract`:

```ts
import type { AnyCtxMap, RouterContract } from '../core/router-contract.js';

export function createClient<
  Route extends { tag: string },
  Map extends AnyCtxMap,
>(
  baseUrl: string,
  router: RouterContract<Route, Map>,
  options?: { fetch?: FetchLike; validate?: Validate },
): TypedClient<Route, Map, Validate>
```

`RouterArg` is deleted. `RequestOpts` switches its constraint to
`HttpContext<any, any, any, any, any, any>`.

### Step 3 — Update `createServer`

The anonymous inline type in `createServer` becomes:

```ts
import type { AnyCtxMap, RouterContract } from '../core/router-contract.js';

export function createServer<
  Route extends { tag: string },
  Map extends AnyCtxMap,
>(
  router: RouterContract<Route, Map>,
  handlers: HandlerMap<Map, Route>,
  options?: { … },
)
```

`HandlerCtx` and `HandlerMap` keep their own constraints; only the router-argument type
changes.

### Step 4 — Simplify `createTestClient`

With both `createClient` and `createServer` accepting `RouterContract<Route, Map>`, the
function body becomes cast-free:

```ts
export function createTestClient<
  Route extends { tag: string },
  Map extends AnyCtxMap,
>(
  router: RouterContract<Route, Map>,
  handlers: HandlerMap<Map, Route>,
  options?: { baseUrl?: string; serverOptions?: CreateServerOptions },
): TypedClient<Route, Map> {
  const server = createServer(router, handlers, options?.serverOptions);
  const baseUrl = options?.baseUrl ?? 'http://localhost';

  const mockFetch: FetchLike = async (url, init) => {
    const bodyText = typeof init.body === 'string' ? init.body : undefined;
    const body = bodyText ? JSON.parse(bodyText) as unknown : undefined;
    const headers: Record<string, string> = {};
    if (init.headers) for (const [k, v] of Object.entries(init.headers)) headers[k] = v;
    const result = await server.handle(new URL(url), init.method, body, headers);
    return { status: result.status, json: () => Promise.resolve(result.body) };
  };

  return createClient(baseUrl, router, { fetch: mockFetch });
}
```

No casts. `RouterShape` (local to `test-client.ts`) is deleted.

### Step 5 — Export from `src/index.ts`

```ts
export type { AnyCtxMap, RouterContract } from './core/router-contract.js';
```

Consumers that wrap the router (middleware layers, custom servers) can use `RouterContract`
as the canonical type rather than duplicating the structural constraint.
