# PLAN.30 — Typed request headers as contract

## Context

Currently the only way to consume request headers is raw extraction inside a
handler (`ctx.req.headers.get('x-api-key')`). This bypasses validation,
skips OpenAPI documentation, and leaks into the typed client — callers have
no compile-time guidance about which headers are required.

ts-rest solves this by making headers a declared schema on the route contract,
on the same level as `body` and `query`. A route that requires
`authorization: string` and `x-tenant-id: string` enforces those at parse
time, documents them in the OpenAPI spec, and tells the client SDK what to
send.

## Goal

Add an optional `headers` Zod schema to the route contract. The server
validates incoming headers before calling the handler. Validated headers flow
into the `ctx` argument alongside `params`, `body`, and `query`.

```typescript
httpRoute({
  method: 'GET',
  path: 'reports/',
  headers: z.object({
    'x-tenant-id': z.string().uuid(),
    'accept-language': z.string().optional(),
  }),
  schema,
  responses: { 200: { body: ReportListSchema } },
});

// Handler ctx gains a typed headers field
handler: async ({ params, body, query, headers, req }) => {
  const tenantId = headers['x-tenant-id']; // string (not string | null)
  ...
},
```

Missing required headers → 400 with a Zod validation error, before the
handler is called.

## Change Surface

1. **`http-context.ts`**: Add optional `headers` Zod schema field to the
   route context shape.
2. **`server.ts`**: Before calling the handler, extract the relevant headers
   from `req.headers`, run `headerSchema.safeParse`, return 400 on failure,
   and merge validated headers into `ctx`.
3. **`ServerHandlers` mapped type**: If `headers` schema is declared, `ctx`
   includes a typed `headers` field; otherwise it is absent/`never`.
4. **Plan 23 addendum**: Declared request headers appear in the OpenAPI spec
   under `parameters` (in: header).
5. Tests: missing required header → 400; valid header → available typed in ctx.

## Dependencies

- Plan 22 (DI context) — `ctx` shape must be settled first.
- Plan 25 (validation error boundary) — same 400-response pattern; align the
  error format.

## Success Criteria

- Omitting a required header declared in the contract returns 400.
- Validated headers are available typed in the handler ctx.
- `npm test && npm run typecheck` clean.
