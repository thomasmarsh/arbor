# PLAN.12 — Validate client responses

## Problem

`createClient` trusts the server's JSON response completely. The
`as ResponseUnion<...>` cast at `client.ts:72` means the runtime type could be
anything. For a library that emphasizes parse-don't-validate, this is a gap.

The Zod response schemas are already available on the route nodes — they're
just not used on the client side.

## Change

1. Add an optional `validate?: boolean` flag to `createClient` options.
2. When enabled, look up the response schema for the matched status code and
   `safeParse` the response body.
3. Return a `Result` (or throw) on validation failure so the caller knows the
   server returned something unexpected.
4. Build the response-schema lookup map alongside the existing method map in
   `buildMethodMap` (or a parallel function).

## Notes

- Default to `validate: false` to avoid breaking existing consumers or adding
  unwanted overhead.
- Consider whether validation failures should return a `Result.failure` or
  throw — throwing is simpler but breaks the pattern used elsewhere.

## Risk

Low. Opt-in, additive. The schema data is already present.
