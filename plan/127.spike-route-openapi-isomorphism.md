# PLAN.127 (Spike) — Servant-Style Route/OpenAPI Isomorphism

## Context

Haskell's Servant library encodes the API as a type — the spec IS the type. `@arbor/router`
is philosophically similar: the route tree is the single source of truth for URL structure,
HTTP methods, response unions, and OpenAPI output. But this isomorphism is currently only
approximate. `generateSpec()` traverses the route tree and produces OpenAPI, but there is
no formal guarantee that the OpenAPI spec matches the TypeScript types. A developer can:

- Add a new status code to an `httpRoute` response union and forget to regenerate the spec.
- Have a `zod` schema refinement (`.refine()`) that the JSON Schema conversion silently drops.
- Export a TypeScript type from `TypedClient` that does not match the OpenAPI `oneOf`.

Servant-style isomorphism would mean: type-level changes in the route tree are mechanically
reflected in the OpenAPI output with no human intervention, and divergence is a compile-time
error — not a CI check or a code review catch.

## Questions to Answer

**Q1 — Where does the current isomorphism hold?**

Walk through the conversion path for each route component:

| Component | TypeScript source | OpenAPI target | Lossless? |
| --- | --- | --- | --- |
| Path params (`:id`) | `string` | `{in: "path", schema: {type: "string"}}` | Yes |
| Numeric params (`#id`) | `number` | `{in: "path", schema: {type: "integer"}}` | Yes |
| Query schema | `z.ZodObject<Q>` | `parameters[in: "query"]` | Partially (refinements lost) |
| Request body | `z.ZodObject<B>` | `requestBody.content["application/json"].schema` | Partially |
| Response union | `HttpResponseUnion<Res>` | `responses[N]` per status code | Yes for status codes |
| Response body (per code) | `z.ZodType<Body>` | JSON Schema | Partially (discriminated unions → `oneOf`) |
| Optional fields | `z.optional()` | `required: []` omission | Yes |
| Zod refinements (`.refine()`) | TypeScript type-level invisible | Lost | **No** |

**Q2 — Can the isomorphism be enforced at the type level?**

Option A: "Spec-first types" — derive TypeScript response types FROM the generated OpenAPI
JSON Schema rather than from Zod. The route definition embeds the OpenAPI schema directly;
Zod is compiled from it for validation. Type-level change ≡ OpenAPI change by construction.

Option B: "Round-trip test" — serialize the TypeScript type to JSON Schema and compare with
the generated OpenAPI. A type-level utility `AssertSpecMatch<Router>` causes a compile error
if any route's OpenAPI output does not match its TypeScript type. Requires a type-level JSON
Schema representation.

Option C: "Tighter Zod → JSON Schema" — improve the `zod-to-json-schema` conversion to
preserve discriminated unions as `oneOf` with a `discriminator` property, preserve refinements
as descriptions or custom keywords, and make `.nullable()` explicit. This narrows the gap
without full isomorphism.

**Q3 — Client code invalidation**

If a response body type changes (new field added, required field removed), how does a
TypeScript client detect the divergence? Currently: only if they re-run `generateSpec()` and
regenerate their client. A formal isomorphism would mean: the TypeScript type already IS the
client type — `TypedClient<Router>` is always in sync because it derives from the same Zod
schemas. The remaining gap is non-TypeScript clients (Go, Python) generated from OpenAPI.

Could `routerDiff` (plan 129) serve as the invalidation mechanism for non-TypeScript clients?
If the OpenAPI output changes, CI catches it and re-generates downstream clients.

**Q4 — Servant's `HasServer` / `HasClient` type class analogy**

Servant's key insight: a single type `API = "users" :> Get '[JSON] [User]` has BOTH a server
instance (implement the handler) AND a client instance (call the endpoint) derived from it.
Arbor already does this: `RouterContract<Route, Map>` is used by both `createServer` and
`createClient`. The remaining step is extending the same derivation to OpenAPI — making
`generateSpec(router)` provably derive from `RouterContract<Route, Map>` in a way that
fails to compile if the derivation is incomplete.

Is this achievable in TypeScript? Servant's approach relies on type classes; TypeScript uses
structural typing + conditional types. Can a `SpecDerivation<RouterContract<R, M>>` type
be constructed that constrains `generateSpec` to return a spec that exactly mirrors `M`?
