# PLAN.169 — Segment Grammar Specification

## Context

The current segment system grew incrementally: `:str` and `#num` from different
traditions, `?` suffix for optional, `*` for wildcard, and `+name` proposed in plan 168
for arrays. There is no unifying grammar — sigils are borrowed from Express, glob, and
regex without a shared principle.

This plan replaces that ad-hoc set with a consistent, regular grammar before the system
grows further. Plan 168 (`+name` array sigil) is superseded by this spec; its
implementation should follow the grammar defined here.

## Goal

Define the canonical grammar for arbor route path segments. The grammar becomes the
specification that all implementation plans reference. No code changes in this plan.

## Change Surface & T-shirt Sizing

**XS** — specification document only.

Files:

## Dependencies

- Plan 166 (trailing slash normalization) — independent; still valid.
- Plan 167 (RFC 3986 matrix params) — verdict unchanged; close the TODO as directed.
- Plan 168 (array params) — superseded; revise to use `:name[]` before implementing.

## Success Criteria

- Grammar is unambiguous (no two valid inputs parse to different meanings).
- Every segment kind maps to exactly one TypeScript type.
- Every segment kind has an unambiguous OpenAPI representation.
- No sigil collides with URL-reserved characters in ways that cause reader confusion.

## Grammar

### BNF

```
path      ::= segment ('/' segment)* '/'?

segment   ::= literal
            | param
            | rest

literal   ::= char+
char      ::= any char except '/' and '\', or '\' followed by any char

param     ::= ':' name qualifier?
qualifier ::= type optional?
            | array optional?
            | optional

type      ::= '<' scalar '>'
scalar    ::= 'int'
            | 'float'
            | value ('|' value)+    -- enum: at least two values
value     ::= [a-zA-Z0-9_-]+

array     ::= '[' sep? ']'
sep       ::= any single non-']' printable character   -- default: ','

optional  ::= '?'

rest      ::= '*' name?
name      ::= [a-zA-Z_][a-zA-Z0-9_]*
```

Trailing slashes in paths are normalized (stripped) by the parser — `users/` and `users`
are identical. See plan 166.

### Qualifier ordering rule

When present, qualifiers always appear in this order: **type/array → optional**. No other
ordering is valid.

## Segment Reference

| Syntax | Kind | TypeScript type | Required? |
|---|---|---|---|
| `users` | `lit` | — | — |
| `:name` | `str` | `string` | yes |
| `:name?` | `opt-str` | `string \| undefined` | no |
| `:name<int>` | `int` | `number` | yes |
| `:name<int>?` | `opt-int` | `number \| undefined` | no |
| `:name<float>` | `float` | `number` | yes |
| `:name<float>?` | `opt-float` | `number \| undefined` | no |
| `:name<a\|b\|c>` | `enum` | `'a' \| 'b' \| 'c'` | yes |
| `:name<a\|b\|c>?` | `opt-enum` | `'a' \| 'b' \| 'c' \| undefined` | no |
| `:name[]` | `array` | `string[]` | yes |
| `:name[;]` | `array` | `string[]` | yes |
| `:name[]?` | `opt-array` | `string[] \| undefined` | no |
| `*name` | `rest` | `string` | — |
| `*` | `rest` | — (unnamed) | — |

### Notes

**Default array separator** is `,` (comma). Specifying `[,]` is valid but redundant.
A single-value segment (no commas) produces a one-element array. An empty segment
produces an empty array.

**`rest` is a structural category**, not a type category. It captures all remaining URL
segments joined with `/`, always as `string`. It cannot be optional (it already matches
zero remaining segments). It cannot carry a type annotation. It must appear last in a
path, and only one `rest` segment is permitted per route path.

**Enum requires at least two values** — a single value is a literal segment. The values
must be non-empty alphanumeric/hyphen/underscore strings; they cannot contain `|`, `>`,
or `/`.

**`float`** matches any string parseable by `parseFloat` that round-trips (`String(parseFloat(s)) === s`). Integers are valid floats. `int` uses `parseInt` with strict
integer validation (`/^-?\d+$/`).

## Routing Behaviour (Disambiguation)

When multiple sibling routes could structurally match the same URL segment, the
resolution order is:

1. `lit` — exact literal match (highest priority)
2. `int` / `float` — numeric match
3. `enum` — constrained string match
4. `str` — any string (lowest priority among single-segment kinds)
5. `array` — single segment containing separator character(s)
6. `rest` — matches remaining (lowest overall priority)

This ordering is enforced at `indexNodes` / `walkParseIndexed` time, not at definition
time. A `lit` route always wins over `:name` for the same path, regardless of definition
order.

---

## Escaping

A backslash before any special character produces the literal character:

| Escape | Produces |
|---|---|
| `\:` | literal `:` in a segment |
| `\*` | literal `*` in a segment |
| `\<` | literal `<` in a segment |
| `\[` | literal `[` in a segment |
| `\\` | literal `\` |

In practice, literal `:`, `*`, `<`, and `[` in URL path segments are rare. The escape
mechanism exists for completeness; the parser should handle it but documentation need
not emphasise it.

## TypeScript Inference

`ExtractPathParams<Path>` must be updated to recognise the new grammar. It currently
only extracts param names (as a string union). The north star is to extract names **and
types** so that `section(':orgId<int>/', [...])` contributes `{ orgId: number }` to the
inherited params, not just `'orgId'`.

This is a two-phase change:

**Phase 1 (name extraction — unblocks plan 166 and all migration work):**

```typescript
// Strip qualifier from a raw param token after the ':' sigil.
// ':id<int>?'  → 'id'
// ':ids[]?'    → 'ids'
// ':id?'       → 'id'
type StripQualifier<S extends string> =
  S extends `${infer N}<${string}>?` ? N :
  S extends `${infer N}<${string}>`  ? N :
  S extends `${infer N}[${string}]?` ? N :
  S extends `${infer N}[${string}]`  ? N :
  S extends `${infer N}?`            ? N :
  S;

// Handles both ':name/' (with trailing slash) and ':name' (without).
export type ExtractPathParams<Path extends string> =
  Path extends `${string}:${infer Raw}/${infer Rest}`
    ? StripQualifier<Raw> | ExtractPathParams<Rest>
    : Path extends `${string}:${infer Raw}`
      ? StripQualifier<Raw>
      : Path extends `${string}*${infer Name}`
        ? Name extends '' ? never : Name
        : never;
```

**Phase 2 (typed extraction — assigns TypeScript types to each param name):**

Deferred. Requires `ExtractPathParams` to return a map `{ name: type }` rather than a
string union. This is a breaking change to `SectionParams` in `RouteNode` and all
downstream types. Scope separately once the name-only phase is stable.

## OpenAPI Mapping

| Kind | OpenAPI parameter shape |
|---|---|
| `lit` | path component only (not a parameter) |
| `str` | `{ in: 'path', required: true, schema: { type: 'string' } }` |
| `opt-str` | `{ in: 'path', required: false, schema: { type: 'string' } }` |
| `int` | `{ in: 'path', required: true, schema: { type: 'integer' } }` |
| `opt-int` | `{ in: 'path', required: false, schema: { type: 'integer' } }` |
| `float` | `{ in: 'path', required: true, schema: { type: 'number' } }` |
| `opt-float` | `{ in: 'path', required: false, schema: { type: 'number' } }` |
| `enum` | `{ in: 'path', required: true, schema: { enum: ['a','b','c'] } }` |
| `opt-enum` | `{ in: 'path', required: false, schema: { enum: ['a','b','c'] } }` |
| `array` | `{ in: 'path', required: true, style: 'simple', explode: false, schema: { type: 'array', items: { type: 'string' } } }` |
| `opt-array` | same with `required: false` |
| `rest` | `{ in: 'path', required: false, schema: { type: 'string' } }` |

`segmentToOpenApi` (path template rendering) is unchanged — all param kinds render as
`{name}` in the OpenAPI path string.

## Migration from Current Syntax

| Current | New | Notes |
|---|---|---|
| `#id` | `:id<int>` | Removes `#` fragment-anchor collision |
| `#id?` | `:id<int>?` | Consistent optional form |
| `:id` | `:id` | Unchanged |
| `:id?` | `:id?` | Unchanged |
| `*rest` | `*rest` | Unchanged |
| `+ids` (plan 168) | `:ids[]` | Plan 168 revised before implementation |

**Breaking change scope**: every call to `route()` or `section()` using `#` sigils must
be updated. This touches `src/test-utils/fixtures.ts`, all test files using `#`, and
all examples. The change is mechanical and grep-able: `rg '#[a-zA-Z]'` in route path
strings finds every instance.

The migration should be done as a single commit after the parser supports both syntaxes
in a transition period, or in one atomic pass after the new parser lands.

## Implementation Plan (downstream)

This spec is the gate for the following implementation tasks (to be planned separately):

1. **Parser update** — extend `parseSegments` to handle `<type>`, `[]`, and `\` escape.
   New `Segment` kinds: `int`, `float`, `enum`, `array`, `opt-int`, `opt-float`,
   `opt-enum`, `opt-array`. Retire `num`, `opt-num` (replaced by `int`, `opt-int`).
   Size: **S**.

2. **`matchSegments` update** — add matching logic for new kinds. Integer matching already
   exists (transplant from `num`); float, enum, and array are new. Size: **S**.

3. **`ExtractPathParams` Phase 1** — name extraction update (see above). Size: **XS**.

4. **`generate-spec.ts` update** — add OpenAPI mappings for new kinds (see table above).
   Size: **XS**.

5. **Migration** — update `#sigil` call sites in fixtures, tests, examples. Size: **XS**
   (mechanical grep-and-replace).

6. **Enum routing** — the disambiguation ordering at `indexNodes` for enum vs str siblings.
   Size: **S** (requires priority-aware segment indexing).

7. **`ExtractPathParams` Phase 2** — typed inference map. Size: **M** (deferred).

These can be sequenced or parallelized after the parser lands (task 1 gates tasks 2–6).
