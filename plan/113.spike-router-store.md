# PLAN.113 (Spike) — Router as TCA `Store`

## Context

`packages/common` ships a TCA-inspired architecture: `Store<S, A, R>` holds valtio-proxied
state; `Reducer<S, A, R>` mutates draft state and returns `Effect<A> | null`; the
`Environment` `R` carries all side-effect capabilities as typed `Effect` values. The `ui`
package already uses this pattern for auth and counter state.

The browser navigation plans (94–112) describe a hooks-based integration. This spike
validates whether the full navigation lifecycle can be expressed as a `Store` instead —
making navigation state an opt-in TCA feature for teams already using the pattern.

## Questions to Answer

**Q1 — State machine fit**: Can `RouterState<Route>` cleanly cover all navigation phases?

```typescript
type RouterState<Route> =
  | { tag: 'idle'; route: Result<Route, string> }
  | { tag: 'navigating'; from: Result<Route, string>; to: Route }
  | { tag: 'committed'; route: Route }
  | { tag: 'error'; route: Route; error: unknown };
```

Does this union handle back/forward (`popstate`), initial load, loader failure, and
replace navigation without requiring extra flags?

**Q2 — Streaming effects**: `Effect<A>` already supports calling `send` multiple times
via `Effect.of((send) => { /* register listener */ })` — the auth `openReauthPopup` uses
this. Can `popstate` subscription be expressed as a non-terminating `Effect<RouterAction>`?

```typescript
subscribePopstate: Effect.of<RouterAction>((send) => {
  window.addEventListener('popstate', () => send({ tag: 'popstate', url: window.location.href }));
}),
```

Does the fiber fork in `unsafeRun` keep listening after the effect-ts sync wrapper returns?
Verify: the `Eff.sync(() => runner(send))` approach fires the registration synchronously
and the listener then fires asynchronously — confirm this works end-to-end.

**Q3 — Cleanup / teardown**: `Effect` has no `dispose` or `cancel` API. For global
singletons (popstate, singleton routerStore) this is fine. For component-scoped SSE/WS
effects, the current `Effect<A>` class cannot be stopped once forked. Is this a blocker?
Options:
  - Accept the limitation for the TCA path (effects are long-lived by design).
  - Add `Effect.stream<A>(setup: (send) => () => void): Effect<A>` where setup returns a
    teardown; teardown is called when the fiber is interrupted. Spike whether effect-ts
    fiber interruption wires to the teardown callback.

**Q4 — Loaders**: Can loader execution be expressed as a single `Effect<RouterAction>`?

```typescript
runLoaders: (route: Route) => Effect.tryPromise(
  () => Promise.all([fetchUser(route.id)]),
  (data) => ({ tag: 'loader-resolved', data }),
  (err) => ({ tag: 'loader-failed', error: err }),
)
```

Does this compose cleanly with the router reducer's `navigating` branch?

**Q5 — Singleton vs. per-component store**: Should `routerStore` be a module-level
singleton (like `authStore`) or created per-`RouterProvider` instance? Navigation state
is global; a singleton is natural. Does valtio subscription work correctly from both
inside and outside React?
