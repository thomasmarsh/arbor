# Plan 150 — Unified API Routing via @arbor/router

## Context

After plan 142, `packages/api/src/index.ts` has two routing styles in parallel:
- Raw Hono (`app.route('/api/hello', hello)`, `app.route('/api/users', users)`,
  `app.get('/healthz', ...)`) — dispatch is Hono-internal, types are untracked
- Arbor router (`app.all('/api/ledger/*', ledgerServer.handleRequest)`) — typed,
  schema-validated, introspectable

The goal is one routing paradigm: every route defined in `@arbor/router`, Hono reduced
to a single `handleRequest` adapter line.

## Goal

All routes in `packages/api` go through `createServer` + `server.handleRequest(c.req.raw)`.
The `hello` and `users` Hono sub-apps are deleted. `index.ts` has no `app.route` or
`app.get` calls beyond the single catch-all and optional health check.

## Design Notes

**DB env threading**: The users server closes over `ApiEnv` at construction time rather
than reading it from a Hono variable:

```typescript
// users.router.ts
export function createUsersServer(env: ApiEnv) {
  return createServer(usersRouter, {
    'users-list': async (_ctx) => {
      const result = await env.db.users.findAll();
      return result.fold(
        (users) => respond(200, users),
        () => respond(500, { error: 'internal' }),
      );
    },
    'users-get': async (ctx) => {
      const result = await env.db.users.findById(ctx.params.id);
      return result.fold(
        (user) => respond(200, user),
        (err) => (err === 'not_found' ? respond(404, { error: 'not_found' }) : respond(500, { error: 'internal' })),
      );
    },
    'users-create': async (ctx) => {
      const result = await env.db.users.create(ctx.body.email);
      return result.fold(
        (user) => respond(201, user),
        () => respond(500, { error: 'internal' }),
      );
    },
  });
}
```

**Request headers in hello**: `ctx.headers['x-arbor-sub']` replaces `c.req.header(...)`.

**Health check**: Either model as a proper arbor route (`tag: 'healthz'`) or keep as the
one remaining Hono `app.get('/healthz', ...)` — a trivial liveness probe with no type
value is acceptable to leave as-is. Decide at implementation time.

**Composition**: `index.ts` creates all servers once, then a single Hono handler
dispatches by calling each server in turn (first match wins), or builds a combined server
if `@arbor/router` supports merging. If merging is not yet supported, `handleRequest` +
a 404 fallback chain is acceptable.
