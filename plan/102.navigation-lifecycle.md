# PLAN.102 — Navigation Lifecycle (Pending / Committed / Error)

## Context

After plan 101 adds loaders, the navigation lifecycle has three states:
`pending` (loaders running), `committed` (loaders done, new route ready to render),
and `error` (a loader threw). The React layer needs to expose these states so
the application can render skeletons, transitions, and error boundaries without
application code managing loading flags.

## Goal

1. Extend `useRoute()` return type with a `status` discriminant:
   ```typescript
   | { status: 'committed'; route: Route; data: LoaderData }
   | { status: 'pending';   route: Route; data: LoaderData } // previous route still active
   | { status: 'error';     route: Route; error: unknown }
   ```
2. `RouterProvider` uses `React.startTransition` so the committed view stays visible
   during pending state (no layout flash).
3. `<NavigationProgress>` component — renders a thin progress bar or spinner when
   `status === 'pending'`. Styled with CSS custom properties; no hardcoded styles.
4. Route-level `<ErrorBoundary>` integration: loader errors are re-thrown as React
   render errors so standard `ErrorBoundary` components catch them naturally.

## Change Surface

- Update: `packages/router-react/src/router-context.tsx` — status in context value
- Update: `packages/router-react/src/use-route.ts` — expose status
- New: `packages/router-react/src/navigation-progress.tsx`
- New tests for each lifecycle transition

## Dependencies

- Plan 101 (data loaders implementation).

## Success Criteria

- `status === 'pending'` while loaders are running; previous route's data still accessible.
- `status === 'committed'` when loaders resolve; new route data accessible.
- `status === 'error'` when any loader throws; error accessible without crashing.
- `<NavigationProgress>` renders during pending; hides on committed/error.
- `React.startTransition` prevents stale UI flash during transition.
