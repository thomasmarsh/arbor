# PLAN.120 — `RouterAdapter` Interface + `StaticRouter`

## Context

Plan 119 (spike) established that `RouterProvider` (plan 97) must be typed against a
`RouterAdapter<Route>` interface, not a concrete `BrowserRouter` class, so that both
`BrowserRouter` (client) and `StaticRouter` (server) plug in identically.

This plan is an **early foundational plan** — it must be done before or alongside plan 97,
not after. The `RouterAdapter` interface becomes the shared contract that all navigation
contexts implement.

## Goal

1. Define `RouterAdapter<Route>` in `packages/router-browser`:

   ```typescript
   export interface RouterAdapter<Route> {
     getRoute(): Result<Route, string>;
     navigate(route: Route): void;
     replace(route: Route): void;
     subscribe(fn: (route: Result<Route, string>) => void): () => void;
   }
   ```

   `navigate`, `replace`, and `subscribe` are REQUIRED on `RouterAdapter`. `StaticRouter`
   provides no-op implementations — on the server there is no navigation. This avoids
   optional-method complexity in consumer code.

2. Update `BrowserRouter` (plan 95) to explicitly implement `RouterAdapter<Route>`.

3. Implement `StaticRouter<Route>`:

   ```typescript
   export class StaticRouter<Route extends { tag: string }> implements RouterAdapter<Route> {
     constructor(router: RouterContract<Route, any>, url: string);
     getRoute(): Result<Route, string>;
     navigate(_route: Route): void {}    // no-op
     replace(_route: Route): void {}    // no-op
     subscribe(_fn: (r: Result<Route, string>) => void): () => void {
       return () => {};                  // no-op; returns unsubscribe
     }
   }
   ```

4. Update plan 97's `RouterProvider` to accept `RouterAdapter<Route>` rather than
   `BrowserRouter<Route, Map>`. This is the key SSR-compatibility change.

## Why `navigate` is not optional

Making `navigate` optional on the interface forces every component that calls
`useNavigate()` to null-check the function before calling it — that is wrong and
verbose. Instead, `StaticRouter.navigate` is a no-op: on a server render there is no
navigation, so silently doing nothing is correct. Components do not need to know they
are in an SSR context.

## Success Criteria

- `StaticRouter` parses a URL and returns the correct route.
- `StaticRouter.navigate()` / `replace()` / `subscribe()` are callable with no error.
- `BrowserRouter` passes `RouterAdapter<Route>` type check.
- `RouterProvider` (plan 97) compiles with either `BrowserRouter` or `StaticRouter`.
- No `any` in the adapter interface.
