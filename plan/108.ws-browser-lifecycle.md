# PLAN.108 — Browser WebSocket Lifecycle (`useWebSocket` hook)

## Context

`createWsClient()` in `@arbor/router` provides a typed WebSocket client. In React, the
connection lifecycle must track component mount/unmount. This plan wraps the client with
hooks, mirroring plan 107 for SSE.

WebSocket adds a bidirectional dimension absent in SSE: the hook must expose a typed
`send` function for the client's outbound message type (derived from the server's `in`
schema — i.e., the dual).

## Goal

1. `useWebSocket(wsRoute, opts)` hook:
   - Opens the WebSocket on mount; closes on unmount.
   - Returns `{ messages, send, status }` where:
     - `messages: OutMsg[]` — received messages, typed from the route's `out` schema.
     - `send: (msg: InMsg) => void` — typed from the route's `in` schema.
     - `status: 'connecting' | 'open' | 'closing' | 'closed' | 'error'`.
   - Reconnects with exponential backoff on unexpected close.
   - Accepts `maxMessages?: number` (default: 200).

2. Queued send: if `send()` is called before the connection is open, queue the message
   and flush when `status === 'open'`. Configurable max queue depth.

## API Surface

```typescript
const { messages, send, status } = useWebSocket(routes.chat, {
  baseUrl: 'wss://api.example.com',
});
// messages: ChatMessage[]   (inferred from wsRoute out schema)
// send: (msg: ChatInput) => void  (inferred from wsRoute in schema)
// status: 'open' | 'connecting' | ...
```

## Success Criteria

- Connection opens on mount; closes on unmount (no `WebSocket` leak).
- `send()` is typed: wrong message type is a compile error.
- `messages` is typed: each entry is the server's outbound schema type.
- Queued messages flush when connection opens.
- Reconnect fires on unexpected close; does not fire on intentional unmount.
