# PLAN.98 — `<Link>` Component

## Context

Navigation must never use raw URL strings. `<Link to={route}>` calls `router.print(route)`
at render time to produce the `href`. This makes broken links a type error: if the route
tree changes and a tag is removed, all `<Link to={{ tag: 'removed-tag', ... }}>` call
sites become compile errors immediately.

## Goal

A `Link` component that:

1. Accepts a typed `to` prop of type `Route` (or a subset narrowed by tag).
2. Renders an `<a>` with `href` derived from `router.print(to)`.
3. Intercepts `onClick` on same-origin navigation: calls `navigate(to)` instead of
   a full page load. Falls back to native navigation for `target="_blank"`, Ctrl/Meta/Shift
   click, and `download` attributes.
4. Accepts `replace?: boolean` — uses `BrowserRouter.replace()` instead of `navigate()`.
5. Exposes `isActive: boolean` in a render prop or `data-active` attribute, derived from
   whether `useRoute().tag === to.tag`.

## API Surface

```typescript
interface LinkProps<Route> extends Omit<React.AnchorHTMLAttributes<HTMLAnchorElement>, 'href'> {
  to: Route;
  replace?: boolean;
  activeClassName?: string;
}

export function Link<Route>({ to, replace, activeClassName, ...rest }: LinkProps<Route>): JSX.Element;
```

## Success Criteria

- `Link` renders correct `href` via `router.print(to)`.
- Clicking the link calls `navigate()` and does not trigger a page reload (no
  `window.location.href` change in jsdom).
- Ctrl+click does not call `navigate()` (falls through to native).
- `data-active` is set when current route tag matches `to.tag`.
- `to` prop is typed — passing a non-route object is a compile error.
