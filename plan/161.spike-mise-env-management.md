# Plan 161 — Spike: `mise` for Local Environment Management

## Context

The root `package.json` currently has five `dev:*` scripts, each threading a different combination of dotenv files:

```json
"dev:mock":    "dotenv -e apps/bff/.env.local -e apps/api/.env.local -e .env.mock  -- turbo dev",
"dev:bff":     "dotenv -e apps/bff/.env.local -e apps/api/.env.local -e .env.bff   -- turbo dev",
"dev:oidc":    "dotenv -e apps/bff/.env.local -e apps/api/.env.local -e .env.oidc  -- turbo dev",
"dev:staging": "dotenv -e apps/bff/.env.staging.local -e apps/api/.env.local -e .env.staging -- turbo dev",
```

Problems:
- Adding a new app means updating all four scripts.
- The right combination of files is non-obvious; the ordering matters (later files win).
- The committed mode-env files (`.env.mock` etc.) are a parallel documentation surface to `turbo.json`'s `globalEnv` array.

`mise` (née `rtx`) is a polyglot runtime + task + env manager. Its `.mise.toml` can declare env vars that activate when you `cd` into the directory, and tasks that run in that environment. This spike evaluates whether replacing the root-level `.env.*` mode files and the `dev:*` script fan-out with a single `mise.toml` is worth the tool adoption cost.

## Goal

Answer the following questions with working prototypes or concrete findings:

1. **Integration with Turbo**: Can `mise task` (or `mise run`) invoke `turbo dev` while passing the correct env? Does `turbo`'s hash include mise-provided env vars? (If not, caching may break.)
2. **CI compatibility**: Does the CI pipeline need `mise` installed, or only the machine-local workflow? Can `dotenv-cli` remain as the CI mechanism while `mise` handles local dev?
3. **Developer buy-in surface**: What does a developer need to install/run after `git clone`? Is `mise trust` an acceptable friction point?
4. **Env layering**: Can `mise` layer machine-local secrets (currently in `apps/*/env.local`) on top of mode flags (currently in `.env.mock` etc.)? Or does it flatten them in a way that conflicts?
5. **`use-builder.ts` example style**: Would scripts in `.mise.toml` replace `pnpm dev:mock`, or co-exist? What's the UX for switching modes — `mise run dev mock` vs `ARBOR_MODE=mock pnpm dev`?

## Proposed Approach

Create a branch. Prototype a `.mise.toml` that:

```toml
[env]
# shared across all modes — sourced from apps/bff/.env.local + apps/api/.env.local
# (mise supports `_.file = ["apps/bff/.env.local", ...]`)
_.file = ["apps/api/.env.local", "apps/bff/.env.local"]

[tasks.dev]
description = "Run the full stack (default: mock auth)"
run = "turbo dev"

[tasks."dev:mock"]
env = { VITE_AUTH_MODE = "mock", ARBOR_AUTH_DISABLED = "true" }
run = "turbo dev"

[tasks."dev:bff"]
env = { VITE_AUTH_MODE = "bff", ARBOR_AUTH_DISABLED = "false" }
run = "turbo dev"
# ... etc
```

Then validate against the five questions above.

## Dependencies

None. This spike is purely additive — if the outcome is "not worth it", no production change is made.

## Success Criteria

The spike is complete when the five questions above have definitive answers (yes/no + rationale). The spike passes if `mise run dev:mock` launches the full stack with the same env as `pnpm dev:mock` currently does, and turbo caching behavior is unaffected.

**Go decision**: Proceed to a deliver plan if:
- CI can stay on `dotenv-cli` (no mise install required in CI).
- The `.env.*` mode files can be deleted (env fully in `.mise.toml`).
- Developer setup requires only `mise install` + `mise trust` after `pnpm install`.

**No-go**: Stay on current approach if turbo cache hash breaks with mise-provided env, or if CI requires significant changes.
