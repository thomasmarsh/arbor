# Plan 148 — Auto-Polling & Manual Refresh

## Context

Multiple terminal sessions and Claude Code may modify `plan/ledger.jsonl` concurrently. This plan
adds a polling effect so the browser table stays fresh without requiring a full page reload.

## Goal

The table auto-refreshes every 5 seconds. The `r` key (added in Plan 147) triggers an immediate
re-fetch. A subtle "last updated" timestamp is shown in the header.

## Change Surface & T-shirt Sizing

**Size: XS**

Modified files:
- `packages/ui/src/ledger/ledger.store.ts` — add `startPolling` action + poll tick effect
- `packages/ui/src/ledger/LedgerTable.tsx` — dispatch `startPolling` on mount, show timestamp

## Polling via Effect.merge

The TCA pattern handles recurring effects through the same `Effect.merge` mechanism already
used in `counterReducer`:

```typescript
case 'fetch': {
  $.state.loadState = { tag: 'loading' };
  return Effect.merge<LedgerAction>(
    env.fetchQueue.map((result) => result.fold(
      (groups) => ({ tag: 'loaded', groups }),
      (err) => ({ tag: 'error', message: err }),
    )),
    // Schedule the next poll tick
    env.pollTick.map(() => ({ tag: 'fetch' })),
  );
}
```

`env.pollTick` is `Effect.sleep(5000)` — the next `fetch` dispatches automatically after 5s.

The `r` key (from Plan 147's `refresh` action) cancels the pending tick and starts a fresh cycle
by dispatching `fetch` immediately (the merge creates a new tick on completion).

## Last-Updated Timestamp

```typescript
// Add to LedgerState
lastUpdated: Date | null;

// In 'loaded' case
$.state.lastUpdated = new Date();
```

Displayed in the component header: `Last updated: 12:34:05`.

## Stopping the Poll

When the component unmounts, the store's cleanup handles outstanding effects. No explicit
stop action needed — the `useStore` hook from `@arbor/common/react` already cancels pending
effects on unmount.

## Success Criteria

- Table refreshes every ~5 seconds without user input.
- `r` key triggers an immediate refresh (visible via timestamp change).
- Wave 29 slice is complete.

## Post-mortem

**Root cause**: The `@typescript-eslint/no-invalid-void-type` rule (enabled via `strictTypeChecked`) forbids `void` as a type argument to *call expressions* even when `allowInGenericTypeArguments: true`. The rule's `invalidGrandParents` list includes `CallExpression`, so `Effect.of<void>(...)` fires the rule despite the outer `<void>` being syntactically a generic type argument.

**What made it worse — flawed structural-assignability reasoning**: After the first fix attempt failed, I tried changing `pollTick: Effect<void>` → `Effect<undefined>` in the interface. I incorrectly predicted `Effect<void>` would be assignable to `Effect<undefined>`. TypeScript checks generic class compatibility directly through the type parameter (covariant position), so `void` not assignable to `undefined` propagates and makes `Effect<void>` incompatible with `Effect<undefined>`. The correction (`Effect<undefined>` assignable to `Effect<void>`) would have worked, but I had the direction backwards, wasting two more edit/revert cycles.

**Actual fix**: Annotate the binding site (`const oncePollEnv: typeof mockLedgerEnv`) so contextual typing drives `Effect.of`'s generic to `void` without any explicit `<void>` syntax in the source. No `void` keyword appears anywhere in a call-expression type-argument position — ESLint rule never fires.

**Pre-flight step that would have caught it**: Check `eslint.config.js` for `strictTypeChecked` before writing any `Effect.of<void>` syntax. Stage 2 already requires reading the ESLint config when the change surface touches generics — this case qualified and the check was skipped.
