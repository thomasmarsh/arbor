# PLAN.136 (Spike) — Schema Scope Audit and Generalization

## Context

Plan 133 treats `RouteNode.schema: z.ZodObject<any, any> | null` as *the* coupling point to
replace. But a quick audit reveals schema usage is far deeper:

| Position            | Type                                       | Used for                                |
| ------------------- | ------------------------------------------ | --------------------------------------- |
| `RouteNode.schema`  | `z.ZodObject<any, any> \| null`            | Path param parsing + type inference      |
| `HttpContextData.querySchema` | `z.ZodObject<any, any>`        | Query string parsing + OpenAPI           |
| `HttpContextData.bodySchema`  | `z.ZodType`                    | Request body validation + OpenAPI        |
| `HttpContextData.headerSchema` | `z.ZodObject<any, any>`       | Request header validation + OpenAPI      |
| `HttpContextData.cookieSchema` | `z.ZodObject<any, any>`       | Cookie validation + OpenAPI              |
| `HttpContextData.responseSchemas` | `Record<number, z.ZodType>` | Response body per status code + OpenAPI |
| `HttpContextData.responseHeaderSchemas` | `Record<number, z.ZodObject<any,any>>` | Response headers per status |
| `HttpContextData.responseCookieSchemas` | `Record<number, z.ZodObject<any,any>>` | Response cookies per status |
| `WsMeta.ZS`         | `z.ZodObject<any, any>`                    | WebSocket message schema                 |
| `session-route.ts`  | `z.ZodObject<any, any>`                    | Session payload schema                   |

That is ten distinct schema attachment points. Plan 133 proposes a native schema type —
but it's unclear whether the replacement should:

(a) Replace each of these independently (minimal coupling, large change surface), or
(b) Introduce a unified `RouteSchema<P, Q, B, R>` type that groups all schemas for a route
    (cleaner, but changes the `httpRoute()` API and `RouteNode`'s type parameters), or
(c) Replace only the most impactful positions first (path params + query + body) and leave
    the rest for a later wave.

The `Segment` type in `segments.ts` is interesting in this context: it is already a
serializable discriminated union for path structure. The schema question is whether the
*value types* of path, query, and body parameters should be described in the same way.

## Goal

Produce a design proposal (not implementation) for a `RouteSchema` shape that:

1. Covers all ten attachment points.
2. Integrates cleanly with the existing `RouteNode` type parameter structure.
3. Identifies which positions need full schema support vs. only an object-shape schema.
4. Identifies which positions are "object-only" (headers, cookies, query) vs. "any type"
   (body, response bodies).

## Questions to Answer

**Q1 — Unified vs. distributed**

A unified `RouteSchema<Params, Query, Body, Response>` type would group all schemas at a
route level, making the contract of a route explicit. Compare to the current distributed
model where schemas are scattered across `RouteNode.schema` (for params) and `_meta`
(for everything else). What is the cost of unification in terms of type parameter count
on `RouteNode` and `httpRoute()`?

**Q2 — Type parameter count**

`RouteNode` currently has 5 type parameters: `R, C, Context, SectionParams, Meta`. A
unified schema approach might add `Schema extends AnyRouteSchema` as a 6th. Does TypeScript
infer a 6th parameter cleanly in the `route()` and `httpRoute()` call sites? Or does this
push into "too many parameters" territory where inference degrades?

**Q3 — Schema as the source of truth for type inference**

Currently, `R` (the phantom route type) is derived from `z.infer<S>` where `S` is the Zod
schema passed to `route()`. If we have a native schema type `Schema<T>`, then `R = Infer<S>`.
But body and response types are currently inferred from the *handler function signature*,
not from a schema. If we move to schema-first, the handler must conform to the schema rather
than the handler defining the type. This is a philosophical question about direction of
inference: does the schema drive the handler type, or does the handler type drive schema
validation? Both are defensible. What are the tradeoffs?

**Q4 — Segments as schema**

`Segment` in `segments.ts` already describes path structure but only has `str` and `num`
(string or number). The comment in that file notes: "this system only permits scalars.
Consider more structured options." Is a `Schema<T>` type the right generalization of
`Segment` — i.e., should segments eventually carry their own type constraints rather than
relying on the top-level schema object to describe path param shapes?

**Q5 — The `_meta` problem**

The current design stores all HTTP-specific schemas in `_meta` (a "plugin metadata bag").
This was intentional — to keep `RouteNode` domain-independent. But it means the HTTP
contract is invisible at the `RouteNode` type level. A unified schema would need to either:
- Remain in `_meta` (preserving domain independence, but opaque to the core)
- Move to a first-class type parameter (visible to the core, better type safety)
- Use a typed accessor pattern (a "schema extractor" that reads `_meta` in a typed way)

Which option best preserves the `RouteNode` design principle (core is domain-independent)?

## Findings (June 2026)

### Q1 — Unified vs. distributed

Keep schemas distributed across attachment points, not unified into a single `RouteSchema<P,Q,B,H,C,R>`
type parameter. The unified form adds 1–3 type parameters to `RouteNode` and `httpRoute()` for
positions that are already well-separated. The current `_meta` bag handles HTTP-specific schemas
cleanly — the only change needed is swapping the type of the param schema from `z.ZodObject<any,any>`
to `ObjectSchema<T>`. Body and response schemas stay in `_meta` as Standard Schema references.

### Q2 — Type parameter count

No new type parameter on `RouteNode`. The param schema type `T` is already captured in `R`
(the phantom route type — `R = Infer<S>` at construction). Storing `S` itself as a type
parameter would be redundant and would push into "too many parameters" territory. The
`schema` field changes from `z.ZodObject<any,any> | null` to `ObjectSchema<T> | null` where
`T` is structurally recovered at the call site, not stored as a phantom.

### Q3 — Schema-first vs. handler-first

Stay handler-first for body and response types. The handler signature drives the TypeScript
type for body/response; the user's schema validates at the boundary. Schema-first would
require the handler to conform to the schema's inferred type — correct in principle but a
breaking change to the current ergonomics. For the router-controlled positions (params,
query, headers, cookies), schema-first is already the model and stays that way.

### Q4 — Segments as schema

Not yet. `Segment` covers structural path matching (`:id` → string, `#id` → number). The
native `ObjectSchema<T>` covers the value types of those params as an independent layer.
Merging them into one type would complicate both. Revisit when session type message schemas
need to describe param-level constraints on protocol messages.

### Q5 — The `_meta` problem

Leave schema references in `_meta`. The `RouteNode` core stays domain-independent. Typed
accessor functions (`getHttpMeta`, `getOpenApiMeta`) already provide safe, typed reads of
`_meta` without exposing its structure to the core. The only change is that those accessors
will reference `StandardSchemaV1<T>` for body/response instead of `z.ZodType`.

## Implementation Plan

This is the gating plan for the native schema implementation. All `src/` changes are blocked
until the scratch file (step 1) passes. Each step has a hard gate: existing tests pass,
typecheck passes.

### Step 1 — Scratch: prove type inference (gate for everything else)

File: `framework/router/scratch/native-schema-spike.ts`

Falsifiable claims that must hold before any `src/` change:

```typescript
import { expectTypeOf } from 'vitest';

// Claim 1: object schema infers its field types
const userSchema = object({ tag: literal('user'), id: string() });
expectTypeOf(userSchema).toEqualTypeOf<ObjectSchema<{ tag: 'user'; id: string }>>();

// Claim 2: Infer<S> extracts T from ObjectSchema<T>
type UserParams = Infer<typeof userSchema>;
expectTypeOf<UserParams>().toEqualTypeOf<{ tag: 'user'; id: string }>();

// Claim 3: optional wraps correctly
const q = object({ page: optional(integer()) });
expectTypeOf<Infer<typeof q>>().toEqualTypeOf<{ page?: number }>();

// Claim 4: route() infers RouteNode<Infer<S>> — mock route() to test inference shape
declare function mockRoute<S extends AnyObjectSchema>(s: S, path: string): { _type: Infer<S> };
const r = mockRoute(userSchema, 'users/:id');
expectTypeOf(r._type).toEqualTypeOf<{ tag: 'user'; id: string }>();
```

Run `tsc --diagnostics` and record instantiation count. If count exceeds 50,000, the type
design has a recursion problem — stop and redesign before proceeding.

Note on `object` recursion risk: `ObjectSchema<T>` references `FieldSchema<T[K]>` for each
field. If `FieldSchema` itself references `ObjectSchema`, this creates mutual recursion.
**Do not allow nested objects in the initial implementation.** The router-controlled positions
are all flat (params, query, headers, cookies are key-value maps of scalars). If nested
objects are needed later, add a depth counter using the same pattern as `FlattenChildrenImpl`.

### Step 2 — Define `ObjectSchema<T>` and `Infer<S>` in `src/core/`

New file: `framework/router/src/core/schema.ts`

No changes to any other file in this step.

```typescript
// Scalar kinds — no functions, no closures, fully serializable
export type ScalarSchema<T> =
  | { kind: 'string' }
  | { kind: 'number' }
  | { kind: 'integer' }
  | { kind: 'boolean' }
  | { kind: 'literal'; value: T }
  | { kind: 'optional'; inner: ScalarSchema<Exclude<T, undefined>> };

export type AnyScalarSchema = ScalarSchema<string | number | boolean | undefined>;

// Object schema — flat only (no nested ObjectSchema in fields; see Step 1 note)
export type ObjectSchema<T extends Record<string, unknown>> = {
  kind: 'object';
  fields: { [K in keyof T]: ScalarSchema<T[K]> };
};

export type AnyObjectSchema = ObjectSchema<Record<string, unknown>>;

// Type-level inference from schema to TypeScript type
export type Infer<S> =
  S extends ObjectSchema<infer T> ? T :
  S extends ScalarSchema<infer T> ? T :
  never;
```

Gate: `pnpm --filter @arbor/router typecheck` passes. No other files changed.

### Step 3 — Factory functions in `src/core/schema.ts`

Add to the same file. No other files changed.

```typescript
export const string   = (): ScalarSchema<string>  => ({ kind: 'string' });
export const number   = (): ScalarSchema<number>  => ({ kind: 'number' });
export const integer  = (): ScalarSchema<number>  => ({ kind: 'integer' });
export const boolean  = (): ScalarSchema<boolean> => ({ kind: 'boolean' });
export const literal  = <T extends string | number | boolean>(value: T): ScalarSchema<T> => ({ kind: 'literal', value });
export const optional = <T>(inner: ScalarSchema<T>): ScalarSchema<T | undefined> => ({ kind: 'optional', inner });
export const object   = <T extends Record<string, unknown>>(
  fields: { [K in keyof T]: ScalarSchema<T[K]> }
): ObjectSchema<T> => ({ kind: 'object', fields });
```

Re-run scratch `expectTypeOf` claims. Gate: all pass, instantiation count unchanged.

### Step 4 — Swap `RouteNode.schema` type

Change one field in `framework/router/src/core/route-node.ts`:

```diff
-  schema: z.ZodObject<any, any> | null;
+  schema: AnyObjectSchema | null;
```

Remove the `import type z from 'zod'` line from `route-node.ts` if it becomes unused.

Gate: `pnpm --filter @arbor/router test && pnpm --filter @arbor/router typecheck`. All
existing tests must pass — if any test constructs a `RouteNode` directly with a Zod schema,
update it to use the factory functions.

### Step 5 — Update `route()` builder

Change `framework/router/src/core/define-routes.ts`:

```diff
-  S extends z.ZodObject<any, any>,
->(schema: S, path: string, children?: [...C]): BuildableRouteNode<RouteNode<z.infer<S>, [...C]>> {
+  S extends AnyObjectSchema,
+>(schema: S, path: string, children?: [...C]): BuildableRouteNode<RouteNode<Infer<S>, [...C]>> {
```

Gate: existing tests pass. The `route()` call sites in `examples/` should continue to work
if examples are updated to use native schema factory functions.

### Step 6 — Update `walk.ts` (remove Zod value import)

Replace `instanceof z.ZodLiteral` with `field.kind === 'literal'`:

```diff
- import z from 'zod';
+ import type { AnyObjectSchema } from './schema.js';

- export function getShape(schema: z.ZodObject<any, any>): Record<string, z.z.ZodType> { ... }
- export function getTag(schema: z.ZodObject<any, any>): string | undefined {
-   const tag = getShape(schema)['tag'];
-   return tag instanceof z.ZodLiteral ? (tag.value as string) : undefined;
- }
+ export function getTag(schema: AnyObjectSchema): string | undefined {
+   const tag = schema.fields['tag'];
+   return tag?.kind === 'literal' ? String(tag.value) : undefined;
+ }
```

`z.core.$ZodIssue` in `ParseDiag` must also be replaced with a plain error type — see below.

Gate: `pnpm --filter @arbor/router test && pnpm --filter @arbor/router typecheck`. This
step removes the last Zod value import from the core path.

### Step 7 — Update `generate-spec.ts` for native schema

The param and query schema generation in OpenAPI currently calls `zodToJsonSchema`. For
native schemas, this becomes a direct mapping: `{ kind: 'string' }` → `{ type: 'string' }`,
etc. Body and response schemas continue to use whatever JSON Schema the user provides.

### Step 8 — Standard Schema for body/response (parallel track)

Update `http-context.ts` to accept `StandardSchemaV1<T>` instead of `z.ZodType` for
`bodySchema`, `responseSchemas`, `headerSchema`, `cookieSchema`, `querySchema`.

This step is independent of steps 4–7 and can be developed in parallel once the scratch
file (step 1) passes. It is the largest step in terms of type signature changes but does
not touch `core/`.

### Error type migration

`ParseDiag` currently uses `z.core.$ZodIssue`. Replace with a plain type:

```typescript
export type SchemaIssue = { message: string; path?: (string | number)[] };
export type ParseDiag =
  | { kind: 'segment-mismatch'; path: string; urlSegments: string[] }
  | { kind: 'schema-error'; path: string; issues: SchemaIssue[] };
```

Standard Schema's error type is compatible with this shape.

### What does NOT change

- `FlattenChildrenImpl` and all depth-counter machinery — untouched
- `CtxMap` and `EmbeddedMap` — untouched
- The `_meta` structure for HTTP context — untouched except type of schema fields
- `createClient`, `createServer`, `createTestClient` public APIs — untouched
- The OpenAPI `generateSpec()` output format — untouched
