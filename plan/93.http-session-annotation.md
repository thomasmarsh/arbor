# Plan 93 — HTTP Session Type Annotation

## Context

Plans 87–90 introduced session types for SSE (`sseRoute`) and WebSocket (`wsRoute`).
Both route factories store a `SessionMeta<S>` phantom in `_meta`, enabling uniform
introspection via `InferSession` / `InferDual`. The HTTP layer (`httpRoute`) was not
revisited: `HttpContextData` does not extend `SessionMeta<S>`, and `createClient().fetch()`
derives its response type through `HttpContext.response`, not through `Dual<S>`.

This plan closes that gap. HTTP request/response **is** a session type — it is the
shortest non-trivial case. The server's session is:

```
Recv<RequestInput, Select<{ [S in StatusCodes]: Send<Res[S], End> }>>
```

and the client's dual (computed by `Dual<S>`) is:

```
Send<RequestInput, Branch<{ [S in StatusCodes]: Recv<Res[S], End> }>>
```

The `Branch` on the client side maps precisely to the `matchResponse` exhaustive-handler
map: `matchResponse` IS the `Branch` combinator for HTTP, and its `ExhaustiveConstraint<R>`
type is isomorphic to the branch cases. The current `HttpResponseUnion<Res>` type is
isomorphic to `BranchInput<Dual<HttpSession<Req, Res>>>`. These are the same structure,
two different names.

### What value this brings

1. **Uniform `_meta` introspection** — any tool (OpenAPI generator, MPST projection engine
   from plan 91, capability system from plan 92) can call `InferSession<Node>` on any route
   node regardless of protocol. Today HTTP routes return `never` from `InferSession`.

2. **Conceptual completeness** — the "every route has a session type" invariant holds across
   all three protocol families. Documentation and examples can use consistent language.

3. **`matchResponse`-as-Branch** — the identity `BranchToUnion<Dual<HttpSession<Req, Res>>>
   ≡ HttpResponseUnion<Res>` is proved via type-level test. Future tooling that composes
   session types across protocol boundaries has a verified bridge to the HTTP response union.

4. **Exported `HttpSession<Req, Res>` and `InferHttpSession<Route>`** — utility types for
   plan 91/92 and downstream use.

### What this does NOT change

- `createClient().fetch()` continues to type its return via `HttpResponseUnion<Map[Tag]['response']>`.
  Deriving it from `Dual<S>` would add instantiation overhead with no ergonomic gain.
- `matchResponse`'s implementation and type signatures are unchanged.
- No server dispatch logic changes.
- No existing public API changes. All changes are additive (new phantom fields, new type exports).

1. Define `HttpSession<Req, Res>` — the canonical session type for a single HTTP endpoint.
2. Extend `HttpContextData` to carry `SessionMeta<HttpSession<...>>` (phantom only).
3. Update `httpRoute` to populate the phantom field with the correct inferred `S`.
4. Prove `BranchToUnion<Dual<HttpSession<Req, Res>>> ≡ HttpResponseUnion<Res>` via a
   type-level test in `http-context.test.ts`.
5. Export `HttpSession`, `InferHttpSession`, and `HttpResponseBranch` from `index.ts`.

## Design Notes

### Encoding RequestInput

The "request input" for a GET route with no body is just `{ query: Q; params: P }`.
For a POST it adds `body: B`. Rather than encoding this complex shape into the session
type's `Recv<T>`, consider two alternatives:

**Option A — Phantom void for request**: Use `Recv<void, HttpResponseSelect<Res>>` for
the server session. This loses request type information in `InferSession` but keeps the
type much simpler and avoids inflating instantiation counts with large `T` shapes.

**Option B — Full input shape**: Use `Recv<{ params: P; body: B; query: Q }, ...>`.
Complete information, but `HttpSession` grows complex for routes with many generic params.

**Recommendation**: Start with Option A (phantom `void` request side). The primary value
is in the response `Select<...>` / `Branch<...>` encoding. The request side is already
captured by `HttpContext<Method, Body, Response, Query, Headers, Cookies>` — duplicating
it in the session annotation adds noise. `InferDual<Route>` is most valuable for the
client-side Branch (which response handlers to provide), not for the request side.

This means `HttpSession<Res>` (dropping `Req` from the public API) =
`Recv<void, HttpResponseSelect<Res>>`, and the dual is
`Send<void, Branch<{ [S in keyof Res]: Recv<Res[S], End> }>>`.

### BranchToUnion utility

To prove the isomorphism, we need a utility type that extracts the response union from
a `Branch<Cases>` session:

```typescript
type BranchCaseToStatusObj<Cases extends Record<string, Session>, K extends keyof Cases> =
  Cases[K] extends Recv<infer T, any> ? { status: K; body: T } : never;

type BranchToUnion<S extends Session> =
  S extends Branch<infer Cases extends Record<string, Session>>
    ? { [K in keyof Cases]: BranchCaseToStatusObj<Cases, K> }[keyof Cases]
    : never;
```

The identity to prove:
```
BranchToUnion<Dual<HttpSession<{ 200: User; 404: Error }>>> ≡ HttpResponseUnion<{ 200: User; 404: Error }>
```

Both sides should equal `{ status: 200; body: User } | { status: 404; body: Error }`.

Note: `keyof Res` is `number` by default for `Record<number, ...>` and needs care:
the `status` field in `HttpResponseUnion` is a literal number type (`200 | 404`), not
`string`. `HttpResponseSelect<Res>` must preserve literal number keys via
`[S in keyof Res & number]` (already in the design above).

### Spike requirement

Before writing `httpRoute` changes, validate the `HttpSession` encoding in a scratch file:
1. Confirm `Dual<HttpSession<{ 200: User; 404: Error }>>` resolves correctly without TS2589.
2. Confirm `BranchToUnion<...>` extracts literal status types (not `string` or `number`).
3. Confirm that extending `HttpContextData` with `SessionMeta<Session>` does not break
   `getHttpMeta`, `collectHttpMaps`, or any `walkCollect` call that types nodes as
   `HttpWalkNode`.

This spike step is mandatory because `HttpContextData` is used widely (server, client,
OpenAPI) — a stray `never` or `any` in `SessionMeta<Session>` could silently widen types.

### No change to client type derivation

`TypedClient<Route, Map>` derives the response type via:
```
Map[Tag]['response'] → HttpResponseUnion<Map[Tag]['response']>
```

This path is correct, efficient, and well-tested. Deriving from `Dual<InferSession<...>>`
would require an extra `BranchToUnion` unwrap step and adds instantiation cost. Leave it.
The session annotation in `_meta` is purely phantom / introspection-facing.
