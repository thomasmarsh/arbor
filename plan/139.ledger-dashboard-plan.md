# Plan 139 — Ledger Web Dashboard: Wave Plan & Ordering

## Context

The project's task ledger lives in `plan/ledger.jsonl` and is currently managed via a Python
Textual TUI (`tools/arbor-cli`). This plan introduces a TypeScript web replacement built on two
principles already proven in the codebase:

- **`@arbor/router`** for the API layer — type-safe route definitions, typed HTTP client, zero
  runtime codegen.
- **TCA reducer/store pattern** (packages/ui) for the frontend — explicit state, typed actions,
  injected effects.

The goal is to ship something visible and runnable at the end of every wave, not to do all
backend-first then all frontend. Each wave completes a thin vertical slice.

## Goal

Replace the Python TUI with a browser-based ledger dashboard built on the project's own
type-safe router and frontend patterns, delivered in three iterative waves.

## Architecture Decisions

### Package structure

| Concern | Location |
|---|---|
| TS ledger types, Zod schemas, file I/O, queue computation | `packages/api/src/ledger/` |
| `@arbor/router` route tree definition + request handlers | `packages/api/src/ledger/routes.ts` |
| Route tree exported as `ledgerRouter` from `packages/api` | `packages/api/src/index.ts` |
| Hono mounts arbor handler at `/api/ledger/*` | `packages/api/src/index.ts` |
| Typed client, TCA store, React component | `packages/ui/src/ledger/` |

### Typed client flow

`packages/ui` adds `@arbor/api` as a `workspace:*` devDependency. The ledger store's env
imports `typeof ledgerRouter` and calls `createClient` with it — all response types flow from
the route tree, no duplication.

### Ledger path

The API reads `plan/ledger.jsonl` from disk. For local dev the path defaults to an env var
`ARBOR_LEDGER_PATH` falling back to the repo root autodetected by walking up from `__dirname`.

### Hono integration

The existing Hono server stays. Arbor routes are registered as a catch-all:

```typescript
app.all('/api/ledger/*', async (c) => {
  const result = await ledgerServer.handle(
    new URL(c.req.url), c.req.method,
    await c.req.json().catch(() => undefined),
  );
  return c.json(result.body, { status: result.status });
});
```

Route paths in the arbor router include the full prefix: `api/ledger/tasks`, `api/ledger/queue`.

## Wave Breakdown

### Wave 27 — Read-only MVP (tasks 140–144)

Delivers: fetch tasks from disk → JSON API → browser table. No mutations.

```
140  TS ledger data layer   (api)  — types, Zod schemas, file reader, queue computation
141  @arbor/router routes   (api)  — route tree + handlers for GET tasks/queue
142  Hono integration       (api)  — mount arbor handler, export ledgerRouter
143  Ledger TCA store       (ui)   — state, reducer, env (fetch only)
144  LedgerTable component  (ui)   — read-only task table, no interactions
```

**Dependencies**: 140 → 141 → 142 (serial); then 143 → 144; 142 blocks 143 (needs route types).

### Wave 28 — Mutations (tasks 145–147)

Delivers: set status (next/done/todo), bump/defer rank — wired through API + store + keyboard UI.

```
145  Mutation routes        (api)  — PATCH /api/ledger/tasks/:id (status, rank)
146  Mutation store actions (ui)   — setStatus, bump, defer actions + LedgerEnv effects
147  Keyboard shortcuts     (ui)   — n/d/b/D/r keys + show-all toggle (combined: they touch the same component)
```

**Dependencies**: 145 after 142; 146 after 143 + 145; 147 after 146.

### Wave 29 — Polish (tasks 148–149)

Delivers: auto-refresh polling, dependency display in blocked column.

```
148  Polling / manual refresh  (ui)  — periodic re-fetch, r key in store
149  Dep-aware display         (ui)  — blocked tasks show pending dep IDs, dim done/canceled
```

**Dependencies**: 148 after 147; 149 after 144.
