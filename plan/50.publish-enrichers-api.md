# PLAN.50 — Publish enrichers to the public API and redirect plans 36–40

## Context

Plan 49 proved that handler enrichers (`withEnricher`, `composeEnrichers`) are a
clean, type-safe replacement for the middleware pipeline (plan 44). The spike left
three open gaps:

1. `src/server/enrichers.ts` is not exported — consumers cannot use it.
2. Plans 36–40 still list plan 44 as a dependency; that plan is now superseded.
3. The plan 44 file still exists as an open plan and needs to be closed.

## Goal

1. Export `Enricher`, `withEnricher`, and `composeEnrichers` from the public API.
2. Mark plan 44 as superseded.
3. Update plans 36–40 plan docs so their dependency and design sections reference
   `withEnricher` instead of the old pipeline approach.

No new runtime behaviour. No changes to `RouteNode`, `createServer`, or existing
tests.

## Change Surface

### 1. `src/server/index.ts`

Add a re-export:

```typescript
export { type Enricher, composeEnrichers, withEnricher } from './enrichers.js';
```

### 2. `src/index.ts`

Add to the `// HTTP` export block:

```typescript
export { type Enricher, composeEnrichers, withEnricher } from './server/enrichers.js';
```

### 3. Plan 44 closure

Rename `plan/44.middleware-pipeline.md` →
`plan/44.SUPERSEDED.middleware-pipeline.md` and prepend a one-line note:

```
> **Superseded by plan 49.** Use `withEnricher` / `composeEnrichers` instead.
```

### 4. Plan docs 36–40 — dependency update

For each plan, replace the plan 44 dependency bullet with:

```
- Plan 50 (enrichers public API) — use `withEnricher` as the mechanism instead
  of a pipeline entry. Each plan delivers a named enricher function.
```

Update the design/implementation section of each plan to describe an enricher
function rather than a pipeline middleware:

| Plan | Enricher name   | Short-circuit condition                                     |
| ---- | --------------- | ----------------------------------------------------------- |
| 36   | `withRateLimit` | rate exceeded → 429                                         |
| 37   | `withMetrics`   | never short-circuits; always passes through, records timing |
| 38   | `withCors`      | preflight → 200 or origin rejected → 403                    |
| 39   | `withSession`   | invalid/missing token → 401                                 |
| 40   | `withRbac`      | insufficient role → 403                                     |

`withMetrics` is a degenerate enricher (ok: true always); its Extra carries
timing metadata. Implementation may prefer a plain wrapper instead — evaluate
during plan 37.

## Dependencies

- Plan 49 ✓ — `withEnricher` / `composeEnrichers` already in `src/server/enrichers.ts`.

## Success Criteria

- `pnpm test && pnpm typecheck && pnpm lint` passes after the export additions.
- `Enricher`, `withEnricher`, `composeEnrichers` are importable from the package root.
- Plan 44 is closed.
- Plans 36–40 each name a concrete enricher and no longer reference plan 44.
