# PLAN.105 (Spike) — TanStack Query Integration

## Context

TanStack Query is the dominant data-fetching library for React. The arbor typed client
already wraps `fetch` calls; TanStack Query manages caching, background refetch,
deduplication, and stale-while-revalidate. The question is whether the two can be composed
cleanly: use the typed arbor client as the `queryFn` and the route object as the
`queryKey`.

Two unknowns:

1. **Query key equality**: TanStack Query uses structural `JSON.stringify` comparison for
   query keys. The route discriminated union objects serialize to JSON naturally (no
   functions, no circular refs, no Symbols). But does the key structure cause accidental
   cache collisions? E.g., `{ tag: 'user', id: '1' }` vs `{ tag: 'user', id: '2' }` must
   be distinct keys; `{ tag: 'user', id: '1' }` navigated twice must hit the same cache.

2. **Type threading**: `useQuery` returns `data: TResult | undefined`. Can `TResult` be
   inferred as `HttpResponseUnion<Map[Tag]['response']>` without explicit annotation?
   Write a `useRouteQuery<Tag>` helper that threads the types.

## Goal

Write a scratch prototype (`packages/router-react/scratch/tanstack-query-spike.ts`) that:

1. Creates an `useRouteQuery(tag, route)` hook that calls `client.fetch(route)` as the
   `queryFn` and uses `route` as the `queryKey`.
2. Verifies `data` is typed as `HttpResponseUnion<Map[Tag]['response']>`.
3. Confirms query key equality works for identity and for different param values.
4. Tests invalidation: `queryClient.invalidateQueries({ queryKey: [route] })` correctly
   narrows to the right query.

## Deliverable

Decision memo (≤ 250 words):
- Does the integration work cleanly?
- Any type-inference gaps requiring `as` casts?
- Recommended `useRouteQuery` signature.
- Go/no-go on a `@arbor/router-query` adapter package.
