# PLAN.95 — Browser Navigation Core (`packages/router-browser`)

## Context

`@arbor/router` already has `parse(url)` and `print(route)`. The browser navigation
runtime is a thin reactive wrapper: subscribe to `popstate`, call `parse()` on the current
URL, expose current route and a typed `navigate()` function. No framework dependency.

## Goal

Implement `packages/router-browser` with a `BrowserRouter` class that:

1. Wraps a `RouterContract<Route, Map>` with History API integration.
2. Calls `router.parse(window.location.pathname + window.location.search)` to get the
   current route.
3. Exposes `navigate(route)` — calls `router.print(route)` + `history.pushState()`.
4. Exposes `subscribe(fn)` — fires `fn(Result<Route, string>)` on every navigation
   (initial + `popstate` + programmatic).
5. Exposes `getRoute()` — synchronously returns the current parsed route.

## API Surface

```typescript
// packages/router-browser/src/browser-router.ts

export class BrowserRouter<Route extends { tag: string }, Map extends AnyCtxMap> {
  constructor(router: RouterContract<Route, Map>);

  getRoute(): Result<Route, string>;
  navigate(route: Route): void;
  replace(route: Route): void;
  subscribe(fn: (route: Result<Route, string>) => void): () => void; // returns unsubscribe
  destroy(): void; // remove popstate listener
}
```

## Success Criteria

- `navigate(route)` calls `history.pushState` with the correct URL.
- `subscribe` fires on `popstate` events.
- `getRoute()` returns the current parsed route.
- `destroy()` removes the `popstate` listener.
- All covered by `jsdom`-based unit tests.
- Type: `navigate` accepts only `Route` (not arbitrary strings).
