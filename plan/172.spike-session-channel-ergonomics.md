# Plan 172 — Spike: Session Channel Ergonomics

## Context

Plans 87 and 88 established the session type primitives (`Send`, `Recv`, `Branch`, `Select`,
`End`, `Dual<S>`, `Channel<S>`) and landed them in `src/core/session.ts`. The current
WebSocket implementation in `ws-context.ts` ignores `Channel<S>` entirely and uses a flat
`WsChannel<S>` that collapses the session type to its first recv/send pair, discarding all
continuations. `Branch`/`Select` are completely wired.

The reason `WsChannel` stayed flat is a real constraint: `Channel<S>` is a
**state-threading** design where each operation returns `Channel<N>` (the next protocol
state). TypeScript has no linear types, so nothing prevents a handler from accidentally
reusing a spent channel reference. Forcing users to write `ch0 → ch1 → ch2` for every
protocol step is ergonomically painful and still doesn't buy compile-time safety against
re-use.

Before committing to any implementation direction, this spike re-examines the design
space from first principles.

## Goal

Produce a scratch file (`scratch/172-session-ergonomics.ts`) that implements and compares
at least two concrete encoding strategies for typed multi-step protocols over a WebSocket,
then records a verdict with rationale.

## Questions to Answer

1. **State-threading (`Channel<S>`)**: Can a runtime implementation satisfy `Channel<S>`
   cleanly via casting? What does a multi-step handler look like? Does the "spent channel"
   footgun feel tolerable in practice?

2. **Discriminated union dispatch**: Instead of sequential state, the server declares a
   union of all message types it can receive (e.g. `| { type: 'auth'; token: string } |
   { type: 'chat'; text: string }`). Dispatch is a typed switch. Does this model
   multi-phase protocols adequately (e.g. auth-first before chat is allowed)?

3. **Phase-gated channels**: A middle path — a small set of named "phases" where each
   phase has its own typed send/recv, and the handler advances phases explicitly
   (e.g. `phase.auth → phase.chat`). Simpler than full sequential types; still encodes
   protocol structure.

4. **Taggged protocol + `Branch`/`Select` only**: Drop `Send`/`Recv` chains; model all
   protocols as a tree of branches (server picks a path) and selects (client picks a
   path). Iteration within a branch becomes a recursive type. Simpler mental model?

## What the Spike Must Produce

The scratch file must implement a single concrete protocol — a **lobby protocol** — under
each candidate encoding:

```
Protocol (English):
  1. Client sends { type: 'join'; username: string }
  2. Server sends { type: 'welcome'; roomId: string } OR { type: 'full'; reason: string }
  3. If welcome: client and server exchange chat messages indefinitely
     - Client sends { type: 'chat'; text: string }
     - Server sends { type: 'chat'; author: string; text: string }
     OR
     - Client sends { type: 'leave' }
     - Server sends { type: 'bye' } → connection ends
  4. If full: connection ends
```

For each encoding, show:
- The route/session type declaration
- The server handler (what the author writes)
- The client call-site (what the consumer writes)
- The type that the compiler enforces at each step

## Success Criteria

- All four (or however many) encodings compile cleanly in the scratch file
- The server handler and client call-site are written out for the lobby protocol under
  each encoding
- A **Verdict** section at the bottom of the scratch file records:
  - Which encoding is chosen and why
  - What it does NOT enforce (honest about limitations)
  - What the `wsRoute` / `ws-context.ts` implementation plan looks like under the winner
- No changes to any `src/` file

## Notes on the Design Space

The key tension:

| Property | State-threading (`ch0→ch1→ch2`) | Discriminated union | Phase-gated |
|---|---|---|---|
| Enforces message ORDER | ✓ (type-level) | ✗ | Partial |
| Enforces message SHAPE | ✓ | ✓ | ✓ |
| Readable handler code | Awkward | Natural | Natural |
| "Spent channel" footgun | Present | N/A | Scoped |
| Branch/Select natural | Yes | Via union | Phase transition |
| `Dual<S>` still useful | Yes | Partial | Yes |

The discriminated union approach is essentially what every other framework does (just with
typed payloads). The differentiator is encoding **order constraints** at the type level.
The spike's job is to determine if we can make that differentiator ergonomic enough to
ship.
