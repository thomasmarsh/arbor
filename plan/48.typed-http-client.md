# PLAN.48 — Typed HTTP Client: options API, response headers, subpath exports

## Context

Plan 31 spike validated three improvements to the existing `createClient`:

1. **Options object**: `client.fetch(route, body?)` becomes `client.fetch(route, {body?, headers?}?)`.
   Necessary to thread typed request headers (Plan 30's `HttpContext['headers']`) without
   a positional explosion. The current positional `body` arg cannot grow further.

2. **Response headers**: The client currently returns `ResponseUnion` (core, no header fields).
   Plan 29 introduced `HttpResponseUnion` which carries `{ body, headers }` for status codes
   that declare response header schemas. Switching the return type exposes this to callers.

3. **Subpath exports**: `package.json` already has `"./openapi"`. Adding `"./client"` and
   `"./server"` lets consumers tree-shake: a browser bundle can `import from '@arbor/router/client'`
   and never pull in `createServer`.

## Goal

Minimal, non-breaking upgrade to `createClient`:

- Update call signature to `{ body?, headers? }` options object
- Switch return type to `HttpResponseUnion`
- Wire request headers into the outgoing `fetch` call
- Add `./client` and `./server` subpath exports

## Change Surface

### `src/client/fetch-client.ts`

Replace `BodyArgs<Ctx>` (body only) with `RequestOpts<Ctx>` (body + headers):

```typescript
type RequestOpts<Ctx extends HttpContext<any, any, any, any, any>> = [Ctx['body']] extends [never]
  ? [Ctx['headers']] extends [never]
    ? {}
    : { headers: Ctx['headers'] }
  : [Ctx['headers']] extends [never]
    ? { body: Ctx['body'] }
    : { body: Ctx['body']; headers?: Ctx['headers'] };
```

Change `fetch` signature from:

```typescript
fetch<Tag>(route, ...args: BodyArgs<Map[Tag]>): Promise<ResponseUnion<...>>
```

to:

```typescript
fetch<Tag>(route, opts?: RequestOpts<Map[Tag]>): Promise<HttpResponseUnion<Map[Tag]['response']>>
```

Runtime: extract `opts?.body` and `opts?.headers`; merge headers with `Content-Type` when body
is present. Use `HttpResponseUnion` (imported from `http-context.ts`) as the response type.

### `src/client/index.ts`

Populate with re-exports:

```typescript
export { createClient, type FetchLike, type TypedClient } from './fetch-client.js';
```

Also export the `TypedClient` utility type so consumers can type variables that hold a client
without repeating the generic parameters.

### `src/server/index.ts`

Populate with re-exports:

```typescript
export { createServer, type HandlerMap, type HandlerCtx } from './server.js';
```

### `package.json`

Add two subpath entries alongside the existing `"./openapi"`:

```json
"./client": {
  "import": "./dist/client/index.js",
  "types": "./dist/client/index.d.ts"
},
"./server": {
  "import": "./dist/server/index.js",
  "types": "./dist/server/index.d.ts"
}
```

`src/index.ts` continues to re-export everything — the flat `@arbor/router` import keeps working.

### `src/client/fetch-client.test.ts`

Update every call site:

- `client.fetch(route, body)` → `client.fetch(route, { body })`
- Add tests for typed request headers (route with `headers` in context)
- Add type test: response type includes `headers` for routes that declare response header schemas

## Dependencies

- Plan 29 ✓ — `HttpResponseUnion` carries `{ body, headers }` for status codes with header schemas
- Plan 30 ✓ — `HttpContext['headers']` is the request header type threaded into `RequestOpts`
- Plan 31 ✓ — spike confirmed the `TypedClient` prototype compiles

## Success Criteria

1. `pnpm test && pnpm typecheck && pnpm lint` passes clean
2. All existing test scenarios still pass (GET, POST, validate mode, URL construction)
3. New type tests:
   - Route with `headers` declared: `client.fetch(route, { headers })` compiles; omitting `headers` is a type error
   - Route without `headers`: `client.fetch(route, { headers: ... })` is a type error
   - Route with response header schema at 200: return type includes `headers` field on the 200 member
4. `import { createClient } from '@arbor/router/client'` resolves after build
5. `import { createServer } from '@arbor/router/server'` resolves after build

## Non-Goal

- No ts-rest-style `client.users.get(...)` proxy API — the existing `client.fetch(route, opts?)`
  ergonomics are sufficient and match the route-object-first pattern of the whole library.
- No response header _validation_ on the client side — the server validates outbound headers;
  the client just surfaces what the server returns.
