# PLAN.164 — Native Schema Core Threading

## Context

Plan 163 (scratch proof) is the gate for this plan. Do not start until all 7 claims in
the scratch file pass and the `tsc --diagnostics` instantiation count is below 50,000.

This plan threads the native schema through `src/` — swapping `z.ZodObject<any,any>` out
of the core path and removing the Zod value import from `walk.ts`. The result: browser
bundles using `createClient` no longer carry ~195KB of Zod runtime.

## Goal

Complete steps 4–7 of the implementation plan in `plan/136.spike-schema-scope.md` findings:

- Step 4: swap `RouteNode.schema` type
- Step 5: update `route()` builder
- Step 6: update `walk.ts` (remove Zod value import; replace `instanceof z.ZodLiteral`)
- Step 7: update `generate-spec.ts` for native schema path params/query

## Step-by-Step

Each step must leave the test suite green before proceeding to the next.

### Step 4 — Add `schema.ts`; swap `RouteNode.schema`

Create `framework/router/src/core/schema.ts` using the type definitions from the scratch
file (plan 163). This is a new file with no imports from other router modules.

Then change one line in `route-node.ts`:

```diff
-  schema: z.ZodObject<any, any> | null;
+  schema: AnyObjectSchema | null;
```

Remove `import type z from 'zod'` from `route-node.ts` if it becomes unused.

Gate: typecheck passes. Tests may fail at this step if they construct `RouteNode` directly
with Zod schemas — update those tests to use factory functions.

### Step 5 — Update `route()` builder

In `define-routes.ts`:

```diff
-  S extends z.ZodObject<any, any>,
->(schema: S, path: string, children?: [...C]): BuildableRouteNode<RouteNode<z.infer<S>, [...C]>> {
+  S extends AnyObjectSchema,
+>(schema: S, path: string, children?: [...C]): BuildableRouteNode<RouteNode<Infer<S>, [...C]>> {
```

Update examples that pass Zod object schemas at the `route()` call site to use the native
factory functions. Body and response schemas in `httpRoute()` are untouched (that is plan 165).

Gate: tests pass, examples run.

### Step 6 — Update `walk.ts`

Replace the Zod value import and `instanceof` checks:

```diff
- import z from 'zod';
+ import type { AnyObjectSchema } from './schema.js';

- export type ParseDiag =
-   | { kind: 'segment-mismatch'; path: string; urlSegments: string[] }
-   | { kind: 'schema-error'; path: string; issues: z.core.$ZodIssue[] };
+ export type SchemaIssue = { message: string; path?: (string | number)[] };
+ export type ParseDiag =
+   | { kind: 'segment-mismatch'; path: string; urlSegments: string[] }
+   | { kind: 'schema-error'; path: string; issues: SchemaIssue[] };

- export function getShape(schema: z.ZodObject<any, any>): Record<string, z.z.ZodType> { ... }
- export function getTag(schema: z.ZodObject<any, any>): string | undefined {
-   const tag = getShape(schema)['tag'];
-   return tag instanceof z.ZodLiteral ? (tag.value as string) : undefined;
- }
+ export function getTag(schema: AnyObjectSchema): string | undefined {
+   const tag = schema.fields['tag'];
+   return tag?.kind === 'literal' ? String(tag.value) : undefined;
+ }
```

`validateSchema` similarly becomes a switch on `kind` for each scalar type, calling the
native validator rather than `schema.safeParse()`. The result type changes from
`Record<string, unknown> | undefined` to `{ ok: true; data: Record<string, unknown> } | { ok: false; issues: SchemaIssue[] }`.

Gate: all tests pass. This is the step that removes the Zod value import from the browser
bundle path — verify with: `grep -r "import z from" framework/router/src/core/`.

### Step 7 — Update `generate-spec.ts`

The existing `zodToJsonSchema` helper is called for path params and query schemas. Replace
it with a `nativeSchemaToJsonSchema` function that maps `ScalarSchema<T>` directly to JSON
Schema keywords (no conversion needed — the constraint vocabulary IS JSON Schema):

```typescript
function nativeToJsonSchema(schema: AnyScalarSchema): Record<string, unknown> {
  switch (schema.kind) {
    case 'string':  return { type: 'string', ...schema.constraints };
    case 'number':  return { type: 'number', ...schema.constraints };
    case 'integer': return { type: 'integer', ...schema.constraints };
    case 'boolean': return { type: 'boolean' };
    case 'literal': return { const: schema.value };
    case 'optional': return nativeToJsonSchema(schema.inner); // required:false handled at field level
    case 'brand':    return nativeToJsonSchema(schema.inner);
  }
}
```

Body and response schemas in `generate-spec.ts` continue to use `z.toJSONSchema()` or the
user-supplied JSON Schema — untouched until plan 165.

Gate: `pnpm --filter @arbor/router run examples` runs and `openapi-output.ts` produces valid output.
