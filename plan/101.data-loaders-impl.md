# PLAN.101 — Data Loaders Implementation

## Context

Plan 100 (spike) produced a decision on loader placement and Suspense vs. explicit state.
This plan implements the chosen design. **Fill in the chosen approach here after spike.**

Placeholder design (to be confirmed by spike 100):

- Loaders live in a `LoaderMap<Route, Map>` — a separate map keyed by route tag, typed by
  route param types. No changes to `@arbor/router`.
- Execution: on navigation, `BrowserRouter` fires all loaders in the active route's
  ancestor chain in parallel (`Promise.all`). Data is stored in a per-route cache keyed
  by serialized route.
- React integration: `useLoaderData(tag)` returns the loader's resolved value for the
  given tag, typed by the loader's return type. The `RouterProvider` drives loading.
- Pending state: exposed as `isPending` in a `useNavigation()` hook; uses
  `React.startTransition` internally so the current view stays visible during transitions.

## Goal

1. `createLoaderMap(router, loaders)` — typed loader registry.
2. `BrowserRouter` integration: run loaders on navigation, cache results.
3. `useLoaderData<Tag>(tag)` hook — typed loader result.
4. `useNavigation()` hook — `{ isPending: boolean }`.
5. Error handling: if a loader throws, expose via `useRoute()` error state.

## Success Criteria

- Loaders for all ancestors fire in parallel on navigation.
- `useLoaderData('user')` returns the correct type for the `user` loader.
- `isPending` is `true` while loaders are in flight; `false` once all resolve.
- Loader errors surface as `useRoute()` error state without crashing the tree.
- Cache invalidates correctly when navigating to a different param value.
