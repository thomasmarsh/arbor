# PLAN.129 — Router Contract Diffing (`routerDiff`)

## Context

The route tree is a first-class value — it can be serialized and compared. Today, detecting
breaking API changes requires manually reviewing diffs or running OpenAPI diffing tools
against a serialized spec. Both approaches are lossy: OpenAPI diffing works on a JSON
document (not the Zod schemas), and code review misses subtle changes (a status code added
to a response union is a non-breaking addition; removing one is breaking).

`routerDiff(oldRouter, newRouter)` produces a typed manifest of API changes, classified
by severity, derived directly from the route tree — not from the generated OpenAPI JSON.
Semver bump type becomes mechanically derivable from the diff output.

## Goal

### 1. Contract serialization

```typescript
// packages/router/src/contract/serialize.ts

export interface SerializedRoute {
  tag:          string;
  path:         string;
  method?:      string;
  params:       Record<string, 'string' | 'number' | 'optional-string' | 'optional-number'>;
  queryShape?:  Record<string, unknown>;  // JSON Schema of query Zod schema
  bodyShape?:   Record<string, unknown>;  // JSON Schema of body Zod schema
  responses:    Array<{ status: number; bodyShape: Record<string, unknown> }>;
}

export function serializeContract(router: RouterContract<any, any>): SerializedRoute[];
```

`queryShape` and `bodyShape` are JSON Schema representations of the Zod schemas — produced
by the same converter used in `generateSpec()`. This is the right level of granularity:
it captures the shape without embedding Zod internals.

### 2. Diff computation

```typescript
// packages/router/src/contract/diff.ts

export type ChangeKind =
  | 'route-added'            // new route tag: non-breaking
  | 'route-removed'          // removed route tag: BREAKING
  | 'path-changed'           // URL path changed: BREAKING
  | 'method-changed'         // HTTP method changed: BREAKING
  | 'param-added-required'   // new required path param: BREAKING
  | 'param-removed'          // removed path param: BREAKING
  | 'param-type-changed'     // param type changed (string→number): BREAKING
  | 'query-field-added-required'  // new required query field: BREAKING
  | 'query-field-removed'    // removed query field: BREAKING
  | 'response-code-added'    // new success status code: non-breaking
  | 'response-code-removed'  // removed status code: BREAKING
  | 'response-body-field-added-optional'  // new optional field: non-breaking
  | 'response-body-field-added-required'  // new required field on existing code: BREAKING
  | 'response-body-field-removed'         // removed field: BREAKING
  | 'response-body-field-type-changed';   // type changed: BREAKING

export interface ContractChange {
  tag:       string;           // route tag this change affects
  kind:      ChangeKind;
  breaking:  boolean;
  detail:    string;           // human-readable description
}

export function routerDiff(
  oldRouter: RouterContract<any, any>,
  newRouter: RouterContract<any, any>,
): ContractChange[];
```

### 3. Semver derivation

```typescript
export type SemverBump = 'major' | 'minor' | 'patch';

export function deriveSemverBump(changes: ContractChange[]): SemverBump;
// major  if any change is breaking
// minor  if only non-breaking additions
// patch  if no route-visible changes (internal only)
```

### 4. CI integration

A script `scripts/check-api-contract.ts`:

```bash
tsx scripts/check-api-contract.ts --old contracts/v1.json --current
```

The script serializes the current router, diffs against a committed baseline
(`contracts/v1.json`), prints a change manifest, and exits non-zero if any breaking
change is detected without a matching version bump.

The baseline file is committed as `contracts/api.json` and updated intentionally
(`pnpm run contract:snapshot`) when a breaking change is acknowledged and the semver
is bumped.

## Success Criteria

- `routerDiff(v1, v2)` correctly classifies all 14 `ChangeKind` variants.
- `deriveSemverBump` returns `major` when any `breaking: true` change exists.
- The CI script exits non-zero on a breaking change with no prior acknowledgement.
- `serializeContract` round-trips through JSON without loss (no functions, no Symbols).
- Tests cover at least 3 breaking and 2 non-breaking change scenarios.
