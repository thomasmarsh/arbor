# PLAN.44 — Type-safe middleware pipeline

## Context

`spec.enhancements.md` Phase 2, item 2. Right now cross-cutting concerns
(auth, rate limiting, logging) are implemented as ad-hoc checks scattered
inside `server.ts`. As the number of such concerns grows (plans 36, 37, 38,
39), each one adds another conditional branch to the dispatch loop.

The middleware pipeline formalizes this as a first-class, composable
abstraction. The key constraint from the architecture doc: implicit middleware
arrays are explicitly forbidden. Middleware is declared on the route or at
the server level, not injected by magic.

The type system must track what a middleware adds to `ctx` so downstream
middleware and the handler can consume it without casts.
kj
## Goal

Allow routes to declare a `middleware` array. Each middleware is a typed
function that receives `ctx` and returns an augmented `ctx`. TypeScript
accumulates the ctx additions across the pipeline.

```typescript
const authMiddleware: Middleware<{}, { session: UserSession }> =
  async (ctx, next) => {
    const session = await resolveSession(ctx.req);
    if (!session) return new Response(null, { status: 401 });
    return next({ ...ctx, session });
  };

const route = httpRoute({
  method: 'GET',
  path: 'me/',
  middleware: [authMiddleware],
  ...
});

// Handler ctx now includes `session: UserSession` — typed, not asserted.
handler: async ({ session }) => { ... }
```

The pipeline replaces the ad-hoc checks in plans 36, 37, 38, 39, 40 with
composable middleware that can be declared once and reused across routes.

## Change Surface

1. **New `src/server/middleware.ts`**: `Middleware<In, Out>` type, plus a
   `runMiddleware(pipeline, ctx)` runner that accumulates the ctx type.
2. **`http-context.ts`**: Add optional `middleware` array to the route context.
3. **`server.ts`**: Replace ad-hoc auth/rate-limit checks with a call to
   `runMiddleware`.
4. **`ServerHandlers` mapped type**: Infer the accumulated ctx type from the
   declared middleware chain. The handler's `ctx` argument reflects all
   middleware additions.
5. Tests:
   - Empty middleware pipeline → handler called with base ctx.
   - Middleware that returns early (401) → handler not called.
   - Middleware that augments ctx → handler receives augmented type.
   - Two middleware in sequence → both additions visible in handler.

## Dependencies

- Plan 22 (DI context) — base `ctx` shape must be settled before adding
  middleware-augmented shapes on top.

## Success Criteria

- Middleware additions to ctx are typed in the handler without casts.
- A middleware that short-circuits the pipeline prevents the handler from
  running.
- `npm test && npm run typecheck` clean.
