# Plan 91 — Spike: Multi-Party Session Types (MPST)

## Context

Multi-Party Session Types (MPST) extend binary session types (plan 87/88) to N
participants. A *global type* describes the entire protocol between all participants
in one place:

```
Alice → Bob  : RequestPayload
Bob   → Carol: ProcessedData
Carol → Alice: FinalResult
```

Each participant's *local type* is derived by *projection*: `Project<G, "Alice">` gives
Alice's local session type — what she sends, what she receives, and when. The key
guarantee: if every participant follows their local type, the global protocol is
automatically satisfied. No runtime coordination needed; the types ARE the proof.

For @arbor/router, MPST could model:
- **Multi-service workflows** — a client, an API server, and a database service all
  participating in a typed protocol; each service's handler is typed to its projection
- **Choreography-first API design** — define the global interaction pattern; derive
  server and client types by projection; OpenAPI becomes a view of the projection
- **Typed webhook chains** — `A → B → C` callback chains typed at the protocol level

This spike validates whether TypeScript can encode global types and compute projections
without hitting instantiation limits, and whether the result is ergonomic enough to be
useful in practice.

## Questions to Answer

1. **Global type encoding** — Can `Comm<From, To, T, Continuation>` be encoded as a
   navigable phantom type tree? Can branching (`ChoiceBy<Decider, Cases>`) be encoded?

2. **Projection** — Does `Project<G, P>` compute the correct local type for participant P
   for global types of depth ≥ 3? Does it correctly generate `End` for participants that
   are not involved in a communication step?

3. **Instantiation limits** — Does projection on a 5-participant, 5-step global type
   exceed TS limits? What depth counter is needed?

4. **Ergonomics** — Is the resulting local type readable? Can it be used directly in a
   handler signature, or does it need a `Channel<LocalType>` wrapper?

5. **Integration path** — Can `wsRoute` (plan 90) become a two-participant special case
   of an MPST route without changing its runtime behaviour?

## Implementation Notes

### Global Type Encoding

```typescript
type Comm<
  From extends string,
  To extends string,
  T,
  Cont extends GlobalType = GEnd
> = { _tag: 'Comm'; _from: From; _to: To; _payload: T; _cont: Cont };

type ChoiceBy<
  Decider extends string,
  Cases extends Record<string, GlobalType>
> = { _tag: 'Choice'; _decider: Decider; _cases: Cases };

type GEnd = { _tag: 'GEnd' };
type GlobalType = Comm<any, any, any, any> | ChoiceBy<any, any> | GEnd;
```

### Projection

```typescript
type Project<G extends GlobalType, P extends string> =
  G extends Comm<infer From, infer To, infer T, infer Cont>
    ? From extends P ? Send<T, Project<Cont, P>>
    : To   extends P ? Recv<T, Project<Cont, P>>
    : Project<Cont, P>  // P not involved — skip this step
  : G extends GEnd ? End
  : never;
```

The tricky case is `ChoiceBy`: when P is the decider, it becomes `Select<...>`; when P
is a receiver of the choice, it becomes `Branch<...>`; when P is neither, it is the
merge of all branches' projections (requires that merge to be well-defined).

### Known Hard Cases

- **Merge operator**: projecting a `ChoiceBy` for a non-deciding participant requires
  merging the projections of all branches. Merge is well-defined only when the branches
  are compatible (same structure up to the point of divergence). TypeScript cannot
  validate this constraint; it would need to be a runtime invariant.
- **Recursive global types**: TypeScript cannot express coinductive/recursive types
  natively. A depth counter will be needed for protocols with loops.
