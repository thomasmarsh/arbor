# Plan 81 — Default handler supervision (let-it-crash safety net)

## Context

`createServer` currently does not wrap handler invocations in any error boundary. An
unhandled exception inside a handler propagates as a rejected `Promise`, which — depending
on the runtime — either silently swallows the error, logs an unhandled rejection, or
crashes the process. The request hangs or produces no response.

The correct behavior for an HTTP server is well-established: uncaught handler errors
become 500 responses with a structured body, and the error is observable via the existing
`withMetrics` telemetry channel. This should be the **default**, not an opt-in wrapper.

Elixir/OTP calls this "let-it-crash" supervision: handlers are isolated; a failure in one
request cannot corrupt another. The equivalent in a sync/async JS server is a try/catch
around each dispatch that maps uncaught errors to structured responses.

## Goal

`createServer` wraps every handler invocation in a supervision boundary. Uncaught errors:

1. Produce a `500` response with a configurable body (default: `{ error: 'internal server error' }`).
2. Are passed to the existing pluggable error mapper before defaulting to 500 — so teams
   that already map `DatabaseError → 409` get that behavior for free.
3. Emit a `RequestMetric` via `withMetrics` if a metrics emitter is configured, marking
   the request as failed.
4. Are re-thrown (or passed to an `onError` callback) for external logging, not silently
   swallowed.

The supervision is **opt-out**: pass `{ supervise: false }` to `createServer` to get the
existing unguarded behavior (useful if the caller has its own outer try/catch).

## Approach

In `executeRoute` (or wherever the handler is called):

```typescript
try {
  const result = await handler(ctx);
  return result;
} catch (err) {
  // 1. Try error mapper
  const mapped = options.errorMap ? tryMapError(err, options.errorMap) : null;
  if (mapped) return { ...mapped, tag };

  // 2. Emit metric (if configured)
  options.onError?.(err, tag);

  // 3. Default 500
  return { status: 500, body: { error: 'internal server error' }, tag };
}
```

The `onError` callback replaces the need to re-throw: callers connect it to their logger.
Default: `console.error`.

## Pay-for-what-you-use note

The supervision boundary adds a single `try/catch` per request — zero allocation cost on
the happy path (V8 optimizes `try` blocks with no throw). The `onError` callback default
is `console.error`, which adds no runtime unless an error actually occurs.

## Change Surface

- `src/server/server.ts` — wrap handler invocation in `executeRoute`; add `onError` and
  `supervise` options to `CreateServerOptions`
- `src/server/server.test.ts` — test that uncaught handler errors produce 500; test that
  the error mapper is consulted first; test `onError` callback receives the error
- `examples/basic-server.ts` — demonstrate `onError` callback

## Dependencies

None. Self-contained in `server.ts`.

## Success Criteria

- Handler that throws produces `{ status: 500, body: { error: 'internal server error' } }`.
- Handler that throws a mapped error type produces the mapped status code.
- `onError` is called with the thrown error and the route tag.
- `{ supervise: false }` disables the boundary; exceptions propagate as before.
- `pnpm test && pnpm typecheck` green.
