# Plan 175 — `wsSessionRoute`: Route Factory + Server Dispatch

## Context

> **North star**: `plan/spec.ws-session-ergonomics.md` — shows the route declaration,
> server handler, and what `createWsSessionServer` looks like end-to-end.

Plans 173–174 add `IxSession` and its primitive operations to `core/`. This plan wires them
into the WebSocket route layer — a new route factory `wsSessionRoute<S>()` that exposes a
fully-typed `IxSessionOps` to the handler.

### Current state

`wsRoute(schema, path, { in: InSchema, out: OutSchema })` produces a flat two-message route.
The handler receives `{ channel: WsChannel<Recv<In, Send<Out>>> }` — no protocol ordering, just
a typed async iterable + send method.

`wsRoute` and `WsChannel` are **unchanged** by this plan. This plan adds a parallel path.

### New API surface

```typescript
// Route factory — S is the server-side session type
export function wsSessionRoute<
  ZS extends AnyObjectSchema,
  S extends Session,
>(
  schema: ZS,
  path: string,
  session: S,  // phantom value — carries S to the type system; undefined at runtime
): BuildableRouteNode<RouteNode<Infer<ZS>, [], WsSessionContext<S>, never, WsSessionMeta<S>>>

// Context type — handler receives this
export interface WsSessionContext<S extends Session> {
  ops: IxSessionOps
  // ops is untyped at the method level (same as buildIxSessionOps return type)
  // The session type S is carried by the route node and checked at the call site
  // via the handler type (WsSessionHandlerMap below)
  _sessionType: S  // phantom — never assigned at runtime; drives client-side Dual<S>
}

// Meta type — stored in the route node's _meta field
export interface WsSessionMeta<S extends Session> extends SessionMeta<S> {
  readonly __wsSession?: S  // phantom sentinel for getWsSessionMeta type guard
}
```

### Server dispatch

The existing `createWsServer` is unchanged. Add a parallel:

```typescript
export function createWsSessionServer<
  Route extends { tag: string },
  Map extends Record<string, WsSessionContext<any>>,
>(
  router: WsRouterContract<Route, Map>,
  handlers: WsSessionHandlerMap<Map, Route>,
): { handleConnection(tag, params, adapter): Promise<void> }

type WsSessionHandlerMap<Map, Routes> = {
  [Tag in keyof Map & string]: (
    ctx: WsSessionHandlerCtx<Routes, Tag, Map[Tag]>
  ) => Promise<void>
}

interface WsSessionHandlerCtx<Routes, Tag extends string, Ctx extends WsSessionContext<any>> {
  params: Omit<Extract<Routes, { tag: Tag }>, 'tag' | 'child' | 'query'>
  ops: IxSessionOps
}
```

The `handleConnection` implementation calls `buildIxSessionOps(adapter)` and passes `ops` to
the handler. The handler is responsible for calling `ops.close()` (or letting the IxSession
chain reach `End`).

```typescript
it('wsSessionRoute: server recv join, send welcome; client verifies', async () => {
  const JoinSchema = z.object({ username: z.string() })
  const WelcomeSchema = z.object({ roomId: z.string() })

  const route = wsSessionRoute(
    object({ tag: literal('ws/lobby') }),
    'ws/lobby',
    undefined as never,
  )
  const router = defineRoutes([route])

  const server = createWsSessionServer(router, {
    'ws/lobby': async ({ ops }) => {
      const join = await ops.recv(JoinSchema).run()
      await ops.send({ roomId: 'r1' }, WelcomeSchema).run()
    },
  })

  const [serverAdapter, clientAdapter] = createWsAdapterPair()
  void server.handleConnection('ws/lobby', {}, serverAdapter)

  const clientOps = buildIxSessionOps(clientAdapter)
  await clientOps.send({ username: 'alice' }, JoinSchema).run()
  const welcome = await clientOps.recv(WelcomeSchema).run()
  expect(welcome).toEqual({ roomId: 'r1' })
})
```

4. All existing `createWsServer` and `wsRoute` tests continue to pass.
5. `pnpm --filter @arbor/router test` passes in full.

## Notes

- The `session` parameter of `wsSessionRoute` is a phantom value — pass `undefined as never as S`
  in tests and `undefined as never as LobbySession` in production. The value is discarded; only
  the type matters. This pattern mirrors `sessionRoute` in `session-route.ts` which takes `_session: S`.
- The `_sessionType` field on `WsSessionContext<S>` is never assigned at runtime
  (`undefined as never`). It exists only to carry `S` into the handler map types so that future
  plans (176) can extract `Dual<S>` for the client. An alternative is using `InferSession` on
  the route node — either is fine as long as `S` is recoverable from the context map type.
- `buildIxSessionOps` is imported from `core/ix-session-ops.ts`. The import is from `core/` into
  `contexts/`, which is the allowed direction.
