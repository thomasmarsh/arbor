# PLAN.117 — Loaders, Meta, and Scroll in the Router Store

## Context

Plans 101 (loaders), 111 (meta tags), and 112 (scroll restoration) describe standalone
subsystems each managed by their own hooks or classes. In the TCA architecture, these are
not separate concerns — they are `RouterEnv` capabilities and `RouterAction` branches.

This plan integrates all three into the `routerReducer` and `RouterEnv`, replacing the
need for separate imperative lifecycle hooks for teams using the TCA path.

## Goal

Extend `RouterEnv` and `routerReducer` (plan 114) with:

### 1. Loaders as `RouterEnv` capability

```typescript
export interface RouterEnv<Route> {
  runLoaders: (route: Route) => Effect<RouterAction<Route>>;
  // ^ already in plan 114; this plan specifies the loader data flow
}

// Loader data lives in RouterState.committed:
type RouterState<Route, Data> =
  | { tag: 'committed'; route: Route; data: Data }
  | ...;
```

`runLoaders` fires all loaders for the ancestor chain of `route` in parallel and resolves
to a single `{ tag: 'loader-resolved', route, data }` action. Loader data is typed as the
intersection of all ancestor loaders' resolved types (same design decision as plan 101 —
follow whichever placement was decided there).

### 2. Meta tags as `RouterEnv` capability

```typescript
export interface RouterEnv<Route> {
  updateMeta?: (title: string, description?: string) => Effect<never>;
}
```

In the `loader-resolved` branch of the reducer:
```typescript
case 'loader-resolved': {
  $.state = { tag: 'committed', route: action.route, data: action.data };
  const meta = metaMap?.[action.route.tag];
  if (meta && env.updateMeta) {
    const { title, description } = meta(action.route);
    return env.updateMeta(title, description);
  }
  return null;
}
```

`liveRouterEnv` provides:
```typescript
updateMeta: (title, description) => Effect.of(() => {
  document.title = title;
  document.querySelector('meta[name="description"]')?.setAttribute('content', description ?? '');
}),
```

### 3. Scroll restoration as `RouterEnv` capability

```typescript
export interface RouterEnv<Route> {
  saveScroll:    (key: string) => Effect<never>;
  restoreScroll: (key: string) => Effect<never>;
}
```

In the `navigate` branch: `saveScroll(currentKey)` before pushing history.
In the `loader-resolved` branch: `restoreScroll(newKey)` after committing.

`liveRouterEnv` provides:
```typescript
saveScroll: (key) => Effect.of(() => {
  sessionStorage.setItem(`scroll:${key}`, String(window.scrollY));
}),
restoreScroll: (key) => Effect.of(() => {
  const y = Number(sessionStorage.getItem(`scroll:${key}`) ?? 0);
  window.scrollTo(0, y);
}),
```

### 4. History stack in `RouterState`

```typescript
type RouterState<Route, Data> =
  | { tag: 'committed'; route: Route; data: Data; history: Route[]; future: Route[] }
  | ...;
```

`navigate` prepends `current` to `history` and clears `future`.
`replace` does NOT modify history.
`popstate` is treated as external navigation — sync from URL, no history modification.

## Success Criteria

- `loader-resolved` triggers `updateMeta` when a meta map is provided.
- `loader-resolved` triggers `restoreScroll` after committing.
- History stack correctly grows on `navigate`; stays the same on `replace`.
- All three capabilities are mockable via `mockRouterEnv` (no `window`/`document` access
  in the reducer itself — only in `liveRouterEnv`).
