# Plan 69 — Enforce optional segment ordering at definition time

## Context

The segment matcher is greedy with no backtracking. An `opt-str` or `opt-num` segment
always consumes the next available URL segment if one exists, regardless of what segment
follows it in the same path pattern. When an optional segment precedes a required one, the
matcher silently produces wrong results.

The test suite documents this explicitly (segments.test.ts line 225):

```ts
it('first optional greedily consumes the lone value', () => {
  // opt-str captures any non-null value, so '42' becomes name, not id
  expect(matchSegments(segs, ['42'], {})).toEqual({ params: { name: '42' }, rest: [] });
});
```

The same problem occurs in section trees. A section with an optional prefix cannot be
followed by children that require segments:

```ts
// Intended: /en/users and /users both match
section(':lang?', [
  route(UserList, 'users'),  // never reachable from /users
])
```

URL `/users` → `opt-str` greedily takes `'users'` as `lang`, `rest = []`, children get
`[]` and fail to match `lit 'users'`. The route 404s silently.

### Why not backtracking?

Full backtracking would fix all cases but adds meaningful complexity to `matchSegments` and
changes its time complexity. The practical need for optional-before-required in a single
path string is narrow — the standard solution in every routing library is to model it as
nested routes instead:

```ts
// Correct: separate section handles the prefix
section(':lang', [route(UserList, 'users')]),  // /en/users
route(UserList, 'users'),                        // /users
```

The cleaner fix is to **make the illegal ordering unrepresentable** at route definition
time, catching the mistake as a thrown error before any requests are handled.

## Goal

`parseSegments` throws if an optional segment (`opt-str`, `opt-num`) appears before any
non-wildcard segment in the same path string. The error message explains the constraint and
suggests the nested-route alternative.

## Allowed orderings

| Pattern | Allowed | Reason |
| ------- | ------- | ------ |
| `users/:id?` | ✓ | optional is last |
| `users/:id?/*rest` | ✓ | optional before wildcard only — wildcard always matches remaining |
| `:lang?/users` | ✗ | optional before required literal |
| `:lang?/:page` | ✗ | optional before required param |
| `:a?/:b?` | ✗ | optional before optional (same greedy ambiguity) |

`opt-str`/`opt-num` before `wildcard` is valid: the wildcard absorbs whatever is left after
the optional segment has had a chance to match, so there is no greedy-consumes-wrong-field
problem.

## Implementation

Add a post-parse validation step inside `parseSegments`:

```ts
export function parseSegments(path: string): Segment[] {
  const segments = path.split('/').filter(Boolean).map(parseSingle);
  validateOptionalOrdering(segments, path);
  return segments;
}

function validateOptionalOrdering(segments: Segment[], path: string): void {
  let sawOptional = false;
  for (const seg of segments) {
    const isOptional = seg.kind === 'opt-str' || seg.kind === 'opt-num';
    const isWildcard = seg.kind === 'wildcard';
    if (isOptional) {
      sawOptional = true;
    } else if (sawOptional && !isWildcard) {
      throw new Error(
        `Invalid path "${path}": optional segment must be last (only a wildcard may follow). ` +
        `Use nested routes to model optional prefixes.`,
      );
    }
  }
}
```

- `src/core/segments.test.ts` — add tests that expect the throw for bad orderings; update

## Dependencies

None. Self-contained in `segments.ts`.

## Success Criteria

- `parseSegments(':lang?/users')` throws with a clear message.
- `parseSegments(':a?/:b?')` throws.
- `parseSegments('users/:id?')` does not throw.
- `parseSegments(':id?/*rest')` does not throw.
- All existing tests that exercise valid orderings continue to pass.
- `pnpm test && pnpm typecheck` green.
