# Order of execution

```text
135 ─┐
     ├──▶ 137 ──▶ 138
136 ─┘
```

**135 and 136 are independent and can be done in either order or parallel sessions.** 137 waits on 135. 138 waits on all three.

## Per-task instructions

**Start with `plan 135` — Landscape Survey**

Pure research. No code changes to `src/`. You'll need to read the docs/source for TypeBox, ArkType, and skim Zod v4's changelog. The deliverable is a decision appended to the plan file answering: adopt an existing library, or build custom? Until this is answered, 137 and 138 are speculative.

Signal at session start: `plan 135`

**Then `plan 136` — Schema Scope Audit (can overlap with 135)**

Pure design. Audit the 10 attachment points already enumerated in the plan (all known — no discovery needed). Design a `RouteSchema` shape sketch. The deliverable is a design decision appended to the plan file, not implementation. Particularly: does the unified schema become a new `RouteNode` type parameter, or stay in \_meta?

Signal at session start: `plan 136`

**Then plan 137 — Refinement Taxonomy**

Design document + ~10-line scratch file (Q5 branded types only). The core question is: for route boundary schemas specifically, do format/bound constraints (email, uuid, min, max) cover the realistic majority of what users need? If yes, owning the schema format is viable. If no, the escape hatch design from Q2 becomes the main deliverable.

Signal at session start: `plan 137`

**Finally `plan 138` — Interop Design**

The last spike. By this point 135 has told us which library (if any) to build on, 136 has told us what `RouteSchema` looks like, and 137 has told us what the escape hatch needs to be. 138 designs the fluent builder and `fromZod()` adapter and proves them with a short scratch file. This is the last gate before any implementation planning.

Signal at session start: `plan 138`

## Exit condition

After all four are done, 133 becomes the synthesis point: write a short implementation plan (or a close-as-not-worth-it recommendation) based on the combined findings. That will be a separate planning task.
