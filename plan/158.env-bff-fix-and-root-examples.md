# Plan 158 — Fix `.env.bff` Typo + Root Mode-Env Documentation

## Context

`.env.bff` has a leading space before `ARBOR_AUTH_DISABLED`:

```
 ARBOR_AUTH_DISABLED=false   ← leading space
```

`dotenv-cli` and most dotenv parsers treat this as an invalid key and silently skip the line, meaning `ARBOR_AUTH_DISABLED` is never set when running `dev:bff`. The stack starts with auth effectively in an unknown state.

Additionally, the four root mode-env files (`.env.mock`, `.env.oidc`, `.env.staging`, `.env.bff`) have no companion `.example` files and no inline documentation explaining which variables they own or what the valid values are. Developers discover the set of recognized `VITE_AUTH_MODE` values by grep or word of mouth.

## Goal

1. Fix the leading-space typo in `.env.bff`.
2. Add a root-level `.env.example` (committed, comprehensive) that lists every variable that appears in the four mode-env files, with valid values and brief comments.

The `.env.example` file serves as a single authoritative reference — `turbo.json`'s `globalEnv` list plus this file should together make the env surface self-documenting.

## Change Surface & T-shirt Sizing

**XS** — two file changes, no logic.

| File | Change |
|------|--------|
| `.env.bff` | Remove leading space from `ARBOR_AUTH_DISABLED=false` |
| `.env.example` | New file — annotated listing of all mode-env variables |

## Dependencies

None.

## Success Criteria

- `grep -c 'ARBOR_AUTH_DISABLED' .env.bff` returns 1 and parsing the file produces the key correctly (`node -e "require('dotenv').config({path:'.env.bff'}); console.log(process.env.ARBOR_AUTH_DISABLED)"` prints `false`).
- `.env.example` exists at the workspace root and covers every variable that appears in `.env.mock`, `.env.oidc`, `.env.staging`, `.env.bff`.
