# Plan 86 — Lint rules: add switch-exhaustiveness-check, no-console, reportUnusedDisableDirectives; fix all suppressions

## Context

The project runs `strictTypeChecked + stylisticTypeChecked` from typescript-eslint — a
comprehensive baseline. Lint currently passes clean. However an audit reveals two
categories of work:

**Missing rule.** `@typescript-eslint/switch-exhaustiveness-check` is not part of either
preset, yet the codebase is built around discriminated unions over segment kinds (`lit`,
`str`, `num`, `opt-str`, `opt-num`, `wildcard`) and parse diagnostics. A missing case in
a switch over these unions produces a silent runtime defect, not a compile error.
Every new segment kind added later is a latent landmine. This rule closes that gap.

**Overly broad suppressions.** 12 source files carry `eslint-disable` directives. 8
of them are file-level `/* eslint-disable @typescript-eslint/no-explicit-any */` on
production files. These are broader than necessary — they silence the rule for the
entire file instead of the specific lines where `any` is structurally required. The
project rule is "do not use `as any` without a documented comment reason"; the
suppressions don't document reasons. Three line-level suppressions have the same
problem.

The fix plan has three independent phases that can be executed in any order.

---

## Goal

1. Add `switch-exhaustiveness-check` to guard discriminated union exhaustiveness.
2. Add `no-console` to block accidental debug logging in production source.
3. Enable `reportUnusedDisableDirectives` to keep suppression comments honest.
4. Convert all file-level `eslint-disable` blocks to targeted line-level comments with
   reasons, or eliminate them by fixing the underlying code.
5. Fix the two `no-unused-vars` suppressions in `route-node.test.ts` by prefixing
   variables with `_` (already in the ignore pattern).

---

## Change Surface

### Phase A — Config changes (`eslint.config.js`)

**A1. Add `switch-exhaustiveness-check`.**

```diff
<<<<<<< SEARCH
      '@typescript-eslint/no-import-type-side-effects': 'error',
=======
      '@typescript-eslint/no-import-type-side-effects': 'error',
      '@typescript-eslint/switch-exhaustiveness-check': [
        'error',
        { considerDefaultExhaustiveForUnions: true },
      ],
>>>>>>> REPLACE
```

`considerDefaultExhaustiveForUnions: true` means a `default:` clause satisfies the
check, so existing switches with `default` (e.g. `segmentToParam` in
`generate-spec.ts`) are fine as-is. The rule flags switches over a union that have
neither exhaustive cases nor a `default`.

**A2. Add `no-console` for source files; relax in tests and examples.**

```diff
<<<<<<< SEARCH
    rules: {
      '@typescript-eslint/no-explicit-any': 'error',
=======
    rules: {
      'no-console': 'error',
      '@typescript-eslint/no-explicit-any': 'error',
>>>>>>> REPLACE
```

Then add a second config object (after the main one) to relax it for non-source files:

```javascript
{
  files: ['packages/**/*.test.ts', 'packages/**/examples/**/*.ts'],
  rules: {
    'no-console': 'off',
  },
},
```

**A3. Enable `reportUnusedDisableDirectives`.**

In the `defineConfig([ ... ])` wrapper, add a top-level option:

```javascript
export default defineConfig([
  { linterOptions: { reportUnusedDisableDirectives: 'error' } },
  // ... existing blocks
]);
```

This means once an underlying violation is fixed, the suppression comment must also
be removed — otherwise lint fails.

### Phase B — Fix production source suppressions

The 8 files listed below carry `/* eslint-disable @typescript-eslint/no-explicit-any */`
at the top. Remove the file-level disable and replace with targeted line-level
comments. The justification pattern to use:

```typescript
// eslint-disable-next-line @typescript-eslint/no-explicit-any -- <reason>
```

| File | Typical reason |
|------|---------------|
| `src/core/walk.ts` | `WalkNode` widened type; `z.ZodObject` requires `any` shape param |
| `src/core/route-node.ts` | `RouteNode` phantom interface uses `any` for variance |
| `src/core/define-routes.ts` | Generic overloads for builder return types |
| `src/core/router-contract.ts` | `RouterContract` needs `any` for covariant children |
| `src/contexts/http-context.ts` | HTTP context generics; Zod `any` shape |
| `src/contexts/openapi-context.ts` | OpenAPI context generics |
| `src/client/fetch-client.ts` | Client generics; response body typing |
| `src/openapi/generate-spec.ts` | Zod shape introspection |

For `src/core/walk.ts` there is an additional line-level suppress at line 30:

```typescript
// eslint-disable-next-line @typescript-eslint/no-unsafe-return, @typescript-eslint/no-unsafe-member-access
return node._meta?.querySchema;
```

This is fixable: `WalkNode._meta` is typed as `any`. Narrow it to:

```typescript
export type WalkNode = RouteNode<
  unknown,
  RouteNode<unknown, any, any, any, any>[],
  any,
  any,
  { querySchema?: z.ZodObject<any, any> } & Record<string, unknown>
>;
```

That gives `node._meta?.querySchema` a known type and eliminates both suppressions.

For `src/index.test.ts` and `src/core/define-routes.test.ts` (also file-level
`no-explicit-any`), the `as any` cast in `define-routes.test.ts:113` is intentionally
testing invalid runtime input. Keep the `as any` but add a line-level comment:

```typescript
// eslint-disable-next-line @typescript-eslint/no-explicit-any, @typescript-eslint/no-unsafe-argument -- deliberate invalid input for runtime test
return router.print(route as any, undefined as any);
```

### Phase C — Fix test file suppressions

**C1. `src/core/route-node.test.ts:35,46` — `no-unused-vars`.**

Both suppressions are on `stubRouter` variables assigned solely for `expectTypeOf`
type extraction. The fix is a one-character rename:

```typescript
// Before
const stubRouter = { ... };
expectTypeOf<InferRoute<typeof stubRouter>>()...

// After — remove suppress comment entirely
const _stubRouter = { ... };
expectTypeOf<InferRoute<typeof _stubRouter>>()...
```

The `varsIgnorePattern: '^_'` config already covers this. No config change needed.

**C2. `src/core/walk.test.ts` and `src/contexts/openapi-context.test.ts` — `no-non-null-assertion`.**

Add a test-file config override in `eslint.config.js` instead of per-file blocks:

```javascript
{
  files: ['packages/**/*.test.ts'],
  rules: {
    '@typescript-eslint/no-non-null-assertion': 'off',
  },
},
```

Then remove the two file-level `/* eslint-disable @typescript-eslint/no-non-null-assertion */`
comments.

## Success Criteria

1. `pnpm lint` exits 0 with the new config applied.
2. No file-level `/* eslint-disable */` blocks remain in `src/`.
3. Every remaining `eslint-disable-next-line` has a `-- <reason>` suffix.
4. `@typescript-eslint/switch-exhaustiveness-check` is active and
   `pnpm typecheck && pnpm test` still pass.
5. `reportUnusedDisableDirectives: 'error'` is active — any future suppression left
   behind after a fix causes lint to fail.
6. `no-console` is active for source files; test files and examples are exempt.
