# 182. Spike ‚Äî DnD Framework Integration Pattern

## Context

Column reordering (plan 183) and potentially task reordering require drag-and-drop. Our component pattern wraps state in a TCA `useStore` + `send` idiom. DnD libraries expose callbacks (`onDragEnd`, `onDrop`) that produce imperative results; we need to bridge those into `send(action)` cleanly, without leaking library internals into store action shapes.

The main design question: what does the minimal adapter surface look like so that DnD usage sites only need to express "when item A drops onto position B, send this action" ‚Äî with no DnD library types appearing at the call site?

Candidate libraries:
- `@dnd-kit/core` + `@dnd-kit/sortable` ‚Äî headless, works cleanly alongside MUI, actively maintained
- `react-beautiful-dnd` ‚Äî good UX defaults but deprecated; large bundle; known MUI conflicts
- Native HTML5 DnD ‚Äî no dependency but poor touch support and janky default visuals

## Goal

1. Choose a library with rationale.
2. Design a minimal adapter hook: something like `useSortable<K>(items: T[], getId: (t: T) => K, onReorder: (fromId: K, toId: K) => void)` that returns MUI-compatible drag handle props and a sortable container ref ‚Äî caller wires `onReorder` to `send(...)`.
3. Prototype the hook against a sortable list wired to a test store; verify it compiles and runs.
4. Commit a design note to `plan/` summarizing the chosen pattern and any constraints discovered.

## Appendix ‚Äî Spike Findings

Scratch file: `apps/ui/src/ledger/_spike-dnd.tsx`

### Library choice: `@dnd-kit/core` + `@dnd-kit/sortable`

**Rationale:**
- Headless ‚Äî no bundled styles; composes cleanly with MUI.
- Actively maintained; `react-beautiful-dnd` is archived/deprecated.
- First-class keyboard navigation via `KeyboardSensor` + `sortableKeyboardCoordinates` out of the box.
- TypeScript types are good; no need for `@types/*`.
- Touch support via `PointerSensor` (unified pointer events model).
- Bundle: ~15 kB gzipped (core + sortable + utilities) vs. ~30 kB for rbd.

### Adapter shape

Two exports; neither leaks any `@dnd-kit` type to callers:

```ts
// 1. Container ‚Äî wraps the sortable region; stateless (store owns order)
function DndSortWrapper<K extends string | number>(props: {
  ids: readonly K[];
  onReorder: (fromId: K, toId: K) => void;
  direction?: 'horizontal' | 'vertical';
  children: React.ReactNode;
}): React.ReactElement

// 2. Per-item hook ‚Äî called once per sortable child inside <DndSortWrapper>
function useDndItem(id: string | number): DragItemProps

interface DragItemProps {
  ref: (node: HTMLElement | null) => void;
  style: React.CSSProperties;        // CSS transform + transition + opacity
  isDragging: boolean;
  handleProps: Record<string, unknown>; // pointer/keyboard listeners + aria attrs
}
```

### Wiring pattern (plan 183 usage site)

```tsx
// Store action (added to LedgerAction):
| { tag: 'reorderColumn'; fromId: ColId; toId: ColId }

// Reducer case:
case 'reorderColumn': {
  const from = $.state.columnOrder.indexOf(action.fromId);
  const to   = $.state.columnOrder.indexOf(action.toId);
  if (from !== -1 && to !== -1)
    $.state.columnOrder = [...arrayMove($.state.columnOrder, from, to)];
  break;
}

// Component:
<DndSortWrapper ids={$.state.columnOrder} onReorder={(from, to) => send({ tag: 'reorderColumn', fromId: from, toId: to })}>
  {$.state.columnOrder.map(id => <SortableColHeader key={id} id={id} />)}
</DndSortWrapper>

// Per-header:
const { ref, style, handleProps } = useDndItem(id);
<TableCell ref={ref} style={style} {...handleProps}>{label}</TableCell>
```

### Footguns discovered

1. **`readonly K[]` from valtio Snapshot** ‚Äî `$.state.columnOrder` is `readonly ColId[]` in JSX. `DndSortWrapper.ids` must accept `readonly K[]`; the internal `SortableContext.items` cast requires `as unknown as (string | number)[]`.

2. **`arrayMove` returns `readonly T[]`** ‚Äî the store mutation `$.state.columnOrder = arrayMove(...)` needs a spread: `[...arrayMove(...)]` to satisfy the mutable target.

3. **`DndSortWrapper` is stateless by design** ‚Äî it holds no local copy of `ids`. During an active drag, `@dnd-kit` moves items visually via CSS transforms, not by re-rendering the list; the store is updated only on `onDragEnd`. This is correct behaviour ‚Äî do not add local `sortedIds` state to the adapter. If intermediate animations are needed, add a `draggingId` field to `LedgerState` instead.

4. **`handleProps` spread on MUI components** ‚Äî `handleProps` is typed as `Record<string, unknown>`. MUI components accept `React.HTMLAttributes`, so cast at the spread site: `{...(handleProps as React.HTMLAttributes<HTMLElement>)}`. An eslint `react/jsx-props-no-spreading` suppression is needed per-site.

5. **Keyboard accessibility** ‚Äî `KeyboardSensor` is included in the adapter's `useSensors` call. No extra work needed at the call site; the table header cells get correct `aria-*` attributes via `handleProps`.
