# PLAN.165 — Standard Schema for Body and Response

## Context

Plan 164 removes Zod from the core routing path (params, query, walk, OpenAPI param
generation). This plan decouples body and response schemas from Zod in `http-context.ts`,
`server.ts`, and `fetch-client.ts`.

This is a parallel track to plan 164 — it can be developed independently once plan 163
(scratch proof) passes. It does not touch `core/`.

The user-facing change: `httpRoute()` body, response, header, and cookie schemas accept
any value implementing `StandardSchemaV1<T>` rather than `z.ZodType` / `z.ZodObject<any,any>`.
Zod v4, Valibot, and ArkType all implement this already — existing call sites that pass
Zod schemas continue to work unchanged.

## Goal

Replace `z.ZodType` and `z.ZodObject<any,any>` in `http-context.ts`, `server.ts`,
`fetch-client.ts`, `sse-context.ts`, and `ws-context.ts` with the minimal `UserSchema<T>`
structural interface defined in plan 138 findings.

## Implementation

### Define `UserSchema<T>` in `schema.ts`

Add alongside the native schema types (do not import from `@standard-schema/spec`):

```typescript
// Minimal structural interface for user-controlled schemas.
// Zod v4, Valibot, and ArkType all satisfy this via structural typing.
export interface UserSchema<T = unknown> {
  readonly '~standard': {
    readonly validate: (value: unknown) =>
      | { readonly value: T }
      | { readonly issues: ReadonlyArray<{ readonly message: string }> };
  };
}

export type AnyUserSchema = UserSchema<unknown>;
```

### Update `http-context.ts`

Replace all `z.ZodType` and `z.ZodObject<any,any>` occurrences with `AnyUserSchema`:

```diff
- bodySchema?: z.ZodType;
- responseSchemas?: Record<number, z.ZodType>;
- querySchema?: z.ZodObject<any, any>;
- headerSchema?: z.ZodObject<any, any>;
- cookieSchema?: z.ZodObject<any, any>;
- responseHeaderSchemas?: Record<number, z.ZodObject<any, any>>;
- responseCookieSchemas?: Record<number, z.ZodObject<any, any>>;
+ bodySchema?: AnyUserSchema;
+ responseSchemas?: Record<number, AnyUserSchema>;
+ querySchema?: AnyUserSchema;
+ headerSchema?: AnyUserSchema;
+ cookieSchema?: AnyUserSchema;
+ responseHeaderSchemas?: Record<number, AnyUserSchema>;
+ responseCookieSchemas?: Record<number, AnyUserSchema>;
```

### Update validation call sites

`server.ts` calls `schema.safeParse(value)`. Replace with the Standard Schema call:

```diff
- const r = schema.safeParse(value);
- if (!r.success) { ... r.error.issues ... }
+ const r = schema['~standard'].validate(value);
+ if ('issues' in r) { ... r.issues ... }
```

`fetch-client.ts` similarly calls `schema.safeParse(responseBody)` — same replacement.

### Update `generate-spec.ts` body/response path

Body and response schemas no longer use `z.toJSONSchema()`. Instead, `httpRoute()` options
accept an optional `jsonSchema` field alongside each user schema, or the schema may expose
a `toJsonSchema()` method (Valibot, TypeBox). If neither is present, the OpenAPI output for
that position is omitted with a comment. This is acceptable — the structural param/query
OpenAPI is lossless; body/response is best-effort.

For the transition period, Zod v4 users can pass `jsonSchema: z.toJSONSchema(mySchema)`
alongside their body schema explicitly. A follow-on plan can autodetect Zod and call
`z.toJSONSchema()` via the `@arbor/router/zod` subpath export.

## Learnings from Plan 164

### walk.ts querySchema path is still fully Zod

`resolveQuerySchema`, `validateSchema`, `extractQueryParams`, and `getShape` in `walk.ts`
are all still Zod. Plan 165 must migrate these to the Standard Schema protocol:

```diff
- const r = schema.safeParse(rawQuery);
- if (!r.success) { diag?.push({ issues: r.error.issues.map(...) }); }
+ const r = schema['~standard'].validate(rawQuery);
+ if ('issues' in r) { diag?.push({ issues: r.issues.map(i => ({ message: i.message })) }); }
```

The `getShape` and `getTag` helpers for Zod are now dead code after 164 (native `getTag`
replaced the Zod one). Only `getShape` survives for `extractQueryParams` — this function
must be rewritten to use the Standard Schema validation result rather than reading the schema
shape directly.

### `InferUserSchema<S>` needed for httpRoute return type

`httpRoute` currently uses `z.infer<Q>`, `z.infer<H>`, `z.infer<CK>` to derive context
types. Add this helper alongside `UserSchema<T>` in `schema.ts`:

```typescript
export type InferUserSchema<S> = S extends UserSchema<infer T> ? T : never;
```

Then replace `z.infer<Q>` → `InferUserSchema<Q>` in `http-context.ts`.

### `buildHeaderParams` requiredness cannot be derived from AnyUserSchema

`generate-spec.ts:buildHeaderParams` currently uses `instanceof z.ZodOptional` to mark
header fields as required or optional. The Standard Schema interface has no structural way
to express this. **Decision required**: either (a) treat all header schemas as `required: true`
by default, or (b) add an optional `required?: Record<string, boolean>` alongside the
header schema in httpRoute options. Option (a) is simpler and correct for auth headers.

### Size estimate is too low

Plan 164 was tagged "M" but touched 40+ files including all test suites. Plan 165 will
similarly need to migrate every test file that currently constructs `z.ZodType` values for
body/response schemas. The functional change is small but the mechanical migration scope is
the same order of magnitude. Re-tag as **M**.

### Build dist explicitly after source changes

`pnpm --filter @arbor/router build` runs `tsc -b` against `tsconfig.json` (`noEmit: true`)
and does NOT update `dist/`. After source changes, rebuild with:
```
tsc -p framework/router/tsconfig.build.json
```
The API imports from `dist/`, so without this step the runtime will use stale exports.
This caused the `integer` export to be missing in the API after plan 164 landed.
