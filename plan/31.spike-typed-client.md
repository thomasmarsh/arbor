# PLAN.31 MEMO — Typed HTTP API Client Architecture

## Q1: Can `_ctxMap` serve as the client's type source without leaking server-only code?

Yes. The client needs exactly one thing from the router: `{ _type: Route; _ctxMap: Map; print(r: Route): string }`. This is already the `RouterArg` shape in `fetch-client.ts`. It contains no reference to `HandlerMap`, `createServer`, or any Node-specific API. The shared boundary is `RouteNode` + `HttpContext`, both of which are pure type definitions with a zod dependency — safe for browser bundles.

The one runtime import the client makes from `walk.ts` (`getTag`) is also browser-safe: it reads `schema.shape` to extract a `ZodLiteral` value. No server-specific code, no `fs`, no Node globals.

**Answer: yes, cleanly separated already.**

## Q2: Which generation strategy preserves the most type fidelity?

The `RouteNode` / `_ctxMap` path already achieves full fidelity for every case that matters:

| Feature | RouteNode path | OpenAPI JSON path |
|---|---|---|
| Path params | ✅ `Extract<Route, { tag: T }>` | ❌ lost (OpenAPI has string schemas) |
| Discriminated response by status | ✅ `HttpResponseUnion<Map[T]['response']>` | ⚠ possible with `openapi-typescript` but lossy |
| Request body | ✅ `Map[T]['body']` | ⚠ requires codegen step |
| Request headers (Plan 30) | ✅ `Map[T]['headers']` | ❌ not representable in standard OpenAPI |
| Response headers (Plan 29) | ✅ via `HttpResponseUnion` (gap: client currently uses `ResponseUnion`) | ❌ |
| Zero codegen | ✅ | ❌ |

**Answer: RouteNode / `_ctxMap`. No codegen, tightest inference, already working.**

## Q3: Recommended package structure and client call signature

### Package structure

Keep the client in `@arbor/router`. Add `"./client"` and `"./server"` subpath exports to `package.json` (the `"./openapi"` subpath already exists). Tree-shaking and subpath imports then prevent server code from reaching browser bundles without a separate package to maintain.

```json
"./client": {
  "import": "./dist/client/index.js",
  "types": "./dist/client/index.d.ts"
},
"./server": {
  "import": "./dist/server/index.js",
  "types": "./dist/server/index.d.ts"
}
```

### Call signature

The current API is `client.fetch(route, body?)`. Consolidate into an `options` object to thread request headers (Plan 30) without a positional explosion:

```typescript
client.fetch(route)                            // GET, no body, no headers
client.fetch(route, { body })                  // POST with body
client.fetch(route, { body, headers })         // POST with body + typed request headers
client.fetch(route, { headers })               // GET with typed request headers
```

### Identified gap

The client currently uses `ResponseUnion` (core, no header fields) instead of `HttpResponseUnion` (http-context, carries response headers for status codes that declare them). The implementation plan should switch this.

## Prototype (type-level only, no runtime)

Verifies the preferred shape is expressible:

```typescript
import type { HttpContext, HttpResponseUnion } from '../contexts/http-context.js';

type RequestOpts<Ctx extends HttpContext<any, any, any, any, any>> =
  [Ctx['body']] extends [never]
    ? [Ctx['headers']] extends [never] ? {} : { headers: Ctx['headers'] }
    : [Ctx['headers']] extends [never]
    ? { body: Ctx['body'] }
    : { body: Ctx['body']; headers?: Ctx['headers'] };

type TypedClient<
  Route extends { tag: string },
  Map extends Record<string, HttpContext<any, any, any, any, any>>,
> = {
  fetch<Tag extends keyof Map & string>(
    route: Extract<Route, { tag: Tag }>,
    options?: RequestOpts<Map[Tag]>,
  ): Promise<HttpResponseUnion<Map[Tag]['response']>>;
};
```

This compiles. The `Extract<Route, { tag: Tag }>` constrains the route value to exactly the params for that tag; `HttpResponseUnion` gives the discriminated response union including response headers.

## Recommendation

Open an implementation plan with scope:
1. Add `"./client"` and `"./server"` subpath exports to `package.json`
2. Change `client.fetch(route, body?)` to `client.fetch(route, options?)` — `{ body?, headers? }`
3. Switch client response type from `ResponseUnion` → `HttpResponseUnion` to expose response headers
4. Move `createClient` export from `src/index.ts` to `src/client/index.ts`; same for `createServer`
