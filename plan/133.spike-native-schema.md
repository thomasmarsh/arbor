# PLAN.133 (Spike) — Native Schema Representation

## Context

`@arbor/router` is a **library**. It does not define schemas — users bring their own. When
a user writes `route(z.object({ id: z.string() }), ':id')`, the Zod schema is authored and
owned by the user. Arbor receives it, extracts the TypeScript type via `z.infer<S>`, and
stores it for runtime validation and OpenAPI generation.

Today arbor **requires** that user schemas be `z.ZodObject`. This constrains what users can
bring: their schema must be a Zod object, not TypeBox, not ArkType, not a plain type. The
question is whether this constraint is worth keeping, and if not, what arbor should require
instead.

Zod schemas play two roles in the user's hands:

1. **Type inference** — `z.infer<S>` extracts the TypeScript type from the schema the user wrote.
2. **Runtime validation** — `.safeParse()` / `.parse()` validates incoming data using the
   user's validation rules, including any `.refine()` calls the user added.

For role 1, Zod is convenient but not special. For role 2, Zod is the user's choice — arbor
is just the messenger that calls `.safeParse()` on their behalf.

Zod's fluent builder API (`z.object({...}).refine(fn)`) is a method chain over a mutable
internal structure — it is not data. Arbor cannot serialize a Zod schema to JSON, compare
two schemas structurally, or generate an OpenAPI spec from `.refine(fn)` calls. It must use
`zod-to-json-schema` to extract what it can, losing any refinements that involve functions.

The question is: **should arbor control the schema representation it accepts**, trading user
flexibility (bring any Zod schema) for introspection power (serialization, contract diffing,
protocol semversioning, OpenAPI accuracy)? And if so, how do we avoid asking users to rewrite
their schemas?

`RouteNode.schema: z.ZodObject<any, any> | null` is where that coupling lives.

## Questions to Answer

**Q1 — What would a native schema look like?**

Candidate: a `Schema<T>` discriminated union that mirrors Zod's types but is a plain
data structure:

```typescript
type Schema<T> =
  | { kind: 'string' }                     // Schema<string>
  | { kind: 'number' }                     // Schema<number>
  | { kind: 'integer' }                    // Schema<number> but integer-constrained
  | { kind: 'boolean' }                    // Schema<boolean>
  | { kind: 'literal'; value: T }          // Schema<T> for string/number/boolean literals
  | { kind: 'object'; fields: FieldMap<T> }// Schema<T> for object shapes
  | { kind: 'array'; item: Schema<E> }     // Schema<E[]>
  | { kind: 'optional'; inner: Schema<T> } // Schema<T | undefined>
  | { kind: 'nullable'; inner: Schema<T> } // Schema<T | null>
  | { kind: 'union'; members: Schema<T>[] }// Schema<T1 | T2 | ...>
  | { kind: 'discriminated'; field: string; members: DiscUnion<T> }
  | { kind: 'enum'; values: T[] }          // Schema<'a' | 'b'>
  | { kind: 'brand'; inner: Schema<T>; brand: string }; // Schema<T & Brand>
```

Can `Schema<T>` achieve the same type inference as `z.ZodType<T>` without Zod? The key
test: does `route(schema, 'users')` infer param types correctly when `schema` is a
`Schema<T>` instead of a `z.ZodObject<T>`?

**Q2 — Compile to Zod**

Can a `toZod(schema: Schema<T>): z.ZodType<T>` function be written that produces an
equivalent Zod schema for runtime validation? If yes:

- The core uses `Schema<T>` for type inference and tree construction.
- `toZod(schema)` is called once at server startup to produce validators.
- `generateSpec()` uses `Schema<T>` directly (no longer needs the `zod-to-json-schema` converter).
- External consumers who already use Zod can still pass Zod schemas via a `fromZod(z)` adapter.

**Q3 — Ergonomics comparison**

Zod today:
```typescript
route(z.object({ id: z.string(), age: z.number().int().min(0) }), 'users/:id')
```

Native schema:
```typescript
route(object({ id: string(), age: integer().min(0) }), 'users/:id')
// or: route({ id: 'string', age: { kind: 'integer', min: 0 } }, 'users/:id')
```

Which form is more readable? Does losing Zod's method chain (`z.string().email().min(5)`)
hurt ergonomics for complex schemas? Would native schema be better for simple route params
but worse for complex body validation?

**Q4 — Migration path**

`RouteNode.schema: z.ZodObject<any, any> | null` is baked into the public type. Any
native schema would need to either:
- Replace `ZodObject` in `RouteNode` — major breaking change.
- Add a second type parameter: `RouteNode<R, C, Ctx, Params, Meta, S extends AnySchema>` — additive.
- Keep `ZodObject` in `RouteNode` but accept `Schema<T>` in the builder functions (compiled
  to Zod before storing) — transparent to the internal type, migration-compatible.

Option 3 (transparent compilation) is the lowest-risk path: external API accepts both
`z.ZodObject<T>` and `Schema<T>` via an overload; internally always uses Zod.

**Q5 — Serialization advantage**

A key motivation: `Schema<T>` is a plain JSON-serializable value (no functions, no class
instances). `serializeContract()` (plan 129) could use `Schema<T>` directly instead of
converting through Zod → JSON Schema. Does this eliminate the lossy `refine()` gap
identified in plan 127?

## Deliverable

Scratch file `packages/router/scratch/native-schema-spike.ts`:

1. Implement `Schema<T>` type definition.
2. Implement `object({ id: string() })` factory functions.
3. Verify TypeScript infers `{ id: string }` from `object({ id: string() })`.
4. Implement `toZod(schema: Schema<T>): z.ZodType<T>` for `object`, `string`, `number`,
   `optional`, `array`, `union`.
5. Use `route(object({ id: string() }), ':id')` and verify param type inference.
6. Compare ergonomics: write the same schema three ways (Zod, native short-form, native
   explicit) and assess readability.

Decision memo (≤ 400 words):
- Is the type inference for Q1 achievable without `any` in the implementation?
- What's lost relative to Zod (`.refine()`, `.transform()`, `.brand()`, error messages)?
- Is the transparent compilation path (Q4 option 3) sufficient, or does it require a
  flag day breaking change?
- Recommendation: proceed as a new primary schema API, proceed as a compatibility layer,
  or close as "Zod is good enough."

## Dependencies

None — independent of all other plans. But results affect:
- Plan 127 (isomorphism) — native schema eliminates the `.refine()` loss.
- Plan 129 (contract diff) — native schema is directly serializable.
- Long term: if adopted, replaces Zod in `RouteNode` and affects every plan that passes
  Zod schemas.

## Decomposition (June 2026)

The problem space is larger than originally scoped. An audit of schema coupling in the
codebase revealed ten distinct attachment points (not just `RouteNode.schema`), and several
additional dimensions worth exploring before writing any code:

- **Library landscape**: TypeBox, ArkType, Valibot, and Zod v4 are all serious candidates.
  Building a custom schema when one of these already fits would be wasteful.
- **Scope generalization**: Schema is used for params, query, body, headers, cookies, and
  responses. A unified `RouteSchema` design is worth considering.
- **Refinement taxonomy**: Most `.refine()` usage is format/bound constraints (serializable);
  a taxonomy defines the serialization boundary.
- **Interop**: Zod can be made opt-in via a fluent builder that produces native schemas, and
  a `fromZod()` adapter for existing Zod users. ArkType's insight — refinements as constraint
  strings (`'string >= 5'`, `'string.email'`) — is worth evaluating for the same purpose.
