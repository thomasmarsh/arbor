# Plan 174 — `IxSession` Primitive Operations + Session Interpreter

## Context

> **North star**: `plan/spec.ws-session-ergonomics.md` — shows the complete ops usage
> in server handler and client. The "Ops Interface" section is what this plan implements.

Plan 173 adds the `IxSession<Before, After, A>` class — a phantom-typed CPS wrapper with `.then()`
for sequencing. This plan adds the **primitive factory operations** that construct `IxSession`
values for each session type step, and the **session interpreter** that binds them to a
`WsAdapter`.

### The interpreter pattern

`IxSession` instances are pure descriptions (like `Effect<A>` in `@arbor/common`). The
WebSocket adapter is injected when constructing the "ops" object, which is what a handler
receives:

```typescript
// Adapter-bound factory: create one per connection
const ops = buildIxSessionOps(adapter)

// Each method returns an IxSession bound to the adapter via closure
ops.recv<JoinMsg>()           // IxSession<Recv<JoinMsg, N>, N, JoinMsg>
ops.send({ type: 'join' })    // IxSession<Send<JoinMsg, N>, N, void>
ops.branchTo('welcome')       // IxSession<Branch<{...}>, C['welcome'], void>
ops.chooseFrom<ChatCases>()   // IxSession<Select<ChatCases>, any, ChoiceResult<ChatCases>>
ops.close()                   // Promise<Done>
```

### Wire protocol for Branch and Select

`Branch` = the holder sends a tag string to the peer, then advances.
`Select` = the holder awaits a tag string from the peer, then routes.

The wire format is `{ tag: string, payload?: unknown }`. The `branchTo` op serialises
the chosen tag; `chooseFrom` deserialises the incoming tag and builds `ChoiceResult<C>`.
Per-step message schemas (the `payload` field) are validated by `recv` / `send`.

### Type signatures (target)

```typescript
export interface IxSessionOps {
  // Receive a typed message. N is inferred from the call-site's contextual type.
  recv<T>(schema: UserSchema<T>): IxSession<Recv<T, any>, any, T>

  // Send a typed message.
  send<T>(v: T, schema: UserSchema<T>): IxSession<Send<T, any>, any, void>

  // Server sends a tag to client and advances to that branch arm.
  branchTo<C extends Record<string, Session>, K extends keyof C & string>(
    k: K
  ): IxSession<Branch<C>, C[K], void>

  // Server awaits a tag from client; returns discriminated ChoiceResult.
  chooseFrom<C extends Record<string, Session>>(): IxSession<Select<C>, any, ChoiceResult<C>>

  // Mark the session complete.
  close(): Promise<Done>
}

export function buildIxSessionOps(adapter: WsAdapter): IxSessionOps
```

**Note on `recv` / `send` schema params**: The current `buildWsChannel` in `ws-context.ts`
validates messages via `syncValidate(schema, JSON.parse(raw))`. Use the same `syncValidate`
utility from `src/core/schema.ts`. The schema param is runtime evidence for validation; the
generic `T` drives compile-time typing.

**Note on `branchTo` / `chooseFrom`**: Tags are sent as `JSON.stringify({ tag: k })` and
parsed by the `chooseFrom` side. The `payload` is separate from the tag envelope — `recv`
reads the next full message, not just the payload field. This keeps the wire format simple:
each message is either a `{ tag }` envelope (for Branch/Select) or a data payload (for
Send/Recv). One pattern to avoid confusion: Branch/Select always precede the data step in the
session type (e.g., `Branch<{ welcome: Send<WelcomeMsg, ...> }>` sends the tag then the
payload in two separate operations). Alternatively, merge tag and payload in one message — the
spike (172-session-ergonomics.ts Encoding A notes) describes both options; pick the simpler
one for this implementation.

3. The following **runtime test** passes (using `createWsAdapterPair` from `ws-context.ts`):

```typescript
// Server: recv join, send welcome, close
// Client: send join, recv welcome, close
it('lobby handshake round-trip via IxSession', async () => {
  const [serverAdapter, clientAdapter] = createWsAdapterPair()
  const serverOps = buildIxSessionOps(serverAdapter)
  const clientOps = buildIxSessionOps(clientAdapter)

  const serverSession =
    serverOps.recv<{ username: string }>(z.object({ username: z.string() })).then(() =>
      serverOps.send({ type: 'welcome', roomId: 'r1' }, z.object({...}))
    )
  const clientSession =
    clientOps.send({ username: 'alice' }, z.object({...})).then(() =>
      clientOps.recv<{ type: 'welcome'; roomId: string }>(z.object({...}))
    )

  const [, received] = await Promise.all([
    serverSession.run(),
    clientSession.run(),
  ])
  expect(received).toEqual({ type: 'welcome', roomId: 'r1' })
})
```

## Notes

- `buildIxSessionOps` must live in `core/` — it imports from `core/schema.ts` for validation.
  It MUST NOT import from `contexts/`, `server/`, or `client/`. It takes a `WsAdapter` (defined
  in `ws-context.ts`) — but `WsAdapter` is a plain interface, so you can forward-declare or
  move it to `core/ws-adapter.ts` to satisfy the no-upward-import constraint. Check the actual
  import graph before placing the file.
- The `any` types in `IxSessionOps` signatures are intentional for the ops interface — the
  specific `N` (next state) is inferred from the session type at the call site via contextual
  typing. The phantom `Before`/`After` on `IxSession` carry the type, not the ops method.
