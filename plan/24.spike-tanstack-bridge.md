# PLAN.24 (Spike) — TanStack Router bridge feasibility

## Status: DEFERRED

**Decision (Q4 + Q6 resolved)**: TanStack bridge work is additive and
deprioritized. Do not start until plans 19–27 are complete and the runtime
safety floor (plan 25) and type coverage (plan 26) are solid. If and when
this spike is revisited, the package boundary question (Q4) is also resolved:
implement as a separate package (`packages/router-tanstack`) to avoid
making `@tanstack/react-router` a peer dep of every `@arbor/router` consumer.

---

## Original Scope (for when this is revisited)

`spec.tanstack-bridge.md` outlines an adapter strategy. Before committing to
implementation, this spike answers three concrete questions:

1. **Type compatibility**: Can `ChildUnion<C>` (Nexus's discriminated union)
   be mapped to TanStack's `RouteOptions` without losing type fidelity? Where
   does `any` creep in?

2. **Path syntax translation**: Nexus uses `:param` and `#param` (hash) syntax.
   TanStack uses `$param`. Is the mapping mechanical (regex replace) or are
   there edge cases (optional segments, wildcards)?

3. **Bundle cost**: Resolved — bridge lives in a separate package
   (`packages/router-tanstack`), not in this package.

## Deliverable (when undeferred)

A short decision memo (≤ 400 words) answering questions 1 and 2 above, plus
a recommended implementation plan or a go/no-go recommendation.

## Time-box

One session. If answers require prototyping a full type-level adapter, escalate
to a full implementation plan.
