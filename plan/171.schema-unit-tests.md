# PLAN.171 — Unit Tests for parseObjectSchema

## Context

Plan 164 introduced `src/core/schema.ts` with `parseObjectSchema` and the internal
`parseScalar` function as the runtime validator for all route-level schemas. These are only
exercised indirectly through `walk.test.ts` and `define-routes.test.ts` — no test file
directly targets `schema.ts` itself.

Uncovered paths:
- `brand` — delegates to inner schema but is never called in any test
- `literal` with number/boolean values (all existing tests use string tag literals)
- `optional` wrapping a non-string type, e.g. `optional(integer())`
- The failure path of `parseObjectSchema` — multiple failing fields, `issues` array shape
- `number` vs `integer` (both check `typeof value === 'number'` but emit different OpenAPI `type`)
- `validateNative` null/non-object input guard (`value != null && typeof value === 'object'`)

## Goal

Add `src/core/schema.test.ts` with direct unit tests for `parseObjectSchema`. Keep the
file small and focused — the goal is coverage of the validator's code paths, not
integration tests (those already exist).

## Change Surface & T-shirt Sizing

**XS** — one new test file, no source changes.

| File | Change |
| ---- | ------ |
| `src/core/schema.test.ts` | **New** |

## Dependencies

- Plan 164 (done — `schema.ts` implemented and in prod)

## Success Criteria

1. `pnpm --filter @arbor/router test` passes with the new file included
2. Each `parseScalar` `case` branch is exercised by at least one test
3. `parseObjectSchema` success path, partial-failure path, and all-failure path each have a test
4. The `null` input guard in `validateNative` has a test (pass `null` as the raw value)
