# Plan 62 — Unify response types and eliminate status `as const`

## Context

There are three parallel structures for HTTP responses across the codebase:

1. **`DispatchResult`** (`server/server.ts`) — the internal shape the server builds
   after running a handler:
   ```typescript
   interface DispatchResult {
     status: number;
     body: unknown;
     headers?: Record<string, string>;
     cookies?: Record<string, string>;
     tag: string;
   }
   ```

2. **`HttpResponseUnion<Resp>`** (`contexts/http-context.ts`) — the type-level shape
   inferred from response descriptors, used to type handler return values:
   ```typescript
   type HttpResponseUnion<Resp> = {
     [S in keyof Resp]: { status: S; body: InferBody<Resp[S]>; headers?: ...; cookies?: ... }
   }[keyof Resp];
   ```

3. **Raw handler return** — what authors actually write in handlers:
   ```typescript
   return { status: 200 as const, body: { id: '123' } };
   ```

None of these three types are the same. `DispatchResult` uses `status: number` while
`HttpResponseUnion` uses `status: S` (literal). The raw handler return requires
`as const` on the status literal to satisfy `HttpResponseUnion`. `DispatchResult`
adds a `tag` field that doesn't belong in the handler's return type.

This produces two concrete ergonomic problems:

1. **`status: N as const` boilerplate** — every handler return in every test and example
   requires `as const`. 30+ occurrences. Easy to forget, produces a confusing type error.

2. **Two response descriptor syntaxes** — routes can define responses as either:
   - Bare schema: `{ 200: UserSchema }` 
   - Descriptor object: `{ 200: { body: UserSchema, headers: HeaderSchema } }`
   The detection uses duck-typing in `httpRoute()` (`'safeParse' in d`, `'body' in d`).
   These look identical at a glance, behave differently, and silently fail if the
   descriptor has a `body`-less object shape.

## Goal

1. Define one canonical `HttpResponse` type used end-to-end.
2. Eliminate `status: N as const` from handler return sites.
3. Replace duck-typing in descriptor unpacking with an explicit discriminant.

## Proposed Approach

### Step 1 — Canonical HttpResponse type

```typescript
// In contexts/http-context.ts
interface HttpResponse<
  Status extends number = number,
  Body = unknown,
  Headers = undefined,
  Cookies = undefined,
> {
  status: Status;
  body: Body;
  headers?: Headers;
  cookies?: Cookies;
}
```

`HttpResponseUnion<Resp>` becomes a mapped union over this type.
`DispatchResult` is either replaced by `HttpResponse<number>` or aliased to it
(removing the `tag` field, which is a routing concern unrelated to the response shape).

### Step 2 — Eliminate `as const` via inference helper

Introduce a `respond` helper that captures the status as a literal without `as const`:

```typescript
function respond<S extends number, B>(status: S, body: B): HttpResponse<S, B>;
function respond<S extends number, B, H, C>(
  status: S, body: B, opts: { headers?: H; cookies?: C }
): HttpResponse<S, B, H, C>;
```

Handler return sites change from:
```typescript
return { status: 200 as const, body: user };
```
to:
```typescript
return respond(200, user);
```

The literal type is inferred from the argument, not asserted. The helper is
a thin wrapper with no runtime overhead (returns `{ status, body, ...opts }`).

### Step 3 — Explicit descriptor discriminant

Replace duck-typing in `httpRoute()` with a typed discriminant. Two options:

**Option A** — Add a `type` field to descriptor objects:
```typescript
type ResponseDescriptor<B, H, C> = { _desc: true; body: B; headers?: H; cookies?: C };
function desc<B, H, C>(body: B, opts?: { headers?: H; cookies?: C }): ResponseDescriptor<B, H, C>
```

Unpacking becomes: `if ('_desc' in d)` — a reliable discriminant.

**Option B** — Require the explicit object form always; remove bare-schema shorthand.

Option A is preferred — it keeps the shorthand ergonomic for simple cases while
making the descriptor form explicit and reliable.

## Change Surface

| File | Change |
|------|--------|
| `contexts/http-context.ts` | Define `HttpResponse`; replace `HttpResponseUnion` and `InferResponseDescriptor` with `HttpResponse`-based types; add `respond` helper; replace duck-typing with `_desc` discriminant |
| `server/server.ts` | Replace `DispatchResult` with `HttpResponse<number>`; remove `tag` field (pass tag separately) |
| `src/index.ts` | Export `respond` and `desc` helpers |
| `examples/*.ts` | Replace `status: N as const` with `respond(N, body)` |
| `src/**/*.test.ts` | Replace `status: N as const` with `respond(N, body)` |

## Dependencies

- Plan 61 (`_meta` typing) should land first — once `getHttpMeta` is type-safe,
  the descriptor unpacking refactor in Step 3 is cleaner.
- Plan 60 (server decompose) should land first or in parallel — `DispatchResult`
  removal affects both plans.
- Can be split: Step 2 (`respond` helper) is independent and low-risk; can land
  before Step 1 and 3 if needed as a quick ergonomic win.

## Success Criteria

- No `status: N as const` in `src/` or `examples/`.
- `HttpResponseUnion` and `DispatchResult` are removed (or aliased to `HttpResponse`).
- `httpRoute()` descriptor unpacking has no `'safeParse' in d` or `'body' in d` checks.
- `respond(200, body)` is exported from `src/index.ts` and used in all examples.
- `pnpm test && pnpm typecheck && pnpm lint && pnpm run examples` clean.
