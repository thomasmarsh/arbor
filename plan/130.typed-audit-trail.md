# PLAN.130 — Typed Audit Trail (`createAuditLogger`)

## Context

Every request to a `createServer()` router maps to a typed route and produces a typed
response. The audit schema IS the route schema — there is no separate audit definition to
maintain. `createAuditLogger(router)` wraps the server and emits a structured, typed audit
event for every dispatched request, without per-handler instrumentation.

This is architecturally distinct from `withMetrics()` (plan 37), which measures request
timing. The audit logger captures *what happened* (who accessed which resource with what
outcome); metrics capture *how long it took*. Both can run simultaneously.

## Goal

```typescript
// packages/router/src/server/audit-logger.ts

export interface AuditEvent<Route extends { tag: string }, Map extends AnyCtxMap> {
  tag:        Route['tag'];        // discriminant — narrows all fields
  route:      Extract<Route, { tag: Route['tag'] }>;  // fully typed route object
  method:     string;
  statusCode: number;
  timestamp:  number;             // Date.now()
  durationMs: number;
  userId?:    string;             // extracted from session context if present
  ip?:        string;             // from request headers (x-forwarded-for)
}

export interface AuditEmitter<Route extends { tag: string }, Map extends AnyCtxMap> {
  emit(event: AuditEvent<Route, Map>): void;
}

export function createAuditLogger<
  Route extends { tag: string },
  Map extends AnyCtxMap,
>(
  router: RouterContract<Route, Map>,
  emitter: AuditEmitter<Route, Map>,
): (server: ReturnType<typeof createServer>) => ReturnType<typeof createServer>;
```

### Usage

```typescript
const server = createServer(router, handlers);

const auditedServer = createAuditLogger(router, {
  emit(event) {
    // event.tag narrows: event.route, event.userId, etc.
    auditStore.write(event);
  },
})(server);

// auditedServer.handleRequest(req) wraps the original dispatch
```

### Typed event narrowing

The `AuditEvent.tag` is a discriminant over all route tags. This means a consumer of audit
events can switch on `event.tag` and get fully typed access to the route's params:

```typescript
switch (event.tag) {
  case 'delete-user':
    // TypeScript knows event.route.id: string
    alert(`User ${event.route.id} deleted by ${event.userId}`);
    break;
}
```

This exhaustive narrowing is the key property: the audit consumer is type-checked against
the same route schema as the server handlers. Adding a route automatically adds a new
case to the discriminant — the audit consumer gets a compile error if it doesn't handle
the new tag (when used with exhaustive checking).

### Session user extraction

`createAuditLogger` accepts an optional `extractUserId` function:

```typescript
extractUserId?: (ctx: HandlerCtx<any>) => string | undefined
```

Default: look for `ctx.session?.sub` (the `withSession` guard injects this). Teams using
custom session shapes provide their own extractor.

### Built-in sinks

- `consoleAuditEmitter()` — structured `console.log` (dev / debug)
- `noopAuditEmitter()` — swallows all events (testing, disabled audit)
- `streamAuditEmitter(writable)` — writes JSON-Lines to any Node.js `Writable`

Teams provide their own for Elasticsearch, S3, CloudWatch, etc.

## Dependencies

- `createServer()` (plan 60 — done).
- `withSession()` (plan 39 — done) — for default `extractUserId`.
- Plan 81 (supervision — done) — the audit logger wraps the supervised dispatch,
  so errors are already mapped to 500 before the audit event is emitted.

## Success Criteria

- Every request (success and error) emits one `AuditEvent`.
- `event.tag` narrows correctly in a `switch` statement.
- `event.route` carries the full typed route object (params, etc.).
- Plan 81's supervision boundary fires before the audit emitter — audit events see the
  final status code (500 for supervised errors), not the raw exception.
- `noopAuditEmitter()` adds zero overhead on the happy path (inlined by V8).
- No `any` in the public `AuditEvent` type.
