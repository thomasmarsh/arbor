# PLAN.119 (Spike) — SSR Foundations Assessment

## Context

Plans 94–118 describe browser integration and TCA bindings assuming a client-side SPA.
The BFF today serves `index.html` as a catch-all (pure SPA). SSR would replace that
catch-all with a React render function: parse the URL on the server, run loaders, render
to HTML, embed data for client hydration.

Before implementing anything, this spike answers which decisions in plans 94–118 must
be made SSR-aware NOW to avoid a refactor later — and what the minimal new surface area is.

## Questions to Answer

**Q1 — `RouterAdapter` interface**: plan 97 (`RouterProvider`) needs to accept either
`BrowserRouter` (client) or `StaticRouter` (server). If plan 97 is typed against a
concrete `BrowserRouter<R, M>`, adding SSR later forces a refactor. What is the minimal
`RouterAdapter<Route>` interface that both classes can implement?

```typescript
interface RouterAdapter<Route> {
  getRoute(): Result<Route, string>;
  navigate?(route: Route): void;   // optional — StaticRouter has none
  replace?(route: Route): void;
  subscribe?(fn: (r: Result<Route, string>) => void): () => void;
}
```

Is this sufficient? Does `RouterProvider` need anything else from the adapter?

**Q2 — `BrowserRouter` import safety**: plan 95 calls `window.addEventListener('popstate', ...)`.
If this call is in the constructor body (not at module top-level), importing
`packages/router-browser` in a Node.js SSR context is safe — the constructor just isn't
called. Verify that no planned code has module-level side effects that reference `window`.

**Q3 — Loader portability**: plan 101 loaders call `fetch`. Does the loader factory
(`createLoaderMap`) capture `globalThis.fetch` at module load time, or lazily at call time?
Edge runtimes may provide a different `fetch` from Node.js or the browser. Loaders should
accept a `fetch` parameter or be injected via `RouterEnv` to remain portable.

**Q4 — valtio on the server**: `Store` uses `valtio`'s `proxy`. Does `valtio` work in
Node.js? (It does — valtio uses Proxy which is available in all modern JS runtimes.)
Does `useSnapshot` work in a server React render context? (No — it uses `useSyncExternalStore`
which throws during `renderToString` without a server-safe snapshot path.) Document this
and propose a fix: either use `getProxyState().state` directly on the server, or provide a
plain-state fallback when `typeof window === 'undefined'`.

**Q5 — React SSR API target**: Three React APIs are available:
- `renderToString` — synchronous, blocks until full render, no streaming
- `renderToPipeableStream` — Node.js streaming, works with `<Suspense>`
- `renderToReadableStream` — WinterCG streaming, works with Hono (BFF)

The BFF uses Hono (WinterCG `fetch`/`Response` model). `renderToReadableStream` integrates
directly with Hono's `c.stream()`. Is `renderToReadableStream` available in the Node.js
version used by the BFF (requires Node 18+)?

**Q6 — Hydration**: `hydrateRoot(container, element)` expects the server HTML and the
client React tree to match exactly. If the server renders a route with SSR data and the
client mounts with `RouterProvider` + `BrowserRouter`, the initial `getRoute()` must
return the same route. Does parsing `window.location` on the client reproduce the
server's parse exactly? (Should be yes — `parse()` is deterministic.)

**Q7 — Head management**: `document.title` writes (plan 111) do nothing on the server.
`<title>` must be emitted into the `<head>` during `renderToReadableStream`. React 19 has
native `<title>` / `<meta>` hoisting; React 18 does not. Which React version does the
UI target? If React 18, we need a head-collection context (similar to React Helmet).

## Deliverable

Decision memo (≤ 400 words) covering:
- Q1: proposed `RouterAdapter` interface (feeds directly into plan 97 and plan 120)
- Q2: confirm no module-level side effects in planned code
- Q3: loader fetch injection strategy
- Q4: valtio server-safety fix
- Q5: React SSR API (`renderToReadableStream` confirmed or `renderToPipeableStream`)
- Q6: hydration confirmed
- Q7: React version + head management approach

No implementation. Amendments to plans 97, 101, 111, 117 if needed.

## Dependencies

None — this is a prerequisite spike for all SSR plans (120–126).

## Why Now

Plan 97 is the first concrete deliverable that touches the abstraction boundary.
If `RouterProvider` is designed with `RouterAdapter<Route>` instead of a concrete
`BrowserRouter` type, SSR is free. If not, it costs a refactor. This spike costs one
session; the refactor could cost five.

## Time-box

One session. No code lands in `src/`.
