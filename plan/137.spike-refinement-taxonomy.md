# PLAN.137 (Spike) — Refinement Taxonomy and the Cost of Owning the Schema

## Context

`@arbor/router` is a **library**. Users author their own schemas and pass them to arbor.
Today a user can write:

```typescript
httpRoute(z.object({
  email: z.string().email(),
  age:   z.number().int().min(0).max(150),
  bio:   z.string().max(500).optional(),
}).refine(d => isValid(d), { message: 'invalid' }), 'POST', '/users', { ... })
```

Arbor calls `.safeParse()` on their behalf — it does not write these schemas. The user's
`.refine()` call is their runtime safety net. If arbor moves to a native schema format, the
user's `.refine(fn)` either:

- **Cannot be expressed** in the native format (function, not data), or
- **Requires an escape hatch** in arbor's schema API, or
- **Requires the user to duplicate validation** elsewhere (handler code, guard layer).

This is the real cost of arbor owning the schema. The question is not "what does arbor use"
but: **if we ask users to adopt our schema format, what do they give up, and is there a
clean escape hatch for what they lose?**

Accepting user-owned schemas (Zod, etc.) means: zero migration burden, full `.refine()` /
`.transform()` / `.brand()` support, runtime safety entirely the user's concern — but arbor
cannot serialize, diff, or structurally compare schemas, and OpenAPI generation is lossy
wherever `.refine(fn)` appears.

Controlling the schema representation means: users bear a migration cost (or use a `fromZod()`
adapter), function-based refinements are not in the schema layer — but arbor gains
serialization, contract diffing, semversioning, and lossless OpenAPI generation for everything
expressed in the schema.

ArkType's insight is relevant here: *it serializes refinements as constraint strings*
(`type('string >= 5')`, `type('string.email')`). This means format and bound constraints
remain data — no functions. The question is whether that coverage is sufficient for users'
actual validation needs on route boundaries.

## Goal

Analyze the **space of refinements users apply to route boundary schemas**, classify them by
whether they can be expressed as serializable data, and determine whether the constraint
vocabulary of a native schema can cover the meaningful majority of user needs. Identify what
the escape hatch should look like for the rest.

This is not an audit of arbor's own code. It is an analysis of what users do when defining
schemas for route params, query strings, request bodies, and response bodies.

## Questions to Answer

**Q1 — What do users actually validate at route boundaries?**

Route boundary schemas (params, query, body, response) are a specific domain. They tend to
be simpler than arbitrary application schemas because they describe wire-format data, not
rich domain objects. Survey the space:

- What fraction of real-world route boundary schemas use only format/bound constraints
  (email, uuid, min, max, int, url, regex)? These are all representable as data.
- What fraction use cross-field `.refine()` — e.g., `password === confirmPassword`,
  `startDate < endDate`? These require functions.
- What fraction use `.transform()` — e.g., coerce a string to a Date? These change the type.
- Are there common patterns that look like refinements but are really type narrowing (e.g.,
  discriminated union on a `type` field)?

The claim to test: for *route boundary* schemas specifically, the function-requiring cases are
a small minority, and arbor's native schema can cover the majority.

### Q2 — The cost for the function-requiring minority

When a user genuinely needs `.refine(fn)` on a route schema, what are their options if arbor
uses a native schema format?

- **Option A — Validate in the handler**: The native schema handles structural validation;
  the user adds `if (!isValid(body)) return 400` in their handler. No arbor API change needed.
  Cost: users lose the declarative ergonomics of `.refine()`.

- **Option B — Arbor provides a `validate` slot alongside the schema**: `httpRoute()` accepts
  a separate `validate: (parsed) => ValidationResult` option, decoupled from the schema itself.
  The schema remains serializable; the validate function is a side-channel that only runs at
  the validation boundary, not stored in the schema.

- **Option C — An `opaque` schema kind**: `{ kind: 'opaque', validate: fn, jsonSchema: {...} }`
  — the user provides both a hand-written JSON Schema description (for OpenAPI) and a validator
  function. Arbor uses the description for serialization and the function for validation.

- **Option D — `fromZod(zodSchema)` as the escape hatch**: Users who need `.refine()` pass a
  Zod schema via `fromZod()`. Arbor uses Zod for validation and falls back to lossy OpenAPI
  generation (same as today). This is the "pay as you go" option.

Which option(s) best preserve the user's ability to achieve runtime safety without requiring
them to abandon their validation logic or write it twice?

**Q3 — Transforms: schema or codec?**

`.transform()` changes the parsed type (e.g., `z.string().transform(Number)` parses as `number`
even though the input was `string`). This is not validation — it is a codec. The input type
and output type differ.

If arbor owns the schema, transforms are problematic because:

- The schema describes the *wire* type, but `Infer<S>` would give the *parsed* type.
- Serializing the schema should produce a JSON Schema for the wire type, not the parsed type.

Should arbor's native schema support transforms at all, or should they be handled as a
separate parsing step layered above the schema? Note: for route boundary data (query strings,
URL params) arbor already does its own coercion (`:id` as a number segment). Are user-level
transforms redundant there?

**Q4 — The constraint vocabulary: what does "good enough" look like?**

If arbor's native schema covers only serializable constraints, define "good enough" for the
majority of users. Specifically: enumerate the constraint metadata fields that would cover
the common cases without becoming a mini-language.

Candidate vocabulary for strings:
`format` (email, url, uuid, cuid, ulid), `minLength`, `maxLength`, `pattern` (regex as string)

Candidate vocabulary for numbers:
`integer`, `minimum`, `maximum`, `exclusiveMinimum`, `exclusiveMaximum`, `multipleOf`

Note these map directly to JSON Schema keywords — this is intentional. If the native schema's
constraint vocabulary IS JSON Schema's constraint vocabulary, then OpenAPI generation is
lossless for the A-class constraints with no conversion step.

Is this vocabulary sufficient for the cases users actually care about on route boundaries?
What's left out that would be a genuine daily frustration?

### Q5 — Branded types

`z.string().brand<'UserId'>()` creates `string & { __brand: 'UserId' }` — a purely type-level
distinction. Runtime value is still a plain string. A native schema can support this without
functions: `{ kind: 'brand', inner: { kind: 'string' }, brand: 'UserId' }` produces
`Schema<string & Brand<'UserId'>>`. No serialization problem. This is worth confirming in
the scratch file (plan 138's spike).

## Deliverable

1. **User need analysis**: What fraction of route boundary schema refinements in practice fall
   into "serializable constraint" vs. "requires function"? Draw from arbor's own examples
   (which demonstrate the library's intended use patterns) plus common real-world patterns.
   Be explicit that this is a representative sample, not a complete audit of user code.

2. **Escape hatch recommendation**: Which of Options A–D (from Q2) provides the best
   user experience for the function-requiring minority? Can they be combined?

3. **Transform verdict** (Q3): In-schema support vs. separate codec layer.

4. **Constraint vocabulary proposal** (Q4): The exact field names and value types needed.
   Note which map to JSON Schema keywords (serialization benefit) and which don't.

5. **Branded type confirmation** (Q5): Does `Schema<T & Brand<B>>` work at the type level?
