# PLAN.116 — `Effect` Streaming + SSE/WS as Long-Lived Effects

## Context

The current `Effect<A>` class wraps effect-ts and calls `send(a)` once when the inner
effect completes. The `Effect.of((send) => ...)` constructor can call `send` multiple times
(nothing prevents it), but there is no teardown mechanism — once an effect is forked via
`unsafeRun`, it cannot be cancelled. For global singleton effects (popstate subscription,
router store) this is acceptable. For component-scoped SSE and WebSocket connections, it
causes resource leaks if the component unmounts.

This plan investigates and extends `Effect` to support a teardown contract, then models
SSE and WebSocket connections as composable streaming effects in the `RouterEnv`.

## Part 1 — `Effect.stream<A>` extension

Add a new constructor to `Effect<A>`:

```typescript
// packages/common/src/store/effect.ts

public static stream<A>(
  setup: (send: Send<A>) => (() => void) | void
): Effect<A>
```

The `setup` function returns an optional teardown callback. When the underlying effect-ts
fiber is interrupted (via `Fiber.interrupt`), the teardown is invoked.

Implementation path:
- `setup` is wrapped in an `Eff.acquireRelease` scope: `acquire` calls `setup(send)` and
  captures the returned teardown; `release` calls the teardown if provided.
- The fiber is started with `Runtime.runFork` as before, but the forked fiber reference is
  returned from `unsafeRun` so callers can interrupt it.
- Update `unsafeRun` signature: `unsafeRun(send: Send<A>): Fiber<void>`.

This is additive — existing callers that ignore the fiber handle are unaffected.

## Part 2 — SSE as `Effect.stream`

```typescript
// packages/router-browser/src/router-store/sse-effect.ts

export function sseEffect<A>(
  url: string,
  onEvent: (data: unknown) => A,
  onError: () => A,
): Effect<A> {
  return Effect.stream<A>((send) => {
    const es = new EventSource(url);
    es.onmessage = (e) => send(onEvent(JSON.parse(e.data)));
    es.onerror   = ()  => send(onError());
    return () => es.close();
  });
}
```

The teardown (`es.close()`) fires when the fiber is interrupted — i.e., when the store
that holds the effect decides to stop it (e.g., on navigation away from the route that
opened the SSE connection).

## Part 3 — WS as `Effect.stream`

```typescript
export function wsEffect<In, Out>(
  url: string,
  onMessage: (data: Out) => RouterAction,
  onClose:   ()            => RouterAction,
): { effect: Effect<RouterAction>; send: (msg: In) => void }
```

The `send` function is a closure over the `WebSocket` instance — callers dispatch messages
via `send` and the effect delivers incoming messages as actions to the store.

## Part 4 — Integration with `RouterEnv`

`RouterEnv` (plan 114) gains optional SSE/WS effect factories:

```typescript
export interface RouterEnv<Route> {
  // ... existing fields ...
  openSseConnection?: (url: string) => Effect<RouterAction<Route>>;
  openWsConnection?:  (url: string) => { effect: Effect<RouterAction<Route>>; send: WsSend };
}
```

A `RouterAction` of `{ tag: 'sse-event'; route: Route; data: unknown }` and
`{ tag: 'ws-message'; route: Route; data: unknown }` feed real-time data into the
navigation state without the component needing a separate `useSSE` / `useWebSocket` hook.
Plans 107/108 remain available as the hooks-based alternative.

## Success Criteria

- `Effect.stream` teardown is invoked when the fiber is interrupted.
- SSE: `es.close()` is called when the streaming effect is interrupted; no event handler
  fires after teardown.
- WS: `ws.close()` is called on teardown; send after teardown is a no-op (not a throw).
- Existing `Effect.of` / `Effect.tryPromise` callers are unaffected (backward compatible).
- Tests pass for both streaming effects under `jsdom` (mock `EventSource` + `WebSocket`).
