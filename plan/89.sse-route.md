# Plan 89 — SSE Route (`sseRoute`)

## Context

SSE (Server-Sent Events) is the simplest practical session type: a unidirectional
server→client stream. The session type is `Send<E, Send<E, ...>>` repeated until the
client disconnects, which collapses in practice to `AsyncIterable<E>`. The key advantage
over an ad-hoc streaming response: the event schema is declared at the route level, the
handler's return type is typed to that schema, and the client's async iterable type is
derived automatically — no casting, no `any`.

This plan delivers `sseRoute()` using the session type foundations from plan 88. The
client side sees the dual: the server declares `Send<E>` (repeatedly), so the client
receives `Recv<E>` (repeatedly), which maps to `AsyncIterable<E>`.

## Goal

Ship, end-to-end:

1. **`sseRoute(schema, path, opts)`** — factory; `opts.events` is a Zod schema for the
   event payload; returns a `RouteNode` with `SseMeta<EventSchema>` in `_meta`
2. **Server dispatch** — `createServer` (or a parallel `createSseServer`) recognises SSE
   routes; calls the handler which returns `AsyncIterable<z.infer<EventSchema>>`; serializes
   to `text/event-stream` format; manages connection lifecycle
3. **Typed SSE client** — `createClient().subscribe(route)` returns
   `AsyncIterable<z.infer<EventSchema>>`; runtime uses `EventSource` or `fetch` with
   streaming body; event data is parsed and validated against `opts.events`
4. **Type safety** — handler type error if returned iterable's element type doesn't match
   `EventSchema`; client iterable type is derived from the route definition
5. **Example** — `examples/sse-stream.ts` demonstrates a typed event stream end-to-end

- `sseRoute(S, 'events/stream', { events: z.object({ msg: z.string() }) })` — handler
  must return `AsyncIterable<{ msg: string }>`; returning `AsyncIterable<number>` is a
  type error
- Client `createClient().subscribe(sseRoute)` returns `AsyncIterable<{ msg: string }>`
- `text/event-stream` response format is valid (passes an RFC 8895 conformance check)
- Connection closes cleanly when the iterator completes
- `pnpm run examples` output shows streamed events
- `pnpm test && pnpm typecheck && pnpm lint` green

## Design Notes

### Event Schema Shape

Events are typed via the `events` Zod schema, which describes a single event payload.
The `data:` field in the SSE wire format carries the JSON-serialized payload:

```typescript
sseRoute(CounterSchema, 'stream/counter', {
  events: z.object({ count: z.number() }),
})

// Handler:
'stream/counter': async function* (ctx) {
  for (let i = 0; i < 10; i++) {
    yield { count: i };  // typed: { count: number }
    await sleep(100);
  }
}
```

### Session Type Mapping

`sseRoute` is syntactic sugar for a session type of the form `Send<E, S>` where S
recurses back to `Send<E, S>` (an infinite stream). In practice we represent it as
`AsyncIterable<E>` on both sides because TypeScript cannot express coinductive (infinite)
types, but the session type metadata in `_meta` carries `Send<E>` as the declared shape
for future tooling.

### WinterCG Compatibility

The serializer must output a `Response` with:
- `Content-Type: text/event-stream`
- `Cache-Control: no-cache`
- `Connection: keep-alive` (where relevant)

Use a `ReadableStream` backed by the handler's async iterator so the response is
WinterCG-compatible. Avoid Node-specific streams.

### Client Transport

`EventSource` is the standard browser API for SSE. For server-side or test use,
fall back to `fetch` with a `ReadableStream` body reader. The client adapter is
injected (same pattern as `createClient`'s `fetch` option), so both are supported.

### Backpressure

SSE has no backpressure — the server pushes; the client buffers. Document this
limitation; do not add flow-control in this plan. It is a potential plan-90+ concern.
