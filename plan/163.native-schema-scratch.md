# PLAN.163 — Native Schema Scratch Proof

## Context

Plans 133–138 resolved the design questions for arbor's native schema system. The decision:
build a custom `ObjectSchema<T>` for router-controlled positions (path params, query, headers,
cookies, session messages); accept `StandardSchemaV1<T>` for user-controlled positions (body,
response). Full design in `plan/136.spike-schema-scope.md` findings.

This plan is the hard gate before any `src/` changes. TypeScript type inference for a custom
schema discriminated union has two specific failure modes:

1. **Circular type references** — `ObjectSchema<T>` referencing `ScalarSchema<T[K]>` can
   cause runaway instantiation if `ScalarSchema` references back to `ObjectSchema`.
2. **Inference breakdown at the call site** — `route(object({ tag: literal('user') }), ':id')`
   must propagate `{ tag: 'user' }` into `RouteNode<R>` without an explicit annotation.

The scratch file proves these risks are resolved before touching production code.

## Goal

A scratch file with explicit falsifiable `expectTypeOf` claims. Every claim must pass `tsc`
and `vitest`. If any claim fails, the type design is redesigned — not the claim.

## Change Surface & T-shirt Sizing

## Scratch File Spec

`framework/router/scratch/native-schema-spike.ts`:

```typescript
import { expectTypeOf } from 'vitest';

// ── Types ────────────────────────────────────────────────────────────────────

type StringConstraints = {
  format?: 'email' | 'url' | 'uuid' | 'cuid' | 'ulid' | 'datetime' | 'date' | 'time';
  minLength?: number;
  maxLength?: number;
  pattern?: string;
};

type NumberConstraints = {
  minimum?: number;
  maximum?: number;
  exclusiveMinimum?: number;
  exclusiveMaximum?: number;
  multipleOf?: number;
};

type ScalarSchema<T> =
  | { kind: 'string';   constraints?: StringConstraints }
  | { kind: 'number';   constraints?: NumberConstraints }
  | { kind: 'integer';  constraints?: NumberConstraints }
  | { kind: 'boolean' }
  | { kind: 'literal';  value: T }
  | { kind: 'optional'; inner: ScalarSchema<Exclude<T, undefined>> }
  | { kind: 'brand';    inner: ScalarSchema<T>; brand: string };

type AnyScalarSchema = ScalarSchema<string | number | boolean | undefined>;

type ObjectSchema<T extends Record<string, unknown>> = {
  kind: 'object';
  fields: { [K in keyof T]: ScalarSchema<T[K]> };
};

type AnyObjectSchema = ObjectSchema<Record<string, unknown>>;

type Infer<S> =
  S extends ObjectSchema<infer T> ? T :
  S extends { kind: 'string' } ? string :
  S extends { kind: 'number' | 'integer' } ? number :
  S extends { kind: 'boolean' } ? boolean :
  S extends { kind: 'literal'; value: infer V } ? V :
  S extends { kind: 'optional'; inner: infer I } ? Infer<I> | undefined :
  S extends { kind: 'brand'; inner: infer I } ? Infer<I> :
  never;

// ── Factories ────────────────────────────────────────────────────────────────

const string   = (constraints?: StringConstraints): ScalarSchema<string>  => ({ kind: 'string', constraints });
const number   = (constraints?: NumberConstraints): ScalarSchema<number>  => ({ kind: 'number', constraints });
const integer  = (constraints?: NumberConstraints): ScalarSchema<number>  => ({ kind: 'integer', constraints });
const boolean  = (): ScalarSchema<boolean> => ({ kind: 'boolean' });
const literal  = <T extends string | number | boolean>(value: T): ScalarSchema<T> => ({ kind: 'literal', value });
const optional = <T>(inner: ScalarSchema<T>): ScalarSchema<T | undefined> => ({ kind: 'optional', inner });
const object   = <T extends Record<string, unknown>>(
  fields: { [K in keyof T]: ScalarSchema<T[K]> }
): ObjectSchema<T> => ({ kind: 'object', fields });

// ── Claims ───────────────────────────────────────────────────────────────────

// Claim 1: object schema infers its field types
const userSchema = object({ tag: literal('user'), id: string() });
expectTypeOf(userSchema).toEqualTypeOf<ObjectSchema<{ tag: 'user'; id: string }>>();

// Claim 2: Infer<S> extracts T from ObjectSchema<T>
type UserParams = Infer<typeof userSchema>;
expectTypeOf<UserParams>().toEqualTypeOf<{ tag: 'user'; id: string }>();

// Claim 3: optional wraps correctly — undefined is added to the field type
const querySchema = object({ page: optional(integer()), q: optional(string()) });
expectTypeOf<Infer<typeof querySchema>>().toEqualTypeOf<{ page: number | undefined; q: string | undefined }>();

// Claim 4: route() infers RouteNode<Infer<S>> — mock route() to test inference without touching src/
declare function mockRoute<S extends AnyObjectSchema>(
  schema: S,
  path: string
): { _type: Infer<S> };

const r = mockRoute(userSchema, 'users/:id');
expectTypeOf(r._type).toEqualTypeOf<{ tag: 'user'; id: string }>();

// Claim 5: constraints do not change the inferred TypeScript type
const s1 = string();
const s2 = string({ format: 'email', maxLength: 100 });
expectTypeOf(s1).toEqualTypeOf(s2);

// Claim 6: literal works for numbers and booleans, not just strings
const n = literal(42);
const b = literal(true);
expectTypeOf<Infer<typeof n>>().toEqualTypeOf<42>();
expectTypeOf<Infer<typeof b>>().toEqualTypeOf<true>();

// Claim 7: UserSchema structural interface — satisfied without importing Zod
// (proves Standard Schema interface works via structural typing)
interface UserSchema<T> {
  readonly '~standard': {
    readonly validate: (value: unknown) =>
      | { readonly value: T }
      | { readonly issues: ReadonlyArray<{ readonly message: string }> };
  };
}

declare const mockZodSchema: UserSchema<{ id: string }>;
declare function mockValidate<T>(schema: UserSchema<T>, value: unknown): T | null;
expectTypeOf(mockValidate(mockZodSchema, {})).toEqualTypeOf<{ id: string } | null>();
```

## Notes

- Do not allow nested `object()` inside `object()` fields in this pass. The router-controlled
  positions (params, query, headers, cookies) are flat. If claims require nested objects,
  that is a scope creep — add a depth counter (same pattern as `FlattenChildrenImpl`) only
  if specifically needed.
- The `brand` kind is defined in the type but not exercised in claims 1–7. Add a Claim 8
  if `withBrand` / branded types are needed for session type message schemas (revisit in
  plans for sessions).
