# PLAN.131 — Supervision + Metrics Integration

## Context

Plan 81 (done) added a supervision boundary to `createServer()`: uncaught handler
exceptions produce a 500 response and call `onError`. It is opt-out via `{ supervise: false }`.

What plan 81 does NOT do: automatically emit a `RequestMetric` from the supervision
boundary. If a team uses `withMetrics()` (plan 37) AND a handler throws, the supervision
boundary fires and returns 500, but the metrics emitter never sees the request — it only
wraps the outer server, which now returns a synthetic 500 from the supervisor, not from
the metrics-wrapped handler path.

The gap: metrics and supervision are independent wrappers applied in an undefined order.
A handler error can fall through the gap between them.

## Goal

Unify the supervision boundary and the metrics channel so that:

1. Every request — including those that throw — emits exactly one `RequestMetric`.
2. The `RequestMetric` for a supervised error includes `{ status: 500, error: true }`.
3. No configuration required: if `withMetrics()` is configured, supervised errors appear
   in its output automatically. If not, nothing changes.

### Design

Move metric emission into `executeRoute()` (where the supervision boundary lives), rather
than in the `withMetrics()` outer wrapper. `withMetrics()` remains the public API for
configuring the emitter, but the emission point moves inside the supervisor.

```typescript
// In executeRoute() — after the try/catch supervision boundary:
const endTime = performance.now();
options.metricsEmitter?.emit({
  tag,
  method,
  status:    result.status,
  durationMs: endTime - startTime,
  error:     result.status >= 500,
});
```

The `options.metricsEmitter` is passed down from `createServer()` — already present since
plan 37. The change is: move the `emit` call from `withMetrics()`'s outer wrapper into
`executeRoute()`'s inner finally block, so it fires regardless of throw.

### `withMetrics()` becomes a configuration helper

`withMetrics(server, emitter)` currently wraps the server. After this plan it becomes:

```typescript
export function withMetrics<R, M>(
  server: Server<R, M>,
  emitter: MetricsEmitter,
): Server<R, M> {
  // Inject the emitter into server's internal options.
  // Emission happens inside executeRoute, not here.
  return server.withMetricsEmitter(emitter);
}
```

The public API is unchanged. Internally, emission is guaranteed for every dispatch.

### OTP connection

In OTP, the supervisor not only catches crashes — it logs them to the SASL error log
automatically. The supervision boundary in `createServer` should behave the same: crash
→ 500 → metric → `onError` log. No configuration required for the metric; the log is
`onError`'s job.

## Change Surface

- Update: `packages/router/src/server/server.ts` — move `emit` into `executeRoute` finally block
- Update: `packages/router/src/server/with-metrics.ts` — change to a configuration helper
- Update: `packages/router/src/server/server.test.ts` — test metric emitted on handler throw
- Update: `packages/router/src/server/with-metrics.test.ts` — update for new emission model

## Dependencies

- Plan 81 (supervision — done, this extends it).
- Plan 37 (`withMetrics` — done, this changes its internals).

## Success Criteria

- A handler that throws emits a `RequestMetric` with `status: 500` and `error: true`.
- The metric is emitted once per request, not twice (no double-emit if both
  `withMetrics` and `onError` are configured).
- `withMetrics()` public API is unchanged — existing callers do not need updating.
- `{ supervise: false }` still disables supervision; metric emission is also skipped
  (if supervision is off, execution is raw — the caller is responsible for metrics).
- All existing `withMetrics` tests pass.
