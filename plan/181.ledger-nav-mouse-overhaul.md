# 181. Ledger Navigation & Mouse Overhaul

## Context

`useLedgerKeyboard.ts` conflates two concerns: navigation (`j/k/↑/↓`) and ad-hoc mutation triggers (`n=next`, `d=done`, `b=bump`, `D=defer`, `a=all`, `r=refresh`). There is no mouse support — clicking a row does nothing. The store's `selectedIndex: number` is fragile: if the visible list changes (filter, reload), the cursor silently jumps to a different task.

## Goal

Replace the current keyboard/selection model with standard conventions:

- Navigation: arrow keys and vi keys (`hjkl`) only; clicking a row also selects it.
- Remove all letter-key mutation shortcuts. Mutations happen through the drawer (plans 179, 186) or toolbar actions.
- `?` opens a help overlay listing all keybindings.
- `Enter` → open detail drawer. `Escape` → close it.
- Selection is stable across list reloads (tracks by task ID, not array index).

## Post-mortem

Verification chain required 5+ passes; four distinct root causes each cost a full iteration.

1. **Valtio Snapshot lint conflicts (2 separate fights).** `$.state.helpOpen` and the `watch` selector both triggered contradictory pairs of `@typescript-eslint` rules (`no-unnecessary-type-assertion` vs `no-unsafe-assignment`/`no-unsafe-return`) because valtio resolves boolean/nullable Snapshot fields to an internal "error type" in strict mode. *Pre-flight that would have caught it*: before adding any new `$.state.*` JSX prop or `watch` selector, check an existing one (e.g. `$.state.showAll`) and confirm the lint pattern — member-access is fine, direct assignment to a typed prop is the problem.

2. **`Store` mutating `initialLedgerState` via valtio proxy.** `proxy({ state: initialState })` writes through to the caller's object; subsequent test renders started with stale `selectedId`. Pre-existing bug, but only surfaced because new tests relied on `selectedId === null` at mount. *Pre-flight*: check whether any test asserts initial state shape (they didn't before this plan), signalling the const-mutation hazard.

3. **ArrowDown/ArrowUp test assumed `selectedIndex: 0`.** Old test worked because row 0 was always pre-selected; new ID model starts with `selectedId: null`. Adapting without re-reading the test led to a structurally different (and fragile) double-keypress sequence. *Pre-flight*: read every test that will change, not just the implementation files.

4. **MUI Dialog leaving `aria-hidden`/`overflow` on `<body>` after test teardown.** Opening the help overlay without closing it in the `?` test contaminated all subsequent renders. *Pre-flight*: any test that opens a Dialog must close it before the test ends; add this as a checklist item when testing MUI Modal components.

## Success Criteria

- Clicking any visible row selects it (MUI `selected` highlight).
- `j/k/↑/↓` navigate; selection follows the same task after a list reload (e.g., hitting `r` keeps cursor on the same task).
- `?` opens a modal; `Escape` or clicking outside closes it.
- No letter key triggers a mutation directly.
- Full test suite passes with no deleted tests.
