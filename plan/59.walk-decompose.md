# Plan 59 — Decompose walkParse and unify walk boilerplate

## Context

`core/walk.ts` contains three tree-walking functions — `walkParse`, `walkCollect`,
`walkPrint` — that all repeat the same structural pattern:

```typescript
function walkXxx(nodes, ...) {
  for (const node of nodes) {
    if (node.schema !== null) {
      const tag = getTag(node.schema);
      // do something with tag
    }
    if (node.children.length > 0) {
      Object.assign(result, walkXxx(node.children as WalkNode[], ...));
    }
  }
}
```

`walkParse` is the worst offender at 92 lines, mixing six concerns in one function:
1. URL segment matching
2. Path parameter accumulation
3. Query schema resolution (with two different branches for lazy vs eager schemas)
4. Zod schema validation (called in multiple branches)
5. Child node recursion
6. Diagnostic collection via `diag?.push()`

`client/fetch-client.ts` adds two more near-identical walkers: `buildMethodMap` and
`buildResponseSchemaMap`, both walking nodes and extracting a single field from
`getHttpMeta(node)`. Both were written independently of `walk.ts` despite doing the
same structural work. Plan 56 already introduced `walkCollect` as a generic collector
in core; these two functions should be replaced by calls to it.

## Goal

1. Extract focused single-concern helpers from `walkParse`.
2. Unify the repeated structural pattern across `walkParse`, `walkCollect`,
   `walkPrint` — each should read as a concise composition of smaller pieces.
3. Replace `buildMethodMap` and `buildResponseSchemaMap` in `fetch-client.ts`
   with `walkCollect` calls (already available from Plan 56 work).

## Proposed Approach

### Step 1 — Extract segment matching

Pull the URL-segment loop out of `walkParse` into a pure helper:

```typescript
function matchSegments(
  segments: Segment[],
  parts: string[],
  params: Record<string, string>,
): { matched: boolean; params: Record<string, string> }
```

No schema access, no recursion, no diagnostics. Returns whether the segments matched
and the accumulated params.

### Step 2 — Extract query schema resolution

```typescript
function resolveQuerySchema(node: WalkNode): z.ZodObject<any, any> | undefined
```

Handles the lazy (function) vs eager (object) schema cases that are currently
duplicated across two branches inside `walkParse`.

### Step 3 — Extract schema validation

```typescript
function validateSchema<T>(
  schema: z.ZodType,
  value: unknown,
  diag?: Diagnostic[],
): T | undefined
```

Wraps `safeParse`, pushes to `diag` on failure, returns the typed value or undefined.
Eliminates the repeated `safeParse` + `diag?.push` + `continue` pattern.

### Step 4 — Simplify walkParse

After extraction, `walkParse` becomes an orchestrator that calls:
1. `matchSegments` — check URL match
2. `resolveQuerySchema` — get the query schema  
3. `validateSchema` — validate body and query
4. Recurse on children if unmatched

Target: under 40 lines.

### Step 5 — Simplify walkCollect and walkPrint

Both currently hand-roll the tag-extraction + recursion pattern. Extract a shared
`forEachTaggedNode(nodes, cb)` traversal utility so each just provides the callback:

```typescript
function walkCollect(nodes, extractor) {
  const map: Record<string, unknown> = {};
  forEachTaggedNode(nodes, (node, tag) => {
    const v = extractor(node, tag);
    if (v !== undefined) map[tag] = v;
  });
  return map;
}
```

### Step 6 — Replace fetch-client walkers

Replace `buildMethodMap` and `buildResponseSchemaMap` with:

```typescript
const methodMap = walkCollect(nodes, (n) => getHttpMeta(n)?.method);
const responseSchemaMap = walkCollect(nodes, (n) => getHttpMeta(n)?.responseSchemas);
```
