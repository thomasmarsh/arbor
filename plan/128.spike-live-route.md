# PLAN.128 (Spike) — LiveRoute: Server-Driven Typed UI Transitions

## Context

Phoenix LiveView pushes HTML diffs over a persistent WebSocket. The client is a thin JS
runtime that applies patches; the server owns all state and renders all HTML. The key
advantage over a pure SPA: no state synchronization problem — there is no client state
to sync with the server.

`@arbor/router` already has the typed bidirectional transport (`wsRoute`, `Dual<S>`).
The route discriminated union models UI state transitions naturally: navigating from
`{ tag: 'user', id: '1' }` to `{ tag: 'user-tab', tab: 'posts' }` is a discrete,
typed event that the server can handle and respond to with a targeted HTML patch.

The key advantage over Phoenix: event types are statically verified against the route
schema — a client can only emit events the server declared in the route's session type.

## Questions to Answer

**Q1 — Session type shape**

What session type models the LiveView protocol? A first sketch:

```typescript
// Server sends HTML patches; client sends DOM events (clicks, input changes, submits)
type LiveSession = Send<HtmlPatch, Recv<DomEvent, LiveSession>>;  // recursive

// HtmlPatch: a morphdom-style diff or a targeted replacement
interface HtmlPatch {
  op: 'replace' | 'append' | 'remove';
  selector: string;
  html: string;
}

// DomEvent: typed based on the route's declared event schema
interface DomEvent { tag: string; payload: unknown }
```

Is a recursive session type `LiveSession = Send<HtmlPatch, Recv<DomEvent, LiveSession>>`
within TypeScript's depth limits? (Plans 87/88 validated recursive types up to some depth.)
Does `Dual<LiveSession>` produce the correct client type?

**Q2 — Route integration via `liveRoute()`**

```typescript
const routes = defineRoutes([
  liveRoute({
    tag: 'counter',
    path: 'counter',
    render: (state: CounterState) => `<div id="count">${state.count}</div>`,
    events: z.discriminatedUnion('tag', [
      z.object({ tag: z.literal('increment') }),
      z.object({ tag: z.literal('decrement') }),
    ]),
    reduce: (state: CounterState, event: CounterEvent) => ({ ...state, count: state.count + 1 }),
  })
]);
```

`liveRoute()` would wrap `wsRoute()` with a session type derived from the `render` /
`events` / `reduce` triple. The server-side handler receives typed events; the client
only needs to send what the schema declares.

**Q3 — Client runtime**

What is the minimum client-side runtime? Requirements:
- Open a WebSocket to the `liveRoute` URL.
- Apply HTML patches using a DOM diffing library (morphdom, idiomorph, or custom).
- Capture DOM events (form submit, button click) and send them typed to the server.
- No client-side routing — the server drives all state changes.

Could this client runtime be a tiny `<script>` (< 5 KB) that boots from the server-rendered
HTML? The `liveRoute()` server handler would include a `<script src="/arbor-live.js">` in
its initial render.

**Q4 — Integration with plan 123 (`renderRoute`)**

A `liveRoute` render function runs on the server. On the initial HTTP request, it should
render the initial HTML via plan 123's `renderRoute()`. On subsequent events over WebSocket,
it renders incremental patches. Can the same render function be used for both?

**Q5 — Feasibility check**

Can the session type `Send<HtmlPatch, Recv<DomEvent, LiveSession>>` be expressed within
TypeScript's current depth limits (validated in plan 87)? Is the recursive `LiveSession`
type instantiation-safe? Prototype in a scratch file.
