# Plan 154 — Relative Route Paths and Section Composition in `@arbor/api`

## Context

After Plans 152–153, the router supports `section(path, RouterContract)` with full
type safety. This plan completes the composition story by migrating `@arbor/api`'s
route path strings from absolute to relative and rewiring `api/index.ts` to use a
single `defineRoutes` + `createServer` with no fallback chain.

Today, every route in the API bakes in its full deployment path:

```
api/ledger/tasks      api/ledger/tasks/:id
api/hello             api/users             api/users/:id
```

This couples each domain module to its deployment topology. The ledger routes cannot
be renamed or moved without editing the route definition files. The UI's typed client
imports `ledgerRouter` and calls `createClient('http://localhost:3001', ledgerRouter)`,
relying on the router's absolute paths to build URLs.

## Goal

Domain modules define routes relative to their sub-tree root. The deployment topology
(`api/`, `api/ledger/`, etc.) lives in exactly one place — `api/index.ts`. The UI
client uses a base URL that includes the mount path so it remains correct without
knowing the router's internal paths.

```typescript
// ledger/router.ts — context-free
httpRoute(GetTasks, 'GET', 'tasks',      { … })
httpRoute(GetTask,  'GET', 'tasks/:id',  { … })
httpRoute(PatchTaskStatus, 'PATCH', 'tasks/:id',       { … })
httpRoute(PatchTaskRank,   'PATCH', 'tasks/:id/rank',  { … })

// routes/hello.router.ts
httpRoute(HelloRoute, 'GET', 'hello', { … })

// routes/users.router.ts
httpRoute(ListUsers,  'GET',  'users',     { … })
httpRoute(GetUser,    'GET',  'users/:id', { … })
httpRoute(CreateUser, 'POST', 'users',     { … })

// api/index.ts — single composed server
const apiServer = createServer(
  defineRoutes([
    section('api', [
      section('hello',  helloRouter),
      section('users',  usersRouter),
      section('ledger', ledgerRouter),
    ]),
  ]),
  {
    hello:                   async (ctx)  => …,
    'users-list':            async (_ctx) => …,
    'users-get':             async (ctx)  => …,
    'users-create':          async (ctx)  => …,
    'ledger-get-tasks':      async (_ctx) => …,
    'ledger-get-task':       async (ctx)  => …,
    'ledger-patch-task-status': async (ctx) => …,
    'ledger-patch-task-rank':   async (ctx) => …,
  },
);

app.all('/api/*', async (c) => {
  const result = await apiServer.handleRequest(c.req.raw);
  return c.json(result.body, result.status as any);
});
```

Note: the section constraint from Plan 153 applies. Sub-routers must have routes with
at least one segment so that after the section consumes its own path, the children
still have remaining URL segments to match against. The restructuring above satisfies
this:

| Section mount    | Sub-router routes       | URL matched          |
|------------------|-------------------------|----------------------|
| `api`/`hello`    | `hello`                 | `/api/hello`         |
| `api`/`users`    | `users`, `users/:id`    | `/api/users[/:id]`   |
| `api`/`ledger`   | `tasks`, `tasks/:id`, … | `/api/ledger/tasks…` |

Wait — the `hello` sub-router only has a single route at `'hello'`. Wrapping it in
`section('hello', helloRouter)` inside the outer `section('api', [...])` would result
in path `api/hello/hello`. Instead, the hello route should be at `''` (empty) under
the `section('hello', ...)` mount, OR the outer section should be just `'api'` and
`helloRouter` should expose `'hello'` as a direct child rather than via an additional
section nesting.

**Revised structure**: `helloRouter` keeps its leaf path as `'hello'`; it is NOT
wrapped in a further `section('hello', …)`. The `section('api', [...routes])` pattern
is used at a single level:

```typescript
defineRoutes([
  section('api', [
    ...helloRouter.children,       // route: 'hello'
    section('users',  usersRouter),  // routes: '', ':id' after consuming 'users'
    section('ledger', ledgerRouter), // routes: 'tasks', 'tasks/:id', …
  ]),
])
```

For `usersRouter`, routes are:
- `''` — matches `/api/users` exactly (after `section('users')` consumes `users`)
- `':id'` — matches `/api/users/123`

This requires the walk to match a zero-segment child route when the section has
consumed all segments up to the resource root. **Verify this works** against the
walk algorithm before committing to this structure; if zero-segment routes under a
section do not match, use `'users'` and `'users/:id'` as the route paths and drop the
`section('users', …)` wrapper (mount `usersRouter` at `'api'` alongside the hello
routes).

The simplest guaranteed-correct structure avoids zero-segment children entirely:

```typescript
defineRoutes([
  section('api', [
    httpRoute(HelloRoute, 'GET', 'hello', { … }),   // inline or from helloRouter.children
    ...usersRouter.children,    // 'users', 'users/:id'
    section('ledger', ledgerRouter),
  ]),
])
```

Implementation should verify and choose the right structure.

## Change Surface & T-shirt Sizing

**Size: S**

Files modified:

- `packages/api/src/ledger/router.ts` — strip `api/ledger/` prefix from all paths
- `packages/api/src/routes/hello.router.ts` — keep `'hello'` path (no change needed if above structure used)
- `packages/api/src/routes/users.router.ts` — strip `api/` prefix: `'users'`, `'users/:id'`
- `packages/api/src/index.ts` — single `defineRoutes + createServer`; single
  `app.all` adapter; remove two-server fallback chain and manual body parse
- `packages/ui/src/ledger/ledger.env.ts` — update `createClient` base URL from
  `'http://localhost:3001'` to `'http://localhost:3001/api/ledger'`

## Dependencies

- Plan 153 — `section()` accepts RouterContract (required for `section('ledger', ledgerRouter)`)

## Success Criteria

- `pnpm --filter @arbor/api typecheck` clean.
- `pnpm --filter @arbor/ui typecheck` clean.
- `GET /api/hello`, `GET /api/users`, `POST /api/users`, `GET /api/users/:id`,
  all ledger endpoints respond correctly (manual smoke test or existing API tests).
- `api/index.ts` contains exactly one `createServer` call and one `app.all` handler.
- No `.children` spread in `api/index.ts`.
- The `ledger/router.ts` route paths contain no `api/` prefix.
