# PLAN.111 — Route-Level Meta Tags (`<title>`, `<meta>`)

## Context

Every navigation should update `document.title` and relevant `<meta>` tags (description,
OG tags). Currently this requires per-component `useEffect` calls that are easy to forget
and hard to keep consistent. Route-level meta declarations make it part of the route
definition and enforce consistency across the application.

## Goal

1. A `withMeta(fn)` route builder (or `_meta` extension) that attaches a typed meta
   function to a route node: `(params: RouteParams) => { title: string; description?: string }`.
2. `BrowserRouter` calls the meta function on every committed navigation and updates
   `document.title` and `<meta name="description">` automatically.
3. `useRouteMeta()` hook — returns the resolved meta for the current route; for SSR or
   for custom `<Head>` components.
4. Meta functions receive the typed route params — a title like
   `(r) => \`User: ${r.id}\`` is type-safe.

## API Surface

```typescript
const router = defineRoutes([
  route({ tag: 'user', path: ':id' })
    .withMeta(({ id }) => ({ title: `User ${id}`, description: 'User profile page' })),
]);

// Or as a separate map (preferred if route-attached is not chosen in plan 100):
const metaMap = createMetaMap(router, {
  user: ({ id }) => ({ title: `User ${id}` }),
});
```

## Open Question

Should meta be route-attached (cleaner co-location) or a separate map (no coupling to
`@arbor/router`)? Follow the same decision made in plan 100/101 for consistency.

## Success Criteria

- `document.title` updates on every committed navigation.
- Meta function receives typed params (no `any`).
- `useRouteMeta()` returns the correct resolved meta object.
- Routes without meta attached do not throw — default to no-op.
