# Plan 189 — Spike: React Lifecycle Cruft Elimination

## Context

The UI components use `useEffect` for three distinct purposes that deserve three different
treatments:

| Category | Example | Right tool |
|---|---|---|
| **Bootstrap** | `useEffect(() => { send({ tag: 'fetch' }); }, [])` | Typed `onMount` in `useStore` |
| **State-change reaction** | `useEffect(() => scrollIntoView(), [selectedIndex])` | Injectable env dep or `watch` |
| **Ongoing subscriptions** | `useLedgerKeyboard` — adds/removes keydown listener | Declarative `Sub<A>` managed by runtime |

Category 3 (`AuthEnvProvider` cleanup `useEffect`s) is correct and untouched.

We explored the full design space before committing:

**Ruled out — `present(selector, fn)` third return from `useStore`:**
Elegant call-site ergonomics but fatally weak on testing. Callbacks defined in component
body can only be exercised by rendering the full component and mocking DOM APIs. The env-
threading approach lets store unit tests spy on a `scroll: vi.fn()` dependency with no DOM
at all.

**Ruled out — Component factory (`createStoreComponent`):**
Fights React's composition model. Suspense, error boundaries, `forwardRef`, context — all
assume plain function components. Framework churn for marginal gain.

**Ruled out — Signals / `useProxy`:**
Fights React's rendering model. `useProxy` + mutable proxy in render is incompatible with
StrictMode double-invocation and makes snapshot-based tests unpredictable.

**Prior art that informs this spike:**
- **MobX `reaction(selector, effect)`** — production-validated selector/effect split
- **Elm `Cmd` vs `Sub`** — one-shot effects vs ongoing subscriptions deserve separate types;
  the runtime diffs subscriptions each render and owns listener lifecycle
- **Vue `watch(source, cb, { flush: 'post' })`** — post-DOM timing is a real primitive, not
  a `useLayoutEffect` workaround
- **RTK `createListenerMiddleware`** — react to transitions (actions), not state values
- **valtio `subscribeKey`** — already in the dep graph; fine-grained key-level subscription

## Goal

Run four time-boxed experiments, each eliminating a class of `useEffect` from component
files. Measure ergonomics, testability, and correctness for each. Decide which to promote
to a proper implementation task.

**Target end-state for a component:**

```ts
// No useEffect import. No useRef for env. No custom keyboard hook.
export function LedgerTable({ env }: Props) {
  const [$, send] = useStore(ledgerReducer, env, initialLedgerState, { tag: 'fetch' });
  // render only
}
```

## Experiments

### A — `onMount` typed action in `useStore`

```ts
useStore(reducer, env, initialState, { tag: 'fetch' })
```

A single `useEffect([], () => send(onMount))` lives inside `useStore`. Components never
write init `useEffect`s again. The business logic of what mount *means* stays in the
reducer where it can be unit-tested as a plain action.

**Test:** `reducer(initialState, { tag: 'fetch' }, testEnv)` — no rendering, no DOM.

**Validate:** StrictMode fires effects twice in dev mode. Mount action should be idempotent
or guarded; confirm the store's single-send semantics hold.

---

### B — Factory env initializer in `useStore`

```ts
useStore(
  reducer,
  () => ({ ...liveLedgerEnv, scroll: () => rowRef.current?.scrollIntoView(...) }),
  initialState,
  { tag: 'fetch' },
)
```

If `environment` is a function, `useStore` calls it once inside the ref initialisation —
identical to the `useState(() => expensiveInit)` lazy-init pattern React developers already
know. The resulting env object is stable across all renders. Components need no `useRef`
boilerplate for env construction.

`scroll: () => void` on the env type means the reducer says *when* to scroll; the env says
*how*. The test env provides `scroll: vi.fn()` — store unit tests assert scroll was triggered
by the correct action with no DOM involved.

**Test:**
```ts
const scrollSpy = vi.fn();
const env = { ...baseTestEnv, scroll: scrollSpy };
store.send({ tag: 'selectRow', index: 2 });
expect(scrollSpy).toHaveBeenCalled();
```

**Validate:** env object identity is stable across re-renders; `setDependencies` receives the
same reference every render; in-flight async effects see a consistent env.

---

### C — `watch(selector, fn)` from `useStore`

```ts
const [$, send, watch] = useStore(reducer, env, initialState, { tag: 'fetch' });

watch(
  s => s.selectedIndex,
  () => rowRef.current?.scrollIntoView({ block: 'nearest' }),
);
```

`watch` is an alternative to env-threading for presentation-layer reactions where injecting
a dep into the env would feel like overreach. Internally backed by valtio `subscribeKey` or
`subscribe` — no `useEffect` in `useStore` for this path, just a reactive subscription on
the proxy. All `watch` registrations are collected each render and diffed against the live
subscription set.

Compare directly with Experiment B on the testability axis:
- B (`scroll` in env): store unit-testable, no DOM, reducer expresses causality
- C (`watch`): component-level, requires rendering, but no env change needed

**Test:** confirm `fn` fires only when `selector` result changes — not on unrelated state
mutations — and that subscriptions are torn down on unmount.

**Decision gate:** if C's testability story is materially weaker than B, drop it. The goal
is not an elegant API in spite of bad tests; it is both.

---

### D — Declarative `Sub<A>` runtime (Elm-style)

The deepest experiment. Keyboard listeners are ongoing subscriptions, not one-shot effects.
`useLedgerKeyboard` encodes this as a `useEffect` in a custom hook — but the *conditions*
under which the subscription is active (loaded state, which keys are valid) are business
logic that belongs in the reducer layer.

Proposed shape:

```ts
// Sub type — a declared ongoing subscription
type Sub<A> =
  | { tag: 'keydown'; handler: (e: KeyboardEvent) => A | null }
  | { tag: 'none' }

// Store descriptor
const ledgerDescriptor = {
  reducer: ledgerReducer,
  subscriptions: (state: LedgerState): Sub<LedgerAction>[] =>
    state.loadState.tag === 'loaded'
      ? [Sub.keydown(e => keyboardAction(e, state))]
      : [],
};

// useStore diffs subscriptions each render; adds/removes listeners
const [$, send] = useStore(ledgerDescriptor, env, initialState, { tag: 'fetch' });
```

`useLedgerKeyboard` disappears entirely. The conditions for keyboard handling are visible in
the store descriptor alongside the reducer — not hidden in a hook. `useStore` becomes the
sole owner of `addEventListener` / `removeEventListener` for all declared subscriptions.

**Test:**
```ts
// Sub declarations are pure data — assert the right Sub is returned for each state:
expect(subscriptions(loadedState)).toEqual([Sub.keydown(expect.any(Function))]);
expect(subscriptions(loadingState)).toEqual([]);
```

**Validate:** subscription diffing is correct across state transitions; cleanup fires on
state change (loaded → loading) and on unmount; no double-bind in StrictMode.

## Post-mortem

**Root cause 1 — missing `expect` import**: First red run on the scroll spy test. Pre-flight check: always grep for `expect` in the import list when writing assertion-heavy tests in files that don't already import it.

**Root cause 2 — `TestStore` mutating shared fixtures via valtio proxy**: The hardest failure. `TestStore(reducer, env, loadedState)` wraps `loadedState` in a valtio proxy without cloning. Filter mutation tests (`setTextFilter`, etc.) wrote through to `loadedState.filters`; rank/status tests replaced `loadedState.loadState` with `emptyGroups`. By the time the new subscription tests ran, the fixture was corrupt. Fix: `structuredClone(initialState)` in `TestStore` constructor — the invariant "TestStore owns its state" should have been there from the start. Pre-flight check for future tests that read shared module-level fixtures: verify no earlier test holds a proxy over the same reference.
