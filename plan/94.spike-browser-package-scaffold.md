# PLAN.94 (Spike) — Browser Package Scaffold

## Context

Before building `packages/router-browser` and `packages/router-react`, we need to know
the minimal package structure and testing setup. Key unknowns: does `vitest` with `jsdom`
adequately simulate `window.history` and `popstate` events for unit tests, or do we need
`vitest --browser` (Playwright-backed)? Do the existing `tsconfig` patterns in the
workspace extend cleanly to browser-targeted packages, or do we need `lib: ['DOM']` opt-in?

## Goal

Produce a validated scaffold for `packages/router-browser` with:

1. `package.json` — correct `exports`, `peerDeps`, workspace deps.
2. `tsconfig.json` — `lib` includes `DOM`; extends workspace base.
3. `vitest.config.ts` — environment: `jsdom`; confirm `window.history.pushState` and
   `popstate` are accessible in tests.
4. A single smoke-test confirming `window.location`, `history.pushState`, and a `popstate`
   event subscription all work as expected in the test harness.

## Deliverable

- Scaffold files for `packages/router-browser` with passing test.
- Short (≤ 200-word) decision memo: jsdom sufficient? any `vitest --browser` edge cases
  encountered? recommended tsconfig `lib` entries.

## Change Surface

- New: `packages/router-browser/package.json`
- New: `packages/router-browser/tsconfig.json`
- New: `packages/router-browser/vitest.config.ts`
- New: `packages/router-browser/src/index.ts` (stub export)
- New: `packages/router-browser/src/scaffold.test.ts`

## Dependencies

None — this spike has no upstream plan dependencies.

## Success Criteria

- `pnpm --filter @arbor/router-browser test` passes.
- `pnpm --filter @arbor/router-browser typecheck` passes.
- Decision memo confirms jsdom is sufficient (or escalates to `vitest --browser`).

## Time-box

One session. Do not implement `BrowserRouter` — scaffold only.

## Decision Memo

**jsdom is sufficient for unit tests.** All four smoke tests pass:
`window.location`, `history.pushState`, `history.replaceState`, and `popstate` via `history.back()` all behave correctly under jsdom 25.

**tsconfig**: `lib: ["ES2022", "DOM", "DOM.Iterable"]` in the package-level tsconfig (extending workspace base) is all that is needed. No changes to the root `tsconfig.base.json`.

**`vitest --browser` not needed at this stage.** jsdom handles History API and event dispatch. The `--browser` mode (Playwright) would be needed only if we test real navigation UX, `<a>` click capture, or scroll restoration — none of which are part of the router-browser core logic.

**Recommended next step**: Plan 95 can implement `BrowserRouter` against this scaffold.
