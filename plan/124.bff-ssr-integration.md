# PLAN.124 — BFF SSR Integration

## Context

The BFF (`packages/bff`) currently serves `index.html` as a catch-all for all non-API
routes. SSR replaces that catch-all with `renderRoute()` (plan 123): the BFF parses the
URL, runs loaders on the server, renders the React tree to streaming HTML, and returns it.

The BFF already uses Hono (`fetch`/`Response` model), which is directly compatible with
`renderRoute()`'s `Response` return type. The wiring is minimal.

## Goal

Replace the production catch-all in `packages/bff/src/index.ts`:

**Before:**
```typescript
app.get('*', serveStatic({ path: path.join(uiDist, 'index.html') }));
```

**After:**
```typescript
import { renderRoute } from '@arbor/router-react/server';
import { App } from '@arbor/ui';          // shared React app component
import { router, loaderMap, metaMap } from '@arbor/ui/router'; // shared exports

app.get('*', async (c) => {
  const response = await renderRoute(c.req.raw, router, App, {
    loaderMap,
    metaMap,
    bootstrapScripts: ['/assets/main.js'], // Vite manifest lookup
  });
  return response;
});
```

## Package Boundary

`App`, `router`, `loaderMap`, and `metaMap` must be exported from `packages/ui` (or a
new shared package) so both the browser entry point (`main.tsx`) and the BFF can import
them. The rule: nothing in these exports can import browser-only globals at module load
time (no `window.*` at the top level; only in function bodies).

This requires auditing `packages/ui` imports and ensuring:
- The router definition has no browser-only side effects.
- Loader functions use injected `fetch` (plan 101 constraint) or `globalThis.fetch` lazily.
- The `App` component renders correctly without `window` (no `useEffect`-only logic that
  crashes on the server — those `useEffect`s simply don't run during SSR).

## Vite Manifest Integration

The BFF needs to serve the correct JS bundle URL in `bootstrapScripts`. In production,
Vite generates a `dist/manifest.json` with hashed asset filenames. The BFF reads this
manifest to find the correct `main.js` path:

```typescript
const manifest = JSON.parse(fs.readFileSync(path.join(uiDist, '.vite/manifest.json')));
const mainScript = manifest['src/main.tsx'].file; // → "assets/main-abc123.js"
```

## Development Mode

In development, Vite runs as a separate server (current setup). SSR in development adds
complexity (Vite's SSR mode with `vite.createServer({ appType: 'custom' })`). Options:

1. **Dev SSR**: wire Vite's SSR transform into the BFF dev server. More complex, accurate.
2. **Dev SPA**: keep the current dev SPA mode; SSR only in production. Simpler; risk of
   SSR-only bugs going undetected in dev.
3. **Hybrid**: add a `VITE_SSR=true` env flag to opt into SSR during development.

Recommended: Option 2 (dev SPA) initially; Option 3 as a follow-up.

## Change Surface

- Update: `packages/bff/src/index.ts` — SSR catch-all
- New: `packages/ui/src/router.ts` — shared exports (`router`, `loaderMap`, `metaMap`)
- Update: `packages/ui/src/main.tsx` — import from `router.ts` instead of defining inline
- Update: `packages/bff/package.json` — add `@arbor/router-react` dependency

## Dependencies

- Plan 123 (`renderRoute` must be implemented).
- Plans 101, 111 — `loaderMap` and `metaMap` must be defined.
- `packages/ui` — must export `App`, `router`, `loaderMap`, `metaMap`.

## Success Criteria

- Production BFF returns server-rendered HTML with the correct route content.
- `curl http://localhost:{PORT}/users/42` returns HTML containing user content (not a blank
  `<div id="root"></div>`).
- Existing API and auth routes are unaffected.
- Client hydration works: `hydrateRoot` picks up the server HTML with no mismatch warnings.
- The JS bundle (bootstrapScript) loads and hydrates correctly.
