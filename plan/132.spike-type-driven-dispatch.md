# PLAN.132 (Spike) — Type-Driven Dispatch Optimization

## Context

Plan 74 (deferred, benchmark-gated) covers a radix tree replacement for `walkParse()`.
This spike takes a different angle: because the route tree is fully typed at compile time
and its structure is statically known, there may be optimization strategies unavailable
to a generic radix tree — approaches inspired by how compilers and type-directed systems
(GHC, Rust, Idris) specialize dispatch at build time.

The key asymmetry: a generic HTTP router (like Fastify's `find-my-way`) must handle
any tree registered at runtime. `@arbor/router`'s tree is fixed at module load time and
never changes. This "closed world" property opens optimization paths that generic routers
cannot use.

## Questions to Answer

**Q1 — Segment type discrimination**

Segments have statically known kinds: `literal`, `string-param`, `num-param`,
`optional-string`, `optional-num`, `wildcard`. For a tree where the first segment
is a literal, the dispatch can short-circuit entirely without walking the tree if
`url.startsWith('/known-prefix')` fails. Is this already happening? If not, a one-pass
prefix filter on the literal root segments costs O(1) and eliminates the majority of
candidates before the walk begins.

For numeric params (`#id`), parsing requires `parseInt` + round-trip check. If a route's
second segment is `#id` and the URL's second segment is not a decimal integer, the entire
subtree can be skipped without descending. Is `walkParse()` currently doing this
short-circuit, or does it attempt all children?

**Q2 — Compile-time dispatch specialization**

At module load time, all route paths are known. A build step (or a `defineRoutes()`
postprocess) could emit a specialized dispatch function — a cascaded `if/switch` instead
of a generic tree walk:

```typescript
// Generated at route definition time (not a code generator — runtime generation):
function specializedParse(segments: string[]): Result<Route, string> {
  if (segments[0] === 'users') {
    if (segments[1] === undefined) return ok({ tag: 'users' });
    const id = segments[1];
    if (segments[2] === undefined) return ok({ tag: 'user', id });
    if (segments[2] === 'posts') return ok({ tag: 'user-posts', id });
    return err('no match');
  }
  if (segments[0] === 'settings') return ok({ tag: 'settings' });
  return err('no match');
}
```

This is not a separate code generator — `defineRoutes()` itself produces this function
at startup (similar to how regex engines compile patterns to automata). The generated
function is a plain closure with no allocation on the hot path.

Is the compilation step (walk tree → emit closure) fast enough that it does not
meaningfully increase `defineRoutes()` startup time?

**Q3 — Comparison with Rust/Axum dispatch**

Axum uses a `MatcherState` that pre-compiles routes into a DFA-like structure. Hono uses
a trie with per-segment discriminated edges. `uWebSockets` uses a compiled regex automaton.
Each exploits closed-world assumptions that `@arbor/router` also has.

Which approach fits best given arbor's segment kinds (`literal`, `:param`, `#param`,
`:opt?`, `#opt?`, `*wildcard`) and the requirement that params be captured by name (not
just position)?

**Q4 — Interaction with the type system**

The type system knows the exact shape of every route. Could the specialized dispatch
function be type-checked against the `ChildUnion<C>` type so that a missing branch is a
compile error? This would mean: the dispatch specialization is not just faster but
provably complete (no missing route case can slip through to a runtime 404).

**Q5 — Benchmark baseline**

Before optimizing: what is the actual cost of `walkParse()` on representative trees?
Measure three sizes: 20 routes (typical small API), 100 routes (medium), 500 routes
(large). Compare: current `walkParse()` vs. a hand-written specialized switch vs. a
`find-my-way` radix tree (from Fastify) used as a reference point.
