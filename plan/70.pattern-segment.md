# Plan 70 — Pattern segment kind for routing-time format constraints

## Context

All string capture currently uses `str` (`:id`) which accepts any non-empty path segment.
Format validation is deferred to the Zod schema. This works but has two practical gaps:

**1. Route disambiguation at parse time.**  
If `/users/me` and `/users/:id` both exist, ordering `lit 'me'` before `:id` works. But
if the distinction is format-based — e.g., `:id` should only match UUIDs while other
strings go elsewhere — there is no routing-level mechanism. Both routes use `str` and order
determines which wins, regardless of the segment's content.

**2. Zod schema shape mismatch.**  
A `z.string().uuid()` refinement on a `:id` param field validates correctly but the error
is a Zod schema failure, not a routing failure. The parse `Result` is an error, not a miss;
the router does not try the next sibling route. The user gets a 400 or a parse failure
instead of routing to the next candidate.

A pattern segment kind (`~id:<regex>` syntax or similar) would let the router reject a
segment at match time and continue trying siblings, the same way `num` already does for
non-numeric values.

## Proposed syntax

```
:id              → str      (any string)
#id              → num      (integer only, after Plan 67)
~id:[0-9a-f]{8} → pattern  (must match /^[0-9a-f]{8}$/)
```

The regex is embedded in the path string, delimited by `:` after the param name.

## Segment type addition

```ts
| { kind: 'pattern'; name: string; regex: RegExp }
```

`parseSegments` extracts the regex string from the path token and compiles it anchored:

```ts
if (s.startsWith('~') && s.includes(':')) {
  const colonIdx = s.indexOf(':');
  const name = s.slice(1, colonIdx);
  const regex = new RegExp(`^${s.slice(colonIdx + 1)}$`);
  return { kind: 'pattern', name, regex };
}
```

`matchSegments` tests the segment:

```ts
case 'pattern':
  if (url == null || !seg.regex.test(url)) return null;
  next[seg.name] = url;
  urlIndex++;
  break;
```

`buildUrl` treats `pattern` the same as `str` (encode the value).

OpenAPI: emit as `{ type: 'string', pattern: '<regex>' }` in path parameter schema.

## Design decisions to resolve before implementation

1. **Syntax**: `~id:[0-9a-f]{8}` is compact but regex syntax in path strings is fragile
   (slashes, special chars). Alternative: a separate `patternRoute()` helper that accepts
   the regex out-of-band rather than embedding it in the path string.

2. **Regex compilation timing**: compile once at `parseSegments` call (route definition
   time) vs. lazily. Once at definition time is correct.

3. **Case sensitivity**: default to case-sensitive. No flag syntax planned.

4. **`opt-pattern`**: optional pattern segment — defer; adds syntax complexity with unclear
   use cases.

5. **TypeScript type for the captured value**: always `string` (the regex validates format,
   not type). No change to route schema inference.

## Dependencies

Plan 67 (integer validation for `num`) establishes the pattern for routing-level type
constraints; Plan 70 extends it to arbitrary regex. Implement 67 first.

## Priority

LOW — this is an enhancement, not a correctness fix. Implement after Plans 67–69.

## Success Criteria

- `parseSegments('users/~id:[0-9]{8}')` produces `[lit 'users', pattern 'id' /^[0-9]{8}$/]`.
- `matchSegments` accepts `'12345678'` and rejects `'abc'` and `'1234567'` (7 digits).
- Routes with `pattern` segments try the next sibling when the segment doesn't match,
  same as `num` does for non-numeric input.
- OpenAPI emits `{ in: 'path', schema: { type: 'string', pattern: '...' } }`.
- `pnpm test && pnpm typecheck` green.
