# Plan 156 — Spike: Method-Dispatched TypedClient API

## Context

The current `TypedClient` exposes a single `fetch()` method:

```typescript
client.fetch({ tag: 'get-me' }, { headers: { authorization: 'Bearer alice' } })
client.fetch({ tag: 'create-post' }, { body: { title: 'Hello world' } })
```

The HTTP method is already encoded in the route definition via `httpRoute(Tag, 'GET', 'path', ...)` but is invisible at the call site. There is no type-level constraint preventing a caller from accidentally using a mutating route as a read. Surfacing the method as the call method (`client.get`, `client.post`, etc.) gives the type system a place to enforce it and aligns with the HTTP/REST vocabulary frontend developers think in.

The precedent for method-first design exists in `packages/common/src/http/client.ts` (the pre-router HTTP client), but that design couples methods to raw URL strings, losing response type inference. This spike validates whether method dispatch and tag-based type inference can be combined cleanly.

## Goal

Replace `TypedClient.fetch()` with per-method dispatch:

```typescript
client.get({ tag: 'get-me' }, { headers: { authorization: 'Bearer alice' } })
client.post({ tag: 'create-post' }, { body: { title: 'Hello world' } })
client.patch({ tag: 'patch-task-status' }, { body: { status: 'done' } })
client.delete({ tag: 'delete-user' })
```

Key invariants:
- `client.get(...)` is a compile error if the route's HTTP method is not `GET`.
- Response type is inferred from the route's `response` schema (same as today).
- `body` and `headers` requirements carry over from the existing `RequestOpts` conditional type.
- No change to `RouterContract`, `httpRoute`, or `RouteNode` shapes.
- `fetch()` is removed — no deprecated fallback.

## Design Sketch

### Type-level method filter

Add a `method` type parameter to `HttpContext` (currently `HttpContext<Path, Schema, Body, Headers, Response, SectionParams, Query>`):

```typescript
type TagsByMethod<Map extends AnyCtxMap, Method extends string> =
  keyof { [K in keyof Map as Map[K]['method'] extends Method ? K : never]: Map[K] };
```

This is additive — `Method extends string = string` defaults cleanly. Verify it doesn't break existing usages of `HttpContext`.

The alternative (traversing the route tree to read `_meta.method` at the type level) is fragile under deep nesting and is not worth pursuing.

### Updated TypedClient interface

```typescript
export interface TypedClient<Route, Map extends AnyCtxMap, Validate extends boolean = false> {
  get<Tag extends TagsByMethod<Map, 'GET'>>(
    route: Extract<FlattenRouteLeaf<Route>, { tag: Tag }>,
    opts?: RequestOpts<Map[Tag]>,
  ): Promise<ResponseType<Map[Tag], Validate>>;

  post<Tag extends TagsByMethod<Map, 'POST'>>(
    route: Extract<FlattenRouteLeaf<Route>, { tag: Tag }>,
    opts?: RequestOpts<Map[Tag]>,
  ): Promise<ResponseType<Map[Tag], Validate>>;

  put<Tag extends TagsByMethod<Map, 'PUT'>>(
    route: Extract<FlattenRouteLeaf<Route>, { tag: Tag }>,
    opts?: RequestOpts<Map[Tag]>,
  ): Promise<ResponseType<Map[Tag], Validate>>;

  patch<Tag extends TagsByMethod<Map, 'PATCH'>>(
    route: Extract<FlattenRouteLeaf<Route>, { tag: Tag }>,
    opts?: RequestOpts<Map[Tag]>,
  ): Promise<ResponseType<Map[Tag], Validate>>;

  delete<Tag extends TagsByMethod<Map, 'DELETE'>>(
    route: Extract<FlattenRouteLeaf<Route>, { tag: Tag }>,
    opts?: RequestOpts<Map[Tag]>,
  ): Promise<ResponseType<Map[Tag], Validate>>;
}
```

`ResponseType<Ctx, Validate>` is the existing conditional on `HttpResponseUnion` already in fetch-client.ts.

### Runtime impact

The `methodMap` runtime lookup is eliminated — the method is known from which function was called. The implementation of each method is identical to the current `fetch()` body with the HTTP method hardcoded.

### Open questions to resolve during spike

1. **Does `HttpContext` expose `method` at the type level today?** Check `buildHttpContext` in `http-context.ts`. If not, adding `Method extends string = string` should be cheap.
2. **Does `TagsByMethod` narrow correctly** when `Map` is a concrete mapped type? Write a scratch `expectTypeOf` test before committing to the interface shape.
