# Plan 190 — TCA scroll reclassification: remove scroll from LedgerEnv

## Context

Plan 189 delivered four experiments (A–D) eliminating `useEffect` from component files.
Experiment B placed `scroll: () => void` in `LedgerEnv` so that `selectUp`/`selectDown`
could return a scroll effect from the reducer, enabling store-level spy tests without DOM
rendering.

During review, this was identified as a **category error**. `scroll` is not a domain
service — it is a view reaction (move the viewport when `selectedIndex` changes). Placing
it in `LedgerEnv` forced the interface to carry a dead stub:

```ts
// liveLedgerEnv — this is never called; always overridden by the component factory
scroll: () => { return; },
```

This stub violates the principle of least astonishment: any reader will ask why the
field exists, and the honest answer is "to satisfy a type constraint," not "because it
does something meaningful."

## Category map

The three categories that emerged from plan 189, and where each concern belongs:

| Concern | Right home | Examples |
|---|---|---|
| Data services | `LedgerEnv` | `fetchQueue`, `setStatus`, `setRank`, `pollTick` |
| Presentation reactions | Component via `watch` | scroll, focus, animation |
| Ongoing subscriptions | `subscriptions` (Exp D) | keyboard, websocket, timer |

`scroll` belongs in the middle row. The `watch(selector, fn)` primitive (Exp C) was built
exactly for this purpose.

## Goal

1. Remove `scroll` from `LedgerEnv` and from `liveLedgerEnv`/`mockLedgerEnv`.
2. Remove the scroll `Effect` returns from `selectUp`/`selectDown` in the reducer.
3. Replace the component factory's `scroll` override with a `watch` call.
4. Remove the store-level scroll spy test — it was testing a presentation concern at the
   wrong layer. The existing component tests cover the full pipeline via the global
   `scrollIntoView` mock.

## Design

```ts
// ledger.env.ts — scroll field gone
export interface LedgerEnv {
  fetchQueue: Effect<Result<DisplayGroupsResponse, string>>;
  setStatus: (id: number, status: TaskStatus) => Effect<undefined>;
  setRank: (id: number, rank: number) => Effect<undefined>;
  pollTick: Effect<undefined>;
  fetchPlanDoc: (taskId: number) => Effect<Result<string, string>>;
}

// ledger.store.ts — selectUp/selectDown return null
case 'selectUp': {
  $.state.selectedIndex = Math.max(0, $.state.selectedIndex - 1);
  return null;
}
case 'selectDown': {
  $.state.selectedIndex = Math.min(action.rowCount - 1, $.state.selectedIndex + 1);
  return null;
}

// LedgerTable.tsx — watch instead of factory scroll override
const [$, send, watch] = useStore(
  withLogging('ledger', ledgerReducer),
  liveLedgerEnv,           // no factory needed; env has no component-local deps
  initialLedgerState,
  { tag: 'fetch' },
  ledgerSubscriptions,
);
watch(s => s.selectedIndex, () => {
  selectedRowRef.current?.scrollIntoView({ block: 'nearest' });
});
```

The factory form of `environment` is no longer needed for `LedgerTable` specifically
(its only reason was to inject the ref-aware scroll). The factory API stays in `useStore`
for cases where an env genuinely needs component-local closure.

## Trade-off acknowledged

Removing `scroll` from `LedgerEnv` means we can no longer write a zero-DOM store test
asserting "scroll fires after `selectDown`." That test was defended on testability grounds
during the spike, but it was testing the wrong layer — the reducer's job is to prove state
transitions are correct (selectedIndex increments), not to prove the view reacts to them.
The view-reaction coverage stays in the component tests via
`window.HTMLElement.prototype.scrollIntoView = vi.fn()`.
