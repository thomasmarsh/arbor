# Plan 160 — Rewrite `setup.sh` to Template from `.env.example`

## Context

`scripts/setup.sh` creates the three machine-local env files by embedding their content as heredocs inside the script itself:

```bash
create_env apps/api/.env.local "\
ARBOR_PG_URL=postgresql://arbor:arbor@localhost:5433/arbor_dev
..."
```

This creates a maintenance divergence: when a new env var is added to `apps/api`, it must be added in **two** places — `apps/api/.env.example` (the canonical template) and `scripts/setup.sh` (the bootstrap copy). In practice the script drifts. The existing `.env.example` files in `apps/api/` and `apps/bff/` are already committed and serve as the single source of truth for what variables exist and what shape they take; setup.sh should derive from them rather than duplicate them.

The one thing setup.sh does that `.env.example` cannot: it generates a cryptographically random `ARBOR_SESSION_SECRET` and splices it in. That logic should be preserved.

## Goal

Rewrite `setup.sh` to:

1. Copy `apps/api/.env.example` → `apps/api/.env.local` (skip if file exists).
2. Copy `apps/bff/.env.example` → `apps/bff/.env.local`, then splice a freshly-generated `ARBOR_SESSION_SECRET` into the copy.
3. Copy `apps/bff/.env.example` → `apps/bff/.env.staging.local` with staging-appropriate placeholder comments intact (skip if exists).

Adding a new env var to an app now requires only editing that app's `.env.example`. The setup script requires no change.

## Change Surface & T-shirt Sizing

**S** — one script rewritten, no other file changes.

| File | Change |
|------|--------|
| `scripts/setup.sh` | Replace heredoc `create_env` calls with `cp`-from-example + `sed` secret injection |

### Implementation sketch

```bash
copy_from_example() {
  local src="$1" dst="$2"
  if [[ -f "$dst" ]]; then
    echo "skipped  $dst (already exists)"
    return
  fi
  cp "$src" "$dst"
  echo "created  $dst"
}

SESSION_SECRET=$(node -e "process.stdout.write(require('crypto').randomBytes(32).toString('hex'))")

copy_from_example apps/api/.env.example        apps/api/.env.local
copy_from_example apps/bff/.env.example        apps/bff/.env.local
copy_from_example apps/bff/.env.example        apps/bff/.env.staging.local

# Splice generated secret into local files
sed -i '' "s/^ARBOR_SESSION_SECRET=.*/ARBOR_SESSION_SECRET=${SESSION_SECRET}/" apps/bff/.env.local
sed -i '' "s/^ARBOR_SESSION_SECRET=.*/ARBOR_SESSION_SECRET=${SESSION_SECRET}/" apps/bff/.env.staging.local
```

Note: `sed -i ''` is macOS syntax; the script already targets macOS (`darwin`). If Linux support is needed later, use a `SED_INPLACE` variable that branches on `$(uname)`.

## Dependencies

- Plan 158 must land first so `.env.example` files are verified complete and accurate before setup.sh trusts them as templates.

## Success Criteria

- Running `setup.sh` on a clean checkout (no `.env.local` files) produces `apps/api/.env.local`, `apps/bff/.env.local`, and `apps/bff/.env.staging.local`.
- Each file has content matching the corresponding `.env.example`, with `ARBOR_SESSION_SECRET` replaced by a 64-char hex string.
- Adding a variable to `apps/bff/.env.example` is immediately reflected the next time `setup.sh` runs on a clean checkout — no script edits required.
- `diff <(grep -v ARBOR_SESSION_SECRET apps/bff/.env.local) <(grep -v ARBOR_SESSION_SECRET apps/bff/.env.example)` produces no output.
