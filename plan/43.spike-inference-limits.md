# PLAN.43 (Spike) — Type inference depth and complexity limits

## Context

The type machinery in this project is built on deeply recursive conditional
types (`Derive`, `ChildUnion`, `Flatten`). TypeScript has two relevant hard
limits: a recursion depth cap (default ~100 levels) and a type instantiation
count limit that can cause "Type instantiation is excessively deep and
possibly infinite" errors. `tsc` also has a practical compiler performance
limit where large trees slow IDE autocomplete to unusable speeds.

We do not currently know:
1. At what route tree depth or breadth inference fails or degrades.
2. Whether the current `Derive`/`ChildUnion` implementation is at risk, or
   whether it uses structural patterns that avoid the recursion limit.
3. What the practical upper bound on a production route tree looks like.

This spike produces that data before we build features that might accelerate
the problem.

## Spike Deliverable

A benchmarking script at `scripts/inference-bench.ts` that:

1. Programmatically generates route trees of increasing depth (2, 5, 10, 20,
   50 levels) and breadth (2, 5, 10, 20 children per node).
2. Calls `tsc --noEmit` on a file that uses `InferRoute<typeof generatedRouter>`
   and records wall-clock time and whether the compiler errors.
3. Outputs a markdown table:

   | Depth | Breadth | tsc time | Status        |
   |-------|---------|----------|---------------|
   | 5     | 10      | 1.2s     | OK            |
   | 10    | 10      | 4.8s     | OK            |
   | 20    | 10      | —        | TS2589 error  |

A short memo (≤ 400 words) interpreting the results:
- Where is the practical limit?
- Is the bottleneck depth, breadth, or instantiation count?
- Are there simple type-level changes (e.g., `& {}` flattening, lazier
  evaluation, switching to mapped types) that push the limit out?
- Does the limit matter for realistic production route trees (typically
  < 200 routes)?

## Time-box

One session. If the benchmark reveals a real problem at realistic tree sizes,
escalate to a full optimization plan (reference `spec.enhancements.md`
Phase 4 — Type Evaluation Cache Engine).

## Dependencies

None — this spike is independent and can run at any point after the core
type machinery is stable (after plan 19).

## Success Criteria

- Benchmark script runs without modification on this repo.
- The memo answers the three questions above with data, not speculation.
- If a limit is found at a tree size likely to be hit in production,
  a new plan is opened to address it.
