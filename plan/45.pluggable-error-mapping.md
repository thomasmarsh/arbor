# PLAN.45 — Pluggable error mapping engine

## Context

`spec.enhancements.md` Phase 2, item 3. Right now the handler error boundary
(plan 25) catches all throws and returns a generic 500. Handlers that want to
return a typed 409 for a database unique-constraint error have to catch the
error themselves and return the appropriate response — boilerplate that every
route repeats.

A pluggable error mapper lets callers declare how application-level exceptions
map to typed HTTP responses, once at the server level, rather than per handler.

## Goal

`createServer` accepts an optional `errorMap` that maps exception classes (or
predicate functions) to typed response factories:

```typescript
const server = createServer(router, handlers, {
  errorMap: [
    {
      match: (e) => e instanceof UniqueConstraintError,
      response: () => ({ status: 409 as const, body: { error: 'conflict' } }),
    },
    {
      match: (e) => e instanceof NotFoundError,
      response: (e) => ({ status: 404 as const, body: { error: e.message } }),
    },
  ],
});
```

The server's handler error boundary (plan 25) checks the error against the
map in order. First match wins. If no entry matches, the existing 500 fallback
applies.

The response factory's return type must be assignable to one of the route's
declared response types — if the route doesn't declare a `409` response, the
`errorMap` entry for that route is a type error.

## Change Surface

1. **`server.ts`**: Extend `createServer` options to accept `errorMap`.
   Update the catch block to check the map before falling back to 500.
2. **`ErrorMap` type**: A discriminated list of `{ match, response }` entries.
   The `response` factory's return type is checked against the route's
   declared response union (best-effort; may need to be a runtime assertion
   if the full type-check is too complex).
3. Tests:
   - Handler throws a mapped error → correct status and body returned.
   - Handler throws an unmapped error → 500.
   - First matching entry wins (order matters).

## Dependencies

- Plan 25 (validation error boundary) — the catch block being extended here
  was introduced in plan 25.
- Plan 44 (middleware pipeline) — error mapping could be expressed as a
  terminal middleware; if 44 lands first, reconsider this as a built-in.

## Success Criteria

- Mapped errors return the declared status code, not 500.
- Unmapped errors still return 500.
- `npm test && npm run typecheck` clean.
