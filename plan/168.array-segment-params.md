# PLAN.168 — Array Segment Parameters

## Context

`segments.ts` TODO (line 1) lists `{ kind: 'array-str'; name: string; separator: string }`
as a candidate extension. Plan 167 assessed RFC 3986 matrix parameters as low-value but
noted separator-based array params (e.g., `/tags/a,b,c`) as practically useful and
independently worth evaluating.

The use case: a path segment whose value is a comma-separated list of scalars that the
router coerces to `string[]`. This is a real pattern in REST APIs (batch-by-ID endpoints,
tag filtering) and maps directly to OpenAPI's `style: simple, explode: false` on a path
parameter. Unlike matrix params, it has broad tooling support.

## Goal

Add a new `array-str` segment kind to `segments.ts` with comma as the default separator
(the OpenAPI `simple` style). The segment captures a single path segment and splits it
into `string[]` at match time. `buildUrl` joins the array with the separator when printing.

## Proposed Sigil

`+name` — the `+` prefix is unambiguous (not used by any existing sigil: `:` str, `#` num,
`*` wildcard) and reads as "one or more values". Examples:

```
route(schema, 'orgs/:orgId/tags/+tagIds/')
// /orgs/acme/tags/typescript,react  →  { orgId: 'acme', tagIds: ['typescript', 'react'] }

route(schema, 'users/+ids/')
// /users/alice,bob,carol  →  { ids: ['alice', 'bob', 'carol'] }
// /users/alice            →  { ids: ['alice'] }  (single-element array)
```

A single value always produces a one-element array — the type is always `string[]`.

## Change Surface & T-shirt Sizing

**S** — 1-3 files; clear design; no new type parameters on `RouteNode`.

| File | Change |
| ---- | ------ |
| `src/core/segments.ts` | Add `{ kind: 'array-str'; name: string; separator: string }` to `Segment`; parse `+name`; match by splitting on separator; update `buildUrl` to join |
| `src/core/segments.test.ts` | New `it.each` cases for parse, match, buildUrl round-trip; PBT coverage |
| `src/openapi/generate-spec.ts` | Add `array-str` case to `segmentToParam` and `segmentToOpenApi` |
| `src/core/route-node.ts` | Update `ExtractPathParams` to recognise `+` sigil (emits the param name into the union) |

No changes to `walk.ts`, `define-routes.ts`, or the native schema system — the segment
captures the array value at match time; the schema (if present) sees a `string[]` in the
params object. Schema validation of array contents is out of scope for this plan.

## Dependencies

None hard. Soft: plan 166 (trailing slash normalization) — if done first, the PBT
generators will already cover both slash forms and this plan's PBT cases can build on them.

## Success Criteria

1. `parseSegments('+ids/')` → `[{ kind: 'array-str', name: 'ids', separator: ',' }]`
2. `matchSegments` on `'+ids/'` against `['a,b,c']` → `{ params: { ids: ['a', 'b', 'c'] }, rest: [] }`
3. Single value: `matchSegments` on `'+ids/'` against `['alice']` → `{ params: { ids: ['alice'] }, rest: [] }`
4. Empty segment (`''`) after split — document behaviour: treat as empty array or skip?
   Recommendation: skip (same treatment as `filter(Boolean)` elsewhere) → `['']` → `[]`.
5. `buildUrl` round-trip: `{ ids: ['a', 'b', 'c'] }` → `/a,b,c` for segment `+ids`
6. `segmentToParam` in `generate-spec.ts` emits:
   ```json
   { "name": "ids", "in": "path", "required": true,
     "style": "simple", "explode": false,
     "schema": { "type": "array", "items": { "type": "string" } } }
   ```
7. `segmentToOpenApi` emits `{ids}` (same as other param kinds — OpenAPI path template syntax)
8. `ExtractPathParams<'users/+ids/'>` → `"ids"` (name enters the section-param union)
9. `pnpm --filter @arbor/router test && pnpm --filter @arbor/router typecheck && pnpm --filter @arbor/router run examples` green

## Design Notes

### Separator configurability

Comma is the only separator in this plan. A different separator (`;`, `|`, ` `) would
require syntax to express it in the path string — e.g., `+name(;)`. Defer until a
concrete use case arises. The `separator` field is included in the `Segment` type to
make future extension non-breaking.

### Interaction with native schema (plan 164)

Plan 164 adds `AnyObjectSchema` to `RouteNode.schema`. Array segment params deliver
`string[]` into the params object. The native `ObjectSchema<T>` today only covers scalar
fields. Two choices:

**(a) Leave unschematized for now** — array params bypass schema validation (schema field
for `ids` is simply absent from the `ObjectSchema`). Safe: the segment itself guarantees
`string[]`; contents are not validated.

**(b) Add `ArraySchema` kind to `schema.ts`** — `{ kind: 'array'; item: AnyScalarSchema }`
with `Infer<ArraySchema<S>>` = `Infer<S>[]`. This gives full type inference for array
params through `Infer<S>` and enables content validation.

**Recommendation: (a) for this plan; plan the ArraySchema extension separately if
validated item types are needed.**

### Optional array params

`+name?` (optional) is not in scope. An optional array segment would match either a
segment of comma-separated values or nothing — similar to `opt-str`. Defer until needed.

### `ExtractPathParams` fix

The existing `ExtractPathParams` TODO (line 12 of `route-node.ts`) notes that params
without a trailing slash are not extracted. This plan adds `+name` support to the template
literal — handle both `+name/` and `+name` (no trailing slash) forms at the same time,
removing that edge case for the `+` sigil. Fixing `:` and `#` forms is covered by plan 166.
