# Plan: Monorepo Structural Separation and CI Test Isolation

> **NOTE:** prefer the intent of this plan over the detailed steps. Treat the specific instructions in this plan with appropriate skepticism as they were written with only the directory tree (`tree -I 'node_modules|dist|plan|tools'`) as a contextual guide.

## Context

The project currently manages framework tools (`@arbor/router`, `@arbor/router-browser`, `@arbor/router-test`), core primitives (`@arbor/common`), and localized servers (`@arbor/api`, `@arbor/bff`, `ui`) inside a single flat `packages/` workspace directory.

This layout presents several structural bottlenecks:

1. **Architectural Leakage**: Engine utilities and app-specific logic are bundled together inside `@arbor/common`, allowing backend schemas to bleed into framework routers.
2. **CI Pipeline Failures**: Running a sweeping test sweep via Vitest runs pure engine tests alongside heavy database integration tests (`packages/api`). These integration tests fail in continuous integration environments because they require active database containers and runtime initialization scripts (`scripts/db.sh`).

## Goal

Restructure the monorepo into clear architectural layers (`framework/` and `apps/`). This will decouple pure framework libraries from the full-stack demo applications. It also separates lightweight unit tests from database integration pipelines so that framework updates do not get blocked by database setups in CI.

## Change Surface & T-shirt Sizing

- **pnpm-workspace.yaml**: Change workspace path configurations to target sub-folders.
- **Directory Matrix**: Create `framework/` and `apps/` base layers. Move existing directories to align with these layers.
- **@arbor/common**: Isolate framework utilities into `framework/common`. Relocate product code and domain schemas to an app-specific shared library.
- **Root Configurations**: Update path mappings inside `tsconfig.json`, `vitest.workspace.ts`, and `turbo.json`.
- **Package Configuration Matrices**: Split test tasks inside application scripts into distinct unit and integration suites.

**T-shirt Sizing**: L (Large — Requires cross-workspace path adjustments, dependency validation across all packages, and continuous integration pipeline reconfiguration).

## Dependencies

- None. This is an infrastructure task that can be executed independently of feature development.

## Step-by-Step Blueprint

### Phase 1: Directory Reorganization & Workspace Configuration

1. **Define Sub-Workspaces**: Modify the root `pnpm-workspace.yaml` to recognize nested directories.
   ```yaml
   packages:
     - 'framework/*'
     - 'apps/*'
   ```
2. **Migrate Framework Engines**: Create the `framework/` root directory. Move the following packages into it:
   - `packages/router` ➔ `framework/router`
   - `packages/router-browser` ➔ `framework/router-browser`
   - `packages/router-test` ➔ `framework/router-test`
3. **Migrate Application Stack**: Create the `apps/` root directory. Move the following packages into it:
   - `packages/api` ➔ `apps/api`
   - `packages/bff` ➔ `apps/bff`
   - `packages/ui` ➔ `apps/ui`

### Phase 2: Untangling `@arbor/common`

1. **Isolate Framework Primitives**: Move `packages/common` to `framework/common`. Strip out any concrete application domains (such as `schemas/hello.ts`). Keep only framework dependencies like `result.ts` and `http/client.ts`.
2. **Create Application Shared Space**: Create a new package at `apps/app-shared` with its own `package.json`. Name this module `@arbor/app-shared`.
3. **Relocate Business Logic**: Move shared application schemas, common state stores, and product-specific constants into `@arbor/app-shared`.
4. **Update Import Vectors**: Use global search-and-replace to update package references across the workspace:
   - Replace any application references that targeted `@arbor/common` with `@arbor/app-shared`.

### Phase 3: Test Isolation via Vitest Workspaces

1. **Refactor Root Workspace**: Update `vitest.workspace.ts` at the root directory to separate fast unit tests from heavy integration suites using distinct configuration filters.

   ```typescript
   import { defineWorkspace } from 'vitest/config';

   export default defineWorkspace([
     {
       test: {
         name: 'unit',
         include: ['framework/**/*.test.ts', 'apps/**/*.test.ts'],
         exclude: ['apps/api/**/*.repository.test.ts', 'apps/api/src/db/**/*'],
       },
     },
     {
       test: {
         name: 'integration',
         include: ['apps/api/**/*.repository.test.ts'],
         globalSetup: './apps/api/src/testing/setup.ts',
       },
     },
   ]);
   ```

2. **Adjust Package Scripts**:
   - Update `apps/api/package.json` to offer separate entry targets:
     ```json
     "test": "vitest run --project unit",
     "test:integration": "vitest run --project integration"
     ```

### Phase 4: Turbo Task Pipeline Adjustments

1. **Re-Architect `turbo.json`**: Update the root `turbo.json` configuration to decouple unit tasks from integration dependencies.
   ```json
   {
     "\$schema": "https://turbo.build",
     "tasks": {
       "build": {
         "dependsOn": ["^build"],
         "outputs": ["dist/**", ".next/**", "tsconfig.tsbuildinfo"]
       },
       "test": {
         "dependsOn": ["build"],
         "cache": true
       },
       "test:integration": {
         "dependsOn": ["build"],
         "cache": false
       }
     }
   }
   ```

### Phase 5: CI Pipeline Integration

1. **Optimize CI Workflows**: Update the repository's CI workflow file (e.g., GitHub Actions, GitLab CI).
2. **Configure Pipeline Stages**:
   - Set the standard PR check stage to run `pnpm turbo run test`. This runs the unit test suite across both the framework and applications without requiring a database.
   - Set up a separate, protected workflow for main merges or explicitly tagged test runs. This stage executes `scripts/db.sh` to spin up a Docker database instance before running the integration suite via `pnpm turbo run test:integration`.

## Success Criteria

1. The project directory matches the new `framework/` and `apps/` layout without broken internal symlinks.
2. Running `pnpm turbo run test` runs successfully across the workspace without booting a database or failing in clean CI systems.
3. Database tests inside `@arbor/api` are isolated and run exclusively when calling `pnpm turbo run test:integration`.
4. `@arbor/router` compiles successfully without importing any product-specific code or application schemas.
