# Plan 176 — `WsSessionClient`: Client-Side IxSession via `Dual<S>`

## Context

> **North star**: `plan/spec.ws-session-ergonomics.md` — the "Client" section shows
> the exact `connectSession` / `chooseFrom` / `branchTo` usage this plan enables.

Plans 173–175 add `IxSession`, `IxSessionOps`, and `wsSessionRoute` on the server side.
The client side currently uses `createWsClient` which returns a flat `WsChannel` via
`buildWsChannel`. This plan adds a parallel `connectSession` method that returns a client-side
`IxSessionOps` bound to a `Dual<S>`-shaped session.

### Key insight: Dual<S> is free

The server session type `S` is stored in `WsSessionContext<S>._sessionType`.
`Dual<S>` from `session.ts` gives the client type mechanically:
- Server `Recv<T, N>` → Client `Send<T, Dual<N>>`
- Server `Send<T, N>` → Client `Recv<T, Dual<N>>`
- Server `Branch<C>` → Client `Select<Dual<C>>`  (server decides = client awaits)
- Server `Select<C>` → Client `Branch<Dual<C>>`  (client decides = server awaits)

Because `IxSessionOps` is untyped at the method level (all methods take `any`), the client
can use the **same `buildIxSessionOps`** regardless of whether the session type is `S` or
`Dual<S>`. The type-level distinction lives in how the route/context is typed at the call
site.

### New API surface

```typescript
// New method on WsClient (alongside existing `connect`)
export interface WsSessionClient<
  Route extends { tag: string },
  Map extends Record<string, WsSessionContext<any>>,
> {
  connectSession<Tag extends keyof Map & string>(
    route: Extract<Route, { tag: Tag }>,
  ): IxSessionOps
  // Returns untyped ops; the session type constraint is enforced externally.
  // A typed version (returning IxSessionOps typed to Dual<S>) can be added later;
  // the untyped ops are correct at runtime and sufficient for a first pass.
}

// Combined client factory
export function createWsSessionClient<
  Route extends { tag: string },
  Map extends Record<string, WsSessionContext<any>>,
>(
  baseUrl: string,
  router: WsRouterLike_Session<Route, Map>,
  opts?: { connect?: WsConnectFn },
): WsSessionClient<Route, Map>
```

`WsRouterLike_Session` is the same shape as `WsRouterLike` but parameterised over
`WsSessionContext` instead of `WsContext`.

```typescript
it('wsSessionRoute round-trip: server recv join, send welcome; client via createWsSessionClient', async () => {
  const JoinSchema = z.object({ username: z.string() })
  const WelcomeSchema = z.object({ roomId: z.string() })

  const route = wsSessionRoute(
    object({ tag: literal('ws/lobby') }),
    'ws/lobby',
    undefined as never,
  )
  const router = defineRoutes([route])

  // Server
  const server = createWsSessionServer(router, {
    'ws/lobby': async ({ ops }) => {
      const join = await ops.recv(JoinSchema).run()
      await ops.send({ roomId: `room-for-${join.username}` }, WelcomeSchema).run()
    },
  })
  const [serverAdapter, clientAdapter] = createWsAdapterPair()
  void server.handleConnection('ws/lobby', {}, serverAdapter)

  // Client using createWsSessionClient
  const client = createWsSessionClient('ws://localhost', router, {
    connect: () => clientAdapter,
  })
  const parsed = router.parse(new URL('http://localhost/ws/lobby'))
  expect(parsed.isOk()).toBe(true)
  if (!parsed.isOk()) return

  const ops = client.connectSession(parsed.value)
  await ops.send({ username: 'alice' }, JoinSchema).run()
  const welcome = await ops.recv(WelcomeSchema).run()
  expect(welcome).toEqual({ roomId: 'room-for-alice' })
})
```

## Notes

- `connectSession` calls `buildIxSessionOps(adapter)` — same as the server path, but with
  `in`/`out` schemas swapped (the client sends what the server receives and vice versa).
  Since `IxSessionOps.send` and `IxSessionOps.recv` take explicit schemas at each call site,
  the swap is handled by the caller, not by `buildIxSessionOps`. No change to
  `buildIxSessionOps` is needed.
- `WsRouterLike_Session` can be defined as an alias for the same shape as `WsRouterLike` but
  with `Map extends Record<string, WsSessionContext<any>>` instead of `WsContext<any, any>`.
  If the two shapes are structurally identical modulo the Map constraint, a single generic
  could cover both — but keep them separate to avoid coupling the two code paths.
- The `defaultConnect` function (creates a `WebSocket`) is shared with `createWsClient` —
  extract it to a module-level constant if it isn't already, so both factories use it.
