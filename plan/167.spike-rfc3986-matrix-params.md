# PLAN.167 (Spike) — RFC 3986 Matrix Parameters

## Context

The TODO comment at the top of `segments.ts` already flags this:

```
// TODO: this system only permits scalars. Consider more structured
// options. See Matrix Parameters (RFC 3986).
// Ideas:
// | { kind: 'array-str'; name: string; separator: string }
// | { kind: 'date'; name: string }
// | { kind: 'custom'; name: string; parse: (val: string) => unknown };
```

RFC 3986 §3.3 defines path segments and the characters allowed within them, including
the sub-delimiters `!$&'()*+,;=`. "Matrix parameters" is a convention (not part of
RFC 3986 proper — it originated with Tim Berners-Lee's 1996 design note and was
popularised by JAX-RS) that uses `;key=value` within a path segment to attach extra
parameters to that segment independently from query parameters. Example:

```
/products;color=blue;size=L/reviews
```

Here `color` and `size` are parameters of the `products` segment, not of the whole URL.

## Goal

Produce a design verdict: does adding matrix parameter support to `segments.ts` provide
meaningful value for the kinds of routes arbor targets? If yes, sketch a minimal
Segment kind and factory. If no, document why and close the TODO.

## Questions to Answer

**Q1 — Is there a use case query params can't serve?**

Matrix parameters differ from query parameters in that they scope to a specific segment.
`/products;color=blue/reviews` vs `/products/reviews?color=blue` — the first says "blue
products' reviews", the second is ambiguous (are we filtering products or reviews?).
This distinction matters when the route tree has multiple collection nodes and filtering
needs to be scoped per node. Is this a real constraint for the HTTP or SSE routes that
arbor targets?

**Q2 — Tooling support**

OpenAPI 3.x does not have a standardised way to describe matrix parameters (the
`matrix` style in OpenAPI is poorly supported across code generators and Swagger UI).
Does poor tooling support eliminate the value gain from type-safe route definitions?

**Q3 — Interaction with the native schema system**

Matrix parameters live inside a segment, not in the query string. The native
`ObjectSchema<T>` describes flat key→value maps. Would matrix params fit naturally into
the existing `Segment` discriminated union, or do they require a parallel schema
attachment point?

**Q4 — Separator-based array parameters**

The TODO also proposes `{ kind: 'array-str'; separator: string }` for comma-separated
values in a single segment (e.g., `/tags/a,b,c`). This is distinct from matrix params
but similarly motivated. Should these be evaluated together or separately?

## Gut Recommendation (pre-research)

**Low value; likely defer or close.**

The use case (per-segment parameter scoping) is real but narrow. The vast majority of
REST and RPC APIs route on path structure alone and use query params for filtering,
regardless of which segment logically "owns" the filter. The ambiguity matrix params
resolve (`/products?color=blue` — which segment does `color` filter?) is better solved
by route tree design (put the filter route under the right node) than by syntax.

The tooling situation is decisive: OpenAPI's `matrix` explode style is not reliably
rendered by Swagger UI or supported by most client generators. Adding matrix params to
arbor's `Segment` type would make those routes un-specifiable in the OpenAPI output
without a custom extension, which undermines the openapi-output.ts example's purpose.

The separator-based array params (`/tags/a,b,c`) are more practically useful — they
are widely used and can be described via OpenAPI's `style: simple` with `explode: false`
on a path parameter. These deserve their own evaluation if needed, independent of matrix
params.

**Verdict: close the matrix-params TODO; evaluate array-segment params separately if a
concrete use case arises.**
