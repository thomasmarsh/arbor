# PLAN.121 — Loader Dehydration / Rehydration

## Context

When the server renders a page with loaders, the loader results exist on the server but
not in the browser. Without dehydration, the client fires all loaders again on first
render — the user sees a flash of loading state even though the server already fetched
the data. Dehydration serializes loader results into the HTML; rehydration seeds the
client loader cache so the first render is instant.

This is the same pattern used by React Query, Remix, and TanStack Router.

## Goal

### Server side — `dehydrate(route, loaderData)`

```typescript
// packages/router-browser/src/dehydration.ts

export function dehydrate<Route>(
  route: Route,
  data: Record<string, unknown>,
): string;
// Returns a JSON string safe to embed in: <script>window.__ARBOR_DATA__ = JSON.parse({...})</script>
// Escapes </script> sequences to prevent XSS.
```

`renderRoute()` (plan 122) embeds the output as:
```html
<script>window.__ARBOR_DATA__ = JSON.parse("...");</script>
```

### Client side — `rehydrate()`

```typescript
export function rehydrate(): Record<string, unknown> | null;
// Reads window.__ARBOR_DATA__ and clears it.
```

`BrowserRouter` (plan 95) gains an optional constructor parameter:

```typescript
new BrowserRouter(router, { dehydratedData?: Record<string, unknown> })
```

When `dehydratedData` is provided, it pre-seeds the loader cache (plan 101) so the first
navigation to any route whose data is already present skips the loader entirely.

### Loader cache seeding

The loader cache (plan 101) must expose a `seed(tag: string, data: unknown): void` method
so `BrowserRouter` can pre-populate it from dehydrated data. The cache treats seeded
entries as fresh (no TTL decrement on first access, but normal TTL thereafter).

### Type safety constraint

Loader data is `unknown` at the dehydration boundary — it must be re-validated on the
client. The typed loader system (plan 101) already runs Zod schema validation on the
loader result; `rehydrate()` provides raw JSON and the loader cache validates it when
first accessed. This means the server and client Zod schemas must agree — a type
mismatch silently falls back to re-fetching.

## Dependencies

- Plan 101 (loader cache must exist and expose `seed()`).
- Plan 122 (`renderRoute`) consumes `dehydrate` on the server.

## Success Criteria

- Dehydrated JSON round-trips through `JSON.parse(dehydrate(...))` exactly.
- `</script>` in loader data is escaped (XSS prevention).
- `BrowserRouter` with `dehydratedData` skips loaders for pre-seeded routes.
- `BrowserRouter` without `dehydratedData` behaves identically to plan 95.
- `rehydrate()` returns `null` when `window.__ARBOR_DATA__` is absent (non-SSR context).
