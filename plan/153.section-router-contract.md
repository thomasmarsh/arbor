# Plan 153 — `section()` Accepts RouterContract

## Context

After Plan 152, `CtxMap` recurses through section nodes, so routes nested under
`section()` are fully visible to `createServer`. The remaining gap is ergonomic:
composing two independently-defined routers currently requires exposing `.children`:

```typescript
// today — leaks internal structure
const apiRouter = defineRoutes([
  section('api', [
    ...helloRouter.children,
    ...usersRouter.children,
    section('ledger', ...ledgerRouter.children),
  ]),
]);
```

`RouterContract.children` is typed as `RouteNode<unknown, any, any, any, any>[]` for
covariance reasons, so spreading it into a section node would widen all type
parameters to `any`. Plan 152's recursive `CtxMap` would then traverse `any`-typed
nodes and produce `any` for handler types — defeating the whole point.

## Goal

`section(path, router: RouterContract)` is a valid overload. The runtime uses
`router.children` as the section's children. The type system produces a fully-typed
handler map — no `any` leakage.

```typescript
const apiRouter = defineRoutes([
  section('api', [
    section('hello',  helloRouter),
    section('users',  usersRouter),
    section('ledger', ledgerRouter),
  ]),
]);

createServer(apiRouter, {
  hello:              async (ctx) => …,
  'users-list':       async (_ctx) => …,
  'ledger-get-tasks': async (_ctx) => …,
  // all keys inferred without any
});
```

## Change Surface & T-shirt Sizing

**Size: S**

Files changed:

- `packages/router/src/core/route-node.ts` — add 6th phantom type parameter
  `EmbeddedMap` to `RouteNode`
- `packages/router/src/core/define-routes.ts` — overload `section()`; extend
  `CtxMap`/`CtxMapDeep` to extract `EmbeddedMap` from section nodes

### Design

**Phantom on RouteNode**

Add a 6th type parameter with default `never`:

```typescript
export interface RouteNode<
  R,
  C extends RouteNode<…>[] = [],
  Context = never,
  SectionParams extends string = never,
  Meta = Record<string, unknown>,
  EmbeddedMap = never,           // new — carries sub-router CtxMap for section nodes
> {
  _embeddedMap?: EmbeddedMap;    // phantom — undefined as never at runtime
  …
}
```

**`section` overload**

```typescript
// existing overload (unchanged)
export function section<Path extends string, C extends RouteNode<…>[]>(
  path: Path,
  children: […C],
): BuildableRouteNode<RouteNode<never, […C], never, …>>;

// new overload
export function section<Path extends string, R, Map extends AnyCtxMap>(
  path: Path,
  router: RouterContract<R, Map>,
): BuildableRouteNode<RouteNode<never, RouteNode<unknown, any, any, any, any>[], never, …, Record<string, unknown>, Map>>;
```

At runtime the new overload does:

```typescript
return buildable({
  _type:    undefined as never,
  schema:   null,
  path,
  segments: parseSegments(path),
  children: router.children,     // widened at runtime, precise via EmbeddedMap phantom
});
```

**`CtxMapDeep` extension (from Plan 152)**

Extend `CtxMapEntry` to check for `EmbeddedMap` before recursing into children:

```typescript
type CtxMapEntry<N extends RouteNode<…>, D extends ValidDepth> =
  [PrevD<D>] extends [never] ? Record<never, never> :
  N extends RouteNode<infer R, infer GC extends RouteNode<…>[], infer Ctx, any, any, infer Embedded>
    ? [R] extends [never]
      ? [Embedded] extends [never]
        ? CtxMapDeep<GC, PrevD<D>>   // section with inline children — recurse (from Plan 152)
        : Embedded                   // section with RouterContract — use precise embedded map
      : [R] extends [{ tag: infer T extends string }]
        ? Record<T, Ctx>
        : Record<never, never>
    : Record<never, never>;
```

When `section(path, router)` is called, the section node carries `EmbeddedMap = Map`
(the RouterContract's precise `_ctxMap`), so `CtxMapEntry` returns that directly
rather than recursing into the widened `children` array.

### Walk algorithm

No changes needed. The existing `walkParseIndexed` recurse-into-children logic handles
section nodes transparently — it already uses `node.children` at runtime regardless of
how they were provided.

### Walk constraint: section path must leave at least one segment for children

A section with segments `['api', 'hello']` that wraps a RouterContract whose sole
route has segments `[]` (empty) would match URL `/api/hello` but then find no
remaining segments to pass to children — the walk would fail to reach the leaf.
This plan does **not** change the walk algorithm. Routes in a RouterContract that is
section-wrapped must have at least one segment. See Plan 154 (path migration) for how
the API routes are restructured to satisfy this constraint.

## Dependencies

- Plan 152 — recursive CtxMap (the `CtxMapDeep`/`CtxMapEntry` types extended here
  are introduced in that plan)

## Success Criteria

- `section('api', ledgerRouter)` where `ledgerRouter` has routes with non-empty

- `pnpm --filter @arbor/router test && typecheck` pass.
- New test: round-trip — `defineRoutes([section('api', subRouter)])`, `createServer`,

- Existing tests unaffected.
