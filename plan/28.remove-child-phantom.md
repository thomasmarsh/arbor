# PLAN.28 — Remove the redundant `_child` phantom field

## Context

Decision from Q7: `_child` is a phantom field on `RouteNode` that carries no
information beyond what the `Child` generic parameter already encodes. Unlike
`_type` (which carries `R` into downstream inference), `_child` merely mirrors
the generic. It adds noise to IDE autocomplete, appears in documentation, and
confuses readers of the type machinery.

## Goal

Delete `_child` from `RouteNode` and every site that assigns it. Verify that
type inference is unchanged — no callers should need updating because `_child`
is never read for inference anywhere.

## Change Surface

1. **`src/core/route-node.ts`**: Remove the `_child: Child` field from the
   `RouteNode` interface.
2. **`src/core/define-routes.ts`**: Remove every `_child: undefined as never`
   assignment in `route()`, `section()`, `httpRoute()`, `openApiRoute()`, etc.
3. **`src/core/route-node.test.ts`**: Remove any `expectTypeOf` assertions
   that reference `_child`. Confirm existing assertions still pass.
4. Run `npm test && npm run typecheck` — if any type error surfaces, that
   means `_child` was actually load-bearing somewhere and must be investigated
   before removing.

## Dependencies

- Best done **after plan 19** (which also touches `RouteNode` to add `_ctx`).
  Doing both in the same file in back-to-back plans avoids merge-style
  conflicts while keeping diffs small.

## Success Criteria

- `RouteNode` has no `_child` field.
- No `as any` or cast introduced to compensate.
- `npm test && npm run typecheck` clean with zero behavior change.
- This should be a net deletion of lines, not an addition.

## Outcome — BLOCKED

Attempted. `_child: Child` cannot be deleted without breaking type inference.

`Derive<N>` uses `infer Child` matched against `_child: Child` to recover the
child union in O(1). Without this field, `Derive` must call `ChildUnion<C>`
directly. `ChildUnion` itself calls `Derive` for each child, creating mutual
recursion that hits TypeScript's instantiation depth limit (`TS2589`) even for
the 3-level test trees already in the suite. The field is not cosmetically
redundant — it is a type-level memoization cache.

A future plan would need to restructure `Derive`/`ChildUnion` to be
non-mutually-recursive before `_child` can be removed. See Q7 in
`plan/questions.md`.
