# 187. Ledger Postgres Schema & Seed

## Context

The ledger is currently stored in `plan/ledger.jsonl` and read by the Python CLI. Postgres becomes the source of truth; the JSONL is the persistent cache and VCS history. Because the DB is reset frequently (dev containers torn down and recreated), `arbor db seed` must be able to restore the DB from the JSONL cache at any time. The Node.js migration runner at `apps/api/src/db/migrate.ts` already reads SQL files from `apps/api/src/db/migrations/pg/`; we add the ledger migration there. `arbor db migrate` delegates to that runner.

## Goal

1. SQL migration creating the four ledger tables with FK constraints.
2. `arbor db seed` — idempotent JSONL → Postgres load; the primary DB-recovery path after a reset.
3. `arbor db migrate` — thin wrapper calling `pnpm --filter @arbor/api migrate`.

## Change Surface & T-shirt Sizing

Size: S

### Migration (`apps/api/src/db/migrations/pg/`)

New file `0002_ledger.sql`:

```sql
CREATE TABLE ledger_epics (
  id    TEXT PRIMARY KEY,
  title TEXT NOT NULL
);

CREATE TABLE ledger_stories (
  id      TEXT PRIMARY KEY,
  epic_id TEXT NOT NULL REFERENCES ledger_epics(id),
  layer   TEXT NOT NULL,
  title   TEXT NOT NULL
);

CREATE TABLE ledger_waves (
  id       TEXT PRIMARY KEY,
  name     TEXT NOT NULL,
  position INT  NOT NULL  -- insertion order; drives sort in work-order queries
);

CREATE TABLE ledger_tasks (
  id       INT  PRIMARY KEY,
  kind     TEXT NOT NULL CHECK (kind IN ('task', 'spike')),
  epic_id  TEXT NOT NULL REFERENCES ledger_epics(id),
  story_id TEXT NOT NULL REFERENCES ledger_stories(id),
  wave_id  TEXT NOT NULL REFERENCES ledger_waves(id),
  layer    TEXT NOT NULL,
  status   TEXT NOT NULL CHECK (status IN ('todo', 'next', 'in_progress', 'done', 'canceled')),
  size     TEXT          CHECK (size IN ('xs', 's', 'm', 'l', 'xl')),
  text     TEXT NOT NULL,
  file     TEXT NOT NULL,
  rank     INT,
  deps     INT[] NOT NULL DEFAULT '{}'
);
```

`deps` stored as a Postgres `INT[]` array — avoids a junction table for this read-mostly use case.

### CLI (`tools/arbor-cli/src/arbor_cli/cmd_db.py`)

| Command | Behaviour |
| --- | --- |
| `arbor db migrate` | `subprocess.run(["pnpm", "--filter", "@arbor/api", "migrate"])` — delegates entirely to the Node runner |
| `arbor db seed` | Parses the JSONL cache; upserts all rows into the four tables via `psycopg2`; idempotent (`ON CONFLICT DO UPDATE`) |

### Dependencies in `pyproject.toml`

Add `psycopg2-binary` (or `psycopg[binary]` for psycopg3) to the CLI's dependencies.

## Dependencies

None — can start immediately (no ledger task is in `next` state yet, so the JSONL seed captures a clean snapshot).

## Success Criteria

- `arbor db migrate` exits clean; `arbor db status` shows `0002_ledger.sql` applied.
- `arbor db seed` idempotently populates all four tables; running it twice produces no duplicates and no errors.
- FK constraints enforce referential integrity (epic/story/wave references from tasks validated on insert).
- `psql` query confirming row counts match the JSONL source.
