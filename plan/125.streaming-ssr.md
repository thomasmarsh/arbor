# PLAN.125 — Streaming SSR + Selective Hydration

## Context

Plan 123's `renderRoute()` uses `renderToReadableStream` which already supports streaming
— the shell HTML is sent immediately and `<Suspense>` boundaries stream in later content.
This plan makes the browser integration *use* that capability: deferred loaders don't
block the HTML shell, and lazy components (plan 104) stream in as their modules resolve.

Selective hydration (React 18) means that interactive components within a `<Suspense>`
boundary become interactive as soon as their JS loads, even if sibling boundaries are
still streaming.

## Goal

### 1. Deferred loaders (non-blocking shell)

The HTML shell is sent immediately with `<Suspense fallback={<Skeleton />}>` wrapping
deferred content. Loaders that complete quickly resolve before the stream ends; slow
loaders stream their content as they resolve.

In plan 123's `renderRoute()`, loaders are currently awaited before rendering starts
(`await runLoaders(route)`). Change to:

```typescript
// Start loaders but don't await — pass as Promises to the component tree
const loaderPromises = startLoaders(route, loaderMap);
renderToReadableStream(<StaticProvider loaderPromises={loaderPromises}><App /></StaticProvider>)
```

The component reads loader data via a `use(promise)` hook (React 19) or via a custom
`<Deferred>` wrapper (React 18 + Suspense).

### 2. Lazy component streaming

Plan 104's `createLazyMap` uses `React.lazy` which integrates with Suspense automatically.
In SSR context, `renderToReadableStream` handles `React.lazy` — it streams the
lazy-loaded component's HTML when the import resolves.

The key requirement: the SSR render must have access to the lazy component's module at
render time (no dynamic imports that only work in the browser). In Node.js, `import()` is
synchronous after the first load. If the component module is loaded before `renderRoute()`
is called (e.g., via a preload step), lazy components render without a fallback.

### 3. Client integration: `hydrateRoot` + Suspense

`hydrateRoot` in React 18 handles selective hydration automatically when the HTML was
produced by `renderToReadableStream`. No changes needed on the client side beyond using
`hydrateRoot` instead of `createRoot`.

Update `packages/ui/src/main.tsx` to check for SSR-rendered HTML:

```typescript
const root = document.getElementById('root')!;
if (root.hasChildNodes()) {
  hydrateRoot(root, <App />);   // SSR-rendered — hydrate
} else {
  createRoot(root).render(<App />);  // CSR — fresh render
}
```

## Constraints

- Deferred loaders in streaming SSR must not hold the HTTP connection open indefinitely.
  Add a per-loader timeout (default 5 s); timed-out loaders fall back to client-side
  fetch (the `<Suspense>` boundary renders the fallback on the server, then the client
  fetches on mount).
- The `Content-Type: text/html` response must set `Transfer-Encoding: chunked` for
  streaming to work with Hono's Node.js adapter.

## Change Surface

- Update: `packages/router-react/src/server.ts` — deferred loader path
- Update: `packages/ui/src/main.tsx` — `hydrateRoot` detection
- New: `packages/router-react/src/deferred.tsx` — `<Deferred promise={p}>` wrapper
  (React 18 path) or document `use(promise)` usage (React 19 path)

## Dependencies

- Plan 123 (`renderRoute` base implementation).
- Plan 104 (code splitting — lazy components).
- Plan 101 (loader execution — needs `startLoaders` non-awaited variant).

## Success Criteria

- HTML shell arrives within 50 ms of request (loaders do not block it).
- Deferred loader content streams in as promises resolve.
- `hydrateRoot` hydrates without mismatch warnings.
- A slow loader (>5 s) falls back to client-side fetch without hanging the response.
