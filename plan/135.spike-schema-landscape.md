# PLAN.135 (Spike) — Schema Library Landscape Survey

## Context

Plan 133 proposes building a native schema representation for arbor. Before committing to that,
we should ask: does an existing library already solve this?

The landscape has changed significantly. Three serious candidates exist:

- **TypeBox** — schemas as plain JSON Schema objects with TypeScript phantom types. `Type.Object({
  id: Type.String() })` produces `{ type: 'object', properties: { id: { type: 'string' } } }` with
  inferred type `{ id: string }`. Natively JSON-serializable. Designed for OpenAPI. No Zod.
- **ArkType** — schemas defined via TypeScript-like string literals: `type('{ id: string }')`.
  Refinements are encoded as constraint expressions: `type('string >= 5')`,
  `type('0 < number < 100')`. This is the only library that serializes refinements without
  functions. Runtime parsing is extremely fast (described as "1000x faster than Zod").
- **Valibot** — modular, tree-shakeable. Uses `pipe(v.string(), v.email())`. Schema objects
  are still function-based (validators), not pure data. Better tree-shaking story than Zod
  but the same serialization limitation.
- **Effect/Schema** — maximally powerful but brings the entire Effect runtime. Too heavy.

The key question: can any of these replace Zod in arbor's current use pattern — type inference
for `RouteNode`, runtime validation at the boundary, and JSON Schema generation for OpenAPI —
without requiring us to build and maintain a new schema system?

## Goal

Produce a comparison matrix of the four candidates against arbor's actual requirements.
Make a go/no-go recommendation for each, and a final recommendation for plan 133's direction.

## Questions to Answer

**Q1 — TypeBox as the schema layer**

TypeBox's `TSchema` objects are already what plan 133 describes: plain JavaScript objects that
carry a TypeScript phantom type. `Type.String()` returns `{ type: 'string', [Kind]: 'String' }`
with phantom type `string`. Can arbor replace `z.ZodObject<any,any>` with `TObject<TProperties>`
without losing the type inference properties that `z.infer<S>` provides? Specifically:

- Does `route(Type.Object({ id: Type.String() }), ':id')` infer `RouteNode<{ id: string }, ...>`?
- Does TypeBox have an equivalent of `z.infer<S>` (answer: `Static<T>` from `@sinclair/typebox`)?
- How does TypeBox handle `.refine()`-style constraints? (Answer: it uses JSON Schema `format`,
  `pattern`, `minimum`, `maximum`. Custom predicates via `Type.Unsafe()` — but that breaks
  serialization in the same way as Zod's `.refine()`.)
- Does `@sinclair/typebox/value` (the TypeBox value library) handle runtime validation? Is
  it adequate for boundary validation or does it need `ajv` underneath?

**Q2 — ArkType for serializable refinements**

ArkType's string-based syntax means that `type('string > 5 & < 100')` is:
- Fully serializable (it's just a string)
- TypeScript-type-inferred (ArkType parses the string at the type level)
- Runtime-validated using the same string

Can ArkType be used as arbor's schema layer? Does its TypeScript API allow structured
composition (an `object()` builder equivalent to `z.object()`), or is the string syntax the
only interface? What happens to complex cross-field refinements — does ArkType support them, or
does it also require a function escape hatch?

**Q3 — Zod v4**

Zod v4 (released 2025) has a significantly different internal architecture. It is much faster,
has a smaller bundle, and — importantly — has changed how schemas are represented internally.
Does Zod v4 produce more data-like schemas that could be inspected, serialized, or compared
structurally? Does `@zod/to-json-schema` (formerly `zod-to-json-schema`) work cleanly with v4?
Is there anything in v4's architecture that closes the serialization gap, or is `.refine(fn)`
still a closure in the schema tree?

**Q4 — Dual-library strategy**

Could arbor use TypeBox as the *schema representation* (for OpenAPI and serialization) while
accepting Zod as the *validation runtime* via an adapter? TypeBox's JSON Schema output is
exactly what OpenAPI needs. A `toTypebox(zodSchema)` or `fromZod()` converter would let
existing Zod users pass their schemas without rewriting.

## Findings (June 2026)

Decision: build a custom schema for router-controlled positions; accept Standard Schema for user-controlled positions.

### Q1 — TypeBox

Ruled out on size. TypeBox unpacked is 1.9MB across eight sub-paths. The relevant subset
(`./type` + `./value`) is larger than Zod v4 core (900KB), not smaller. TypeBox's unit of
abstraction is JSON Schema — which is the right serialization format for OpenAPI but not the
right granularity for session type duality (`Dual<Send<T, Recv<U, End>>>`). Adopting TypeBox
would trade one external coupling for a larger one.

### Q2 — ArkType

Interesting for its constraint-string insight (`type('string >= 5')`), but the string AST is
not a suitable internal data model for arbor's schema layer. Structural operations (duality,
diffing) would require parsing the AST. The API surface is also divergent enough that it
would impose ArkType's mental model on arbor users.

### Q3 — Zod v4

Already in use (confirmed: `z.core.$ZodIssue`, `z.toJSONSchema()` are v4 APIs). Zod v4 closes
the `zod-to-json-schema` gap (JSON Schema generation is now built-in), but `.refine(fn)` is
still a closure — the structural serialization gap remains. Zod exits the router core but
remains the recommended user-facing schema for body and response validation (it implements
Standard Schema, so it works transparently).

### Q4 — Dual-library strategy

Superseded by the clean split below.

### Decision

Router-controlled positions (path params, query, headers, cookies, session message types):
native `ObjectSchema<T>` — a narrow discriminated union covering `string`, `number`, `integer`,
`boolean`, `literal<T>`, `optional<S>`, `object<S>`. Serializable, dualizable, zero external
dependencies, ~140 lines of implementation.

User-controlled positions (request body, response bodies): accept `StandardSchemaV1<T>`.
Zod v4, Valibot, and ArkType all implement this already. Arbor calls `schema['~standard'].validate(v)`
at the boundary. OpenAPI generation for these positions accepts an optional `toJsonSchema()` method
or a bare JSON Schema object alongside the schema.

Size outcome: browser bundles drop ~195KB of Zod runtime (currently pulled in via
`walk.ts`'s `instanceof z.ZodLiteral`). The native schema replaces this with ~5KB.
