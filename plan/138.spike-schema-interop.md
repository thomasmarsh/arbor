# PLAN.138 (Spike) — Schema Interop: Zod as Opt-in Adapter and Fluent Builder Design

## Context

Two competing pressures:

1. **Remove Zod from the core** — make `@arbor/router` independent of Zod so that the schema
   layer is portable, serializable, and not tied to one library's release cycle.
2. **Keep Zod users happy** — arbor has users who already have large Zod schemas. Forcing a
   rewrite to a new schema syntax is a breaking change and a maintenance burden.

The user's proposal: "we might even provide Zod syntax compatibility where fluent constructors
simply assemble types in our schemas." This is the key insight. The Zod-like API (`.string()`,
`.object()`, `.min()`, `.email()`) could be a thin fluent builder that *produces our native
schema type* rather than a Zod internal object. Users who write `s.string().email().min(5)` get
back a plain `{ kind: 'string', format: 'email', minLength: 5 }` — not a Zod class instance.

The `fromZod()` adapter is the other direction: for users who already *have* a Zod schema and
want to pass it to arbor, `fromZod(z.object({ id: z.string() }))` would introspect Zod's
internal structure (`.shape`, `._def`) and produce a native schema.

## Goal

Design the fluent builder API and the `fromZod()` adapter. Determine:

1. Whether the fluent builder pattern can produce a native schema type without losing type
   inference (the key claim: `s.object({ id: s.string() })` must infer `Schema<{ id: string }>`).
2. Whether `fromZod()` can round-trip a Zod schema to native without losing structural
   information (for the A/B-class refinements from plan 137).
3. What the opt-in Zod architecture looks like in terms of package structure and exports.

## Questions to Answer

**Q1 — Fluent builder type inference**

The challenge: a fluent API means method chaining (`s.string().email().min(5)`). Each method
call returns a new schema value. TypeScript must infer the constraint accumulation at the type
level. Can we design `StringSchema` such that:

```typescript
const s1 = s.string()              // StringSchema (no constraints)
const s2 = s.string().email()      // StringSchema with format constraint
const s3 = s.string().min(5)       // StringSchema with min constraint
const s4 = s.object({ id: s3 })    // ObjectSchema<{ id: string }>
```

...where `s4` produces `Schema<{ id: string }>` and `Infer<typeof s4>` is `{ id: string }`?
The constraints (email, min, max) do not change the inferred TypeScript type — they are runtime
metadata only. This is cleaner than Zod's approach where `.email()` produces a different
branded type.

**Q2 — The `fromZod()` feasibility problem**

Zod schemas expose their internal structure via `._def` (the definition object). For example,
`z.string().email()._def` contains `{ typeName: 'ZodString', checks: [{ kind: 'email' }] }`.
This is technically inspectable. But it is an internal API that Zod does not commit to
stabilizing between versions.

Can `fromZod()` be written against the *documented* Zod API (`schema._type`, `schema.shape`
for ZodObject, etc.) rather than `._def`? If not, is `fromZod()` fragile (breaks on Zod minor
versions)? What is the minimal set of structural information we need to extract?

**Q3 — The `.refine()` escape hatch**

When a user passes a Zod schema with `.refine(fn)` through `fromZod()`, we cannot serialize
the function. Options:

- **Drop it silently**: treat the schema as if the refinement doesn't exist. (Bad — silent
  loss of validation.)
- **Preserve it opaquely**: store the Zod schema alongside the native schema as a runtime
  validator, but produce the native schema for OpenAPI/serialization. (Dual representation.)
- **Error at construction time**: `fromZod()` throws if the Zod schema contains `.refine()`.
  (Too strict — common Zod pattern.)
- **Warn and degrade gracefully**: emit the OpenAPI schema without the refinement, log a
  warning, use the Zod schema for runtime validation.

The dual representation option is interesting: `{ native: NativeSchema, runtime: ZodSchema }`.
The native schema is used for serialization and OpenAPI; the runtime schema is used for
validation. Both live on the route node. This is a "best of both worlds" but adds complexity.

**Q4 — Package structure for opt-in Zod**

If Zod is made optional, what does the package structure look like?

Option A: **Subpath export** — `@arbor/router/zod` exports `fromZod`, `toZod`. The core
`@arbor/router` has no Zod dependency. Users who want Zod import the subpath.

Option B: **Separate package** — `@arbor/router-zod` is a separate npm package that wraps
arbor and adds Zod integration. Cleaner dependency graph but more maintenance overhead.

Option C: **Peer dependency with feature detection** — Zod remains a peer dep, but is
optional. The `fromZod()` adapter is available if Zod is installed. If not, only native
schemas work.

Which option minimizes breaking changes for current users (who currently get Zod transitively
through `@arbor/router`) while enabling a Zod-free core for future users?

**Q5 — The Zod v4 compatibility question**

Zod v4 changed its package name to `zod/v4` (while `zod` remains as a compatibility shim) and
rewrote internals. If `fromZod()` inspects `._def`, it needs separate implementations for
Zod v3 and v4. If arbor migrates to Zod v4, the adapter must change. Does building `fromZod()`
on `._def` create a maintenance burden tied to Zod's version? Or is there a higher-level
introspection API that is stable across versions?

## Findings (June 2026)

### Q1 — Factory function DX (fluent chaining rejected)

Decision: no fluent chaining. The native schema uses options-object factories:

```typescript
string({ format: 'email', maxLength: 100 })
integer({ minimum: 0, maximum: 150 })
uuid()   // convenience alias for string({ format: 'uuid' })
email()  // convenience alias for string({ format: 'email' })
```

Fluent chaining (`string().email().maxLength(100)`) was considered and rejected. The
constraint vocabulary is small, fixed, and maps directly to JSON Schema keywords — there
is no benefit to building up constraints incrementally via method calls. Options objects
are readable at a glance, require no knowledge of what methods exist on a builder, and
match the serializable data model (the constraints object IS the JSON Schema fragment).

Convenience factories for the common format cases avoid the need to remember the `format`
string:

```typescript
export const email  = (c?: Omit<StringConstraints, 'format'>) => string({ format: 'email',  ...c });
export const uuid   = (c?: Omit<StringConstraints, 'format'>) => string({ format: 'uuid',   ...c });
export const url    = (c?: Omit<StringConstraints, 'format'>) => string({ format: 'url',    ...c });
export const cuid   = (c?: Omit<StringConstraints, 'format'>) => string({ format: 'cuid',   ...c });
export const ulid   = (c?: Omit<StringConstraints, 'format'>) => string({ format: 'ulid',   ...c });
```

Claim 5 in the scratch file (plan 163) is updated to reflect options-object factories:

```typescript
// Claim 5: constraints do not change inferred TypeScript type
const s1 = string();
const s2 = string({ format: 'email', maxLength: 100 });
expectTypeOf(s1).toEqualTypeOf(s2);  // both ScalarSchema<string>
```

### Q2 — `fromZod()` feasibility

Viable with caveats. Zod v4 exposes `schema._zod.def` internally — but this is not a
committed public API. The safer surface is `schema.def` which in Zod v4 is the canonical
definition object. Even this may shift between minor versions.

Decision: do not implement `fromZod()` in the initial wave. The reasons:

1. The router-controlled positions (path params, query, headers, cookies) are where the
   native schema matters most. Users writing new routes will use the native builder directly.
   `fromZod()` is a migration convenience, not a prerequisite for the feature to work.

2. Zod's internal API is a moving target. Building on it now means a maintenance burden
   tied to Zod's release cadence. Better to wait until the native schema is stable and the
   community has settled on `fromZod()` patterns.

3. For body and response positions, users keep their existing Zod schemas unchanged — those
   positions accept Standard Schema, and Zod v4 implements it. No adapter needed there.

`fromZod()` can be revisited as a follow-on plan once the core native schema is shipped.

### Q3 — The `.refine()` escape hatch

Handled by the `validate` slot (plan 137, Q2). `fromZod()` is not needed to expose this —
users who need cross-field validation on a native schema query position pass `validateQuery`
alongside the schema. This is simpler than a Zod adapter and explicit in the API.

### Q4 — Package structure

Option A: subpath export `@arbor/router/zod` for any future Zod-specific utilities
(`fromZod`, `toZod`). The core `@arbor/router` has zero Zod dependency after the migration.

Rationale: current users who pass Zod schemas for body/response positions are unaffected
(Standard Schema is transparent). Users who pass Zod schemas at path param positions will
need to migrate to the native builder — but this is a small surface (only `route()` call
sites, not `httpRoute()` body/response). The subpath keeps the dependency graph clean and
avoids making Zod a peer dep for users who don't want it.

The migration path for existing users:

```typescript
// Before
route(z.object({ tag: z.literal('user'), id: z.string() }), 'users/:id')

// After
import { object, literal, string } from '@arbor/router';
route(object({ tag: literal('user'), id: string() }), 'users/:id')
```

This is a mechanical find-and-replace for most route definitions.

### Q5 — Zod v3 vs. v4

Not relevant. Arbor is already on Zod v4 (`z.core.$ZodIssue`, `z.toJSONSchema()` are v4
APIs). No v3 compatibility is required. After the migration, Zod is no longer a dependency
of `@arbor/router` at all — v3/v4 compatibility is entirely the user's concern for their
own body/response schemas.

### Standard Schema interface

The interface arbor depends on for user-controlled positions:

```typescript
// Minimal subset of StandardSchemaV1 that arbor calls
interface UserSchema<T = unknown> {
  readonly '~standard': {
    readonly validate: (value: unknown) =>
      | { readonly value: T }
      | { readonly issues: ReadonlyArray<{ readonly message: string }> };
  };
}
```

Zod v4, Valibot, and ArkType all satisfy this. Arbor does not import `StandardSchemaV1`
from any external package — it defines its own minimal interface and relies on structural
typing. This avoids a dependency on `@standard-schema/spec`.
