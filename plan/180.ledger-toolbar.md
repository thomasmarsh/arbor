# Plan 180 — LedgerToolbar: Search, Filter, and Drill-down

## Context

The current ledger view shows every task in a single undifferentiated list. There is no way
to narrow by wave, status, or keyword without reading every row. The only toggle is
`showAll` (a/key) which reveals done/canceled tasks.

This plan adds a `LedgerToolbar` component above the table and the corresponding filter state
to the reducer. All filtering is **client-side** — the full dataset is already in memory from
the polling fetch. No new API calls or env methods.

### New reducer state

```typescript
export interface LedgerFilters {
  text: string;
  wave: string | null;
  status: TaskStatus | null;
  kind: 'task' | 'spike' | null;
}

const initialFilters: LedgerFilters = {
  text: '',
  wave: null,
  status: null,
  kind: null,
};

// Added to LedgerState:
filters: LedgerFilters;
```

`selectedIndex` resets to `0` whenever any filter changes (prevents out-of-bounds selection).

### New actions

```typescript
| { tag: 'setTextFilter';   text: string }
| { tag: 'setWaveFilter';   wave: string | null }
| { tag: 'setStatusFilter'; status: TaskStatus | null }
| { tag: 'setKindFilter';   kind: 'task' | 'spike' | null }
| { tag: 'clearFilters' }
```

All return `null` (no effects).

### Filtering logic in LedgerTable

Filtering runs in the component, applied to `visibleRows` before rendering. A task matches
when **all** active filters match:

```typescript
function applyFilters(tasks: TaskEntry[], filters: LedgerFilters): TaskEntry[] {
  return tasks.filter((t) => {
    if (filters.wave   && t.wave !== filters.wave) return false;
    if (filters.status && t.status !== filters.status) return false;
    if (filters.kind   && t.kind !== filters.kind) return false;
    if (filters.text) {
      const q = filters.text.toLowerCase();
      if (!t.text.toLowerCase().includes(q) && !String(t.id).includes(q)) return false;
    }
    return true;
  });
}
```

`applyFilters` is a pure function; it lives at the module level in `LedgerTable.tsx` (or a
co-located `ledger.filters.ts` if it grows). Filtering is applied **before** group
construction so group separator rows still reflect real grouping.

### LedgerToolbar layout

```
┌─────────────────────────────────────────────────────────────┐
│ [🔍 Search tasks...    ]  [Wave ▼]  [Status ▼]  [Kind ▼]   │
│                                  [× Clear]  Last: 12:34:01  │
└─────────────────────────────────────────────────────────────┘
```

- MUI `<Toolbar disableGutters sx={{ gap: 1, flexWrap: 'wrap' }}>` inside a `<Paper>`.
- **Text search** — `<TextField size="small" placeholder="Search tasks…">` — debounce with
  a local `useState` + `useEffect` of 200 ms before dispatching `setTextFilter` (debounce
  lives in the component, not the reducer — it is a presentation concern).
- **Wave dropdown** — `<Select size="small">` populated from unique wave IDs derived from
  all loaded tasks (sorted by wave order in the response). Null option = "All waves".
- **Status dropdown** — `<Select size="small">` with the five `TaskStatus` values + "All".
- **Kind toggle** — MUI `<ToggleButtonGroup exclusive>` with "All / Task / Spike".
- **Clear button** — shown only when any filter is active; dispatches `clearFilters`.
- **Last updated** — the existing `lastUpdated` timestamp moved into the toolbar (removed
  from the freestanding `<p>` in `LedgerTable`).
- **Show-all toggle** — move the existing `showAll` concept into the toolbar as a labelled
  `<Switch>` ("Show done"). The `a` keyboard shortcut remains.

`LedgerToolbar` receives `$`, `send`, `groups` as props — no second `useStore` call. The
parent `LedgerTable` is the single store owner and passes the relevant state slices down.

## Goal

1. Add `filters: LedgerFilters` to `LedgerState` and `initialLedgerState` in
