# PLAN.33 — Per-route CORS policy

## Context

Plan 38 (SEC1) adds a server-level CORS wrapper. But a single policy for all
routes is often too coarse: a public read endpoint (`GET /posts/`) might allow
`*`, while an admin mutation endpoint restricts to an internal origin. ts-rest
and tRPC both allow CORS overrides at the route level.

## Goal

Allow individual routes to declare CORS policy overrides that take precedence
over the server-level defaults from plan 38.

```typescript
httpRoute({
  method: 'GET',
  path: 'posts/',
  cors: { origins: ['*'], methods: ['GET'] },   // public
  ...
});

httpRoute({
  method: 'DELETE',
  path: 'admin/posts/:id/',
  cors: { origins: ['https://internal.app'], credentials: true },
  ...
});
```

The server's CORS wrapper reads the matched route's `cors` field and applies
it instead of the server-level config. If no route-level policy is declared,
the server-level default applies.

## Change Surface

1. **`http-context.ts`**: Add optional `cors` config field to the route
   context.
2. **`server.ts` / CORS wrapper** (plan 38): After route matching but before
   handler dispatch, check the matched route for a `cors` override and apply
   it to the response headers.
3. Tests: route-level `cors` overrides server-level `cors`; routes with no
   `cors` field fall back to server default.

## Dependencies

- Plan 38 (CORS/CSRF server wrapper) — the server-level CORS wrapper must
  exist first; this plan extends it.
- Plan 44 (middleware pipeline) — CORS is effectively pre-handler middleware;
  if a pipeline exists it may be cleaner to model it that way.

## Success Criteria

- A route with `cors: { origins: ['*'] }` returns `Access-Control-Allow-Origin: *`
  regardless of the server-level config.
- A route without a `cors` field inherits the server-level policy.
- `npm test && npm run typecheck` clean.
