# PLAN.106 (Spike) — TanStack Router Bridge (Undefer Plan 24)

## Context

Plan 24 deferred the TanStack Router bridge until the API surface stabilized. Plans
18–93 have now stabilized the surface. This spike re-examines three questions from
plan 24 with fresh eyes:

1. **Type compatibility**: Can `ChildUnion<C>` (arbor's flat discriminated union) be
   mapped to TanStack Router's `RouteOptions`? TanStack uses a different tree structure
   (parent → children array) vs. arbor's phantom-typed node. Where does `any` appear?

2. **Path syntax**: Arbor uses `:param` (string), `#param` (numeric), `:param?` (optional),
   `*rest` (wildcard). TanStack uses `$param`, `$param?`, `$` (wildcard). Is the mapping
   purely mechanical (regex translate on segment strings), or do edge cases require
   runtime path rewriting?

3. **Value proposition**: Now that `packages/router-react` exists (plans 95–104), is a
   TanStack bridge still desirable? When would a team choose the TanStack bridge over the
   native arbor React bindings? Write an honest comparison.

## Approach

Write `packages/router-tanstack/scratch/bridge-spike.ts`:

1. Attempt `adaptToTanStackTree(router)` → `TanStackRouteTree`.
2. Check: do path params infer correctly through TanStack's `useParams()`?
3. Check: does the loader system in TanStack (`loader` property on `RouteOptions`) accept
   the arbor loader types from plan 101?
