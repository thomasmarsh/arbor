# Plan 151 — Export Inferred Types from `@arbor/api/ledger`

## Context

Plan 143 introduced a `_ctxMap` drill-through in `packages/ui/src/ledger/ledger.store.ts` to
extract the display-groups response type:

```typescript
type QueueCtx = LedgerRouter['_ctxMap']['ledger-get-queue'];
export type DisplayGroupsResponse = InferSingleSuccessBody<QueueCtx['response']>;
```

The original plan 151 proposed solving this by adding `InferRouteBody<Router, Tag>` to
`@arbor/router`. That is the wrong layer. The root cause is that `packages/api/src/ledger/router.ts`
already defines `DisplayGroupsResponse` as a Zod schema — it just doesn't export the inferred type.
The UI derives the same shape through `_ctxMap` because the direct type was unavailable.

There is no ergonomic gap in `@arbor/router`. The `fetch-client.ts` call site is already
ergonomic (`client.fetch({ tag: 'ledger-get-queue' })`). The gap is in the API layer not surfacing
its types.

## Goal

Export inferred types from `packages/api/src/ledger/router.ts` so consumers can import them
directly, and remove the `_ctxMap` drill-through from the UI.

## Change Surface & T-shirt Sizing

**Size: XS**

Two files:

- `packages/api/src/ledger/router.ts` — add `export type` declarations for `DisplayGroupsResponse`
  (and `TasksResponse`, `TaskEntry` re-export if callers need them)
- `packages/ui/src/ledger/ledger.store.ts` — replace the `_ctxMap`/`InferSingleSuccessBody`
  pattern with a direct import

No changes to `@arbor/router`. `InferRouteBody<Router, Tag>` is not added — it was treating a
symptom. If a future external consumer receives only a `RouterContract` and has no access to the
raw schemas, `_ctxMap` + `InferSingleSuccessBody` is the correct escape hatch at that point, not
a new helper.
